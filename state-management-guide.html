01 — The Foundation

Fundamentals of State Management

State is any data that changes over time and influences what the user sees. Understanding what state is — and what it is not — is the prerequisite for every sound architectural decision in frontend engineering.

What Is State?

State is the data that your application holds in memory at any given moment. A login status is state. A list of fetched products is state. Whether a modal is open is state. The text typed into a form field is state. Anything that, when it changes, could require the UI to update — that is state.

What is not state: static configuration, pure functions, constants, and computed values that can be derived from other state. One of the most common sources of state complexity is storing derived values as state instead of computing them. If you store both a list of todos and a completedCount, you have two sources of truth that can drift out of sync. Store the list; compute the count.

🖼️

UI State

Controls visual appearance and interaction: modal visibility, selected tab, sidebar open/closed, loading spinners, hover state. Short-lived. Rarely leaves a single component.

🌐

Application State

Shared data driving multiple parts of the application: authenticated user, shopping cart, notification list, theme. Lives longer, accessed from many places.

🗄️

Server State

Data that lives on a remote server and is fetched, cached, and synchronised: users, orders, products, reports. Has a lifecycle — loading, stale, error — that pure client state does not.

📍

URL State

State encoded in the URL: current page, applied filters, search query, selected item ID. Shareable, bookmarkable. Often overlooked as a state location, leading to unnecessary global state.

The Source of Truth Problem

The most fundamental principle in state management is: each piece of state should have exactly one owner. When the same data lives in two places — a Redux store and a local component state, or a React hook and a URL parameter — they will eventually diverge. Every synchronisation bug in frontend history comes from state with two sources of truth.

JavaScript — State Synchronisation Problem
// ❌ Two sources of truth — they will drift
const [user, setUser] = useState(store.getUser())  // local copy
const storeUser = useSelector(state => state.user)  // store copy
// Update one → the other lags. Bug inevitable.

// ✅ Single source of truth — derive, don't duplicate
const user = useSelector(state => state.user)  // one owner: the store

// ✅ Derive — never store what you can compute
const todos = useSelector(state => state.todos)
const completedCount = useMemo(
  () => todos.filter(t => t.done).length,
  [todos]
)
// completedCount can never disagree with todos — it IS derived from todos.

State Lifecycle

State has a lifecycle: it is created (initialised), read (rendered), updated (mutated via actions or setters), and destroyed (cleared when a component unmounts or a session ends). Managing this lifecycle correctly — especially destruction — prevents memory leaks, stale data, and security issues where one user's data bleeds into another's session.

02 — The DOM Era

Early Frontend Architecture

Before state management existed as a concept, the DOM was the state. The value in an input field was state. Whether a div had a class was state. Reading state meant querying the DOM; updating state meant mutating it directly. This worked until applications grew complex enough that the DOM could no longer be trusted as a reliable source of truth.

JavaScript — The DOM as State, circa 2008
// "State" was scattered across DOM nodes.
// To know if the user was logged in, you queried the DOM.
function isLoggedIn() {
  return document.getElementById('logout-btn') !== null
  // Depends on UI rendering. Fragile. Completely wrong.
}

// jQuery-era: state was a mix of DOM attributes, CSS classes,
// and JavaScript variables scattered across dozens of event handlers.
var cartCount = 0  // global variable — the "state"

$('.add-to-cart').on('click', function() {
  cartCount++
  $('#cart-badge').text(cartCount)   // update DOM to match variable
  $('#cart-empty-msg').hide()        // update related DOM
  $('#checkout-btn').show()         // update another related DOM
  // What if one of these fails? State (variable) and UI (DOM) drift.
})

// The fundamental problem: state lived in JavaScript variables,
// but the UI lived in the DOM. Keeping them in sync was entirely
// manual and entirely the developer's responsibility.
// Bugs accumulated as applications grew.

The jQuery era produced an entire class of bugs that are essentially invisible in modern applications: the variable says X but the DOM shows Y. A developer updated the JavaScript state but forgot to update one DOM element. Another developer updated a DOM element directly without going through the JavaScript state. The two drifted apart, and the application showed stale or inconsistent data to the user.

03 — Separation of Concerns

MVC Architecture

Model-View-Controller gave state a formal home: the Model. By separating what the data is from how it is displayed and how user interactions change it, MVC frameworks reduced the class of state/DOM drift bugs that plagued the jQuery era.

JavaScript — Backbone.js: Models as State Containers
// Model: owns the state, emits events when it changes
const Cart = Backbone.Model.extend({
  defaults: { items: [], total: 0 },

  addItem(product) {
    const items = [...this.get('items'), product]
    this.set({
      items,
      total: items.reduce((sum, i) => sum + i.price, 0)
    })
    // 'change' event fires automatically — any listening View re-renders
  }
})

// View: listens to Model events, re-renders on change
const CartView = Backbone.View.extend({
  initialize() {
    this.listenTo(this.model, 'change', this.render)
  },
  render() {
    this.$el.html(this.template(this.model.toJSON()))
  }
})

// Progress: state now lives in a dedicated Model, not scattered in the DOM.
// Problem: multiple Views can modify the same Model.
// Who changed it? When? Why? Tracing state changes is hard.
// This leads to tangled event chains in large apps.

MVC was a genuine improvement over jQuery-era state management. State had a formal home (the Model), and the UI was responsible only for rendering and forwarding interactions. But client-side MVC exposed a new class of problems: in a sufficiently complex application, multiple Models could modify each other, Views could trigger Model changes that triggered other View updates, and tracing the flow of a state change required following a web of event listeners.

04 — Automatic Synchronisation

Two-Way Data Binding

AngularJS's two-way data binding eliminated the manual synchronisation code between model and view. Change the model, the UI updates. Change the UI, the model updates. It felt like magic — and like most magic, understanding what was happening beneath the surface revealed the cost.

JavaScript — AngularJS Digest Cycle: Automatic But Expensive
// The digest cycle: how AngularJS knew what changed
// Every binding creates a "watcher" — a function that returns the current value.
// On every digest cycle, Angular runs ALL watchers.
// If a value changed since last cycle → update the DOM.
// If a watcher's change caused another change → run the cycle again.
// Repeat until stable (max 10 iterations → "infinite digest" error).

// Example: ng-model="user.name" creates TWO watchers:
//   1. $scope.user.name → input.value  (model to DOM)
//   2. input.value → $scope.user.name  (DOM to model)

// Performance: every watcher runs on every digest.
// 2,000 watchers is the practical ceiling before sluggishness.
// A list of 100 items with 5 bindings each = 500 watchers immediately.

// Angular had to expose $scope.$apply() for code outside Angular's awareness:
setTimeout(() => {
  $scope.$apply(() => {   // manually trigger digest cycle
    $scope.message = 'Updated'
  })
}, 1000)
// Forgetting $scope.$apply was one of the most common AngularJS bugs.

// Mitigation attempts:
// - One-time bindings (::value) — watcher removed after first render
// - track by in ng-repeat — avoid recreating DOM nodes
// - Manual $scope.$digest() on specific subtrees
// All of these required deep framework knowledge and undermined the "automatic" promise.

Two-way binding's legacy is important: it proved that developers want automatic synchronisation between data and UI. The cost — unbounded change propagation, performance ceilings, debugging complexity — is what motivated every subsequent approach. React, Vue, and Svelte each found different ways to preserve the benefit (automatic UI updates) while eliminating the cost.

05 — The Paradigm Shift

React's State Revolution

React didn't just change how state was managed — it changed the fundamental mental model of what rendering was. Instead of describing transitions (do X to get from state A to state B), React described outcomes (given state B, the UI looks like this). This shift made state bugs an order of magnitude easier to reason about.

UI = f(state). Given the same state, the same UI is always produced. Every time. Without exception.
JavaScript — React's State Model: Immutable, Declarative, Predictable
// React's key insight: rendering is a pure function of state.
// State → deterministic UI. No side effects. No DOM queries.

// Local state: useState hook
function CartBadge() {
  const [count, setCount] = useState(0)
  return <span>{count}</span>
  // React guarantees: if count is 3, the span shows 3. Always.
}

// State updates are scheduled, not immediate.
// React batches multiple setState calls into one re-render.
function handleBulkUpdate() {
  setA(1)   // batched
  setB(2)   // batched
  setC(3)   // batched → ONE re-render, not three
}

// Functional updates: always use the previous state, never the closure value
const increment = () => setCount(prev => prev + 1)
// ❌ setCount(count + 1) — stale closure in async callbacks
// ✅ setCount(prev => prev + 1) — always gets current state

// useReducer: for state with complex transition logic
const cartReducer = (state, action) => {
  switch (action.type) {
    case 'ADD':    return { items: [...state.items, action.item] }
    case 'REMOVE': return { items: state.items.filter(i => i.id !== action.id) }
    case 'CLEAR':  return { items: [] }
    default:      return state
  }
}
const [cart, dispatch] = useReducer(cartReducer, { items: [] })
06 — Data Flow Patterns

Component Communication

React's unidirectional data flow is elegant for isolated components. As applications grow and unrelated components need the same data, the patterns for sharing state become the primary source of architectural complexity.

JavaScript — State Lifting & the Prop Drilling Problem
// ── State Lifting: the correct first approach ─────────────────
// When two siblings need the same state, lift it to their common ancestor.
function FilteredList() {
  const [query, setQuery] = useState('')
  return (
    <div>
      <SearchInput query={query} onChange={setQuery} />
      <ResultList query={query} />
    </div>
  )
}
// ✅ Correct. One owner. Props carry state down, callbacks carry events up.

// ── Prop Drilling: when lifting state too high ─────────────────
function App() {
  const [user, setUser] = useState(null)
  return <Layout user={user} />
}
function Layout({ user }) {
  return <Sidebar user={user} />  // Layout doesn't USE user — just passes it
}
function Sidebar({ user }) {
  return <UserAvatar user={user} />  // Sidebar doesn't USE user — just passes it
}
function UserAvatar({ user }) {
  return <img src={user?.avatar} />  // only UserAvatar needs user
}
// ❌ Prop drilling: user passes through Layout and Sidebar even though
// neither uses it. Adding a new prop means updating 3 components.
// This is what global state solutions exist to solve.
07 — Application-Wide Data

Global State Concepts

Global state is data that multiple components across the component tree need access to without the overhead of prop drilling. The problem is not accessing global state — it is managing its mutations, preventing stale reads, and keeping components in sync when it changes.

🔐

Authentication

Current user identity, roles, permissions. Every protected route, every conditional UI element, every API call depends on it. Classic global state.

🎨

Theme

Light/dark mode, accent colour, font size preferences. Used by every component that renders. Classic global state with simple structure — often a good Context candidate.

🛒

Shopping Cart

Accessed from product pages, cart drawer, checkout, header badge. Mutated from anywhere. Complex state with business rules — typically warrants a dedicated store.

🔔

Notifications

Pushed from multiple sources: API responses, WebSocket events, scheduled reminders. Consumed by a toast system. Global, append-only with read/dismiss transitions.

ℹ️ The Global State Trap
The most common mistake in state architecture is making state global when it should be local. Before reaching for a global store, ask: how many components actually need this? If the answer is two adjacent siblings, lift state to their parent. URL state (React Router's useSearchParams) often eliminates the need for global state for filtering, pagination, and selection entirely.
08 — Flux to Redux

Redux Architecture

Redux (2015) was Dan Abramov and Andrew Clark's implementation of Facebook's Flux architecture, distilled into three principles: single source of truth, read-only state, and pure functions as reducers. It gave the ecosystem predictable state management at the cost of significant ceremony.

JavaScript — Redux: Flux Architecture Implemented
// Redux's three principles in code:
// 1. Single source of truth — one store for the whole app
// 2. State is read-only — only actions can trigger changes
// 3. Changes via pure functions — reducers: (state, action) → state

// ── Classic Redux (2015–2018): maximum explicitness ───────────
// Action types
const ADD_TO_CART    = 'cart/ADD_TO_CART'
const REMOVE_FROM_CART = 'cart/REMOVE_FROM_CART'

// Action creators
const addToCart    = (item) => ({ type: ADD_TO_CART,    payload: item })
const removeFromCart = (id)   => ({ type: REMOVE_FROM_CART, payload: id })

// Reducer — pure function, no side effects, no mutations
function cartReducer(state = { items: [] }, action) {
  switch (action.type) {
    case ADD_TO_CART:
      return { ...state, items: [...state.items, action.payload] }
    case REMOVE_FROM_CART:
      return { ...state, items: state.items.filter(i => i.id !== action.payload) }
    default:
      return state
  }
}
// Selectors
const selectCartItems = state => state.cart.items
const selectCartTotal = createSelector(selectCartItems,
  items => items.reduce((s, i) => s + i.price, 0)
)

// ── Redux Toolkit (2019): same principles, far less boilerplate ─
const cartSlice = createSlice({
  name: 'cart',
  initialState: { items: [] },
  reducers: {
    addItem:    (state, action) => { state.items.push(action.payload) },     // Immer!
    removeItem: (state, action) => {
      state.items = state.items.filter(i => i.id !== action.payload)
    },
  },
})
// Time travel: Redux DevTools replays actions to any past state.
// Invaluable for debugging complex state transitions.

Redux's key contribution beyond the architecture was the DevTools experience: every state change was an inspectable, replayable action. You could travel back in time to any previous application state, replay a user's exact sequence of actions to reproduce a bug, and export/import state for testing. No other state manager matched this debuggability for years.

09 — Functional Foundations

State Immutability

Immutability is the requirement that state is never mutated in place — every update produces a new object. This constraint sounds like a limitation but is actually the key that enables efficient change detection, time-travel debugging, and predictable rendering.

JavaScript — Why Immutability Enables Everything
// ── Mutable update: change detection is expensive ─────────────
const prevState = { user: { name: 'Alice' } }
prevState.user.name = 'Bob'  // mutation in place
// prevState === prevState (same reference)
// To know what changed, you must deep-compare the ENTIRE tree.
// O(n) where n is the total number of properties. Slow.

// ── Immutable update: change detection is O(1) ────────────────
const prevState2 = { user: { name: 'Alice' } }
const nextState  = { ...prevState2, user: { name: 'Bob' } }
// prevState2 !== nextState (different reference) → changed!
// prevState2.user !== nextState.user → user changed!
// Reference equality check is O(1). React.memo, useSelector,
// PureComponent all depend on this.

// ── Structural sharing: immutable ≠ full copies ───────────────
// Immer (used by Redux Toolkit) uses Proxy to let you write
// mutating syntax while producing immutable output.
import { produce } from 'immer'

const nextCart = produce(cart, draft => {
  draft.items.push(newItem)      // looks like mutation
  draft.total += newItem.price  // looks like mutation
})
// Immer internally applies the draft mutations to produce a NEW object.
// Unchanged subtrees are structurally shared (same reference).
// You get mutable syntax with immutable semantics.
10 — Scale Concerns

Enterprise State Management

At enterprise scale — hundreds of developers, thousands of components, multi-year codebases — state management is primarily a governance and auditability problem, not a performance problem. The question shifts from "how do we make state updates fast?" to "how do we make state changes traceable, reviewable, and safe?"

📋

Auditability

Every state change should have a traceable cause. Redux's action log is the canonical answer. You can replay any user's session and see exactly what happened and in what order.

🔒

Predictability

Given the same sequence of actions, the store always reaches the same state. Pure reducers guarantee this. It makes testing trivial: test the reducer, not the UI.

👥

Team Coordination

Multiple teams touching the same state. Namespaced slices, clearly owned actions, and documented state shape prevent the "who changed the cart state?" debugging sessions.

🧪

Testability

A Redux reducer is the easiest code to test in a frontend application. Pure function: given this state and this action, assert this output. No DOM, no framework, no mocking.

TypeScript — Enterprise Patterns: Typed Actions & RTK Query
// ── RTK Query: server state management built into Redux ────────
const api = createApi({
  reducerPath: 'api',
  baseQuery:   fetchBaseQuery({ baseUrl: '/api' }),
  endpoints: build => ({
    getUsers: build.query<User[], void>({
      query: () => '/users',
      providesTags: ['User'],                       // for cache invalidation
    }),
    createUser: build.mutation<User, CreateUserDto>({
      query: body => ({ url: '/users', method: 'POST', body }),
      invalidatesTags: ['User'],                    // auto re-fetches getUsers
    }),
  }),
})
// RTK Query generates hooks, manages loading/error states,
// handles caching, deduplicates concurrent requests, and
// automatically refetches stale data — all within Redux.
11 — Observable State

Reactive State Management

Where Redux made state changes explicit and traceable, MobX made them automatic and invisible. Observable state, computed values, and reactions — the same concepts from AngularJS's digest cycle, reimagined with fine-grained precision using ES5 proxies and later ES6 Proxies.

JavaScript — MobX & Signals: Fine-Grained Reactivity
// ── MobX: observable state with automatic dependency tracking ──
import { makeAutoObservable, computed, reaction } from 'mobx'

class CartStore {
  items = []

  constructor() {
    makeAutoObservable(this)  // makes all properties observable
  }

  get total() {  // computed — re-runs only when items changes
    return this.items.reduce((s, i) => s + i.price, 0)
  }

  addItem(item) {
    this.items.push(item)  // direct mutation — MobX intercepts via Proxy
  }
}
// MobX tracks which observables each reaction/component accessed.
// Only components that accessed cart.total re-render when total changes.
// This is fine-grained reactivity — surgical, automatic, precise.

// ── Signals: MobX's ideas, framework-native ───────────────────
// Preact Signals / SolidJS / Angular Signals / Vue refs
import { signal, computed, effect } from '@preact/signals-react'

const items = signal([])
const total = computed(() =>
  items.value.reduce((s, i) => s + i.price, 0)  // auto-tracked
)

effect(() => {
  console.log(`Total: ${total.value}`)  // re-runs only when total changes
})
// Signals are the performance ceiling of reactivity:
// the smallest possible unit re-renders when the smallest possible state changes.

Signals represent the convergence of the entire frontend ecosystem's thinking on reactivity. Angular adopted Signals in v16. Vue's ref() is a signal. Svelte Runes are signals. SolidJS is built entirely on signals. The concept — a primitive value that notifies subscribers when it changes and tracks its own dependents — has proven to be the most efficient and composable reactivity primitive.

12 — React's Built-In Solution

React Context API

React Context (introduced properly in React 16.3) solved prop drilling without an external library. It provides dependency injection for React components — a value placed in a Provider is accessible to any descendant Consumer, however deep. Understanding its limitations is as important as understanding its use cases.

JavaScript — Context: Correct Use & The Re-Render Problem
// ── Correct use: infrequently-changing global values ──────────
const ThemeContext = createContext({ mode: 'light' })

function App() {
  const [theme, setTheme] = useState({ mode: 'light' })
  return (
    <ThemeContext.Provider value={theme}>
      <AppContent />
    </ThemeContext.Provider>
  )
}
// ✅ Theme changes rarely — the re-render cost is acceptable.

// ── The re-render problem: why Context ≠ state manager ────────
const CartContext = createContext(null)

function CartProvider({ children }) {
  const [cart, setCart] = useState({ items: [], user: null, total: 0 })
  // ❌ EVERY consumer re-renders on ANY cart change.
  // A component only reading cart.user re-renders when cart.items changes.
  // Context has no subscription granularity.
  // For frequently-changing objects, this tanks performance.

  return <CartContext.Provider value={{ cart, setCart }}>{children}</CartContext.Provider>
}

// ── Splitting context by update frequency ─────────────────────
// Solution: separate rarely-changing from frequently-changing state
const CartItemsContext = createContext([])       // changes often
const CartActionsContext = createContext(null)  // stable reference (useCallback)
// Components that only need actions don't re-render on items changes.
⚠️ Context Is Not a State Manager
Context is a dependency injection mechanism, not a state management solution. It solves where state is accessible but not how it is managed. Pairing Context with useReducer approximates Redux without the DevTools. For high-frequency updates, always prefer Zustand, Jotai, or Redux over Context — the re-render behaviour will become a performance problem at scale.
13 — The Most Important Insight

State Classification

The single most impactful insight in a decade of state management evolution: not all state is the same. Local UI state, global client state, and server state are fundamentally different problems that require fundamentally different solutions. Using the same tool for all three is the source of most state management complexity.

🖼️ Local UI State
Modal open/closed
Selected tab index
Form field value
Accordion expanded
Hover state
Scroll position
Owner: The component
Tool: useState / useReducer
🌐 Global Client State
Authenticated user
Theme preference
Shopping cart
Notifications
User permissions
Feature flags
Owner: Client store
Tool: Zustand / Redux / Pinia
🗄️ Server State
Users list
Orders & products
Reports & analytics
Search results
Paginated feeds
Real-time data

JavaScript — State Classification in Practice
// ── Before classification: everything in Redux ────────────────
// Redux store circa 2017: a massive object holding everything
{
  auth:         { user: null, token: null },          // global client ✓
  users:        { loading: false, data: [], error: null },  // server state ✗
  orders:       { loading: false, data: [], error: null },  // server state ✗
  modalOpen:    false,                                  // local UI state ✗
  selectedTab:  0,                                      // local UI state ✗
  theme:        'light',                                // global client ✓
}

// ── After classification: each type gets its correct tool ─────
// Local UI state → stays in component
const [modalOpen, setModalOpen] = useState(false)
const [selectedTab, setTab]   = useState(0)

// Global client state → Zustand (lightweight, no boilerplate)
const useAuthStore = create((set) => ({
  user:  null,
  theme: 'light',
  setUser:  user  => set({ user }),
  setTheme: theme => set({ theme }),
}))

// Server state → TanStack Query (caching, refetching, sync)
const { data: users }  = useQuery({ queryKey: ['users'],  queryFn: fetchUsers })
const { data: orders } = useQuery({ queryKey: ['orders'], queryFn: fetchOrders })
// Each query has its own loading/error/stale state — no manual management.
14 — Remote Data Lifecycle

Server State Management

Server state is fundamentally different from client state. It lives on a remote server. It can change without your application's knowledge. It has a staleness concept — data fetched 5 minutes ago may no longer be accurate. Managing it correctly requires caching, revalidation, deduplication, and synchronisation that pure client state libraries were never designed to handle.

JavaScript — TanStack Query: Server State Done Right
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query'

// ── Fetching with automatic caching + background refetch ──────
function UserList() {
  const { data, isLoading, isError, isFetching } = useQuery({
    queryKey:  ['users'],
    queryFn:   fetchUsers,
    staleTime: 5 * 60 * 1000,   // 5 min — don't refetch if data is fresh
    retry:     3,               // retry failed requests 3 times
  })
  // isFetching = background refetch in progress (show subtle indicator)
  // isLoading  = first fetch, no cached data yet (show skeleton)
}

// ── Mutations with optimistic updates ─────────────────────────
function DeleteUser({ userId }) {
  const queryClient = useQueryClient()
  const mutation = useMutation({
    mutationFn: (id) => deleteUser(id),

    // Optimistic update: remove from cache before server confirms
    onMutate: async (id) => {
      await queryClient.cancelQueries({ queryKey: ['users'] })
      const previous = queryClient.getQueryData(['users'])
      queryClient.setQueryData(['users'], users =>
        users.filter(u => u.id !== id)
      )
      return { previous }  // snapshot for rollback
    },

    // Rollback on error
    onError: (err, id, ctx) =>
      queryClient.setQueryData(['users'], ctx.previous),

    // Refetch to ensure cache matches server
    onSettled: () =>
      queryClient.invalidateQueries({ queryKey: ['users'] }),
  })
  return <button onClick={() => mutation.mutate(userId)}>Delete</button>
}
15 — Minimal Architecture

Modern Lightweight Stores

The pendulum swung from jQuery's stateless chaos through Redux's maximum ceremony to a new equilibrium: lightweight stores that are small enough to understand completely, large enough to cover real use cases, and simple enough to adopt without a week of learning.

JavaScript — Zustand: The Modern Sweet Spot
import { create } from 'zustand'
import { persist } from 'zustand/middleware'

// ── A complete auth store — no boilerplate, fully typed ────────
const useAuthStore = create(
  persist(
    (set, get) => ({
      user:    null,
      token:   null,
      loading: false,

      login: async (email, password) => {
        set({ loading: true })
        try {
          const { user, token } = await authApi.login(email, password)
          set({ user, token, loading: false })
        } catch (err) {
          set({ loading: false })
          throw err
        }
      },

      logout: () => set({ user: null, token: null }),

      isAdmin: () => get().user?.roles.includes('admin'),
    }),
    { name: 'auth-storage' }  // persist to localStorage
  )
)

// Usage: subscribe to only what you need — prevents unnecessary re-renders
const user    = useAuthStore(state => state.user)
const login   = useAuthStore(state => state.login)
const isAdmin = useAuthStore(state => state.isAdmin())
// Each selector is independent — changing user doesn't re-render
// components that only subscribed to isAdmin().
16 — The Vue Ecosystem

Vue State Evolution

Vue's state management evolution mirrors React's but arrived at different conclusions, particularly around the relationship between reactivity and stores. Pinia's design — built directly on Vue's Composition API — demonstrates what a state manager looks like when the framework and the library share the same reactive primitives.

JavaScript — Vuex vs Pinia: Vue's State Management Arc
// ── Vuex (2015–2022): Redux-inspired, Vue-flavoured ───────────
// Strict mutation/action separation. Modules for namespace.
// Served Vue 2 well. Felt like fighting the framework in Vue 3.
const cartModule = {
  namespaced: true,
  state:    () => ({ items: [] }),
  getters:  { total: s => s.items.reduce((t,i) => t+i.price, 0) },
  mutations:{ ADD: (s,item) => s.items.push(item) },  // synchronous only
  actions:  { addAsync: ({commit},item) => {
    // async logic here, then commit a mutation
    commit('ADD', item)
  } }
}

// ── Pinia (2021–present): the Vue 3 state manager ─────────────
// No mutations. No strict action/mutation split. Just Composition API.
import { defineStore } from 'pinia'
import { ref, computed } from <'vue'

export const useCartStore = defineStore('cart', () => {
  const items = ref([])
  const total = computed(() =>
    items.value.reduce((s, i) => s + i.price, 0)
  )

  async function addItem(item) {
    items.value.push(item)                  // direct mutation via Vue's Proxy
    await syncCartWithServer(items.value)   // async in same function, no split
  }

  return { items, total, addItem }
})
// Pinia = Composition API applied to stores.
// Same mental model as component code. No new concepts to learn.
// Full TypeScript inference. Vue DevTools support. SSR-compatible.
17 — Server-First State

Meta-Framework State Architecture

Next.js, Nuxt, and SvelteKit introduced a new state concern: server-rendered state that must survive the transition to the client. The hydration problem — making the server-rendered HTML interactive without flickering, double-fetching, or state mismatch — is the defining state management challenge of the meta-framework era.

JavaScript — Next.js App Router: Server-Side State & Hydration
// ── Server Component: state fetched on the server, serialised to HTML ──
async function ProductPage({ params }) {
  // This runs on the server. Data is fetched ONCE.
  // Result is serialised into the HTML stream.
  // Client never re-fetches this — zero waterfall.
  const product = await db.products.findById(params.id)

  return (
    <div>
      <h1>{product.name}</h1>
      {/* QuantitySelector is a client component — it has local UI state */}
      <QuantitySelector productId={product.id} defaultQty={1} />
    </div>
  )
}

// ── SvelteKit: load functions and page stores ──────────────────
// +page.server.ts — runs on server, returns data for page
export async function load({ params, locals }) {
  const product = await db.products.findById(params.id)
  return { product }  // serialised, available as page.data in component
}

// Key insight: in meta-frameworks, the DATA is the state.
// Server fetches it; framework serialises it; client picks it up.
// No separate API call from client. No loading state for initial render.
// State management shifts from "how do we fetch and cache this?"
// to "what fetches it, and when is it stale?"
18 — Server-Side State

React Server Components & State

React Server Components represent the most significant rethinking of where state lives since Redux centralised it in 2015. RSC's bet: most state that applications put in client-side stores should never have been client-side state at all — it was always server state that was being awkwardly cached locally.

JavaScript — RSC: Reducing Client State to Its Irreducible Minimum
// ── What RSC eliminates: the entire server-state client side ───
// Before RSC: this was in a Redux store or TanStack Query
const { data: user }         = useQuery(['user'],      fetchCurrentUser)
const { data: permissions }  = useQuery(['permissions'], fetchPermissions)
const { data: notifications }= useQuery(['notifications'], fetchNotifications)

// After RSC: server components access this directly from the DB/session
async function AppShell() {
  const session       = await getSession()
  const permissions   = await db.permissions.findByUser(session.userId)
  const notifications = await db.notifications.findUnread(session.userId)
  // Zero client-side state. Zero API round trips. Zero loading states.
  // This data renders as static HTML. Client JS budget: zero for this.
  return <Shell user={session.user} permissions={permissions} notifications={notifications} />
}

// ── What remains as client state in an RSC architecture ────────
// 1. Optimistic UI state — cart additions before server confirms
// 2. Ephemeral UI state — modal open/closed, selected tab
// 3. Real-time state — WebSocket/SSE data pushed from server
// 4. Form draft state — unsaved form input before submission
//
// This is the irreducible minimum of client state.
// Everything else belongs on the server.

20 — Hidden Complexity

Distributed State Concepts

Every frontend state management library, whether its authors know it or not, is solving a subset of distributed systems problems. Understanding the underlying CS concepts explains why the problems are hard and why no solution is perfect.

🔄

Cache Coherence

When the same data exists in multiple places (Redux store + TanStack Query cache + component state), they can diverge. Cache coherence protocols (invalidation, TTL, versioning) are the solutions. TanStack Query's invalidateQueries is exactly cache invalidation.

⏱️

Data Freshness & Staleness

TanStack Query's staleTime, SWR's revalidation — these are implementations of TTL-based cache expiry from distributed caching systems. The concept predates React by decades.

🏃

Optimistic Concurrency

Optimistic UI updates assume success and roll back on failure. This is the same pattern as optimistic locking in databases. Both solve the "assume good, reconcile on conflict" problem.

📡

Event Propagation

Redux's event sourcing model (append-only action log, state derived from replaying events) is the same architecture as CQRS/Event Sourcing in backend systems. The store is a projection of the event stream.

🔁

Eventual Consistency

A client cache is eventually consistent with the server. After an update, there is a window where the client and server disagree. TanStack Query's background refetch is a convergence mechanism.

🔒

Synchronisation

When multiple browser tabs share a user session, they need state synchronisation. Zustand's persist middleware with StorageEvent listeners, BroadcastChannel, and SharedWorkers are the client-side equivalents of distributed consensus.

21 — The Architecture

State Management Design Principles

Twenty years of evolution and thousands of production applications have distilled the practice of state management into a small set of durable principles. These are not framework-specific — they apply regardless of what tools you choose.

Principle 01
State should have a single owner
Every piece of state belongs to exactly one store, hook, or component. No duplicates. No synchronisation code. When you find yourself copying state between two places or writing code to keep two pieces of state in sync, you have two owners — collapse them into one.
Principle 02
Not all state is global
The default for any new piece of state should be local. Promote to shared state only when a second consumer needs it. Promote to global only when the component tree makes sharing via props impractical. Most state should never leave the component that created it.
Principle 03
Server state is different from client state
Data that lives on a server has a lifecycle — loading, stale, error, background refreshing — that client state libraries are not designed to manage. Use TanStack Query, SWR, or RTK Query for server state. Never put loading/error wrappers around server data in your Redux or Zustand store.
Principle 04
Derive, don't duplicate
Computed values, filtered lists, aggregate totals — these should be computed from state, not stored alongside it. Storing derived state creates two sources of truth and the inevitable synchronisation bug when they drift. useMemo, computed(), and selector functions exist for this reason.
Principle 05
Simpler solutions should be preferred
Use the least powerful tool that solves the problem. useState before useReducer. useReducer before Zustand. Zustand before Redux. Every layer of abstraction has a cost in learning, debugging, and onboarding. Choose the tool that matches the actual complexity of the problem, not the potential future complexity.
Principle 06
Architecture should follow state classification
Your state architecture should mirror your state classification: local UI state stays in components, global client state goes in a lightweight store, server state gets a dedicated query library. Resist the urge to pick one tool and use it for everything — that is the original sin of Redux-for-all-state.
Principle 07
State management is fundamentally a data synchronisation problem
Every state bug — stale data, inconsistent UI, race conditions, optimistic update failures — is a synchronisation failure. When you understand this, you understand why cache invalidation is one of the hardest problems in computing, why TanStack Query's design is correct, and why RSC's approach of eliminating client-side caches where possible is architecturally sound.
"The evolution from global variables to React Server Components is the story of gradually understanding where data belongs, who owns it, and how little of it needs to live in the browser at all."
