01 — The Foundation

How the Query Planner Works

The PostgreSQL query planner is a cost-based optimiser. It enumerates possible execution plans, estimates the cost of each using table statistics, and picks the cheapest one. Understanding this process is the key to every optimisation technique.

Query Execution Pipeline
Parser
SQL text → Parse Tree. Checks syntax. Converts your SQL string into an internal tree structure. Syntax errors are caught here. Fast — no table lookups yet.
Analyser
Parse Tree → Query Tree. Resolves names against the catalog: does the table exist? Are columns valid? What are the data types? Attaches OIDs and type information.
Rewriter
Query Tree → Query Tree. Applies rewrite rules — most importantly, expands VIEWs into their underlying SQL. Rule-based, deterministic.
Planner
Query Tree → Plan Tree. The heart. Generates candidate plans (scan types, join orders, join methods), estimates cost for each using pg_statistic data, selects the cheapest. Most of the optimisation happens here.
Executor
Plan Tree → Result Rows. Walks the plan tree top-down, pulls rows from leaf nodes (scans), feeds them through operators (filters, joins, sorts, aggregates), returns results to the client.

Cost Units — What "Cost" Actually Means

PostgreSQL expresses plan cost in abstract "cost units" — not milliseconds. The units are calibrated around two constants: reading a sequential page costs 1.0, reading a random page costs 4.0 (on spinning disks — set to 1.1 for SSDs). CPU operations are tiny fractions.

cost-constants.sql
-- Key planner cost constants (postgresql.conf)
-- seq_page_cost       = 1.0  (baseline — sequential disk read)
-- random_page_cost    = 4.0  (random disk read — SET TO 1.1 FOR SSD!)
-- cpu_tuple_cost      = 0.01 (processing one row)
-- cpu_index_tuple_cost= 0.005(processing one index entry)
-- cpu_operator_cost   = 0.0025 (evaluating an operator/function)
-- parallel_tuple_cost = 0.1  (passing a tuple between parallel workers)
-- parallel_setup_cost = 1000 (overhead of launching parallel workers)

-- Example: a sequential scan of 10,000 pages =
-- startup_cost: 0
-- total_cost  : 10000 * 1.0 + 50000 * 0.01 = 10500

-- Change for SSDs — critical for index vs seqscan decisions:
ALTER SYSTEM SET random_page_cost = 1.1;
ALTER SYSTEM SET effective_cache_size = '12GB'; -- hint: how much OS cache exists
SELECT pg_reload_conf();

The Three Scan Types

  • Seq Scan
    Sequential Scan — reads every page of the table O(n) where n = table pages. Optimal when fetching a large fraction of the table (>5–10%). The planner prefers it over an index scan when the table is small or when many rows match. Fast I/O thanks to prefetching.
  • Index Scan
    Index Scan — traverses index B-tree, fetches heap rows O(log n + k) where k = matching rows. Optimal for fetching a small fraction of the table. Each index entry requires a random heap page fetch — expensive on HDDs, cheap on SSDs.
  • Index Only
    Index Only Scan — all data served from the index The fastest possible scan when the index contains all needed columns (via INCLUDE or natural coverage). No heap fetch. Requires the visibility map to be up-to-date (VACUUM helps). Check for these with EXPLAIN ANALYZE — "Heap Fetches: 0" is ideal.
  • Bitmap
    Bitmap Heap Scan — batches index lookups, sorts by page Two-phase: first builds a bitmap of matching heap pages from one or more indexes (Bitmap Index Scan), then fetches pages in physical order (Bitmap Heap Scan). Eliminates duplicate page fetches. Used for moderate result sets or when combining multiple indexes with OR/AND.
02 — Reading Plans

EXPLAIN Deep Dive — Reading Every Node

EXPLAIN is the most important tool a PostgreSQL developer has. Most developers only read the top-level numbers. Learning to read the full plan tree transforms your ability to diagnose slow queries.

explain-anatomy.sql
-- Always use these options together:
EXPLAIN (
  ANALYZE,    -- actually execute the query (WARNING: runs DML!)
  BUFFERS,    -- show cache hit/miss counts
  VERBOSE,    -- show output columns, schema-qualified names
  FORMAT JSON  -- machine-readable (use with explain.dalibo.com)
)
SELECT
  u.id, u.email,
  COUNT(p.id) AS posts
FROM   users  u
JOIN   posts  p ON p.author_id = u.id
WHERE  u.created_at > '2025-01-01'
GROUP BY u.id;

💡 The Most Important Red Flags
  1. Estimated rows vs actual rows off by 10x+ → run ANALYZE or increase statistics target
  2. Seq Scan on a large table with a filter → missing index on filter column
  3. Nested Loop with high loop count → multiply actual time by loops — it's much worse than it looks
  4. Hash Join Batches > 1 → increase work_mem or the join is too large for memory
  5. Sort with external merge → increasing work_mem avoids disk spilling
explain-tricks.sql
-- ── Temporarily increase work_mem for one query ──
SET work_mem = '256MB';
EXPLAIN ANALYZE SELECT ...;  -- does it go from Batches:4 to Batches:1?
RESET work_mem;

-- ── Force/disable specific plan strategies ───────
SET enable_seqscan    = OFF;   -- force index usage (testing only!)
SET enable_hashjoin   = OFF;   -- force merge/nested loop join
SET enable_nestloop   = OFF;   -- force hash/merge join
SET enable_mergejoin  = OFF;
RESET ALL;                      -- always reset after testing!

-- ── Lock stats during EXPLAIN (prevents plan change)
BEGIN;
SET LOCAL enable_seqscan = OFF;
EXPLAIN ANALYZE SELECT ...;
ROLLBACK;  -- rolls back the SET, not just the explain

-- ── View plan cache (prepared statements) ────────
SELECT name, statement, calls, total_exec_time/calls AS avg_ms
FROM   pg_prepared_statements;

-- ── Detect parameter sniffing issues ─────────────
-- Generic plan (used after 5 executions of a prepared stmt)
PREPARE get_posts(uuid) AS
  SELECT * FROM posts WHERE author_id = $1;

EXPLAIN EXECUTE get_posts('uuid-here');
-- After 5 calls, PG switches to a generic plan
-- Force custom plan per execution:
SET plan_cache_mode = 'force_custom_plan';
03 — Planner Statistics

Statistics — The Planner's Source of Truth

The planner's cost estimates are only as good as its statistics. Stale or low-resolution statistics are the root cause of most bad plan choices. This section shows how to diagnose and fix them.

statistics.sql
-- ── What statistics does PG store? ───────────────
SELECT
  attname                  AS column,
  n_distinct,              -- estimated distinct values (-1 = all unique)
  correlation,             -- physical vs logical order: 1=perfect, -1=reverse
  most_common_vals,        -- array of most common values
  most_common_freqs,       -- their frequencies (must sum ≤ 1)
  histogram_bounds         -- bucket boundaries for range estimates
FROM   pg_stats
WHERE  tablename = 'posts'
  AND  attname   = 'status';

-- ── Default statistics target is 100 buckets ─────
-- For skewed columns (status, country, category) increase it:
ALTER TABLE posts
  ALTER COLUMN status SET STATISTICS 500;  -- max 10000

ALTER TABLE orders
  ALTER COLUMN created_at SET STATISTICS 500;

ANALYZE posts (status, created_at);  -- re-collect for specific columns

-- ── Multi-column statistics (pg 10+) ─────────────
-- The planner assumes column independence by default.
-- If WHERE status='published' AND country='US' is correlated,
-- create extended statistics to capture that:
CREATE STATISTICS stats_posts_status_country
  (dependencies, ndistinct, mcv)  -- types of stats to collect
  ON status, country
  FROM posts;

ANALYZE posts;

-- Verify it's being used:
SELECT stxname, stxkeys, stxdependencies
FROM   pg_statistic_ext_data
JOIN   pg_statistic_ext ON oid = stxoid
WHERE  stxname = 'stats_posts_status_country';

-- ── Detect stale statistics ───────────────────────
SELECT
  relname                               AS table,
  n_live_tup                            AS live_rows,
  n_dead_tup                            AS dead_rows,
  ROUND(n_dead_tup::numeric/
    NULLIF(n_live_tup+n_dead_tup,0)*100,1) AS bloat_pct,
  last_analyze,
  last_autoanalyze
FROM   pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT  20;
04 — Index Mastery

Index Optimisation In Depth

index-strategies.sql
-- ── Rule 1: Selectivity determines index value ───
-- An index is useful when it eliminates most rows.
-- Low-cardinality columns (boolean, status with 3 values)
-- are often NOT worth indexing alone. Combine them.

-- Bad: status has 3 values, index rarely selective enough
CREATE INDEX idx_bad ON posts(status);

-- Good: combine with a high-cardinality column
CREATE INDEX idx_good ON posts(status, created_at DESC);

-- ── Rule 2: Column order in composite indexes ─────
-- Most selective / most filtered column goes FIRST.
-- This index serves all three query patterns:
CREATE INDEX idx_orders_composite
  ON orders(user_id, status, created_at DESC);

-- Served:  WHERE user_id=?
-- Served:  WHERE user_id=? AND status=?
-- Served:  WHERE user_id=? AND status=? ORDER BY created_at DESC
-- NOT:     WHERE status=? (skips first column)
-- NOT:     WHERE created_at>? (skips first two columns)

-- ── Rule 3: Index-only scans via INCLUDE ─────────
-- Query: SELECT title, published_at FROM posts WHERE author_id=?
-- Without INCLUDE: index scan + heap fetch for title, published_at
-- With INCLUDE:    index-only scan — no heap touch at all
CREATE INDEX idx_posts_author_covering
  ON posts(author_id)
  INCLUDE (title, published_at, status);

-- ── Rule 4: Partial indexes for hot subsets ──────
-- Only 2% of orders are 'pending' but they get 90% of queries
CREATE INDEX idx_orders_pending
  ON orders(created_at, user_id)
  WHERE status = 'pending';
-- Tiny index, matches exactly the queries that need it fast

-- ── Rule 5: Expression indexes for function calls ─
-- These queries DON'T use a plain index on email:
-- WHERE LOWER(email) = 'alice@example.com'
-- WHERE email ILIKE 'alice%'
-- Solution: index the expression itself
CREATE INDEX idx_users_email_ci ON users(LOWER(email));
-- Query MUST use the same expression to match:
-- WHERE LOWER(email) = LOWER($1)  ← uses index
-- WHERE email = $1                ← does NOT use this index

-- ── Index audit queries ────────────────────────────
-- Find unused indexes (wasting write overhead):
SELECT
  schemaname, tablename, indexname,
  pg_size_pretty(pg_relation_size(indexrelid)) AS size,
  idx_scan, idx_tup_read, idx_tup_fetch
FROM   pg_stat_user_indexes
WHERE  idx_scan < 100
  AND  pg_relation_size(indexrelid) > 1024*1024  -- >1MB
ORDER BY pg_relation_size(indexrelid) DESC;

-- Find duplicate / redundant indexes:
SELECT
  a.indexname AS index_a, b.indexname AS index_b,
  a.tablename, a.indexdef
FROM   pg_indexes a
JOIN   pg_indexes b
  ON   a.tablename = b.tablename
 AND   a.indexname < b.indexname
 AND   a.indexdef  = b.indexdef;

-- Find missing indexes via sequential scans on large tables:
SELECT
  relname, seq_scan, seq_tup_read,
  idx_scan,
  pg_size_pretty(pg_total_relation_size(relid)) AS size
FROM   pg_stat_user_tables
WHERE  seq_scan > 100
  AND  pg_total_relation_size(relid) > 10*1024*1024  -- >10MB
ORDER BY seq_tup_read DESC;
05 — Anti-patterns

Query Anti-patterns & Their Fixes

These patterns appear in almost every production codebase. Each one can silently cause a 10x–1000x performance regression as data grows.

1. Functions on Indexed Columns

❌ Index not used — full scan
-- Wrapping a column in a function
-- prevents index use
SELECT * FROM users
WHERE DATE(created_at) = '2026-01-15';

SELECT * FROM users
WHERE EXTRACT(YEAR FROM created_at) = 2026;

SELECT * FROM users
WHERE LOWER(email) = 'alice@example.com';
-- All do seq scan despite index on created_at/email
✅ Index used — range scan
-- Rewrite to keep the column bare
SELECT * FROM users
WHERE created_at BETWEEN
  '2026-01-15' AND '2026-01-16';

SELECT * FROM users
WHERE created_at >= '2026-01-01'
  AND created_at  <  '2027-01-01';

SELECT * FROM users
WHERE email = LOWER($1);
-- Or create: INDEX ON users(LOWER(email))

2. SELECT * in Production

❌ Fetches everything including blobs
-- SELECT * fetches every column:
-- - prevents index-only scans
-- - wastes network bandwidth
-- - fetches large TEXT/JSONB/bytea cols
-- - breaks when columns are added/reordered
SELECT * FROM posts
WHERE author_id = $1
LIMIT 20;
✅ Fetch only what you need
-- Explicit columns enable:
-- - index-only scans (if in INCLUDE)
-- - smaller result sets
-- - explicit contracts in code
SELECT id, title, slug,
       published_at, view_count
FROM   posts
WHERE  author_id = $1
ORDER BY published_at DESC
LIMIT  20;

3. N+1 Queries

❌ N+1 — one query per row
-- Application code (pseudocode):
-- 1 query to get posts
posts = query("SELECT * FROM posts LIMIT 20")

-- Then 20 more queries for authors
FOR post IN posts:
  author = query(
    "SELECT * FROM users WHERE id=$1",
    post.author_id
  )
-- Total: 21 round-trips to DB
✅ Single join — one query
-- One query with a join:
SELECT
  p.id, p.title, p.created_at,
  u.id   AS author_id,
  u.name AS author_name
FROM   posts  p
JOIN   users  u
  ON   u.id = p.author_id
ORDER BY p.created_at DESC
LIMIT  20;
-- Total: 1 round-trip

4. OFFSET Pagination on Large Tables

❌ OFFSET — gets slower on every page
-- Page 1: reads rows 1-20
SELECT * FROM posts
ORDER BY created_at DESC
LIMIT 20 OFFSET 0;

-- Page 500: reads rows 1-10020, discards 10000
SELECT * FROM posts
ORDER BY created_at DESC
LIMIT 20 OFFSET 10000; -- ← very slow!
✅ Keyset — O(log n) on any page
-- First page (no cursor)
SELECT id, title, created_at
FROM   posts
ORDER BY created_at DESC, id DESC
LIMIT  20;

-- Next page (cursor = last row's values)
SELECT id, title, created_at
FROM   posts
WHERE (created_at, id) < ($last_ts, $last_id)
ORDER BY created_at DESC, id DESC
LIMIT  20; -- always fast!

5. COUNT(*) on Large Tables

count-alternatives.sql
-- ❌ Exact COUNT(*) on a 100M row table = full scan = slow
SELECT COUNT(*) FROM events;  -- sequential scan every time

-- ✅ Option 1: Fast approximate count from statistics
SELECT reltuples::BIGINT AS approx_count
FROM   pg_class
WHERE  relname = 'events';
-- Instant. Updated by ANALYZE. Good enough for "showing 2.4M results"

-- ✅ Option 2: Maintain a counter table
CREATE TABLE table_counts (
  table_name TEXT PRIMARY KEY,
  count      BIGINT NOT NULL DEFAULT 0
);

-- Trigger to keep it updated:
CREATE OR REPLACE FUNCTION update_table_count()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  IF TG_OP = 'INSERT' THEN
    UPDATE table_counts SET count = count + 1
    WHERE  table_name = TG_TABLE_NAME;
  ELSIF TG_OP = 'DELETE' THEN
    UPDATE table_counts SET count = count - 1
    WHERE  table_name = TG_TABLE_NAME;
  END IF;
  RETURN NULL;
END;
$$;

-- ✅ Option 3: COUNT(*) with a filter on a partial index
-- CREATE INDEX idx_events_active ON events(id) WHERE deleted_at IS NULL;
SELECT COUNT(*) FROM events WHERE deleted_at IS NULL;
-- Uses the partial index — much faster than full scan

6. IN vs EXISTS vs ANY

in-vs-exists.sql
-- IN with subquery — can be slow (executes subquery for each row)
SELECT * FROM users
WHERE id IN (SELECT author_id FROM posts WHERE status = 'published');

-- EXISTS — stops as soon as one match is found (often faster)
SELECT * FROM users u
WHERE EXISTS (
  SELECT 1 FROM posts WHERE author_id = u.id AND status = 'published'
);

-- NOT IN is especially dangerous with NULLs!
-- If subquery returns ANY NULL, NOT IN returns zero rows (SQL logic)
SELECT * FROM users
WHERE id NOT IN (SELECT author_id FROM posts);
-- ⚠ If author_id is nullable → returns NOTHING

-- Safe alternative: NOT EXISTS
SELECT * FROM users u
WHERE NOT EXISTS (
  SELECT 1 FROM posts WHERE author_id = u.id
);

-- Or LEFT JOIN anti-join (often the fastest):
SELECT u.*
FROM   users u
LEFT JOIN posts p ON p.author_id = u.id
WHERE  p.id IS NULL;  -- users with no posts
06 — Parallel Query

Parallel Query Execution

PostgreSQL 9.6+ can use multiple CPU cores for a single query. On large analytical queries this can give 4–8x speedups. Understanding when and how it kicks in lets you tune it effectively.

parallel-query.sql
-- ── Configuration ─────────────────────────────────
-- max_parallel_workers_per_gather = 4  (workers per query node)
-- max_parallel_workers             = 8  (total across all queries)
-- max_worker_processes             = 16 (total background workers)
-- parallel_tuple_cost              = 0.1
-- parallel_setup_cost              = 1000 (amortised over large scans)
-- min_parallel_table_scan_size     = 8MB (min table size for parallel)

-- ── Check if parallel is used ─────────────────────
EXPLAIN (ANALYZE, BUFFERS)
SELECT status, COUNT(*), SUM(amount)
FROM   large_orders
GROUP BY status;

-- Look for these nodes in the plan:
-- Gather              ← leader collects from workers
-- Gather Merge        ← like Gather but preserves sort order
-- Partial HashAggregate  ← each worker does partial aggregation
-- Parallel Seq Scan   ← workers each scan a segment
-- Parallel Index Scan ← workers each handle index ranges

-- ── Force parallel for testing ────────────────────
SET max_parallel_workers_per_gather = 4;
SET parallel_setup_cost = 0;
SET parallel_tuple_cost = 0;

-- ── Per-table parallel settings ───────────────────
-- Override globally for specific tables:
ALTER TABLE large_events SET (parallel_workers = 8);

-- ── When parallel query DOESN'T kick in ──────────
-- - Query is inside a transaction with non-default isolation
-- - Query calls non-parallel-safe functions
-- - Table has a row-level security policy
-- - Query uses cursors or FOR loops
-- - Result set is too small (planner deems parallel overhead not worth it)

-- ── Mark custom functions as parallel-safe ────────
CREATE OR REPLACE FUNCTION safe_divide(a NUMERIC, b NUMERIC)
RETURNS NUMERIC LANGUAGE sql
PARALLEL SAFE   -- allows use in parallel nodes
IMMUTABLE AS $$
  SELECT CASE WHEN b = 0 THEN NULL ELSE a/b END
$$;
07 — Query Rewriting

Query Rewriting for Performance

query-rewrites.sql
-- ── 1. DISTINCT vs GROUP BY ───────────────────────
-- DISTINCT often performs worse than GROUP BY for deduplication:
SELECT DISTINCT author_id FROM posts;   -- sort-based dedup
SELECT author_id FROM posts GROUP BY author_id; -- hash-based, faster

-- ── 2. CTE vs Subquery — materialisation fences ──
-- In PG 12+, CTEs are NOT automatically materialised (fenced).
-- The planner can inline them. But MATERIALIZED forces a fence:

-- Inline CTE (planner can optimise through it):
WITH recent AS (
  SELECT * FROM posts WHERE created_at > NOW() - '30d'::interval
)
SELECT * FROM recent WHERE status = 'published';
-- Planner pushes status filter down into the CTE

-- Materialised fence (executes CTE once, stores result):
WITH expensive_calc AS MATERIALIZED (
  SELECT user_id, SUM(amount) AS total
  FROM   transactions
  GROUP BY user_id
)
SELECT u.*, ec.total
FROM   users u
JOIN   expensive_calc ec ON ec.user_id = u.id;
-- CTE runs once; join against the materialised result

-- ── 3. LATERAL replaces correlated subqueries ─────
-- ❌ Correlated subquery — runs once PER outer row
SELECT
  u.id, u.name,
  (SELECT COUNT(*) FROM posts WHERE author_id = u.id) AS cnt,
  (SELECT title FROM posts WHERE author_id = u.id
   ORDER BY created_at DESC LIMIT 1)  AS latest_title
FROM users u;

-- ✅ LATERAL — one join, evaluated once per outer row
SELECT u.id, u.name, p.cnt, p.latest_title
FROM   users u
LEFT JOIN LATERAL (
  SELECT
    COUNT(*) AS cnt,
    MAX(title) FILTER (WHERE created_at = MAX(created_at)) AS latest_title
  FROM posts WHERE author_id = u.id
) p ON TRUE;

-- ── 4. Batch updates instead of row-by-row ────────
-- ❌ Slow: 10,000 individual updates (10k WAL records)
FOR user_id IN list:
  UPDATE users SET tier='gold' WHERE id=user_id

-- ✅ Fast: one UPDATE with a VALUES list
UPDATE users SET tier = v.tier
FROM  (VALUES
  ('uuid-1'::uuid, 'gold'),
  ('uuid-2'::uuid, 'silver')
) v(id, tier)
WHERE users.id = v.id;

-- ── 5. First / Last per group with DISTINCT ON ────
-- ❌ Slow: subquery + rank window function
SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY author_id ORDER BY created_at DESC) AS rn
  FROM posts
) t WHERE rn = 1;

-- ✅ Fast: DISTINCT ON (PostgreSQL-specific)
SELECT DISTINCT ON (author_id)
  author_id, title, created_at
FROM   posts
ORDER BY author_id, created_at DESC;
-- One index scan on (author_id, created_at DESC) = very fast
08 — Join Algorithms

Join Optimisation

PostgreSQL has three join algorithms. The planner chooses based on row counts and available indexes. Knowing when each is appropriate lets you guide the planner when it goes wrong.

AlgorithmBest WhenMemoryRequires
Nested LoopSmall outer set + indexed innerO(1)Index on inner join column
Hash JoinMedium/large sets, no useful sortO(smaller table)work_mem for hash table
Merge JoinBoth sides already sortedO(1)Both sides sorted on join key
join-tuning.sql
-- ── Help the planner with join order ──────────────
-- join_collapse_limit controls how many tables the planner
-- considers reordering. Default 8. With 12+ tables it may give up.
SET join_collapse_limit = 12;
SET from_collapse_limit = 12;

-- ── Force join order with explicit subqueries ─────
-- PG won't reorder across subquery boundaries:
SELECT *
FROM (
  SELECT * FROM small_table   -- always hashed first
) small
JOIN large_table ON large_table.id = small.ref_id;

-- ── Hash join spilling — increase work_mem ────────
-- If EXPLAIN shows: Hash Join  Batches: 8  (not 1)
-- The hash table spilled to disk — slow!
-- Fix: increase work_mem for this session:
SET work_mem = '256MB';

-- ── Avoid implicit cross joins (cartesian products) ─
-- ❌ Comma-join with missing WHERE → millions of rows
SELECT * FROM users, posts;   -- CROSS JOIN if no condition!

-- ✅ Always explicit JOIN with ON condition
SELECT * FROM users JOIN posts ON posts.author_id = users.id;

-- ── Join on mismatched types kills indexes ─────────
-- posts.author_id is UUID, joining to TEXT variable
SELECT * FROM posts JOIN users ON users.id = posts.author_id::TEXT;
-- Cast breaks the index! Always join on the same types.

-- ── Joining on expressions — use indexes ─────────
-- If you must join on an expression, create an expression index:
CREATE INDEX idx_posts_author_text ON posts(author_id::TEXT);
-- Then: JOIN ON users.id::TEXT = posts.author_id::TEXT uses it
09 — Partitioning

Table Partitioning

Partitioning splits one logical table into multiple physical tables (partitions). Done right, it enables partition pruning — the planner skips irrelevant partitions entirely, turning a billion-row scan into a million-row scan.

partitioning.sql
-- ── Range partitioning by month ───────────────────
CREATE TABLE events (
  id         BIGINT GENERATED ALWAYS AS IDENTITY,
  user_id    UUID        NOT NULL,
  event_type TEXT        NOT NULL,
  payload    JSONB,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (created_at);

-- Create partitions (automate this in production with pg_partman):
CREATE TABLE events_2026_01
  PARTITION OF events
  FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');

CREATE TABLE events_2026_02
  PARTITION OF events
  FOR VALUES FROM ('2026-02-01') TO ('2026-03-01');

-- Default partition catches everything else:
CREATE TABLE events_default
  PARTITION OF events DEFAULT;

-- ── Partition pruning — the key benefit ──────────
EXPLAIN
SELECT * FROM events
WHERE  created_at BETWEEN '2026-01-01' AND '2026-01-31';
-- → Seq Scan on events_2026_01 (pruning skips all other partitions)

-- ── Indexes on partitions ─────────────────────────
-- Indexes created on the parent apply to all existing + future partitions:
CREATE INDEX idx_events_user_id  ON events(user_id);
CREATE INDEX idx_events_type     ON events(event_type, created_at DESC);

-- ── List partitioning (by category/country) ──────
CREATE TABLE users_by_region (
  id     UUID PRIMARY KEY,
  email  TEXT NOT NULL,
  region TEXT NOT NULL
) PARTITION BY LIST (region);

CREATE TABLE users_emea PARTITION OF users_by_region
  FOR VALUES IN ('UK', 'DE', 'FR', 'NL');

CREATE TABLE users_amer PARTITION OF users_by_region
  FOR VALUES IN ('US', 'CA', 'MX', 'BR');

-- ── Drop old partitions instantly ────────────────
-- vs DELETE which is slow and bloats the table:
ALTER TABLE events DETACH PARTITION events_2024_01;
DROP TABLE events_2024_01;  -- instant, no bloat, no vacuum needed

-- ── Check partition pruning is working ────────────
SELECT * FROM pg_partition_tree('events');
EXPLAIN SELECT * FROM events WHERE created_at = NOW();
-- Should show only 1 partition in the plan, not all of them
10 — Materialised Views

Materialized Views for Pre-computation

A materialised view stores the result of a query on disk — like a cached query result. For expensive aggregations or dashboards that don't need real-time data, they're a game-changer.

materialized-views.sql
-- ── Create a materialised view ────────────────────
-- This expensive aggregation runs once:
CREATE MATERIALIZED VIEW author_stats AS
SELECT
  u.id                                         AS author_id,
  u.username,
  u.email,
  COUNT(p.id)                                  AS total_posts,
  COUNT(p.id) FILTER (WHERE p.status='published') AS published_posts,
  COALESCE(SUM(p.view_count), 0)              AS total_views,
  MAX(p.created_at)                            AS last_post_at
FROM      users  u
LEFT JOIN posts  p ON p.author_id = u.id
GROUP BY u.id
WITH DATA;  -- populate immediately (WITH NO DATA = empty until first refresh)

-- ── Index the materialised view like any table ────
CREATE UNIQUE INDEX ON author_stats(author_id);
CREATE INDEX        ON author_stats(total_views DESC);

-- ── Refresh strategies ────────────────────────────
-- Full refresh (blocks reads, replaces all data):
REFRESH MATERIALIZED VIEW author_stats;

-- Concurrent refresh (does NOT block reads — requires unique index):
REFRESH MATERIALIZED VIEW CONCURRENTLY author_stats;

-- ── Automate refresh with pg_cron ────────────────
CREATE EXTENSION IF NOT EXISTS pg_cron;

-- Refresh every 15 minutes:
SELECT cron.schedule(
  'refresh_author_stats',
  '*/15 * * * *',
  $$REFRESH MATERIALIZED VIEW CONCURRENTLY author_stats$$
);

-- ── Incremental refresh with trigger (when pg_cron isn't enough) ─
-- Track which rows changed, only recompute those:
CREATE TABLE author_stats_dirty (author_id UUID PRIMARY KEY);

CREATE OR REPLACE FUNCTION mark_author_dirty()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
  INSERT INTO author_stats_dirty(author_id)
  VALUES(COALESCE(NEW.author_id, OLD.author_id))
  ON CONFLICT DO NOTHING;
  RETURN NULL;
END; $$;

CREATE TRIGGER posts_dirty
AFTER INSERT OR UPDATE OR DELETE ON posts
FOR EACH ROW EXECUTE FUNCTION mark_author_dirty();
11 — Configuration

Configuration Tuning for Performance

postgresql.conf — production settings
# ── Memory (most impactful settings) ─────────────────────────────

# shared_buffers: PostgreSQL's own page cache
# Rule: 25% of total RAM. Don't set higher — OS cache does the rest.
shared_buffers = 8GB               # for 32GB RAM server

# effective_cache_size: HINT to planner about total available cache
# (shared_buffers + OS file cache). Affects index vs seqscan decisions.
effective_cache_size = 24GB        # ~75% of RAM

# work_mem: memory per sort/hash operation PER QUERY PER NODE
# WARNING: max usage = max_connections * nodes_per_query * work_mem
# Start conservative, increase for specific slow queries via SET
work_mem = 32MB

# maintenance_work_mem: for VACUUM, CREATE INDEX, ALTER TABLE
maintenance_work_mem = 1GB

# ── WAL (Write-Ahead Log) ─────────────────────────────────────────
wal_buffers = 64MB                 # usually 1/32 of shared_buffers
checkpoint_completion_target = 0.9  # spread checkpoint I/O over 90% of interval
wal_compression = on               # compress WAL (saves disk + network for replication)
max_wal_size = 4GB                 # larger = fewer checkpoints = less I/O

# ── Query planner ─────────────────────────────────────────────────
random_page_cost = 1.1             # SET TO 1.1 FOR SSD (default 4.0 assumes HDD)
effective_io_concurrency = 200     # parallel I/O requests (200 for SSD, 2 for HDD)
default_statistics_target = 200    # more buckets = better estimates (default 100)

# ── Parallelism ───────────────────────────────────────────────────
max_worker_processes = 16
max_parallel_workers = 8
max_parallel_workers_per_gather = 4
max_parallel_maintenance_workers = 4  # parallel CREATE INDEX, VACUUM

# ── Autovacuum — keep this healthy ────────────────────────────────
autovacuum_vacuum_scale_factor = 0.05   # trigger at 5% dead rows (default 20%)
autovacuum_analyze_scale_factor = 0.02  # analyze at 2% changed rows (default 10%)
autovacuum_vacuum_cost_delay = 2ms      # less throttling for large tables
autovacuum_max_workers = 6             # more workers for busy DB

# ── Connections ───────────────────────────────────────────────────
# Use PgBouncer in front of Postgres. Each connection costs ~5MB RAM.
max_connections = 200              # PgBouncer pool → fewer actual PG connections

# ── Logging for slow query analysis ──────────────────────────────
log_min_duration_statement = 100   # log queries taking >100ms
log_autovacuum_min_duration = 0    # log all autovacuum runs (for tuning)
log_lock_waits = on                # log when queries wait on locks
log_temp_files = 0                 # log all temp file creation (work_mem spills)
⚠ work_mem is Per Node, Not Per Query
A complex query with 3 sort nodes can use 3 × work_mem — and each connection can have a separate query running. 200 connections × 3 nodes × 64MB = 38GB. Set it conservatively globally and increase per-query with SET LOCAL work_mem = '256MB' inside a transaction when needed.
12 — Monitoring

Monitoring, Profiling & the Query Optimisation Workflow

monitoring.sql
-- ── pg_stat_statements — your #1 tool ────────────
-- Install once per database:
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
-- Add to postgresql.conf: shared_preload_libraries = 'pg_stat_statements'

-- Top 20 slowest queries by total time:
SELECT
  LEFT(query, 100)                              AS query,
  calls,
  ROUND((total_exec_time / calls)::numeric, 2) AS avg_ms,
  ROUND(total_exec_time::numeric, 2)           AS total_ms,
  rows / NULLIF(calls, 0)                      AS avg_rows,
  shared_blks_hit + shared_blks_read            AS total_blocks,
  ROUND(
    shared_blks_hit::numeric /
    NULLIF(shared_blks_hit+shared_blks_read, 0) * 100, 1
  )                                               AS cache_hit_pct,
  temp_blks_written                               AS spilled_to_disk
FROM   pg_stat_statements
WHERE  calls > 10
ORDER BY total_exec_time DESC
LIMIT  20;

-- Top queries by I/O (disk reads):
SELECT
  LEFT(query, 100) AS query,
  shared_blks_read AS disk_reads,
  calls
FROM   pg_stat_statements
ORDER BY shared_blks_read DESC
LIMIT  10;

-- Reset stats (do at the start of a benchmarking session):
SELECT pg_stat_statements_reset();

-- ── Active queries & lock waits ───────────────────
SELECT
  pid, now() - pg_stat_activity.query_start AS duration,
  query, state, wait_event_type, wait_event
FROM   pg_stat_activity
WHERE  (now() - pg_stat_activity.query_start) > '5s'::interval
  AND  state <> 'idle'
ORDER BY duration DESC;

-- ── Find blocking queries ─────────────────────────
SELECT
  blocker.pid                    AS blocker_pid,
  LEFT(blocker.query, 60)       AS blocker_query,
  blocked.pid                    AS blocked_pid,
  LEFT(blocked.query, 60)       AS blocked_query,
  now() - blocked.query_start   AS wait_duration
FROM   pg_stat_activity blocked
JOIN   pg_stat_activity blocker
  ON   blocker.pid = ANY(pg_blocking_pids(blocked.pid))
WHERE  cardinality(pg_blocking_pids(blocked.pid)) > 0;

-- ── Cache hit rate — should be >99% in production ─
SELECT
  SUM(heap_blks_hit)  AS heap_hits,
  SUM(heap_blks_read) AS heap_reads,
  ROUND(
    SUM(heap_blks_hit)::numeric /
    NULLIF(SUM(heap_blks_hit)+SUM(heap_blks_read),0) * 100, 2
  )                   AS cache_hit_ratio
FROM pg_statio_user_tables;

-- ── Table / index bloat ───────────────────────────
SELECT
  relname, n_dead_tup,
  pg_size_pretty(pg_total_relation_size(relid)) AS total_size,
  ROUND(n_dead_tup::numeric/NULLIF(n_live_tup+n_dead_tup,0)*100,1) AS bloat_pct
FROM   pg_stat_user_tables
WHERE  n_dead_tup > 10000
ORDER BY n_dead_tup DESC;

The Optimisation Workflow

  • Step 1
    Identify — find the slow query Use pg_stat_statements to find queries with the highest total time or most disk reads. Use log_min_duration_statement = 100 to capture slow queries in your log file.
  • Step 2
    Diagnose — read the query plan Run EXPLAIN (ANALYZE, BUFFERS) on the slow query. Paste the JSON output to explain.dalibo.com for a visual plan. Identify the slowest node and the root cause.
  • Step 3
    Fix — apply the right technique Missing index? Seq scan on filtered column? → Add index. Stale stats? Estimates wildly off? → ANALYZE or increase statistics_target. Large sort spilling? → Increase work_mem. Bad query pattern? → Rewrite.
  • Step 4
    Verify — measure the improvement Run EXPLAIN ANALYZE again and compare the plans. Check pg_stat_statements after the fix is live to confirm the improvement in production. Track query performance over time with a monitoring tool.
  • Step 5
    Monitor — prevent regressions Set up log_min_duration_statement, alerting on p99 query latency, and a weekly review of the top 20 slowest queries from pg_stat_statements. Catch regressions before users do.
"Every slow query has an explanation. EXPLAIN tells you what it is — the skill is knowing how to read it."
