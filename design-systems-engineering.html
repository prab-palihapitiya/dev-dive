01 — The Problem

Why Design Systems Exist

A design system is most often mistaken for a component library or a Figma file. Neither is correct. A design system is the platform that standardises how every product in an organisation is designed, built, tested, and shipped — and it exists because the cost of inconsistency compounds with every team you add.

The Cost of Inconsistent UI

Without a shared system, every team independently rebuilds the same primitives — and they rarely match. Team A's button has 4px border radius and a 200ms transition. Team B's has 8px radius and no transition. Team C copy-pasted from a tutorial and never updated it when the brand refreshed.

Team A → Button: 4px radius, #1976d2, 200ms transition
Team B → Button: 8px radius, #1565c0, no transition
Team C → Button: 0px radius, #2196f3, 350ms transition

Same product. Three different "primary" colours. Three accessibility audits.
Three maintenance burdens. Zero shared improvements.
🔁

Duplicate Work

Every team builds, tests, and maintains its own version of buttons, inputs, modals, and tables — multiplying engineering cost without multiplying value.

🎭

Inconsistent UX

Users experience a fragmented product when navigating between features built by different teams — eroding trust in brand and product quality.

♿

Accessibility Gaps

Without centralised, audited components, every team must independently solve keyboard navigation, screen reader support, and focus management — and most won't.

🐌

Slower Delivery

Building from scratch every time means every feature ships slower than it would with a battle-tested component library to compose from.

What a Design System Actually Is

A design system is not Bootstrap, Material UI, Tailwind CSS, or any single tool. It is a composite of disciplines that together standardise how interfaces get built:

Design Principles — the "why" behind every decision
+ Design Language — visual identity, voice, interaction feel
+ Design Tokens — the atomic values: color, spacing, type
+ Components — the reusable building blocks
+ Accessibility — built in, not bolted on
+ Documentation — usage guidance, anti-patterns
+ Developer Tooling — CLI, linters, code generators
+ Governance — versioning, contribution, ownership
= Design System

The Evolution Toward Platforms

Static HTML → hand-written markup, no reuse
    ↓
CSS Frameworks → Bootstrap — shared styles, not components
    ↓
Component Libraries → Material UI, Chakra — shared components, generic
    ↓
Design Systems → brand-specific tokens + components + governance
    ↓
Frontend Platforms → CLI, codegen, linting, CI/CD, analytics built on top
"A design system isn't a collection of reusable UI components. It's a product platform that standardises how interfaces are designed, built, tested, documented, and evolved across an organisation."
02 — Foundations

Design Tokens

Design tokens are the atomic values of a design system — every colour, spacing unit, font size, and shadow expressed as a named, reusable variable rather than a hardcoded value. They are what makes theming, multi-brand support, and dark mode possible without rewriting components.

Why Hardcoded Values Fail

CSS✗ Hardcoded
/* Scattered across 200 files — changing the brand colour means a global find-replace */
.button { background: #1976d2; }
.link    { color: #1976d2; }
.badge   { border: 1px solid #1976d2; }
CSS✓ Token-Based
/* One source of truth — change once, propagates everywhere */
:root {
  --color-primary-500: #1976d2;
}
.button { background: var(--color-primary-500); }
.link    { color: var(--color-primary-500); }
.badge   { border: 1px solid var(--color-primary-500); }

The Three-Layer Token Architecture

Mature design systems organise tokens in three layers, each adding semantic meaning to the layer below. This indirection is what makes theming and rebranding possible without touching component code.

Primitive Tokens — raw values, no meaning
    blue-500: #1976d2, spacing-4: 16px, radius-md: 8px
    ↓ referenced by
Semantic Tokens — contextual meaning
    color-action-primary: blue-500, space-gap-default: spacing-4
    ↓ referenced by
Component Tokens — component-specific
    button-primary-bg: color-action-primary, button-gap: space-gap-default

This indirection means a rebrand changes one primitive value, which cascades through every semantic and component token automatically — without touching a single component file.

Token Categories

CategoryExamplesWhy It's Tokenised
Colorprimary, danger, surface, border, text-mutedTheming, dark mode, multi-brand, accessibility contrast control
Typographyfont-size, line-height, font-weight, letter-spacingConsistent vertical rhythm and reading experience across products
Spacingspace-1 through space-12 (4px scale)Predictable layout rhythm; prevents arbitrary pixel values
Radiusradius-sm, radius-md, radius-fullConsistent corner treatment signals visual hierarchy and brand feel
Shadowshadow-sm, shadow-md, shadow-lgConsistent elevation system communicates layering and depth
Motionduration-fast, duration-slow, ease-standardConsistent animation feel; respects prefers-reduced-motion
JSON — Token Definition (Style Dictionary format)
{
  "color": {
    "primitive": {
      "blue": { "500": { "value": "#1976d2" } }
    },
    "semantic": {
      "action": {
        "primary": { "value": "{color.primitive.blue.500}" }
      }
    },
    "component": {
      "button": {
        "primary": {
          "background": { "value": "{color.semantic.action.primary}" }
        }
      }
    }
  }
}
// Style Dictionary builds this into CSS variables, JS constants, iOS/Android resources
Tools for token pipelines
Style Dictionary (Amazon) and Tokens Studio (Figma plugin) are the standard tools for managing tokens as code and syncing between design and engineering. Define tokens once in JSON, generate CSS variables, JS/TS constants, and even native iOS/Android resources from the same source of truth.
03 — Component Engineering

Component Engineering

A component is not just JSX with props. A well-engineered component has a defined purpose, a stable API, complete state coverage, accessibility built in, and a composition strategy that scales beyond its original use case.

What Every Component Must Define

🎯

Purpose

What problem does this solve, and what does it explicitly not solve? A Button is not a Link; conflating them creates accessibility and SEO problems.

🔌

API

Props that are predictable, typed, and minimal. Every prop should have an obvious purpose — resist the urge to add a prop "just in case."

🎭

States

Default, hover, active, focus, disabled, loading, error — and how they compose. A disabled+loading button is a real state that must be designed, not an edge case.

♿

Accessibility

Keyboard interaction, ARIA roles, focus management, screen reader announcements — defined as part of the spec, not added after a complaint.

Component Anatomy — Button Example

Even a "simple" button has more structure than it appears. Decomposing it into named parts clarifies styling boundaries and makes the component easier to extend.

Button Container — padding, border, background, focus ring
    ├── Leading Icon (optional)
    ├── Label — text content, truncation behaviour
    ├── Trailing Icon (optional)
    └── Loading Indicator — replaces label, preserves width to avoid layout shift

States — The Overlooked Discipline

Most component bugs in production come from unhandled state combinations, not from the "default" state. A button can be simultaneously disabled and loading, focused and hovered, or in an error state while loading a retry. Document every combination explicitly.

StateVisual TreatmentInteraction
DefaultBase styling per variantClickable, focusable
HoverBackground/border shift, cursor: pointerVisual feedback only — no state change
FocusVisible focus ring (never outline: none without a replacement)Keyboard-navigable; ring must meet 3:1 contrast
ActivePressed visual state (scale, darker background)During pointer-down, before release
DisabledReduced opacity, no pointer cursoraria-disabled, removed from tab order or non-interactive
LoadingSpinner replaces label; same width to prevent layout shiftNon-interactive; aria-busy="true"
ErrorBorder/text colour shift to danger tokenAssociated error message via aria-describedby

Composition Over Mega-Components

A common anti-pattern: a single <Table> component with 40 props controlling every possible variation. This becomes unmaintainable. The compound component pattern composes smaller pieces that share implicit context — far more flexible and far easier to reason about.

TypeScript — Compound Component Pattern
// ❌ Mega-component: every variation is a new prop
<Table
  data={rows}
  showHeader={true}
  sortable={true}
  selectable={true}
  stickyHeader={true}
  expandable={true}
  // ...30 more props for every possible combination
/>

// ✅ Compound components: compose only what you need
<Table>
  <Table.Header sticky>
    <Table.SortableColumn field="name">Name</Table.SortableColumn>
  </Table.Header>
  <Table.Body>
    {rows.map(row => (
      <Table.Row key={row.id} selectable>
        <Table.Cell>{row.name}</Table.Cell>
      </Table.Row>
    ))}
  </Table.Body>
</Table>
// Each piece shares context via React Context internally —
// consumer composes exactly what their use case needs
04 — Accessibility Engineering

Accessibility Engineering

Accessibility built into a design system propagates automatically to every product built on top of it. Accessibility retrofitted feature-by-feature is ten times more expensive and produces inconsistent results. This is why accessibility belongs at the platform level, not the feature level.

Accessibility by Default

🏗️

Semantic HTML First

Use <button>, not <div onClick>. Native elements provide keyboard handling, focus management, and screen reader semantics for free — before you write a single line of ARIA.

🏷️

ARIA as a Last Resort

"No ARIA is better than bad ARIA." Use ARIA attributes only when semantic HTML cannot express the pattern — custom dropdowns, comboboxes, tab panels.

⌨️

Keyboard Navigation

Every interactive component must be fully operable by keyboard alone: Tab, Shift+Tab, Enter, Space, Escape, and arrow keys for composite widgets like menus and tabs.

🔊

Screen Reader Testing

Test with VoiceOver (Mac), NVDA (Windows, free), or JAWS — not just automated tools. Automated checkers catch ~30-40% of real accessibility issues.

Inclusive Design Beyond Screen Readers

ConsiderationDesign Implication
Motor impairmentsLarge touch targets (44x44px minimum), no time-limited interactions without extensions, drag alternatives
Cognitive accessibilityClear language, consistent navigation patterns, avoid unnecessary complexity, predictable behaviour
Color blindnessNever rely on colour alone to convey state — pair with icons, text, or patterns (8% of men are colour blind)
Vestibular disordersRespect prefers-reduced-motion; avoid large parallax or auto-playing motion
Low visionSupport 200% browser zoom without breaking layout; sufficient contrast at all sizes

Accessibility Testing Strategy

TypeScript — Automated A11y Testing with axe-core
import { render } from '@testing-library/react'
import { axe, toHaveNoViolations } from 'jest-axe'

expect.extend(toHaveNoViolations)

it('Button has no accessibility violations', async () => {
  const { container } = render(<Button>Save</Button>)
  const results = await axe(container)
  expect(results).toHaveNoViolations()
})

// Run automated checks on EVERY component in CI — non-negotiable
// Complement with Lighthouse CI for full-page audits
// Complement with manual screen reader testing for complex widgets
Automated tools catch only ~30-40% of issues
axe-core and Lighthouse catch missing labels, contrast violations, and structural issues reliably. They cannot evaluate whether the reading order makes sense, whether focus management during a modal open/close is correct, or whether your custom dropdown is actually operable with a screen reader. Manual testing is mandatory for any custom interactive component.
05 — Monorepo Architecture

Monorepo & Package Architecture

Enterprise design systems almost always live in a monorepo. This is not a stylistic preference — it is the only architecture that allows tokens, components, and consuming applications to evolve together with atomic commits and coordinated versioning.

Folder Structure

design-system/
  packages/
    tokens/ <-- @org/tokens — raw + semantic + component tokens
    icons/ <-- @org/icons — SVG sprite + React icon components
    components/ <-- @org/components — Button, Input, Modal...
    hooks/ <-- @org/hooks — useDisclosure, useFocusTrap
    themes/ <-- @org/themes — light, dark, brand variants
    utils/ <-- @org/utils — shared formatting, a11y helpers
  apps/
    storybook/ <-- component documentation site
    docs-site/ <-- design principles, guidelines, contribution docs
  tooling/
    eslint-config/ <-- @org/eslint-config — enforces token usage
    cli/ <-- @org/cli — scaffolds new components

Why Separate Packages, Not One Big Package

Splitting tokens, icons, components, and hooks into separate npm packages allows independent versioning and selective adoption. A team that only needs the design tokens for a marketing site does not need to install the entire component library and its dependency tree.

📦

Independent Versioning

Tokens can bump to v3.0 (breaking colour palette change) without forcing a major version bump on components that haven't changed.

⚖️

Bundle Size Control

Consumers import only what they need. A static marketing site can use just @org/tokens with vanilla CSS — no React runtime required.

🔗

Clear Dependency Graph

Components depend on tokens; tokens depend on nothing. This unidirectional flow prevents circular dependencies and clarifies what changes affect what.

🚀

Tooling Standardisation

Turborepo or Nx orchestrate builds across packages with caching — only rebuild what actually changed, dramatically speeding up CI in large monorepos.

JSON — Workspace Package Configuration
// packages/components/package.json
{
  "name": "@org/components",
  "version": "4.2.0",
  "peerDependencies": {
    "react": ">=18"
  },
  "dependencies": {
    "@org/tokens": "workspace:*",
    "@org/icons": "workspace:*"
  },
  "exports": {
    ".": "./dist/index.js",
    "./Button": "./dist/Button.js"
  }
}
06 — Theming & White-Label

Theming, Dark Mode & White-Label

Theming is the direct payoff of a well-built token architecture. If every colour, spacing, and radius value flows through semantic tokens rather than being hardcoded, switching themes is a CSS variable swap — not a component rewrite.

Dark Mode via CSS Variables

CSS — Theme-Aware Tokens
:root {
  --color-bg-surface: #ffffff;
  --color-text-primary: #1a1a1a;
  --color-border: #e2e8f0;
}

[data-theme="dark"] {
  --color-bg-surface: #1a1a1a;
  --color-text-primary: #f5f5f5;
  --color-border: #334155;
}

/* Components reference semantic tokens — zero changes needed for dark mode */
.card {
  background: var(--color-bg-surface);
  color: var(--color-text-primary);
  border: 1px solid var(--color-border);
}

White-Label Architecture for Multi-Tenant SaaS

Enterprise SaaS platforms often need per-customer branding — different logos, primary colours, and sometimes typography — without maintaining separate codebases. The architecture: runtime theme injection driven by tenant configuration, layered over the same semantic token structure.

Tenant Config (from API) → { primaryColor: "#7c3aed", logo: "...", fontFamily: "Inter" }
    ↓
Theme Generator → derives full token set from a single brand colour (tints, shades, contrast-safe text)
    ↓
CSS Variables Injected at Runtime → document.documentElement.style.setProperty(...)
    ↓
Components Render — zero awareness of which tenant is active
Generate full palettes from one brand colour
Tools like Radix Colors and the OKLCH colour space let you generate an accessible 12-step palette (with guaranteed contrast ratios) from a single customer-provided brand colour. This is essential for white-label products — you cannot ask every customer to define 12 colour shades manually.
07 — Tooling & Documentation

Design System Tooling

A design system without documentation and visual testing tooling will drift from its own specification within months. Storybook and visual regression testing are not optional extras — they are the mechanisms that keep the system honest.

Storybook — The Living Documentation

TypeScript — Storybook Story with Controls
import type { Meta, StoryObj } from '@storybook/react'
import { Button } from './Button'

const meta: Meta<typeof Button> = {
  title: 'Components/Button',
  component: Button,
  argTypes: {
    variant: { control: 'select', options: ['primary', 'secondary', 'danger'] },
    size:    { control: 'select', options: ['sm', 'md', 'lg'] },
    disabled: { control: 'boolean' },
  },
}
export default meta

type Story = StoryObj<typeof Button>

export const Primary: Story = {
  args: { variant: 'primary', children: 'Save Changes' },
}

export const LoadingState: Story = {
  args: { variant: 'primary', isLoading: true, children: 'Saving...' },
}

// Storybook auto-generates a controls panel — designers and PMs
// can explore every variant/state combination without reading code

Visual Regression Testing

Code review catches logic bugs. It does not catch "this button is now 2px taller than yesterday." Visual regression testing renders every component story, screenshots it, and diffs against a baseline — catching unintended visual changes before they reach production.

Component Documentation Standard

Every component's documentation page should answer the same questions, consistently, so consumers know exactly where to look:

📋

Props Table

Auto-generated from TypeScript types. Every prop with its type, default value, and description.

🎨

Variants

Visual grid showing every variant and size combination side by side for quick comparison.

♿

Accessibility Notes

Keyboard shortcuts, ARIA roles used, and screen reader behaviour explicitly documented.

🚫

Anti-Patterns

Explicit "don't do this" examples — the most valuable and most often skipped documentation.

08 — Governance

Versioning & Governance

A design system used by dozens of teams needs explicit rules for how it changes. Without governance, every breaking change becomes a cross-team incident, and every team eventually forks the system rather than dealing with the coordination overhead.

Semantic Versioning Discipline

Semantic Versioning for Design Systems
MAJOR.MINOR.PATCH

// PATCH (4.2.0 → 4.2.1): bug fixes, no API change
// Fixed: Button focus ring not visible in Safari

// MINOR (4.2.0 → 4.3.0): new features, backward compatible
// Added: Button now supports a new "ghost" variant

// MAJOR (4.2.0 → 5.0.0): breaking changes
// Changed: Button's `color` prop renamed to `variant`
// Migration guide + codemod provided

Deprecation Strategy

Never remove an API in the same release that introduces its replacement. Mark it deprecated, log a console warning, provide the migration path, and give consumers at least one full major version cycle to migrate before actual removal.

TypeScript — Graceful Deprecation
interface ButtonProps {
  variant: 'primary' | 'secondary' | 'danger'
  /** @deprecated Use `variant` instead. Will be removed in v6.0.0 */
  color?: 'primary' | 'secondary'
}

function Button({ variant, color, ...props }: ButtonProps) {
  if (color && process.env.NODE_ENV === 'development') {
    console.warn(
      '[Button] `color` prop is deprecated and will be removed in v6.0.0. Use `variant` instead.'
    )
  }
  const resolvedVariant = variant ?? color ?? 'primary'
  // ... render using resolvedVariant
}

The Contribution Model

Proposal — RFC describing the problem, not just the solution
    ↓
Design Review — does it fit the design language? Is there a simpler existing pattern?
    ↓
Engineering Review — API design, accessibility, performance implications
    ↓
Implementation — built by proposer or core team, reviewed against the RFC
    ↓
Documentation — Storybook stories, usage guidance, a11y notes — required, not optional
    ↓
Release — versioned, changelogged, announced to consuming teams
Federated contribution beats centralised bottleneck
A single core team approving every component change becomes a bottleneck at scale. Mature systems allow any team to propose and even implement components, with the core team focused on API review, accessibility sign-off, and consistency — not writing every line themselves.
09 — Advanced Patterns

Advanced Component API Design

At the staff level, design system work shifts from building components to designing component APIs that remain stable across years of evolution. Headless architecture, polymorphic components, and cross-framework strategy are the tools that make this possible.

Headless Component Architecture

Headless components separate behaviour (state management, keyboard handling, ARIA wiring) from presentation (visual styling). The same dropdown logic can power a Tailwind-styled dropdown in one product and a completely different visual treatment in another — without duplicating the accessibility-critical behaviour code.

Headless Logic Layer (Radix UI, Headless UI, Ariakit, React Aria)
    ├── State management (open/closed, selected index)
    ├── Keyboard interaction (arrow keys, Escape, type-ahead)
    └── ARIA attributes and focus management
    ↓ consumed by
Your Design System's Presentation Layer
    └── Brand-specific styling, animations, layout
TypeScript — Headless Dropdown via Radix Primitives
import * as DropdownPrimitive from '@radix-ui/react-dropdown-menu'

// Radix handles: focus trap, keyboard nav, ARIA, positioning
// Your design system handles: visual styling only
export function Dropdown({ trigger, children }: DropdownProps) {
  return (
    <DropdownPrimitive.Root>
      <DropdownPrimitive.Trigger asChild>
        {trigger}
      </DropdownPrimitive.Trigger>
      <DropdownPrimitive.Content className="ds-dropdown-content">
        {children}
      </DropdownPrimitive.Content>
    </DropdownPrimitive.Root>
  )
}
// 100% of accessibility-critical logic comes from Radix, battle-tested
// 100% of visual identity comes from your CSS classes

Polymorphic Components

A Button that renders as an <a> when given an href, but as a <button> otherwise, avoids forcing consumers to choose between two nearly identical components. TypeScript's generic constraints make this type-safe.

TypeScript — Polymorphic "as" Prop
type ButtonOwnProps<E extends React.ElementType> = {
  as?: E
  variant?: 'primary' | 'secondary'
}

type ButtonProps<E extends React.ElementType> =
  ButtonOwnProps<E> & Omit<React.ComponentPropsWithoutRef<E>, keyof ButtonOwnProps<E>>

function Button<E extends React.ElementType = 'button'>({
  as, variant = 'primary', ...props
}: ButtonProps<E>) {
  const Component = as ?? 'button'
  return <Component className={cx('ds-button', variant)} {...props} />
}

// Type-safe in both directions:
<Button>Submit</Button>                            // renders <button>
<Button as="a" href="/pricing">View Plans</Button>  // renders <a>, href is type-checked

Cross-Framework Strategy

Organisations with React, Angular, and Vue products simultaneously face a hard choice: maintain three parallel component libraries, or build on Web Components for true framework portability. Web Components provide native browser-level encapsulation but lose some React-specific ergonomics (synthetic events, easy prop typing).

StrategyApproachTrade-off
Per-Framework LibrariesSeparate React, Vue, Angular component packages sharing only design tokensBest DX per framework; 3x maintenance burden
Web ComponentsFramework-agnostic custom elements (Lit, Stencil) wrapped per frameworkTrue portability; loses some framework-specific idioms
Design Tokens OnlyShare only tokens; each framework builds its own componentsMinimal shared code; visual consistency without component reuse

Internationalisation-Ready Components

↔️

RTL Support

Use CSS logical properties (margin-inline-start instead of margin-left) so layouts flip automatically for Arabic, Hebrew, and other RTL languages.

📏

Variable Text Length

German text runs ~30% longer than English for the same content. Components must handle text overflow gracefully — never hardcode fixed widths around text.

📅

Locale-Aware Formatting

Dates, numbers, and currency must format per locale (Intl.NumberFormat, Intl.DateTimeFormat) — never hardcode "MM/DD/YYYY" or "$".

🔤

Font Fallbacks

Latin-optimised fonts often lack glyphs for CJK or Arabic scripts. Define font stacks with appropriate fallbacks per script to avoid missing-character boxes.

10 — Enterprise Systems

Enterprise Design System Patterns

ERP, FinTech, and survey platforms push design systems beyond marketing-site components into territory most public component libraries don't cover: dense data grids, complex permission-aware forms, and domain-specific widgets.

📊 ERP Systems
Data GridsPermissions

Virtualized tables handling 100k+ rows, multi-step wizard forms, role-aware field visibility, and workflow status components (approval chains, audit trails).

📈 Dashboards
ChartsKPIs

Chart component wrappers around D3/Recharts with consistent token-based theming, KPI cards, time range pickers, and filter bar patterns shared across every reporting view.

💳 FinTech
CurrencyRisk

Currency-aware input components with locale formatting, transaction history tables with status badges, audit timeline components, and risk indicator scales with accessible colour encoding.

📋 Survey Platforms
Form BuilderLogic

Drag-and-drop form builder primitives, conditional logic editors (if-this-then-show patterns), question type libraries, and embedded analytics widgets.

Anti-pattern: encoding business logic into the design system
A design system component should never know about "invoices" or "tenants" — that is domain logic and belongs in the application layer. The design system provides a DataTable with sorting and virtualization; the application composes it into an InvoiceTable with business-specific columns and actions.
11 — Scaling & Adoption

Scaling Design Systems

Building a design system is the easy part. Getting dozens of teams with existing legacy applications to adopt it, measuring whether it's actually working, and avoiding the anti-patterns that erode trust — that is where most design system initiatives succeed or fail.

Adoption Strategy

💥 Big Bang Migration

  • Replace all legacy components at once in a coordinated release
  • Fast but extremely high risk
  • Only viable for small, single-team applications
  • Almost always fails at enterprise scale

🪜 Incremental / Strangler Pattern

  • New features use the design system; legacy code migrates opportunistically
  • Lower risk, slower visible progress
  • Both systems coexist during transition — needs clear boundaries
  • The proven approach for large, multi-team codebases

Measuring Success

MetricWhat It Tells YouHow to Measure
Component Adoption Rate% of UI built with design system components vs custom codeStatic analysis of imports across the codebase
Duplicate Component CountHow much reinvention is still happeningSearch for component names/patterns repeated across repos
Accessibility ScoreAggregate a11y health across consuming productsLighthouse CI scores tracked over time per product
Time-to-ShipWhether the system actually accelerates deliveryCompare feature delivery time before/after adoption
Developer SatisfactionWhether teams want to use the system or are forced toQuarterly DX survey — NPS-style scoring
Bug Rate in DS ComponentsQuality of the shared layer — bugs here multiply across productsIssue tracker labels filtered to design-system-owned components

Organisational Challenges

🏛️

Platform Ownership

Someone must own the roadmap, prioritise requests, and say no to scope creep. Without clear ownership, the system becomes everyone's responsibility and nobody's priority.

⚖️

Standardisation vs Autonomy

Teams resist being told exactly how to build everything. The system should standardise primitives while leaving room for product-specific composition and innovation.

📜

RFC Process

A lightweight, written proposal process for significant changes prevents both bottlenecks (everything needs approval) and chaos (anyone changes anything).

👥

Dedicated Team

At meaningful scale (50+ engineers, multiple products), a design system needs a dedicated team — not a side project for whoever has spare time.

Anti-Patterns to Avoid

Anti-PatternWhy It Hurts
Over-engineering every componentA Button doesn't need 40 props for every imaginable use case — most will never be used and all of them add maintenance burden
Exposing unstable APIsInternal implementation details leaking into the public API lock you into supporting them forever
Encoding business logicA design system component that knows about "invoices" can't be reused for "orders" — keep it domain-agnostic
Too many variantsFive button variants that look nearly identical create decision paralysis, not flexibility
Premature abstractionBuilding a flexible, generic API before you have 2-3 real use cases guarantees you'll guess wrong
Ignoring accessibility until launchRetrofitting a11y is dramatically more expensive than building it in from day one
Poor or absent documentationAn undocumented component gets reimplemented by every team that doesn't know it exists

AI's Role in Design Systems

AI tooling is changing how design systems are maintained — not by replacing human design judgment, but by automating the repetitive parts of building and validating components.

🤖

AI Component Scaffolding

Generate boilerplate component structure, Storybook stories, and test files from a spec — freeing engineers to focus on the actual API design.

🔍

AI Design Review

Automated checks for token usage, missing states, and consistency violations before a PR even reaches human review.

📝

AI Documentation

Draft usage guidance and prop descriptions from component code — a human reviews and refines rather than writing from scratch.

🎨

AI Theme Generation

Generate accessible full token palettes from a single brand colour or natural-language description, with contrast validation built in.

Core practice: Design tokens, Storybook, accessibility fundamentals (semantic HTML, ARIA basics, keyboard navigation), basic theming with CSS variables.
Advanced
System building: Monorepo architecture, package versioning, cross-team governance, component API design (compound components, polymorphic components), performance optimization, visual regression testing.
Staff+
Platform & strategy: Frontend platform engineering (CLI, codegen, linting), enterprise multi-brand architecture, cross-framework component strategy, organisational governance, developer experience metrics, AI-assisted design system automation, and design system ROI measurement that justifies continued investment to leadership.

    ↓

    ↓
