01 — The First Gate

Code Review Techniques

Code review is not a quality gate — it is a knowledge-sharing, mentoring, and design-alignment ritual. The best reviews catch architectural issues, not missing semicolons. Leave that to linters.

🎯

Review the Right Things

Logic correctness, security implications, performance at scale, error handling completeness, test coverage. Not formatting — automate that with Prettier/ESLint.

🔍

Understand Before Commenting

Read the linked ticket, understand the intent. Don't comment on style choices that are valid alternatives. Ask questions before making assertions.

📖

Review in Layers

First pass: architecture and design (10 min). Second pass: logic and edge cases. Third pass: tests. Don't mix these — you'll miss macro issues chasing micro ones.

💬

Comment with Context

Explain the why, not the what. "This will cause N+1 queries at scale because..." is useful. "Bad" is not.

The Code Review Checklist

  • 1
    Does it solve the right problem? Re-read the ticket. Is the implementation aligned with the requirement, or has scope crept in? Is there an over-engineered solution to a simple problem?
  • 2
    Is it correct? Walk through edge cases mentally: empty inputs, null/undefined, race conditions, integer overflow, off-by-one. What happens when a downstream service fails?
  • 3
    Is it secure? Scan for unvalidated inputs, SQL/NoSQL injection, hardcoded secrets, broken auth checks, IDOR, missing rate limits. Security issues in review cost 10× less than in production.
  • 4
    Will it scale? Identify N+1 queries, unbounded loops, missing pagination, synchronous I/O in hot paths, memory leaks (event listeners not removed, closures holding references).
  • 5
    Is it tested properly? Are happy paths, error paths, and edge cases covered? Are tests testing behaviour or implementation details? Would the tests catch a real regression?
  • 6
    Is it maintainable? Will the next engineer understand this in 6 months? Complex logic needs comments explaining intent. Long functions need extraction. Magic numbers need named constants.

Reviewing as an Author

❌ Makes reviewing hard
- 47 files changed, 2,400 additions
- Mix of refactor + feature + bugfix
- No description in PR body
- Force-pushed over reviewer comments
- "LGTM" as self-approval
✅ Makes reviewing easy
- < 400 lines of meaningful diff
- Single concern per PR
- PR body: why, what, how to test
- Responds to each comment
- Marks conversations resolved
- Links to ticket + relevant context
💡 The 200-Line Rule
Research from SmartBear shows review effectiveness drops sharply after 400 lines of diff, with the steepest drop at 200. Above 400 lines, reviewers miss up to 60% of defects. Break large features into stacked PRs: data model → business logic → API → UI. Each review stays sharp.
02 — Confidence at Every Layer

Testing Pyramid & Test Design

The testing pyramid is not a bureaucratic rule — it is an economic model. Fast, cheap unit tests at the base; fewer, slower integration tests in the middle; a handful of E2E tests at the top. Inverting the pyramid means slow, flaky, expensive CI.

🔬

Unit Tests

Test one function or class in isolation. All dependencies mocked. Run in milliseconds. Should cover every logic branch. The foundation — aim for 70%+ of your test suite.

🔗

Integration Tests

Test that components work together correctly: API handler + middleware + DB. Real database (testcontainers). Slower (~100ms each) but catches wiring bugs unit tests miss.

🌐

E2E Tests

Drive the full application via the browser or API gateway. Extremely valuable but slow and flaky. Keep to critical user journeys: signup, checkout, login.

📸

Contract Tests

Verify that a consumer and provider agree on an API contract (Pact). Catches breaking changes before they reach production. Essential in microservice architectures.

JavaScript — Test Design: Behaviour, Not Implementation
import { describe, it, expect, vi, beforeEach } from 'vitest'

// ── ❌ Testing implementation details ─────────────
// Coupled to private method names; breaks on safe refactor
it('calls _hashPassword and _saveToDb', async () => {
  const spy1 = vi.spyOn(service, '_hashPassword' as any)
  const spy2 = vi.spyOn(service, '_saveToDb' as any)
  await service.register(input)
  expect(spy1).toHaveBeenCalled()
  expect(spy2).toHaveBeenCalled()
})

// ── ✅ Testing observable behaviour ──────────────
// Survives any refactor that preserves the contract
describe('UserService.register', () => {
  let service: UserService
  let mockDb: MockUserRepo

  beforeEach(() => {
    mockDb  = { create: vi.fn().mockResolvedValue({ id: 'u1', email: 'a@b.com' }) }
    service = new UserService(mockDb)
  })

  it('returns the created user without the password hash', async () => {
    const result = await service.register({ email: 'a@b.com', password: 'Secret1!' })
    expect(result).toMatchObject({ id: 'u1', email: 'a@b.com' })
    expect(result).not.toHaveProperty('passwordHash')
  })

  it('stores a hashed password, not plaintext', async () => {
    await service.register({ email: 'a@b.com', password: 'Secret1!' })
    const storedHash = mockDb.create.mock.calls[0][0].passwordHash
    expect(storedHash).not.toBe('Secret1!')  // not plaintext
    expect(storedHash).toMatch(/^\$argon2/)        // argon2 hash
  })

  it('throws if the email is already registered', async () => {
    mockDb.create.mockRejectedValue(
      Object.assign(new Error(), { code: 'P2002' })   // Prisma unique violation
    )
    await expect(service.register({ email: 'a@b.com', password: 'Secret1!' }))
      .rejects.toThrow('Email already registered')
  })
})

// ── Property-based testing — find edge cases automatically ─
import { fc, test } from '@fast-check/vitest'

test.prop([fc.string({ minLength: 1, maxLength: 200 })])(
  'slug is always lowercase with only hyphens and alphanumeric',
  (title) => {
    const slug = toSlug(title)
    expect(slug).toMatch(/^[a-z0-9-]*$/)
  }
)

Test Naming Convention

❌ Opaque test names
it('works')
it('test 1')
it('UserService')
it('handles error')
✅ Spec-style: subject + scenario + outcome
it('returns 401 when no token is provided')
it('sends welcome email after successful registration')
it('throws DuplicateEmailError when email is taken')
it('returns empty array when no products match filter')
💡 Testing Trophy
Kent C. Dodds argues for the "Testing Trophy" — heavier emphasis on integration tests than the classic pyramid, because they give the best confidence-to-cost ratio for typical web apps. Unit tests for pure logic; integration tests for service + DB; E2E for the three most critical paths. Write tests that resemble how the software is used.
03 — Foundation of Design

SOLID Principles

SOLID is not a checklist to follow religiously — it is a set of heuristics for writing code that is easier to change. Apply them when they reduce complexity, not to show architectural sophistication.

S
Single Responsibility Principle
A module should have one and only one reason to change. "Reason to change" means one stakeholder or concern. A class that handles both business logic and database persistence has two reasons — split it. Fat controllers are the most common SRP violation in Node.js.
O
Open/Closed Principle
Software entities should be open for extension, closed for modification. Add new behaviour by adding new code (new classes, new functions), not by changing existing, tested code. Strategy pattern and composition over inheritance are the main mechanisms.
L
Liskov Substitution Principle
Subtypes must be substitutable for their base types without altering correctness. If a function accepts a Bird, passing a Penguin that throws on .fly() violates LSP. Design inheritance so every override preserves the parent's contract.
I
Interface Segregation Principle
No client should be forced to depend on methods it doesn't use. Fat interfaces should be split into smaller, role-specific ones. In TypeScript, prefer small focused interfaces over large, grab-bag types.
D
Dependency Inversion Principle
High-level modules should not depend on low-level modules; both should depend on abstractions. Depend on interfaces, not concrete implementations. This is what makes code testable — you can inject mocks because dependencies are injected, not instantiated internally.
TypeScript — SOLID in Practice: DIP + ISP
// ── ❌ Violates DIP: hard dependency on concrete class ─
class OrderService {
  private mailer = new SendGridMailer()        // can't swap, can't test
  private db     = new PrismaClient()          // requires real DB in tests
}

// ── ✅ Depends on abstractions — injectable + testable ─
interface IMailer {
  send(to: string, subject: string, body: string): Promise<void>
}
interface IOrderRepository {
  create(data: CreateOrderDto):            Promise<Order>
  findById(id: string):                   Promise<Order | null>
}

class OrderService {
  constructor(
    private readonly orders: IOrderRepository,
    private readonly mailer: IMailer,
  ) {}

  async place(data: CreateOrderDto) {
    const order = await this.orders.create(data)
    await this.mailer.send(data.email, 'Order confirmed', orderTemplate(order))
    return order
  }
}

// Production wiring (DI container or manual)
const service = new OrderService(new PrismaOrderRepo(), new SendGridMailer())
// Test wiring
const svc     = new OrderService(mockOrderRepo, mockMailer)
04 — Continuous Improvement

Refactoring Patterns

Refactoring is changing the structure of code without changing its observable behaviour. The discipline: never refactor without tests. Never refactor and add features in the same commit. Small, safe steps.

TypeScript — Extract Function & Replace Conditional with Polymorphism
// ── Refactor 1: Extract Function ──────────────────
// Before: long function doing too many things
async function processOrder(order: Order) {
  // 20 lines calculating discount...
  let discount = 0
  if (order.user.tier === 'gold')   discount = 0.15
  if (order.user.tier === 'silver') discount = 0.10
  if (order.total > 500)           discount += 0.05
  // 40 more lines...
}

// After: extracted, named, testable
function calculateDiscount(user: User, orderTotal: number): number {
  const tierDiscount = { gold: 0.15, silver: 0.10, bronze: 0 }
  const bulkDiscount  = orderTotal > 500 ? 0.05 : 0
  return (tierDiscount[user.tier] ?? 0) + bulkDiscount
}

// ── Refactor 2: Replace Conditional with Strategy ──
// Before: giant switch statement, violates OCP
function generateReport(type: string, data: any) {
  switch (type) {
    case 'pdf':   return generatePDF(data)
    case 'csv':   return generateCSV(data)
    case 'excel': return generateExcel(data)
    // Adding a new type means editing this function
  }
}

// After: registry of strategies — extend without modifying
interface ReportGenerator {
  generate(data: ReportData): Promise<Buffer>
}
const generators = new Map<string, ReportGenerator>([
  ['pdf',   new PDFGenerator()],
  ['csv',   new CSVGenerator()],
  ['excel', new ExcelGenerator()],
])
async function generateReport(type: string, data: ReportData) {
  const gen = generators.get(type)
  if (!gen) throw new Error(`Unknown report type: ${type}`)
  return gen.generate(data)
}

05 — Contract Design

API Design

An API is a contract between systems. Once published, breaking it is expensive — it breaks every consumer. Design for clarity, consistency, and evolvability from day one.

RESTful Resource Design

HTTP — REST API Design Principles
# ── Resource naming — nouns, plural, lowercase ────────────────
GET    /api/v1/users           # list users
POST   /api/v1/users           # create user
GET    /api/v1/users/:id       # get one user
PATCH  /api/v1/users/:id       # partial update (prefer over PUT)
DELETE /api/v1/users/:id       # delete user

# Nested resources — max one level of nesting
GET  /api/v1/users/:id/orders  # user's orders
POST /api/v1/users/:id/orders  # create order for user

# ❌ Avoid verb-in-path anti-patterns
POST /api/v1/createUser
POST /api/v1/getUsers
GET  /api/v1/deleteUser?id=123

# ── Status codes — be precise ─────────────────────────────────
200 OK              # successful GET, PATCH, DELETE
201 Created         # successful POST — include Location header
204 No Content      # successful DELETE with no response body
400 Bad Request     # validation error — include which fields failed
401 Unauthorized    # no valid auth credentials
403 Forbidden       # authenticated but not authorised for THIS resource
404 Not Found       # resource doesn't exist
409 Conflict        # duplicate email, version conflict
422 Unprocessable   # semantically invalid input (not just format)
429 Too Many Req    # rate limit exceeded
500 Server Error    # unexpected — never return stack traces
TypeScript — Consistent Error & Response Envelopes
// ── Consistent success envelope ───────────────────
interface ApiResponse<T> {
  data:       T
  meta?: {
    page:       number
    pageSize:   number
    totalCount: number
    totalPages: number
  }
}

// ── Consistent error envelope — always the same shape ─
interface ApiError {
  error: {
    code:      string              // machine-readable: "VALIDATION_ERROR"
    message:   string              // human-readable
    details?: Record<string, string>  // field-level validation errors
    requestId: string              // for support correlation
  }
}

// ── Pagination — cursor-based for large datasets ──
interface CursorPage<T> {
  data:       T[]
  nextCursor: string | null   // null = no more pages
  hasMore:    boolean
}

06 — Structural Thinking

Clean Architecture

Clean Architecture (Uncle Bob) organises code into concentric layers where dependencies only point inward. The core business logic has zero knowledge of the HTTP framework, the database, or any external library.

💠

Entities (Domain)

Pure business objects and rules. No framework imports. No database. The most stable layer — changes only when core business rules change.

⚙️

Use Cases (Application)

Orchestrate entities to fulfil one business scenario. PlaceOrderUseCase, RegisterUserUseCase. Depend on repository interfaces, not implementations.

🔌

Interface Adapters

Controllers, presenters, repository implementations. Convert data between use-case format and framework/DB format. Express handlers live here.

🌐

Frameworks & Drivers

Express, Prisma, Redis, S3. The outermost ring — swappable. Replacing Express with Fastify should not touch a single use case.

TypeScript — Clean Architecture: Project Structure
// src/
// ├── domain/                   ← innermost: pure business logic
// │   ├── entities/
// │   │   └── Order.ts          ← Order value object, business rules
// │   └── repositories/
// │       └── IOrderRepository.ts  ← interface only, no import from DB
// │
// ├── application/              ← use cases, orchestration
// │   └── use-cases/
// │       ├── PlaceOrder.ts     ← imports domain entities + interfaces
// │       └── CancelOrder.ts
// │
// ├── infrastructure/           ← implementations (Prisma, Redis, S3)
// │   ├── repositories/
// │   │   └── PrismaOrderRepo.ts  ← implements IOrderRepository
// │   └── mail/
// │       └── SendGridMailer.ts
// │
// └── interfaces/               ← HTTP, CLI, workers
//     └── http/
//         ├── routes/
//         │   └── orderRoutes.ts  ← thin: validate → call use case → respond
//         └── middleware/

// ── domain/entities/Order.ts — zero external imports ──
export class Order {
  constructor(
    public readonly id:      string,
    public readonly items:   OrderItem[],
    private          status: OrderStatus,
  ) {
    if (items.length === 0) throw new Error('Order must have at least one item')
  }

  cancel(): void {
    if (this.status === 'shipped')
      throw new Error('Cannot cancel a shipped order')
    this.status = 'cancelled'
  }

  get total() {
    return this.items.reduce((sum, i) => sum + i.price * i.qty, 0)
  }
}

// ── interfaces/http/routes/orderRoutes.ts — thin controller ─
router.post('/orders', authenticate, async (req, res) => {
  const dto = PlaceOrderSchema.parse(req.body)       // validate
  const order = await placeOrderUseCase.execute(dto) // delegate
  res.status(201).json({ data: order })               // respond
})
07 — Reusable Solutions

Design Patterns

Design patterns are named solutions to recurring design problems. They give teams a shared vocabulary. The goal is never to implement patterns for the sake of it — it is to recognise when a problem matches a known solution.

TypeScript — Factory, Strategy & Adapter Patterns
// ── FACTORY — centralise creation, hide complexity ─
interface StorageProvider {
  upload(key: string, data: Buffer): Promise<string>
  download(key: string):               Promise<Buffer>
}

class StorageFactory {
  static create(): StorageProvider {
    switch (process.env.STORAGE_PROVIDER) {
      case 's3':    return new S3Storage(process.env.S3_BUCKET!)
      case 'gcs':   return new GCSStorage(process.env.GCS_BUCKET!)
      case 'local': return new LocalStorage('./uploads')
      default:      throw new Error('Unknown storage provider')
    }
  }
}

// ── STRATEGY — swap algorithms at runtime ─────────
interface PricingStrategy {
  calculate(basePrice: number, user: User): number
}

class PremiumPricing implements PricingStrategy {
  calculate(base: number) { return base * 0.80 }  // 20% off
}
class StandardPricing implements PricingStrategy {
  calculate(base: number) { return base }
}
class PricingEngine {
  private strategies = new Map<string, PricingStrategy>([
    ['premium',  new PremiumPricing()],
    ['standard', new StandardPricing()],
  ])
  price(base: number, user: User) {
    const strategy = this.strategies.get(user.tier) ?? new StandardPricing()
    return strategy.calculate(base, user)
  }
}

// ── ADAPTER — wrap incompatible interfaces ─────────
// Third-party library has a different interface than our app expects
interface ILogger {
  info(msg: string, meta?: object): void
  error(msg: string, err?: Error):  void
}
class PinoAdapter implements ILogger {
  constructor(private logger: pino.Logger) {}
  info(msg: string, meta?: object) { this.logger.info(meta ?? {}, msg) }
  error(msg: string, err?: Error)  { this.logger.error({ err }, msg)   }
}
// Now you can swap pino for winston or any logger without touching app code

// ── OBSERVER — decouple event producer from consumers ─
class EventBus {
  private handlers = new Map<string, Function[]>()

  on(event: string, handler: Function) {
    if (!this.handlers.has(event)) this.handlers.set(event, [])
    this.handlers.get(event)!.push(handler)
  }
  async emit(event: string, payload: unknown) {
    const fns = this.handlers.get(event) ?? []
    await Promise.all(fns.map(fn => fn(payload)))
  }
}

Dependency Management

Every dependency you add is code you didn't write, can't fully control, and are responsible for. The discipline of dependency management is knowing when to add, when to write your own, and how to keep what you have healthy.

09 — Institutional Memory

Documentation & Architecture Decision Records

The most valuable documentation explains decisions made and the context that drove them — not what the code does (it shows that), but why it does it that way. ADRs are the format that captures this permanently.

Architecture Decision Records (ADRs)

An ADR is a short document capturing a significant architectural decision — what was decided, the context that led to it, the options considered, and the consequences. They go in docs/decisions/ in version control alongside the code they describe.

Markdown — ADR Template
# ADR-0023: Use Cursor-Based Pagination for the Feed API

## Status
Accepted — 2026-05-14

## Context
The `/feed` endpoint currently uses offset pagination (`?page=N&limit=20`).
At scale (users with 10k+ follows), OFFSET 100000 scans 100k rows on every 
request, causing p99 latency of 4,200ms. Complaints from power users tripled 
in the last sprint.

## Decision
Replace offset pagination with cursor-based pagination using an opaque Base64 
cursor encoding the last seen `(createdAt, id)` tuple.

## Options Considered
1. **Offset pagination (status quo)** — simple but degrades at depth
2. **Cursor-based pagination** — O(log n) with a (createdAt, id) compound index
3. **Keyset pagination with seekable cursor** — same as 2, different UX
4. **Elasticsearch for feed** — dramatically overengineered for this problem

## Consequences
**Positive:**
- p99 latency drops from 4,200ms to ~12ms at any depth
- No "page drift" when items are inserted mid-pagination

**Negative:**
- Cannot jump to an arbitrary page (page 47 of results)
- Consumer API change: `nextCursor` replaces `page` in response
- Existing mobile clients need updating (2-week deprecation window)

## References
- https://use-the-index-luke.com/no-offset
- Spike branch: feature/cursor-pagination-spike

What to Document and What Not To

Document thisDon't document this
Why you chose library X over YWhat a function does (use clear naming)
Non-obvious business rules and constraintsHow a for-loop iterates
Known limitations and TODOs with contextComments that restate the code
External system integration quirksOutdated docs that no one maintains
Runbook for on-call incidentsAuto-generated API docs (generate those)
Performance benchmarks and reasoningAnything that changes faster than the docs
💡 Docs as Code
Store all documentation — ADRs, runbooks, architecture diagrams (Mermaid/PlantUML as text) — in the same Git repository as the code. They version together, get reviewed together, and die together (no orphaned Confluence pages from 2019).
10 — Thinking Well Under Pressure

Technical Decision Making

Technical decisions are made under uncertainty, with incomplete information, and often with time pressure. A structured approach doesn't slow you down — it prevents you from having to reverse expensive mistakes six months later.

"The most important skill in engineering is not knowing the right answer — it is knowing how to find it, and knowing when you need to."

The Decision Framework

  • 1
    Define the actual problem. Is the stated problem the real problem? "We need Kafka" is usually a solution masquerading as a problem. "We need to decouple order processing from payment processing" is the actual problem. Write the problem in one sentence without mentioning a technology.
  • 2
    Identify your constraints. Timeline, team expertise, operational burden, budget, compliance requirements. A technically superior solution your team can't operate is worse than a simpler one they can. Constraints are not excuses — they are inputs.
  • 3
    Generate at least three options. One that minimises change (do less), one that solves it correctly, and one ambitious stretch option. Single-option decisions are not decisions — they are announcements. Document all options even if you know the answer.
  • 4
    Define your evaluation criteria. Write the criteria before evaluating. Operational complexity, time to implement, performance at 10× current load, reversibility. Agree on what matters before bias kicks in.
  • 5
    Timebox the decision. Set a decision date. "We'll decide by Thursday" prevents analysis paralysis. For reversible decisions (library choice, API design), decide in hours. For irreversible ones (database engine, cloud provider), take a week and build a spike.
  • 6
    Write it down as an ADR. Include the context, the options, the chosen option, and the reasoning. Future-you and your teammates will thank present-you.

Reversible vs Irreversible Decisions

Decision TypeExamplesProcessUrgency
Reversible (Type 2)API route names, variable naming, library choice for utilitiesMake it, document it, move on. Easy to undo.Hours
Partially ReversibleDatabase schema, API contracts with consumersSpike + design doc + team reviewDays
Irreversible (Type 1)Cloud provider, database engine, public API breaking changeFull RFC process, explicit stakeholder sign-off, ADRWeeks
ℹ️ Disagree and Commit
Not every decision needs consensus. Once a decision is made through a fair process, team members who disagree should disagree on record (in the ADR) and then fully commit. Passive non-compliance — implementing half-heartedly or relitigating in every code review — is more damaging than the original disagreement.
11 — Team Practices

Code Ownership & Team Practices

Code ownership, on-call practices, and technical debt management are engineering disciplines as important as system design. Teams that neglect them build fast until they don't.

INI — CODEOWNERS: Enforce Review by Domain Owners
# .github/CODEOWNERS
# Every PR touching these paths requires review from the specified owner(s)

# Payments team owns all billing/payment code
/src/features/billing/     @myorg/payments-team
/src/features/checkout/    @myorg/payments-team

# Auth team owns identity-critical paths
/src/features/auth/        @myorg/auth-team @security-lead
/src/middleware/auth*.ts   @myorg/auth-team

# Infrastructure — any infra change needs SRE review
/infra/                    @myorg/sre-team
/Dockerfile                @myorg/sre-team
/.github/workflows/        @myorg/sre-team

# API contracts — changes need API council review
/src/interfaces/http/      @myorg/api-council

# Global: senior engineer reviews all changes
*                          @myorg/senior-engineers

Managing Technical Debt Deliberately

Technical debt is not inherently bad — it is a financing decision. The problem is debt you don't know you have, or debt you can't repay. The practice of explicit debt management makes it visible and controllable.

❌ Debt invisibility
// TODO: fix this later
// HACK: this is terrible but ship it
// FIXME: don't ask

// Debt accumulates silently.
// No ticket. No owner. No timeline.
// "Later" is never.

// Acceptable at current scale (< 1000 items).
// Owner: @alice | Due: 2026-Q3

12 — Ship with Confidence

Production Readiness Checklist

Every feature that touches production should pass this gate. Not as bureaucracy — as the minimum bar for not waking someone up at 3am.

  • ✅
    Tests cover the happy path and all significant error paths. Passing tests in CI — not "it works on my machine". Coverage report reviewed, not just percentage.
  • ✅
    All inputs validated. No req.body.anything reaching the database unchecked. Zod/Valibot schemas on every API boundary. Output encoding where appropriate.
  • ✅
    Errors handled and logged with context. No silent catch blocks. No console.log in production code. Structured logs with requestId for correlation.
  • ✅
    No secrets or PII in code or logs. Git history scanned (gitleaks). No tokens in environment variable names that end up logged. PII redacted before logging.
  • ✅
    Performance considered. New DB queries have EXPLAIN run. No unbounded queries without limits. N+1 patterns identified and resolved. Caching strategy for hot reads.
  • ✅
    Feature flags or gradual rollout for risky changes. New features behind a flag for A/B testing and safe rollback. Database migrations are backward-compatible (add column before removing old one).
  • ✅
    Observability in place. New code paths have metrics, logs, and alerts. On-call runbook updated. SLO impact assessed — does this change affect latency or error rate?
  • ✅
    ADR written for significant decisions. Why this approach was chosen over alternatives. Linked in the PR description. Stored in docs/decisions/.
  • ✅
    Rollback plan documented. Can this be reverted with a deploy? Does it require a DB migration rollback? How long would rollback take? Who needs to be notified?
  • ✅
    Reviewed by someone who didn't write it. At least one reviewer who understands the domain. CODEOWNERS satisfied. Security-critical paths reviewed by a security-aware engineer.
