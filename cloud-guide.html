01 — Network Foundation

VPCs, Subnets & Firewalls

A Virtual Private Cloud is your isolated network slice inside the cloud provider. Everything else — compute, databases, services — lives inside it. Getting your network design right from the start prevents painful re-architecture later.

Core Concepts

🌐

VPC / VNet

Logically isolated network. You define the CIDR block (e.g. 10.0.0.0/16), which gives you 65,536 IP addresses to subnet.

📦

Subnets

Sub-divisions of a VPC. Public subnets have a route to the Internet Gateway. Private subnets don't — they're for databases and internal services.

🛡️

Security Groups / NSGs

Stateful firewalls at the instance/resource level. Only specify what traffic is allowed; everything else is denied by default.

📋

Network ACLs

Stateless subnet-level firewall. Evaluated in rule-number order. Rarely needed beyond the default — Security Groups cover most use cases.

🔀

Route Tables

Define where traffic is sent. Public subnet route table has 0.0.0.0/0 → IGW. Private subnet routes through a NAT Gateway for outbound internet.

🔗

VPC Peering / Private Link

Connect two VPCs privately without traffic leaving the provider backbone. Private Link exposes a service endpoint without full peering.

Terraform — Production 3-Tier VPC 
# VPC with public, private-app, and private-data tiers across 3 AZs
resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true
  enable_dns_support   = true
  tags = { Name = "prod-vpc" }
}

# Public subnets — one per AZ, hosts ALB + NAT gateways
resource "aws_subnet" "public" {
  count             = 3
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet("10.0.0.0/16", 8, count.index)  # /24 each
  availability_zone = data.aws_availability_zones.available.names[count.index]
  map_public_ip_on_launch = true
}

# Private app subnets — EC2/ECS/EKS nodes
resource "aws_subnet" "private_app" {
  count             = 3
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet("10.0.0.0/16", 8, count.index + 10)
  availability_zone = data.aws_availability_zones.available.names[count.index]
}

# Private data subnets — RDS, ElastiCache, no internet access ever
resource "aws_subnet" "private_data" {
  count             = 3
  vpc_id            = aws_vpc.main.id
  cidr_block        = cidrsubnet("10.0.0.0/16", 8, count.index + 20)
  availability_zone = data.aws_availability_zones.available.names[count.index]
}

# Security group — web tier, allow HTTPS from anywhere, egress all
resource "aws_security_group" "web" {
  name   = "web-sg"
  vpc_id = aws_vpc.main.id
  ingress { from_port = 443; to_port = 443; protocol = "tcp"; cidr_blocks = ["0.0.0.0/0"] }
  ingress { from_port = 80;  to_port = 80;  protocol = "tcp"; cidr_blocks = ["0.0.0.0/0"] }
  egress  { from_port = 0;   to_port = 0;   protocol = "-1";  cidr_blocks = ["0.0.0.0/0"] }
}

# App SG — only accept traffic from web SG
resource "aws_security_group" "app" {
  name   = "app-sg"
  vpc_id = aws_vpc.main.id
  ingress {
    from_port       = 3000; to_port = 3000; protocol = "tcp"
    security_groups = [aws_security_group.web.id]   # source is a SG, not CIDR
  }
}
⚠️ GCP VPC is Global
GCP VPCs span all regions by default — subnets are regional but the VPC itself is global. This is unlike AWS/Azure where a VPC/VNet is region-scoped. In GCP you can have a single VPC with subnets in every region, simplifying multi-region networking significantly.
02 — Containerisation

Docker & Containers

Containers package your application and all its dependencies into a portable, reproducible unit. Every cloud provider runs containers natively — understanding Docker deeply is the prerequisite for everything from ECS to Kubernetes.

Dockerfile — Production Node.js Multi-Stage 
# Stage 1: install deps (cached layer)
FROM node:22-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production --ignore-scripts

# Stage 2: build (TypeScript compile, asset bundling)
FROM node:22-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 3: final — minimal runtime image
FROM node:22-alpine AS runner
RUN addgroup -S app && adduser -S app -G app
WORKDIR /app
ENV NODE_ENV=production PORT=3000
COPY --from=deps     /app/node_modules ./node_modules
COPY --from=builder  /app/dist         ./dist
USER app
EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1
CMD ["node", "dist/server.js"]
# Result: ~120MB image vs ~1.2GB if built in single stage

Always scan images for CVEs before pushing to production. 03 — Container Orchestration

Kubernetes Fundamentals

Kubernetes has become the universal container orchestration standard. Every major cloud offers a managed control plane — EKS, GKE, AKS — so you write the same YAML and let them handle master node upgrades, etcd backups, and control plane HA.

Core Resource Types

📦

Pod

Smallest deployable unit. One or more containers sharing a network namespace and volumes. Ephemeral — never manage directly.

🔄

Deployment

Manages a ReplicaSet. Declarative rolling updates, rollback. The standard way to run stateless applications.

🗂️

StatefulSet

Like Deployment but pods get stable network identities and persistent volumes. Use for databases, Kafka, Redis clusters.

🌐

Service

Stable virtual IP (ClusterIP) or external endpoint (LoadBalancer/NodePort). Discovers pods via label selectors.

🚪

Ingress

HTTP/HTTPS routing to Services. One load balancer → many services via hostname and path rules. Replaces a fleet of ALBs.

⚙️

ConfigMap / Secret

Decouple config from images. ConfigMap for non-sensitive data. Secret for credentials (base64-encoded, use External Secrets Operator for real secrets management).

YAML — Production Deployment with HPA 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
  namespace: production
spec:
  replicas: 3
  selector:
    matchLabels: { app: api-server }
  strategy:
    type: RollingUpdate
    rollingUpdate: { maxSurge: 1, maxUnavailable: 0 }  # zero-downtime
  template:
    metadata:
      labels: { app: api-server }
    spec:
      serviceAccountName: api-server-sa   # IRSA / Workload Identity
      containers:
      - name: api
        image: 123456.dkr.ecr.eu-west-1.amazonaws.com/api:v2.4.1
        ports: [{containerPort: 3000}]
        resources:
          requests: { cpu: "100m", memory: "256Mi" }
          limits:   { cpu: "500m", memory: "512Mi" }
        readinessProbe:
          httpGet: { path: /health, port: 3000 }
          initialDelaySeconds: 5; periodSeconds: 10
        livenessProbe:
          httpGet: { path: /health, port: 3000 }
          initialDelaySeconds: 15; periodSeconds: 20
        envFrom:
        - configMapRef:  { name: api-config }
        - secretRef:     { name: api-secrets }
      topologySpreadConstraints:           # spread across AZs
      - maxSkew: 1
        topologyKey: topology.kubernetes.io/zone
        whenUnsatisfiable: DoNotSchedule
---
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-server-hpa
spec:
  scaleTargetRef: { apiVersion: apps/v1, kind: Deployment, name: api-server }
  minReplicas: 3; maxReplicas: 20
  metrics:
  - type: Resource
    resource: { name: cpu, target: { type: Utilization, averageUtilization: 70 } }

06 — Async Communication

Message Queues & Event Streaming

Async messaging decouples producers from consumers, buffers traffic spikes, and enables reliable event-driven architectures. The choice between a queue, pub/sub topic, and event stream depends on your delivery semantics.

Pattern ▲ AWS ● GCP ■ Azure Best for
Queue (point-to-point) SQS Cloud Tasks Service Bus Queues Work queues, job offloading
Pub/Sub fanout SNS → SQS Pub/Sub Service Bus Topics Notifications, fanout events
Event streaming Kinesis / MSK (Kafka) Pub/Sub (streaming) / Dataflow Event Hubs (Kafka-compatible) High-throughput, replay, analytics
Event bus EventBridge Eventarc Event Grid Serverless event routing, SaaS integration
JavaScript — AWS SQS Consumer with Visibility Timeout 
import { SQSClient, ReceiveMessageCommand, DeleteMessageCommand,
         ChangeMessageVisibilityCommand } from '@aws-sdk/client-sqs'

const sqs = new SQSClient({ region: 'eu-west-1' })

async function pollQueue(queueUrl, handler) {
  while (true) {
    const { Messages = [] } = await sqs.send(new ReceiveMessageCommand({
      QueueUrl:            queueUrl,
      MaxNumberOfMessages:  10,
      WaitTimeSeconds:      20,   // long-polling — saves cost
      VisibilityTimeout:    60,   // processing must complete within 60s
    }))

    await Promise.allSettled(Messages.map(async (msg) => {
      try {
        await handler(JSON.parse(msg.Body))
        // Delete only after successful processing
        await sqs.send(new DeleteMessageCommand({
          QueueUrl: queueUrl, ReceiptHandle: msg.ReceiptHandle
        }))
      } catch (err) {
        logger.error(err, 'Message processing failed — will retry')
        // Extend visibility so other consumers don't pick it up yet
        await sqs.send(new ChangeMessageVisibilityCommand({
          QueueUrl: queueUrl, ReceiptHandle: msg.ReceiptHandle, VisibilityTimeout: 0
        }))  // return to queue immediately for retry
      }
    }))
  }
}
07 — Identity & Security

IAM & Cloud Security

Cloud IAM controls who can do what on which resource. Every permission not explicitly granted is denied. The blast radius of a misconfigured IAM role can be catastrophic — least privilege is not optional.

Terraform — IRSA: Pod-level IAM Roles for EKS 
# IRSA: IAM Roles for Service Accounts — no long-lived credentials in pods
data "aws_iam_policy_document" "api_assume_role" {
  statement {
    actions = ["sts:AssumeRoleWithWebIdentity"]
    principals {
      type        = "Federated"
      identifiers = [module.eks.oidc_provider_arn]
    }
    condition {
      test     = "StringEquals"
      variable = "${module.eks.oidc_provider}:sub"
      values   = ["system:serviceaccount:production:api-server-sa"]
    }
  }
}

resource "aws_iam_role" "api" {
  name               = "eks-api-server"
  assume_role_policy = data.aws_iam_policy_document.api_assume_role.json
}

# Scoped policy — only what the pod actually needs
resource "aws_iam_role_policy" "api_policy" {
  name = "api-permissions"; role = aws_iam_role.api.id
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      { Effect   = "Allow"
        Action   = ["s3:GetObject", "s3:PutObject"]
        Resource = "arn:aws:s3:::${var.uploads_bucket}/uploads/*" },
      { Effect   = "Allow"
        Action   = ["secretsmanager:GetSecretValue"]
        Resource = "arn:aws:secretsmanager:eu-west-1:*:secret:prod/api/*" },
    ]
  })
}

08 — Visibility

Monitoring & Observability

Observability in the cloud is the three pillars — metrics, logs, and traces — plus alerting, dashboards, and anomaly detection. Every provider has native tooling, but OpenTelemetry lets you stay vendor-neutral.

JavaScript — OpenTelemetry (vendor-neutral instrumentation) 
import { NodeSDK }             from '@opentelemetry/sdk-node'
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node'
import { OTLPTraceExporter }   from '@opentelemetry/exporter-trace-otlp-http'
import { OTLPMetricExporter }  from '@opentelemetry/exporter-metrics-otlp-http'
import { PeriodicExportingMetricReader } from '@opentelemetry/sdk-metrics'

// Initialise BEFORE importing app code — auto-instruments HTTP, DB, etc.
const sdk = new NodeSDK({
  traceExporter: new OTLPTraceExporter({
    url: process.env.OTEL_EXPORTER_OTLP_ENDPOINT  // → Datadog / Grafana / X-Ray
  }),
  metricReader: new PeriodicExportingMetricReader({
    exporter:     new OTLPMetricExporter(),
    exportIntervalMillis: 30_000,
  }),
  instrumentations: [getNodeAutoInstrumentations()],  // HTTP, express, MongoDB, pg...
  serviceName: 'api-server',
})
sdk.start()

// Custom span example
import { trace } from '@opentelemetry/api'
const tracer = trace.getTracer('api-server')

async function processOrder(orderId) {
  return tracer.startActiveSpan('processOrder', async (span) => {
    span.setAttributes({ 'order.id': orderId, 'order.service': 'checkout' })
    try {
      const result = await chargeAndFulfil(orderId)
      span.setStatus({ code: 1 })  // OK
      return result
    } catch (err) {
      span.recordException(err)
      span.setStatus({ code: 2, message: err.message })  // ERROR
      throw err
    } finally { span.end() }
  })
}

09 — Delivery Pipeline

CI/CD Pipelines

A mature CI/CD pipeline is the backbone of reliable software delivery — it automates testing, building, security scanning, and deployment so every merge to main can ship to production safely.

YAML — GitHub Actions: Build → Scan → Push → Deploy to EKS 
name: Production Deploy
on:
  push: { branches: [main] }

env:
  AWS_REGION: eu-west-1
  ECR_REPO:   123456.dkr.ecr.eu-west-1.amazonaws.com/api
  EKS_CLUSTER: prod-cluster

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with: { node-version: '22', cache: 'npm' }
    - run: npm ci && npm run test:ci && npm audit --audit-level=high

  build-push:
    needs: test
    runs-on: ubuntu-latest
    permissions:
      id-token: write   # OIDC — no long-lived AWS keys in secrets
      contents: read
    outputs:
      image-tag: ${{ steps.meta.outputs.version }}
    steps:
    - uses: actions/checkout@v4
    - uses: aws-actions/configure-aws-credentials@v4
      with:
        role-to-assume: arn:aws:iam::123456:role/github-actions-deploy
        aws-region:     ${{ env.AWS_REGION }}
    - id: meta
      uses: docker/metadata-action@v5
      with: { images: "${{ env.ECR_REPO }}", tags: "type=sha,prefix=" }
    - uses: docker/build-push-action@v5
      with:
        push: true
        tags: ${{ steps.meta.outputs.tags }}
        cache-from: type=gha
        cache-to:   type=gha,mode=max
    - name: Scan image (Trivy)
      uses: aquasecurity/trivy-action@master
      with:
        image-ref:    "${{ env.ECR_REPO }}:${{ steps.meta.outputs.version }}"
        severity:     CRITICAL,HIGH
        exit-code:    '1'   # fail on critical CVE

  deploy:
    needs: build-push
    runs-on: ubuntu-latest
    environment: production   # requires manual approval in GitHub UI
    steps:
    - uses: aws-actions/configure-aws-credentials@v4
      with:
        role-to-assume: arn:aws:iam::123456:role/github-actions-deploy
        aws-region:     ${{ env.AWS_REGION }}
    - run: |
        aws eks update-kubeconfig --name $EKS_CLUSTER
        kubectl set image deployment/api-server \
          api=$ECR_REPO:${{ needs.build-push.outputs.image-tag }} \
          -n production
        kubectl rollout status deployment/api-server -n production --timeout=5m
10 — Growth Patterns

Scalability Patterns

Scalability is the ability to handle growth — more users, more data, more requests — without proportional growth in cost or operational complexity. The patterns differ based on whether your bottleneck is compute, data, or network.

↔️

Horizontal Scaling

Add more instances. Stateless services scale horizontally trivially. Use HPA in Kubernetes, Auto Scaling Groups in EC2, or min/max replicas in Cloud Run.

↕️

Vertical Scaling

Increase instance size (more CPU/RAM). Limited by max instance size. Use for databases and stateful services where horizontal is hard. Schedule downtime window.

🗂️

Database Sharding

Partition data across multiple DB instances by a shard key. Distributes write load. Complex to implement — prefer managed services (Aurora, Spanner) first.

📖

Read Replicas

Direct read traffic to replicas, writes to primary. Cheap horizontal read scale. Beware replication lag on replicas — not suitable for read-your-writes workloads.

💾

Caching Layers

Redis/Memcached in front of the database. Cache hot data at application, CDN, or database query result level. Reduces DB load by 80%+ on read-heavy workloads.

📡

CDN & Edge

Serve static assets and cached API responses from edge nodes close to users. CloudFront, Cloud CDN, Azure CDN. Reduces origin load and latency globally.

11 — Resilience

High Availability & Disaster Recovery

High Availability means the system stays up despite component failures. Disaster Recovery means you can recover when an entire region fails. They are different problems requiring different solutions.

"Plan for everything to fail. Design so individual failures don't become system failures."

HA Design Patterns

  • 1
    Multi-AZ Deployment — Spread instances across ≥3 Availability Zones. AZs are physically separate data centres within a region. An AZ outage (rare but real) should cause zero downtime. All managed services (RDS Multi-AZ, ElastiCache with replicas, EKS with nodes across AZs) do this automatically.
  • 2
    Health Checks & Auto-Healing — Load balancer health checks remove unhealthy instances from rotation. ASG/Kubernetes replaces them. Target: detect failure in <30s, replace in <5min. Test with Chaos Engineering (AWS Fault Injection Service, GCP Fault Injection).
  • 3
    Circuit Breakers — Prevent cascade failures when a downstream service degrades. Open the circuit after N consecutive failures; probe again after a timeout. Libraries: opossum (Node.js), resilience4j (JVM). Service meshes (Istio, Linkerd) do this transparently.
  • 4
    Database HA — RDS Multi-AZ uses synchronous replication with automatic failover (60–120s). Aurora has <30s failover with Global Database for cross-region. GCP Cloud SQL HA and Azure Flexible Server work similarly.

DR Strategies by RTO/RPO

JavaScript — Circuit Breaker with opossum 
import CircuitBreaker from 'opossum'

const breaker = new CircuitBreaker(callPaymentService, {
  timeout:              3000,    // call must complete within 3s
  errorThresholdPercentage: 50,  // open if >50% of calls fail
  resetTimeout:         10000,   // try again after 10s
  volumeThreshold:      5,       // need ≥5 calls to evaluate
})

// Fallback when circuit is open
breaker.fallback((orderId) => ({
  status:  'queued',
  message: 'Payment service degraded — order queued for retry',
  orderId
}))

breaker.on('open',     () => logger.warn('Payment circuit OPEN'))
breaker.on('halfOpen', () => logger.info('Payment circuit probing...'))
breaker.on('close',    () => logger.info('Payment circuit CLOSED — recovered'))
12 — Economics

Cost Optimisation

Cloud bills grow silently. The most effective cost reductions — Reserved Instances, Spot usage, right-sizing, and storage tiering — require intentional engineering decisions, not retroactive cleanup.

📋

Reserved Capacity

Commit 1–3 years for 30–72% off on-demand. AWS RIs / Savings Plans, GCP CUDs, Azure Reserved Instances. Use for baseline always-on capacity.

⚡

Spot / Preemptible

Up to 90% off for interruptible workloads. Batch jobs, CI runners, non-critical workers. Always build your app to handle graceful shutdown in 30s.

📏

Right-Sizing

Most instances are over-provisioned by 2–4×. Use CloudWatch/Cloud Monitoring to find p95 CPU/memory usage. Downsize instances that never exceed 30% CPU.

🗄️

Storage Tiering

S3 Intelligent-Tiering, GCS autoclass, Azure lifecycle policies automatically move infrequently accessed data to cheaper tiers. Can reduce storage costs 60–80%.

📡

Data Transfer

Egress is expensive (~$0.08/GB). Minimise cross-AZ traffic, use VPC endpoints for S3/DynamoDB (free), and CDN to serve content without hitting origin.

🔮

Scale to Zero

Serverless (Lambda, Cloud Run, Container Apps) has zero cost at zero traffic. Move batch jobs, APIs with variable traffic, and dev/staging environments to scale-to-zero.

14 — Event-Driven Compute

Serverless & Edge

Serverless eliminates server management — you deploy functions or containers, the cloud handles the rest: scaling, patching, HA, and billing per invocation. It is not the right choice for every workload, but for the right ones it dramatically reduces operational overhead.
