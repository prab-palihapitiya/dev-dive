01 — Traffic Distribution

Load Balancing

A load balancer is the front door of every scalable system. Its job is to distribute incoming traffic across a pool of servers so that no single server becomes the bottleneck — and to route around unhealthy instances transparently.

The fundamental requirement is that your application servers be stateless. If server A holds a user's session in memory, routing their next request to server B fails. Stateless servers delegate all shared state — sessions, uploads, locks — to external systems (Redis, S3, PostgreSQL), making every instance identical and interchangeable.

Algorithms

Algorithm How it works Best for Weakness
Round Robin Requests cycle through servers in order Homogeneous servers, uniform request cost Ignores server load; slow requests pile up on one server
Least Connections Routes to server with fewest active connections Varying request durations (API gateways, streaming) Requires connection tracking overhead
Weighted Round Robin Higher-capacity servers get proportionally more traffic Mixed server specs; gradual canary rollouts Static weights don't adapt to real-time load
IP Hash Same client IP always routes to same server Sticky sessions when shared state is unavoidable Uneven distribution if IPs are skewed; breaks on IP change
Least Response Time Routes to server with lowest latency + fewest connections Latency-sensitive workloads More complex; requires active latency probing

L4 vs L7

Layer 4 load balancers operate on TCP/UDP — they see IP addresses and ports, nothing more. Fast (no packet inspection), but routing decisions are limited. Layer 7 load balancers understand HTTP — they can route on URL path, headers, cookies, and request body. This enables path-based routing (/api/* → API servers, /static/* → CDN), A/B testing via header injection, and WebSocket upgrade handling. Use L7 for all HTTP workloads; reserve L4 for raw TCP (gRPC, database proxies).

Nginx — L7 load balancer with health checks + canary 
upstream api_v2 {
  # Canary: 5% traffic to new version
  server api-v2-canary:3000 weight=5;
  server api-stable-1:3000 weight=95;
  server api-stable-2:3000 weight=95;

  # Remove unhealthy server after 3 failures; restore after 30s
  server api-stable-3:3000
    max_fails=3 fail_timeout=30s;

  keepalive 64;  # reuse connections — critical for high RPS
}

server {
  location /api {
    proxy_pass         http://api_v2;
    proxy_next_upstream error timeout http_500;
    proxy_http_version  1.1;
    proxy_set_header    Connection "";  # enable keepalive
  }
}
Health check depth
Don't health check /ping that returns 200 unconditionally. Check /health that verifies the server can reach its database and cache. A server that can't talk to Postgres is not healthy, even if its process is running.
02 — Performance Multiplier

Caching

Caching is the most impactful performance tool in distributed systems. A cache hit eliminates the most expensive operation in your stack — a database round-trip. The engineering challenge is making caches correct, not just fast.

Cache Layers

🌐

CDN

Static assets, public pages. Closest to users globally. Zero origin load for cache hits. TTL-based invalidation.

🔀

Reverse Proxy

Nginx / Varnish cache full HTTP responses. Useful for public API responses that are expensive to compute.

⚡

Application Cache

Redis / Memcached. Shared across all app servers. Flexible — cache any serialisable object, result, or computed value.

🗄️

Database Cache

Query plan cache and buffer pool. Automatic. Tune shared_buffers (Postgres) to fit your hot dataset in RAM.

📦

In-Process

Node.js Map / LRU cache. Zero network overhead. Invalidation requires coordination across instances — only for truly immutable or short-TTL data.

Strategies & Eviction

Strategy Flow Consistency Use case
Cache-aside Read: check cache → miss → DB → populate. Write: invalidate cache. Good Default for most read-heavy data
Write-through Every write goes to cache AND DB simultaneously. Strong Data that's always read after write (user profile)
Write-behind Write to cache immediately; async flush to DB. Eventual High-write, loss-tolerant (view counters, analytics)
Read-through Cache fetches from DB on miss; caller always hits cache. Good Simplifies application code; cache library handles misses

Eviction policies: LRU (Least Recently Used) is the correct default — evict the data least likely to be needed. LFU (Least Frequently Used) is better for hotspot-heavy workloads where certain keys are always hot. TTL expiry is always required alongside an eviction policy — don't let stale data survive indefinitely even if it's frequently accessed.

TypeScript — Multi-layer cache with L1 in-process + L2 Redis 
import { LRUCache } from 'lru-cache'

// L1: in-process LRU — zero latency, bounded size
const l1 = new LRUCache<string, unknown>({
  max: 1000, ttl: 30_000  // 30s — accept slight stale for hot items
})

async function get<T>(key: string, fetch: () => Promise<T>): Promise<T> {
  // L1 hit — no network at all
  const l1hit = l1.get(key)
  if (l1hit !== undefined) return l1hit as T

  // L2 hit — Redis (~0.5ms)
  const l2hit = await redis.get(key)
  if (l2hit) {
    const val = JSON.parse(l2hit)
    l1.set(key, val)   // promote to L1
    return val
  }

  // Miss — database (~5–50ms)
  const val = await fetch()
  await redis.setex(key, 300, JSON.stringify(val))
  l1.set(key, val)
  return val
}
Cache invalidation is the hardest problem
Never rely solely on TTL for correctness. When data changes, invalidate the key immediately. For complex dependency chains (a product price change invalidates 30 cached search result pages), maintain a tag-based invalidation index: every cache entry is tagged with its dependencies, and invalidating a tag purges all tagged entries at once.
03 — Data Storage

Databases

There is no universally correct database. There is only the database whose consistency model, query model, and scaling characteristics best match the specific data access patterns of your domain.

Type Model Strengths Weaknesses Reach for when
Relational
PostgreSQL, MySQL 		Tables + relations ACID, rich queries, joins, foreign keys, mature tooling Horizontal scaling hard; schema migrations require care Transactional data, complex relationships, reporting
Document
MongoDB, Firestore 		JSON documents Flexible schema, natural nesting, easy horizontal scale No joins, consistency trade-offs, duplicate data Content management, product catalogues, user profiles with variable shape
Wide-Column
Cassandra, DynamoDB 		Row key + columns Extreme write throughput, linear horizontal scaling, no SPOF No joins, limited queries, eventual consistency, data modelling is hard Time-series, IoT, audit logs, messaging at massive scale
Graph
Neo4j, Amazon Neptune 		Nodes + edges Relationship traversal is fast and natural Poor for aggregate queries; niche tooling Social graphs, fraud detection, recommendation engines
Time-Series
InfluxDB, TimescaleDB 		Timestamped metrics Compressed storage, fast range queries, auto-rollup Poor for relational queries outside time dimension Monitoring metrics, IoT sensor data, financial tick data
Search
Elasticsearch, Typesense 		Inverted index Full-text search, fuzzy matching, faceting, aggregations Not a source of truth — sync lag, eventual consistency Product search, log search, full-text queries
Polyglot persistence
Production systems routinely use multiple databases. Orders in PostgreSQL (ACID required). Product catalogue in MongoDB (flexible schema). Search in Elasticsearch. Session state in Redis. Metrics in TimescaleDB. Each data type lives in the store that best serves its access pattern. The cost: synchronisation between stores and operational complexity — justified when the performance or flexibility gains are real, not speculative.
04 — Query Acceleration

Indexes

An index is a trade: you pay write overhead and storage to make reads fast. Every index you add slows every write. The art is indexing exactly the queries that need it — no more.

Index Types

Type Structure Supports Common use
B-Tree Balanced tree — sorted =, <, >, BETWEEN, ORDER BY, LIKE 'prefix%' Default. 95% of indexes should be B-Tree.
Hash Hash table = only (faster than B-Tree for equality-only) Lookup-only tables: user sessions, token maps
GiST / GIN Generalised search tree Full-text, JSON, geometric, array operators JSONB querying, @@ full-text search in PostgreSQL
Partial B-Tree with WHERE clause Same as B-Tree, but smaller and faster for filtered subsets WHERE deleted_at IS NULL — 95% of queries only touch live rows
Covering B-Tree including extra columns Index-only scans — no heap access needed High-frequency queries that always fetch the same columns
Composite B-Tree on multiple columns Leftmost prefix queries (user_id, created_at DESC) for per-user sorted lists
SQL — Strategic index design patterns 
-- Partial index: only live orders (avoids indexing archived rows)
CREATE INDEX CONCURRENTLY idx_orders_live
  ON orders (user_id, created_at DESC)
  WHERE deleted_at IS NULL;

-- Covering index: query returns only indexed columns → no heap hit
CREATE INDEX idx_products_search
  ON products (category_id, price)
  INCLUDE (name, slug, thumbnail_url);
-- Query: SELECT name, slug, thumbnail_url FROM products
--         WHERE category_id=5 ORDER BY price → index-only scan

-- GIN index for JSONB querying
CREATE INDEX idx_events_metadata
  ON events USING gin(metadata);
-- Enables: WHERE metadata @> '{"type": "purchase"}'

-- ALWAYS use CONCURRENTLY in production — avoids table lock
-- EXPLAIN ANALYZE before and after to verify the index is used
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)
SELECT name, slug FROM products
WHERE category_id = 5 ORDER BY price LIMIT 20;
Index bloat
Indexes grow monotonically. Deleted rows leave dead index entries. On high-churn tables, regular VACUUM and periodic REINDEX CONCURRENTLY keep indexes compact. An index twice the size of the table it serves is a performance liability, not an asset.
05 — Async Decoupling

Queues

A queue decouples the rate at which work is produced from the rate at which it is consumed. It is the correct answer to: "what do we do when the downstream service can't keep up with inbound traffic?"

The fundamental contract of a queue: producers write messages without waiting for consumers to process them. Consumers process messages at their own pace. If consumers fall behind, messages accumulate — and queue depth becomes the leading indicator of capacity problems.

Delivery Semantics

Semantic Guarantee Risk When to use
At-most-once Message delivered 0 or 1 time Loss — message may never be processed Metrics, logs, non-critical notifications
At-least-once Message delivered 1+ times Duplicates — consumer must be idempotent Default for most workloads: emails, webhooks, order processing
Exactly-once Delivered exactly once High cost — requires distributed transactions Financial transfers, inventory deduction (use idempotency keys instead)

Backpressure is a queue's most important operational property. When the queue grows faster than consumers drain it, the system must either: add consumers (scale out), drop messages (at-most-once, acceptable for metrics), or apply backpressure to producers (slow down ingestion). Without an explicit backpressure strategy, a queue becomes an unbounded memory sink that eventually exhausts the broker.

TypeScript — BullMQ producer + idempotent consumer 
import { Queue, Worker, Job } from 'bullmq'

// Producer: fire and forget
const emailQ = new Queue('email', {
  defaultJobOptions: {
    attempts: 5,
    backoff: { type: 'exponential', delay: 2000 },
    removeOnComplete: { count: 100 },
    removeOnFail: false  // keep failed jobs for inspection
  }
})

await emailQ.add('welcome',
  { userId: 'usr_123', email: 'a@b.com' },
  { jobId: `welcome:usr_123` }  // deduplication key
)

// Consumer: idempotent — safe to run twice
new Worker('email', async (job: Job) => {
  // Check if already sent — idempotency guard
  const sent = await redis.get(`sent:${job.id}`)
  if (sent) return  // already processed — safe noop

  await sendWelcomeEmail(job.data)
  await redis.setex(`sent:${job.id}`, 86400 * 7, '1')
}, { concurrency: 10 })
06 — Reactive Architecture

Events

Events represent facts — things that have already happened. An event is immutable; it cannot be undone. This immutability is the foundation of event-driven architectures, event sourcing, and audit trails.

Event-driven architecture separates producers from consumers through an event bus. A PaymentCompleted event published to Kafka is consumed independently by the fulfilment service, the email service, the fraud service, and the analytics pipeline — all without the payment service knowing they exist. Adding a new consumer requires no change to the producer.

Event sourcing takes this further: the database is the event log. Current state is derived by replaying events. Every state change is captured as a fact: OrderPlaced, ItemAdded, PaymentTaken, OrderShipped. The full history of every entity is preserved. Time-travel queries ("what was this order's state on Tuesday?") are free. Rebuilding a new read model from history is possible at any time.

TypeScript — Event sourcing: append event + rebuild state 
type OrderEvent =
  | { type: 'OrderPlaced';  customerId: string; total: number }
  | { type: 'PaymentTaken'; amount: number }
  | { type: 'OrderShipped'; trackingId: string }
  | { type: 'OrderCancelled'; reason: string }

interface OrderState {
  status: 'pending' | 'paid' | 'shipped' | 'cancelled'
  total: number
  paidAmount: number
}

// Rebuild current state by replaying all events (pure function)
function rehydrate(events: OrderEvent[]): OrderState {
  return events.reduce((state, e) => {
    switch (e.type) {
      case 'OrderPlaced':   return { ...state, status: 'pending', total: e.total }
      case 'PaymentTaken':  return { ...state, status: 'paid', paidAmount: e.amount }
      case 'OrderShipped':  return { ...state, status: 'shipped' }
      case 'OrderCancelled':return { ...state, status: 'cancelled' }
    }
  }, { status: 'pending', total: 0, paidAmount: 0 } as OrderState)
}

// Replay all historical events → build fraud score projection
Events vs Commands
Events are facts: OrderPlaced, PaymentFailed. Commands are intentions: PlaceOrder, RefundPayment. Events are past tense; commands are imperative. A consumer of an event cannot reject it — it already happened. A handler of a command can reject it — it hasn't happened yet. Keep this distinction clear in your naming and your mental model.
07 — Finding Things

Search

Database LIKE '%query%' is not search. It's a full table scan that ignores relevance, can't handle typos, and collapses under any meaningful data volume. Real search requires an inverted index.

An inverted index maps each unique term to the list of documents containing it. To search for "wireless headphones", the engine looks up both terms in the index, finds the documents containing each, intersects the lists, and scores by relevance — all in milliseconds regardless of dataset size. This is the data structure powering Elasticsearch, Typesense, Meilisearch, and PostgreSQL's full-text search.

Relevance Scoring

TF-IDF (Term Frequency – Inverse Document Frequency) scores documents by how often a term appears in the document (TF) weighted against how rare the term is across all documents (IDF). "The" appearing 50 times scores lower than "acetylcholine" appearing once — because "the" is everywhere. BM25 is the modern improvement on TF-IDF (used by Elasticsearch by default) — it adds document length normalisation, preventing long documents from always winning.

Search Architecture

Search is always a secondary index — the canonical data lives in a primary database. A sync pipeline keeps the search index current: on write, publish a message to a queue; a consumer applies the delta to the search index. This eventual consistency (typically <1s lag) is acceptable for search. It is not acceptable to write directly to only the search engine — it is not a database of record.

TypeScript — Typesense: index product + multi-field search 
import Typesense from 'typesense'

const client = new Typesense.Client({
  nodes: [{ host: 'localhost', port: 8108, protocol: 'http' }],
  apiKey: process.env.TYPESENSE_KEY!
})

// Schema — define which fields are searchable + filterable
await client.collections().create({
  name: 'products',
  fields: [
    { name: 'name',       type: 'string'},
    { name: 'description', type: 'string'},
    { name: 'category',   type: 'string', facet: true },
    { name: 'price',       type: 'float',  facet: true },
    { name: 'in_stock',   type: 'bool',   facet: true }
  ],
  default_sorting_field: 'price'
})

// Search: typo-tolerant, facets, filtered, sorted
const results = await client
  .collections('products')
  .documents.search({
    q:             'wirelss headphons',  // typo-tolerant
    query_by:      'name,description',
    filter_by:     'in_stock:true && price:<=200',
    facet_by:      'category,price',
    sort_by:       '_text_match:desc,price:asc',
    per_page:      20
  })
Search is not a database
Never use Elasticsearch or Typesense as your primary data store. Use them as a derived index only. Write to your primary database first; sync to the search engine asynchronously. If the search index is lost or corrupted, reindex from the database. If the primary database is lost — you have a much worse problem than search.
08 — Durability & Read Scale

Replication

Replication copies data from one database node to one or more others. It serves two distinct goals that are often conflated: durability (survive hardware failure) and read scaling (distribute read load). The replication strategy for each differs significantly.

Replication Modes

Mode How it works Write latency Data loss on failure Use case
Synchronous Primary waits for replica acknowledgement before committing Higher (+RTT between nodes) Zero — replica is always current Financial data, anything where loss is unacceptable
Asynchronous Primary commits immediately; replica applies changes later Minimal Replication lag window (ms to seconds) Analytics replicas, reporting, global read replicas
Semi-synchronous Wait for at least one replica; others async Medium At most one node's lag Balance between durability and write performance

Replication Lag & Read-Your-Writes

Replication lag — the delay between a write on the primary and its appearance on replicas — causes the most subtle bugs in production. A user creates a resource; they are redirected to a page that reads from a replica; the resource appears missing. The user refreshes — it appears. This is replication lag at human timescales.

Solutions: read-your-writes consistency routes reads for a user to the primary for a short window after that user's write. Monotonic reads ensure a user never sees older data on a second read than on the first — implemented by tagging reads with a "read position" and routing to a replica at or ahead of that position.

TypeScript — Routing reads: primary vs replica by context 
class DbRouter {
  constructor(
    private primary: Pool,
    private replicas: Pool[]
  ) {}

  // Read-your-writes: use primary for N seconds after a write
  async read(userId: string, sql: string) {
    const wrote = await redis.get(`wrote:${userId}`)
    const pool  = wrote ? this.primary : this.nextReplica()
    return pool.query(sql)
  }

  async write(userId: string, sql: string) {
    const result = await this.primary.query(sql)
    // Mark this user as "just wrote" for 5 seconds
    await redis.setex(`wrote:${userId}`, 5, '1')
    return result
  }

  private nextReplica() {
    return this.replicas[Math.floor(Math.random() * this.replicas.length)]
  }
}
09 — Horizontal Data Scale

Sharding

Sharding splits data across multiple database nodes so no single node holds the entire dataset. It is the last resort for data scaling — resort to it only after vertical scaling, read replicas, and caching are exhausted.

Every sharding strategy must answer two questions: how is a row assigned to a shard? and how does a query find the right shard? The answers determine the trade-offs you inherit.

Sharding Strategies

Strategy Assignment rule Advantage Problem
Range Shard 1: user_id 1–1M, Shard 2: 1M–2M… Simple; range queries stay on one shard Hotspots — newest users all land on the last shard
Hash shard = hash(user_id) % N Even distribution; no hotspots Range queries scatter across all shards; resharding changes every key
Consistent Hash Keys and shards placed on a ring; key goes to next shard clockwise Adding/removing shards only remaps ~1/N keys More complex; requires virtual nodes for even distribution
Directory Lookup table maps entity → shard Fully flexible; can move entities between shards Lookup table is a bottleneck and single point of failure

Cross-shard queries are the defining cost of sharding. A query joining users across shards requires scatter-gather: send the query to all shards in parallel, merge results in the application layer. This is expensive and complex. Design your sharding key so that your most common queries live within a single shard. In a multi-tenant SaaS, sharding by tenant_id keeps all a tenant's data on one shard — all tenant-scoped queries are single-shard.

Shard before you need to
Retrofitting sharding onto an existing schema is extremely painful. If you know you will need to shard (because your data growth trajectory makes it inevitable), design the sharding key into your schema from day one — even if you start with a single shard. The column exists; the routing logic is a no-op. Adding shards later is an operational exercise, not a schema migration.
10 — Traffic Control

Rate Limiting

Rate limiting protects a system from being overwhelmed by any single client — malicious or otherwise. Without it, one runaway API consumer can degrade service for all others. With it, you guarantee minimum quality of service across the board.

Algorithms

Algorithm How it works Burst handling Best for
Token Bucket Bucket fills at rate R; each request consumes one token. Burst allowed up to bucket capacity. ✅ Allows bursts API rate limiting where short bursts are acceptable
Leaky Bucket Requests enter a queue; processed at fixed rate. Queue overflow = rejected. ❌ No bursts — smooths traffic Outbound rate limiting to protect downstream services
Fixed Window Count requests in current time window (e.g. 1 min). Reset at window boundary. ✅ Burst at window boundary Simple to implement; acceptable for coarse limits
Sliding Window Count requests in a rolling window. No boundary spike. ✅ Smooth Accurate per-user rate limits; higher implementation cost
TypeScript — Sliding window rate limiter with Redis 
// Sliding window: count requests in last 60 seconds using sorted set
async function isRateLimited(
  userId: string,
  limit: number = 100,   // max requests
  window: number = 60    // per N seconds
): Promise<{ limited: boolean; remaining: number }> {
  const now     = Date.now()
  const cutoff  = now - window * 1000
  const key     = `rl:${userId}`

  const [, count] = await redis
    .multi()
    .zremrangebyscore(key, 0, cutoff)        // remove old entries
    .zcard(key)                               // count remaining
    .zadd(key, now, `${now}-${Math.random()}`)  // record this request
    .expire(key, window)                      // auto-cleanup
    .exec()

  const c = count as number
  return { limited: c > limit, remaining: Math.max(0, limit - c) }
}

// Return rate limit headers so clients can self-throttle
const { limited, remaining } = await isRateLimited(userId)
res.setHeader('X-RateLimit-Remaining', remaining)
if (limited) return res.status(429).json({ error: 'Too Many Requests' })
11 — Coordination

Distributed Locks

A distributed lock prevents multiple processes from executing a critical section simultaneously — across a fleet of servers where in-process mutexes have no reach. It is the solution to the distributed read-modify-write problem.

Classic scenarios: only one server should send a scheduled report at 9am; only one worker should process a given order; only one node should run a database migration. Without a distributed lock, all of them try simultaneously — duplicates, data corruption, or worse.

Redis Redlock Algorithm

The Redlock algorithm acquires a lock by setting a key with SET key value NX PX ttl — set only if not exists, with a TTL so locks expire if the holder crashes. The TTL must be longer than the critical section takes to execute, but short enough that a crashed holder doesn't block for too long.

TypeScript — Distributed lock with fencing token 
import { randomUUID } from 'crypto'

async function withLock<T>(
  resource: string,
  ttlMs: number,
  fn: () => Promise<T>
): Promise<T> {
  const token = randomUUID()  // unique — only holder can release
  const key   = `lock:${resource}`

  // Acquire: SET NX PX — atomic, no race condition
  const acquired = await redis.set(key, token, 'PX', ttlMs, 'NX')
  if (!acquired) throw new Error(`Resource ${resource} is locked`)

  try {
    return await fn()
  } finally {
    // Release: only if WE still hold it (Lua script — atomic)
    await redis.eval(
      `if redis.call("get",KEYS[1])==ARGV[1]
         then return redis.call("del",KEYS[1])
         else return 0 end`,
      1, key, token
    )
  }
}

// Usage: safe scheduled job — only one instance runs
await withLock('daily-report', 60_000, async () => {
  await generateAndSendDailyReport()
})
Lock TTL must exceed critical section duration
If the critical section takes longer than the TTL, the lock expires while the holder is still running. Another process acquires the lock. Now two processes are in the critical section simultaneously — the failure mode you were trying to prevent. Set TTL to 5–10× the expected duration, and monitor for lock expiry events as an alert signal.
12 — Understanding Running Systems

Observability

A system you cannot observe is a system you cannot operate. Observability is the property that lets you answer "what is happening inside my system right now?" from its external outputs alone.

The three pillars — logs, metrics, and traces — are not redundant. Each answers a different question. Metrics tell you something is wrong. Logs tell you what happened. Traces tell you where in the request path it happened. You need all three.

Logs

Structured JSON logs are searchable; freeform text is not. Every log entry must include a correlation ID tying it to the originating request. Log at the right level — ERROR for failures requiring human attention, WARN for recoverable anomalies, INFO for significant business events, DEBUG for development only. Never log PII or secrets.

Metrics & the Four Golden Signals

Signal What it measures Alert threshold example
Latency How long requests take — track p50, p95, p99 (not average) p99 > 2s for 5 consecutive minutes
Traffic Requests per second — rate of demand on the system Traffic drops >30% vs 7-day baseline
Errors Fraction of requests resulting in errors 5xx error rate > 1% for 2 minutes
Saturation How full the system is — CPU, memory, queue depth, connection pool DB connection pool > 90% utilisation

Distributed Tracing

A trace represents the entire journey of one request through all services. Each service adds a span with its start time, duration, and any errors. The full trace shows you which service contributed which latency — making performance investigations in microservices tractable. OpenTelemetry is the standard instrumentation library; backends include Jaeger, Tempo, and Honeycomb.

TypeScript — OpenTelemetry trace + structured log with correlation ID 
import { trace, context } from '@opentelemetry/api'
import pino from 'pino'

const logger = pino({ level: 'info' })

function logWithTrace(level: string, msg: string, extra?: object) {
  const span    = trace.getActiveSpan()
  const spanCtx = span?.spanContext()

  logger[level]({
    ...extra,
    traceId: spanCtx?.traceId,   // ties log to Jaeger trace
    spanId:  spanCtx?.spanId,
  }, msg)
}

// In a request handler:
const tracer = trace.getTracer('api')

app.get('/orders/:id', async (req, res) => {
  await tracer.startActiveSpan('get-order', async (span) => {
    span.setAttributes({ 'order.id': req.params.id })
    try {
      const order = await fetchOrder(req.params.id)
      logWithTrace('info', 'Order fetched', { orderId: order.id })
      res.json(order)
    } catch (err) {
      span.recordException(err as Error)
      logWithTrace('error', 'Order fetch failed', { err })
      res.status(500).json({ error: 'Internal error' })
    } finally {
      span.end()
    }
  })
})
Synthesis — Putting It Together

How These Building Blocks Assemble

No building block operates in isolation. Understanding how they compose — and how the choice in one constrains the choices in others — is the final skill.

Client ↓ [ CDN ] → static assets, cached public pages (Caching Layer 1) ↓ [ Load Balancer ] → L7, weighted routing, health checks, canary ↓ [ Rate Limiter ] → sliding window per user/API key (Redis) ↓ [ API Servers ] → stateless, horizontally scaled ↓ ↓ ↓ [ App Cache ] [ Queue ] [ Search ] (Redis L2) (BullMQ/Kafka) (Typesense/ES) ↓ [ Primary DB ] → sharded by tenant_id, indexed, ACID ↓ [ Replica(s) ] → read scale, async replication, lag monitored ↓ [ Distributed Lock ] → Redis, guards scheduled jobs + critical sections ↓ [ Observability ] → OTel traces + Pino logs + Prometheus metrics

The assembly above is not a prescription — it is a starting point. A banking platform would promote Distributed Locks and Replication to first-class concerns and constrain Caching aggressively (stale balances are unacceptable). A social network would promote Caching and Sharding, and accept eventual consistency broadly. An e-commerce marketplace would promote Search and Queues. The building blocks are universal; the emphasis is domain-specific.

"A senior engineer's job is not to memorise architectures. It is to understand the trade-offs behind these building blocks and assemble them into a system that satisfies the unique requirements of the business domain."

The Decision Checklist

  • ⚖️Load Balancing: Are my servers stateless? Which algorithm matches my traffic pattern? Do my health checks validate real dependencies?
  • ⚖️Caching: What is the consistency requirement? Which layer is appropriate? How do I invalidate on write?
  • ⚖️Databases: What is the access pattern? Which consistency model is required? Am I fitting the data to the right store?
  • ⚖️Indexes: Have I read the execution plan? Am I indexing queries — not tables? Am I paying attention to write overhead?
  • ⚖️Queues: Do consumers need to be idempotent? What is my backpressure strategy? Is the DLQ monitored?
  • ⚖️Events: Am I using events for facts (past tense) and commands for intentions? Is my consumer decoupled from the producer?
  • ⚖️Search: Is the search index a secondary store, not a primary? Is the sync pipeline monitored for lag?
  • ⚖️Replication: Have I handled replication lag in my read paths? Are financial writes synchronous?
  • ⚖️Sharding: Is sharding actually needed? Is the shard key in the schema even before I need it? Can common queries stay single-shard?
  • ⚖️Rate Limiting: Am I returning standard 429 headers? Is the limiter distributed (Redis) not per-server?
  • ⚖️Distributed Locks: Is the TTL longer than the critical section? Is the release atomic (Lua script)?
  • ⚖️Observability: Can I answer "is the system healthy?" from dashboards alone? Are alerts on symptoms, not causes? Do traces span all services?
