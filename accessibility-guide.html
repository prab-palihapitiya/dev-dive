01 — The Foundation

Semantic HTML & the Accessibility Tree

Accessibility begins before a single line of JavaScript runs. Every HTML element you choose either helps or hinders the browser's ability to build an accurate Accessibility Tree — the parallel structure screen readers and other assistive technologies navigate instead of the visual DOM.

Modern browsers expose two parallel representations of your page: the render tree (pixels on screen) and the accessibility tree (semantic roles, names, and states). A <button> element automatically carries role="button", keyboard focusability, and an implicit label from its text content. A <div onclick> carries none of these — you must reconstruct every affordance by hand.

🏭

Landmarks

Elements like <main>, <nav>, <aside>, and <header> create jump points that screen reader users rely on to skip large content blocks.

📋

Heading Hierarchy

One <h1> per page. <h2>–<h6> must descend logically — never skip a level. Screen readers offer keyboard shortcuts to navigate headings like a table of contents.

⚓

Interactive Elements

Use <button> for actions and <a href> for navigation. The distinction matters: buttons trigger behaviour; links change location.

📑

Lists & Tables

Wrap related items in <ul>/<ol>. Use <th scope> and <caption> on data tables so screen readers announce structure, not raw cells.

❌ Div soup — no tree
// Nothing is conveyed to AT
function UserProfile() {
  return (
    <div className="card">
      <div className="title">Prabath</div>
      <div
        className="btn"
        onClick={save}
      >Save</div>
    </div>
  )
}
✓ Semantic — full tree
// Role, name, state — all implicit
function UserProfile() {
  return (
    <article className="card">
      <h2>Prabath</h2>
      <button
        type="button"
        onClick={save}
      >Save</button>
    </article>
  )
}
💡 Use React Fragments to avoid wrapper divs
When a component needs to return multiple elements but a wrapping <div> would break the document structure (e.g. inside a <ul>), reach for <>...</> or <React.Fragment>. No DOM node is added, no accessibility tree is polluted.
02 — SPA Challenge

Dynamic Focus Management

In a traditional multi-page website, every navigation triggers a full reload and the browser resets focus to the document top. SPAs silently swap content without reloading — which means focus silently gets lost, trapped on deleted elements, or left marooned in a now-invisible component.

Route Change Focus Reset

When a user navigates to a new view, manually shift focus to the primary heading or page wrapper of the incoming route. Pair this with an aria-live announcement so screen readers know a navigation occurred.

JSX — Route focus hook 
import { useEffect, useRef } from 'react'
import { useLocation } from 'react-router-dom'

// ── Drop this at the root of your router outlet ──
function RouteAnnouncer() {
  const headingRef = useRef(null)
  const location = useLocation()

  useEffect(() => {
    // Small delay lets React finish painting the new route
    const id = setTimeout(() => {
      headingRef.current?.focus()
    }, 50)
    return () => clearTimeout(id)
  }, [location.pathname])

  return (
    <h1
      ref={headingRef}
      // tabIndex={-1}: programmatically focusable, not in tab order
      tabIndex={-1}
      style={{ outline: 'none' }}
    >
      {getPageTitle(location.pathname)}
    </h1>
  )
}

Modal Focus Trapping

When a modal or dialog is open, keyboard focus must be contained inside it. A user tabbing to the last focusable element should cycle back to the first — never escape into the inert background page.

JSX — Accessible modal with focus trap 
import { useEffect, useRef } from 'react'

const FOCUSABLE = [
  'a[href]', 'button:not([disabled])',
  'input:not([disabled])', 'select',
  'textarea', '[tabindex]:not([tabindex="-1"])'
].join(',')

function Modal({ title, onClose, children }) {
  const dialogRef = useRef(null)

  useEffect(() => {
    const el = dialogRef.current
    const focusable = [...el.querySelectorAll(FOCUSABLE)]
    const first = focusable[0]
    const last  = focusable[focusable.length - 1]

    first?.focus()  // move focus into modal on open

    const handleKeyDown = (e) => {
      if (e.key === 'Escape') { onClose(); return }
      if (e.key !== 'Tab') return

      if (e.shiftKey && document.activeElement === first) {
        e.preventDefault(); last?.focus()   // wrap to last
      } else if (!e.shiftKey && document.activeElement === last) {
        e.preventDefault(); first?.focus()  // wrap to first
      }
    }
    document.addEventListener('keydown', handleKeyDown)
    return () => document.removeEventListener('keydown', handleKeyDown)
  }, [])

  return (
    <div role="dialog" aria-modal="true" aria-labelledby="modal-title" ref={dialogRef}}>
      <h2 id="modal-title">{title}</h2>
      {children}
      <button type="button" onClick={onClose}>Close</button>
    </div>
  )
}
📚 Production Recommendation
For production code, reach for react-focus-lock or focus-trap-react instead of hand-rolling focus trap logic. These libraries handle edge cases like dynamically added focusable elements, portals, and shadow DOM that manual implementations routinely miss.
03 — The Vocabulary

ARIA Roles, Labels & States

ARIA (Accessible Rich Internet Applications) is a set of HTML attributes that supplement — never replace — semantic markup. The first rule of ARIA: if a native HTML element already communicates the right semantics, use it instead of adding ARIA.

ARIA has three categories of attributes. Roles define what a widget is (role="tab", role="combobox"). Properties describe relationships (aria-labelledby, aria-controls). States reflect dynamic conditions (aria-expanded, aria-selected, aria-busy).

AttributePurposeExample
aria-label Provides an accessible name when visible text isn't present Icon-only close button
aria-labelledby Points to another element's text as the accessible name Dialog labelled by its <h2>
aria-describedby Links supplementary descriptive text to an element Input linked to its validation error
aria-expanded Indicates whether a collapsible section is open or closed Accordion, dropdown menu
aria-hidden Removes an element from the accessibility tree entirely Decorative icons, duplicate text
aria-live Marks a region whose content updates should be announced Toast notifications, search results
aria-busy Signals a region is being updated; suppress announcements until false Loading skeleton containers

Custom Accordion Component

JSX — Accessible accordion 
import { useState } from 'react'

function Accordion({ items }) {
  const [openIndex, setOpenIndex] = useState(null)

  return (
    <div role="region" aria-label="FAQ">
      {items.map((item, i) => {
        const isOpen = openIndex === i
        const panelId = `panel-${i}`
        const headingId = `heading-${i}`

        return (
          <div key={i}>
            <h3>
              <button
                id={headingId}
                aria-expanded={isOpen}
                aria-controls={panelId}
                onClick={() => setOpenIndex(isOpen ? null : i)}
              >
                {item.question}
                <span aria-hidden="true">{isOpen ? '▲' : '▼'}</span>
              </button>
            </h3>
            <div
              id={panelId}
              role="region"
              aria-labelledby={headingId}
              hidden={!isOpen}
            >
              {item.answer}
            </div>
          </div>
        )
      })}
    </div>
  )
}
⚠ ARIA Anti-patterns to Avoid
Never apply aria-hidden="true" to a focused element or an element that contains focusable children — keyboard users will land on invisible elements. Also avoid nesting interactive roles (role="button" inside role="link"); screen readers expose only the outermost role in many implementations.
04 — User Input

Forms & Accessible Error Handling

Form fields are where most a11y failures live in production. The problems are predictable: unlabelled inputs, validation errors that appear visually but are never announced, and required-field conventions that exist only as red asterisks sighted users learn to recognise.

JSX — Fully accessible form field 
function FormField({ id, label, error, hint, required, ...props }) {
  const errorId = `${id}-error`
  const hintId  = `${id}-hint`

  // Build described-by list only for present elements
  const describedBy = [
    hint  && hintId,
    error && errorId,
  ].filter(Boolean).join(' ') || undefined

  return (
    <div className="field">
      <label htmlFor={id}>
        {label}
        {required && (
          <span
            aria-hidden="true"
            className="required-star"
          > *</span>  // visual only
        )}
      </label>

      {hint && <p id={hintId} className="field-hint">{hint}</p>}

      <input
        id={id}
        required={required}
        aria-required={required}
        aria-invalid={error ? 'true' : undefined}
        aria-describedby={describedBy}
        {...props}
      />

      {error && (
        <p
          id={errorId}
          role="alert"       // assertive live region — announces immediately
          className="field-error"
        >
          {error}
        </p>
      )}
    </div>
  )
}

Form Validation Summary

When a form is submitted with multiple errors, never rely on inline error messages alone. Render a validation summary at the top of the form, move focus to it, and list each error as a link that jumps to the offending field.

JSX — Validation summary with focus 
function ValidationSummary({ errors }) {
  const ref = useRef(null)

  useEffect(() => {
    if (errors.length) ref.current?.focus()
  }, [errors.length])

  if (!errors.length) return null

  return (
    <div
      ref={ref}
      tabIndex={-1}
      role="alert"
      aria-label="Form submission failed"
    >
      <h2>Please fix the following errors</h2>
      <ul>
        {errors.map((e) => (
          <li key={e.field}>
            <a href={`#${e.field}`}>{e.message}</a>
          </li>
        ))}
      </ul>
    </div>
  )
}
05 — Input Independence

Keyboard Navigation Patterns

WCAG 2.1 Success Criterion 2.1.1 mandates that all functionality is operable through a keyboard alone. This doesn't just mean "don't block Tab key" — it means complex widgets like tab panels, tree views, and data grids must implement standardised keyboard interaction models defined by the ARIA Authoring Practices Guide.

The Roving tabIndex Pattern

For composite widgets like tab lists, radio groups, or toolbars, using a native tabIndex on every child produces a poor UX: keyboard users must tab through every item. The roving tabIndex pattern keeps only the active item in the tab order (tabIndex={0}) and sets all others to tabIndex={-1}, using arrow keys to move within the group.

JSX — Tab list with roving tabIndex 
import { useState, useRef } from 'react'

function TabPanel({ tabs }) {
  const [activeTab, setActiveTab] = useState(0)
  const tabRefs = useRef([])

  const handleKeyDown = (e, index) => {
    let next = index

    if (e.key === 'ArrowRight')
      next = (index + 1) % tabs.length
    else if (e.key === 'ArrowLeft')
      next = (index - 1 + tabs.length) % tabs.length
    else if (e.key === 'Home') next = 0
    else if (e.key === 'End')  next = tabs.length - 1
    else return

    e.preventDefault()
    setActiveTab(next)
    tabRefs.current[next]?.focus()  // move DOM focus with state
  }

  return (
    <>
      <div role="tablist">
        {tabs.map((tab, i) => (
          <button
            key={i}
            role="tab"
            ref={el => tabRefs.current[i] = el}
            aria-selected={activeTab === i}
            aria-controls={`panel-${i}`}
            tabIndex={activeTab === i ? 0 : -1}
            onClick={() => setActiveTab(i)}
            onKeyDown={e => handleKeyDown(e, i)}
          >
            {tab.label}
          </button>
        ))}
      </div>
      {tabs.map((tab, i) => (
        <div
          key={i}
          id={`panel-${i}`}
          role="tabpanel"
          aria-labelledby={`tab-${i}`}
          hidden={activeTab !== i}
        >
          {tab.content}
        </div>
      ))}
    </>
  )
}
💡 Skip Navigation Links
Add a visually-hidden "Skip to main content" link as the very first focusable element on every page. Keyboard users and screen reader users can then bypass repeated nav blocks without tabbing through 20 links. Make it visible on focus with :focus { clip: auto; position: static }.
06 — Dynamic Announcements

Live Regions & Async Updates

When your React app modifies the DOM in response to a user action — a toast pops up, search results reload, a cart updates — sighted users perceive the change visually. Screen readers do not automatically know the page changed unless you declare an ARIA live region.

JSX — Announcement manager hook 
import { useState, useCallback } from 'react'

// ── A single global announcer mounted once in your app root ──
function Announcer({ message, priority = 'polite' }) {
  return (
    <div
      aria-live={priority}
      aria-atomic="true"
      style={{
        /* Visually hidden but present in accessibility tree */
        position: 'absolute',
        width: '1px',
        height: '1px',
        overflow: 'hidden',
        clip: 'rect(0,0,0,0)',
        whiteSpace: 'nowrap',
      }}
    >
      {message}
    </div>
  )
}

// ── Hook for any component to trigger announcements ──
function useAnnounce() {
  const [msg, setMsg] = useState('')

  const announce = useCallback((text) => {
    setMsg('')  // reset first — same string won't re-announce
    requestAnimationFrame(() => setMsg(text))
  }, [])

  return { announce, Announcer: () => <Announcer message={msg} /> }
}

// Usage
function SearchBox() {
  const { announce, Announcer } = useAnnounce()

  const handleSearch = async (query) => {
    const results = await fetchResults(query)
    announce(`${results.length} results found for ${query}`)
  }

  return (
    <>
      <Announcer />  // invisible, speaks to screen readers
      <input type="search" onChange={e => handleSearch(e.target.value)} />
    </>
  )
}
ValueInterrupts?Best For
aria-live="polite" No — waits for idle Success messages, search result counts, cart updates
aria-live="assertive" Yes — interrupts immediately Critical errors, session timeouts, payment failures
role="status" No (implies polite) Persistent status bars, loading states
role="alert" Yes (implies assertive) Error messages, inline validation failures
⚠ Use assertive sparingly
aria-live="assertive" stops the screen reader mid-sentence to announce your message. Used carelessly on frequent updates (every keystroke, every poll result), it becomes an unusable torrent of interruptions. Reserve it for genuinely critical, time-sensitive information.
07 — Visual Design

Colour Contrast & Reduced Motion

WCAG 2.1 AA requires a contrast ratio of at least 4.5:1 for normal text and 3:1 for large text and UI component boundaries. WCAG AAA raises this to 7:1 and 4.5:1 respectively. Contrast failures are among the most common a11y violations in production because they're invisible to sighted developers with typical vision.

CSS — Contrast-safe design tokens + reduced motion 
/* ── Colour tokens with documented contrast ratios ── */
:root {
  /* Primary text on white: 16.1:1  ✅ AAA */
  --text-primary: #1a1a1a;

  /* Secondary text on white: 5.9:1  ✅ AA */
  --text-secondary: #595959;

  /* ⚠️  Muted text on white: 2.8:1  ❌ FAIL */
  /* --text-muted: #9a9a9a; — never use for meaningful text */

  /* Brand accent on white: 4.6:1  ✅ AA */
  --accent: #7c3aed;

  /* Error red on white: 4.9:1  ✅ AA */
  --error: #b91c1c;
}

/* ── Respect user's motion preference ── */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration:   0.01ms !important;
    animation-iteration-count: 1     !important;
    transition-duration:  0.01ms !important;
  }
}

/* ── Always show a visible focus ring ── */
:focus-visible {
  outline: 3px solid var(--accent);
  outline-offset: 2px;
  border-radius: 3px;
}
/* Remove outline only for pointer users, not keyboard */
:focus:not(:focus-visible) {
  outline: none;
}
"Removing focus rings to make an interface look cleaner is one of the most common accessibility failures we encounter in audits. If you can't see where focus is, you can't use the keyboard."
🔍 Contrast Checking Tools
Colour Contrast Analyser (TPGi) is the industry-standard desktop tool. For browser-based workflows, Chrome DevTools now shows contrast ratios inline in the colour picker. The axe DevTools browser extension will flag failing elements automatically during development.
08 — Automated Guardrails

Testing & Automated Tooling

Automated tools can catch roughly 30–40% of WCAG failures — the mechanical ones: missing alt text, empty form labels, insufficient contrast. The remaining 60% requires manual keyboard testing and screen reader verification. A good a11y workflow stacks both.

A. Linting — eslint-plugin-jsx-a11y

This plugin statically analyses your JSX and warns in your editor before code is even saved. It catches missing alt attributes, click handlers on non-interactive elements, and disconnected form labels.

.eslintrc.json 
{
  "plugins": ["jsx-a11y"],
  "extends": [
    "plugin:jsx-a11y/recommended"
  ],
  "rules": {
    // Upgrade click-on-div to an error (default is warn)
    "jsx-a11y/click-events-have-key-events": "error",
    "jsx-a11y/no-static-element-interactions": "error"
  }
}

B. Runtime Audits — @axe-core/react

Axe analyses the rendered DOM and logs violations to your browser console during development, surfacing issues the linter can't catch (like contrast ratios and ARIA relationship correctness).

JavaScript — main.jsx / index.js 
import React from 'react'
import ReactDOM from 'react-dom'

if (process.env.NODE_ENV !== 'production') {
  import('@axe-core/react').then((axe) => {
    axe.default(React, ReactDOM, 1000)
    // Violations printed to DevTools console with WCAG references
  })
}

C. Component Testing — jest-axe

JavaScript — Accessibility unit test 
import { render } from '@testing-library/react'
import { axe, toHaveNoViolations } from 'jest-axe'

expect.extend(toHaveNoViolations)

test('FormField has no axe violations', async () => {
  const { container } = render(
    <FormField
      id="email"
      label="Email address"
      required
    />
  )
  const results = await axe(container)
  expect(results).toHaveNoViolations()
})

Implementation Checklist

  • ✅ Every interactive element is a <button> or <a href>. Custom widgets have the correct role, tabIndex={0}, and keyboard handlers.
  • ✅ All images have alt text. Decorative images use alt="". Icon-only buttons have aria-label.
  • ✅ Every form input has a visible, programmatically associated <label> via htmlFor. Placeholder text is never the sole label.
  • ✅ Validation errors are linked to their inputs via aria-describedby and aria-invalid. A validation summary receives focus on submit.
  • ✅ Route changes and modal opens shift focus programmatically. Modals trap focus. Closing a modal returns focus to the trigger element.
  • ✅ All live DOM changes (toasts, loading states, search results) are announced via aria-live or role="alert".
  • ✅ Text contrast meets WCAG AA (4.5:1). UI component boundaries meet 3:1. Contrast is verified with a dedicated tool, not eyeballed.
  • ✅ A visible focus ring is always shown on keyboard focus via :focus-visible. The ring is never suppressed globally.
  • ✅ Animations respect prefers-reduced-motion. All transitions disabled or slowed to near-zero for users who opt out.
  • ✅ eslint-plugin-jsx-a11y runs in CI. jest-axe tests cover critical components. Manual keyboard walk-through on every new page/flow.
