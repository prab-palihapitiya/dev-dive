01 — The Foundation

Why Databases Exist

Every database exists to solve one fundamental problem: application state must outlive a single process. But survival is just the beginning — concurrency, consistency, and query performance are what separate a database from a file.

Before databases, applications stored state in flat files. The problems that emerged with any non-trivial scale were predictable and severe.

⚡

Concurrent Writes

Two processes write to the same file simultaneously. One overwrites the other. Data is lost with no error.

💥

Data Corruption

A crash mid-write leaves a half-written file. The application starts in an inconsistent state with no recovery path.

🐢

Slow Searches

Finding a user in a 10M-row JSON file requires reading every line. O(n) for every lookup, always.

🔗

No Relationships

Linking an order to a user means duplicating user data in every order record, or implementing your own join logic.

🔒

No Transactions

A bank transfer deducts money from Account A. The process crashes before crediting Account B. Money vanishes.

🔑

No Security Model

File-level permissions are all-or-nothing. You cannot grant one user read access to some columns but not others.

A Database Management System (DBMS) abstracts all of this behind a structured interface — typically SQL. The internal architecture that makes this possible has layers of responsibility:

Application
    │ Issues SQL statements
SQL Parser & Analyser
    │ Validates syntax, resolves table/column names
Query Optimizer
    │ Generates the cheapest execution plan
Execution Engine
    │ Runs the plan: scans, joins, aggregates
Storage Engine
    │ Manages pages, buffer pool, WAL
Disk / SSD
Senior Mindset
Every layer in that stack is a trade-off surface. The query optimizer can make the wrong choice; the storage engine has its own flush policies; the execution engine handles joins differently per database. Understanding the stack — not just SQL — is what makes you dangerous with databases.
02 — Core Concepts

Relational Database Fundamentals

The relational model, invented by Edgar F. Codd in 1970, is built on one elegant idea: all data is stored in relations (tables), and all queries are expressed as operations on those relations. Fifty years later, it remains the dominant data model.

Keys and Relationships

Every table needs a way to uniquely identify each row, and a way to reference rows in other tables. These two concepts — primary keys and foreign keys — are the skeleton of every relational schema.

SQL — Key Types 
-- Primary key: unique identifier per row
CREATE TABLE users (
  id        BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  email     TEXT    NOT NULL UNIQUE,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Foreign key: references primary key in another table
CREATE TABLE orders (
  id      BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  user_id BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  total   NUMERIC(12,2) NOT NULL,
  status  TEXT    DEFAULT 'pending'
);

-- Composite primary key (junction table)
CREATE TABLE order_items (
  order_id   BIGINT REFERENCES orders(id),
  product_id BIGINT REFERENCES products(id),
  quantity   INT    NOT NULL,
  unit_price NUMERIC(10,2) NOT NULL,
  PRIMARY KEY (order_id, product_id)
);

Normalization

Normalization is the process of structuring tables to eliminate redundancy and prevent update anomalies. The goal is not maximum normalization — it is appropriate normalization. Over-normalized schemas can require excessive joins; under-normalized schemas duplicate data and create consistency problems.

Normal Form Rule Violation Example
1NF Each column holds atomic (indivisible) values; no repeating groups Storing tag1,tag2,tag3 in a single tags column
2NF No partial dependency — non-key columns depend on the whole primary key In an order_items table keyed by (order_id, product_id), storing product_name that depends only on product_id
3NF No transitive dependency — non-key columns depend directly on the primary key, not on another non-key column Storing zip_code and city in the same row, where city depends on zip_code, not on the user
BCNF Every determinant is a candidate key — handles edge cases 3NF misses when there are multiple composite keys Rare in practice; appears in teaching schemas with overlapping candidate keys
Denormalization is a deliberate choice
High-read systems often intentionally denormalize — storing computed or duplicated data to avoid expensive joins at query time. This is a valid production pattern as long as you own the consistency implications and have a clear update strategy.

SQL Mastery — Beyond CRUD

CRUD (INSERT, SELECT, UPDATE, DELETE) is the entry level. Senior engineers reach for the full SQL toolkit: window functions, CTEs, lateral joins, and recursive queries.

SQL — Window Functions, CTEs, Lateral Join 
-- Window function: rank orders per user by amount
SELECT
  user_id,
  id          AS order_id,
  total,
  RANK() OVER (PARTITION BY user_id ORDER BY total DESC) AS spend_rank
FROM orders;

-- CTE: reusable subquery with a name
WITH monthly_revenue AS (
  SELECT
    DATE_TRUNC('month', created_at) AS month,
    SUM(total)                           AS revenue
  FROM orders
  GROUP BY 1
)
SELECT month, revenue,
  revenue - LAG(revenue) OVER (ORDER BY month) AS mom_delta
FROM monthly_revenue;

-- Lateral join: per-row subquery (PostgreSQL)
SELECT u.id, u.email, recent.total
FROM users u
CROSS JOIN LATERAL (
  SELECT total FROM orders
  WHERE user_id = u.id
  ORDER BY created_at DESC
  LIMIT 1
) recent;
03 — Performance

Query Optimization & Indexing

A query optimizer's job is to find the cheapest execution plan for a SQL statement. "Cheapest" means fewest disk reads, fewest CPU operations, fewest memory allocations. The optimizer does not always win — your job is to understand when it fails and why.

Index Types

Not all indexes are the same data structure or serve the same purpose. Choosing the wrong index type is one of the most common performance mistakes in production systems.

Index Type Data Structure Best For Notes
B-Tree Balanced tree Equality, range, ORDER BY, LIKE prefix Default index type. O(log n) lookup. Works for most cases.
Hash Hash table Equality lookups only (=) Faster than B-Tree for exact matches; useless for ranges. PostgreSQL only persists these since v10.
GIN Inverted index Full-text search, JSONB, arrays, tsvector Good for contains/overlaps queries. Slower writes due to posting lists.
GiST Generalized search tree Geometric types, full-text (PostgreSQL) Extensible; used by PostGIS for spatial queries.
Partial B-Tree on a subset Filtered data (e.g., only WHERE status = 'active') Much smaller index, faster writes. Ideal when only a fraction of rows is queried.
Covering B-Tree with included columns Index-only scans — avoid heap fetch entirely PostgreSQL: INCLUDE (col1, col2) in CREATE INDEX.
SQL — Practical Index Patterns 
-- Composite index: column order matters
-- (user_id, created_at) supports:  WHERE user_id = ?
--                                   WHERE user_id = ? AND created_at > ?
-- But NOT:                          WHERE created_at > ? (leading column missing)
CREATE INDEX idx_orders_user_date
  ON orders (user_id, created_at DESC);

-- Partial index: only index rows where work happens
CREATE INDEX idx_orders_pending
  ON orders (created_at)
  WHERE status = 'pending';

-- Covering index: include columns to avoid heap fetch
CREATE INDEX idx_orders_covering
  ON orders (user_id)
  INCLUDE (total, status, created_at);

-- Expression index: index a computed value
CREATE INDEX idx_users_lower_email
  ON users (LOWER(email));
-- Now this query hits the index:
-- SELECT * FROM users WHERE LOWER(email) = 'john@example.com';

Reading EXPLAIN ANALYZE

The single most important skill for database performance work is reading query plans. EXPLAIN ANALYZE runs the query and shows you exactly what happened — not what the optimizer predicted, but what actually occurred.

SQL — EXPLAIN ANALYZE Output 
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT o.id, o.total, u.email
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.created_at > now() - INTERVAL '7 days';

-- Sample output (key lines to read):
-- Hash Join  (cost=1234..5678 rows=3200 width=48)
--            (actual time=12.4..89.2 rows=3105 loops=1)
--   Buffers: shared hit=820 read=142
--   -->  Index Scan using idx_orders_user_date on orders
--         (actual time=0.04..45.1 rows=3105)
--   -->  Hash  (cost=410..410 rows=10200)
--         Buckets: 16384  Batches: 1  Memory Usage: 880kB
🎯

cost=X..Y

Planner estimate: startup cost..total cost. High startup cost is normal for sorts and hashes. Compare to actual time.

⚠️

rows= estimate

If estimated rows differ wildly from actual, run ANALYZE to update table statistics. Stale stats cause bad plans.

💾

Buffers: read=N

Disk reads. A high "read" vs "hit" ratio means your working set doesn't fit in shared_buffers — memory needs tuning.

🔄

loops=N

Nested loop: inner side executed N times. If N is large and the inner side does index scans, it multiplies the cost.

Three Questions for Every EXPLAIN
Is my index actually being used? (Look for "Index Scan" vs "Seq Scan".) Are the row estimates accurate? (Compare "rows=estimate" to "rows=actual".) Where is the most time being spent? (Look for the node with the highest "actual time".)
04 — Correctness

Transactions & ACID

A transaction is the fundamental unit of correctness in a relational database. It groups multiple operations into an all-or-nothing unit. ACID is not marketing — it is a precise specification of the guarantees a database must provide for transactions to be trustworthy.

SQL — Bank Transfer Transaction 
BEGIN;

-- Deduct from sender
UPDATE accounts
  SET balance = balance - 500.00
  WHERE id = 42 AND balance >= 500.00;

-- Verify the deduction actually happened
DO $$ BEGIN
  IF NOT FOUND THEN
    RAISE EXCEPTION 'Insufficient funds or account not found';
  END IF;
END $$;

-- Credit the receiver
UPDATE accounts
  SET balance = balance + 500.00
  WHERE id = 99;

-- Persist both changes atomically
COMMIT;

-- If anything above fails, the DB rolls back both operations
⚛️

Atomicity

All operations in a transaction succeed, or none do. A crash mid-transaction leaves no partial state on disk.

✅

Consistency

A transaction moves the database from one valid state to another. Constraints, foreign keys, and triggers are enforced throughout.

🏝️

Isolation

Concurrent transactions do not see each other's intermediate state. Isolation level determines how strictly this is enforced.

🪨

Durability

Once a transaction is committed, it survives crashes. This is achieved through the Write-Ahead Log (WAL) — changes are written to the log before being applied to data pages.

The Write-Ahead Log (WAL)

Durability is not magic — it is the WAL. Before any data page is modified, the change is appended to a sequential log on disk. If the database crashes and restarts, it replays the WAL to recover committed transactions and discard uncommitted ones. Sequential writes to the WAL are fast; random writes to data pages happen lazily in the background.

WAL is also replication
PostgreSQL streaming replication works by shipping WAL records from the primary to replicas. The replica replays the same log, keeping itself in sync. Logical replication uses a decoded form of the WAL that supports cross-version and cross-database replication.
05 — Concurrency Control

Isolation Levels

Isolation sounds binary — transactions are either isolated or they aren't. In reality, isolation is a spectrum. Higher isolation costs more (locks, version chains, aborts). The SQL standard defines four levels, each permitting or preventing specific anomalies.

Isolation Level Dirty Read Non-Repeatable Read Phantom Read Default In
Read Uncommitted ✅ Possible ✅ Possible ✅ Possible —
Read Committed ❌ Prevented ✅ Possible ✅ Possible PostgreSQL, Oracle
Repeatable Read ❌ Prevented ❌ Prevented ✅ Possible* MySQL InnoDB
Serializable ❌ Prevented ❌ Prevented ❌ Prevented Explicit only

* PostgreSQL's Repeatable Read actually prevents phantom reads too, making it stricter than the SQL standard requires.

The Three Anomalies Explained

👻

Dirty Read

Transaction A reads data written by Transaction B before B commits. If B rolls back, A read data that never existed.

🔄

Non-Repeatable Read

Transaction A reads a row. Transaction B updates and commits it. A reads the same row again and gets a different value.

👥

Phantom Read

Transaction A queries a set of rows. Transaction B inserts a new matching row and commits. A re-runs the query and sees an extra "phantom" row.

MVCC — How PostgreSQL Implements Isolation

PostgreSQL uses Multi-Version Concurrency Control (MVCC). Instead of locking rows for reads, it keeps multiple versions of each row. A transaction sees a snapshot of data as it existed at transaction start. Readers never block writers, and writers never block readers — a critical advantage over lock-based systems.

Row (id=1): balance = 1000

Transaction A starts → sees balance = 1000 (snapshot)
Transaction B updates → creates new row version: balance = 800
Transaction B commits
Transaction A re-reads → still sees balance = 1000 (its snapshot)

Old row version is later vacuumed (cleaned up) by autovacuum
VACUUM is not optional
MVCC creates dead row versions (tuples) that accumulate over time. PostgreSQL's autovacuum reclaims this space. Disabling or neglecting autovacuum on high-write tables causes table bloat and eventual transaction ID wraparound — a production emergency that requires downtime to recover from.
06 — MySQL Deep Dive

MySQL Architecture & InnoDB

MySQL became the world's most deployed open-source database by being exactly what most applications needed: fast, simple, and easy to operate. Understanding its architecture reveals why it excels at what it does — and where it hits walls.

Client Connection (mysql protocol / pooler)
    │
SQL Layer — Parser → Optimizer → Executor
    │ Pluggable storage engine interface
InnoDB Storage Engine
    ├── Buffer Pool (data pages in RAM)
    ├── Change Buffer (deferred secondary index writes)
    ├── Redo Log (WAL equivalent — crash recovery)
    └── Undo Log (MVCC + ROLLBACK support)
    │
Disk / SSD (.ibd files, one per table with innodb_file_per_table)

InnoDB Internals Worth Knowing

📄

Clustered Index

InnoDB stores the table data in the primary key's B-Tree leaf nodes. There is no separate heap file — the PK IS the table. Secondary indexes store the primary key value as their row pointer.

💡

Buffer Pool

InnoDB's most critical tuning parameter. Set innodb_buffer_pool_size to 70-80% of available RAM for dedicated database servers. Cache hits eliminate disk I/O entirely.

🔒

Row-Level Locking

InnoDB locks individual rows (not pages, not tables) for writes under Read Committed and Repeatable Read. This enables high write concurrency.

🔑

Gap Locks

Under Repeatable Read, InnoDB uses gap locks to prevent phantoms. A gap lock on the range (10, 20) prevents any INSERT of a value in that range by concurrent transactions.

MySQL — Key Configuration (my.cnf) 
[mysqld]
# Memory — most impactful single setting
innodb_buffer_pool_size         = 4G
innodb_buffer_pool_instances    = 4    # 1 per GB of buffer pool

# Durability vs performance trade-off
# 1 = fully durable (default), 2 = flush once/sec (faster, tiny crash risk)
innodb_flush_log_at_trx_commit  = 1

# Redo log — larger = fewer checkpoints = better write throughput
innodb_redo_log_capacity        = 2G   # MySQL 8.0.30+

# Connections
max_connections                 = 500
thread_cache_size               = 50

# Query cache is removed in MySQL 8 — use ProxySQL or app-level caching
slow_query_log                  = 1
long_query_time                 = 1    # Log queries slower than 1s

When MySQL Wins

MySQL's sweet spot is high-throughput simple operations: single-table reads, CRUD applications, LAMP-stack workloads. It powers WordPress, Shopify's core transactional layer, and most CMS platforms for good reason — its optimizer is well-tuned for these patterns and its operational simplicity is a genuine advantage at small team scale.

Use BIGINT for primary keys, not INT
INT maxes out at ~2.1 billion. A busy e-commerce site can exhaust INT primary keys within years. BIGINT costs 4 extra bytes per row but buys you 9.2 × 10¹⁸ values. Always use BIGINT UNSIGNED or equivalent from day one.
07 — PostgreSQL Deep Dive

PostgreSQL — The Advanced Open-Source Database

PostgreSQL is often described as "the most advanced open-source relational database." That is not marketing. It offers features that rival (and sometimes exceed) Oracle and SQL Server: native JSONB, full-text search, custom types, table inheritance, logical replication, and one of the most sophisticated query planners in existence.

Client (libpq, pgBouncer, pg driver)
    │ One backend process per connection
Postmaster — forks backend processes
    │
Backend Process
    ├── Parser → Analyser → Rewriter → Planner → Executor
    └── Accesses shared memory (shared_buffers, WAL buffers)
    │
Storage
    ├── Heap Files (table data — unordered, no clustered PK)
    ├── Index Files (B-Tree, GIN, GiST, BRIN, Hash)
    ├── WAL (pg_wal directory — sequential write log)
    └── pg_toast (overflow storage for large column values)

PostgreSQL's Unique Strengths

📦

JSONB

Binary JSON stored natively. Supports GIN indexing, JSONPath queries, and hybrid relational-document modelling without a separate NoSQL store.

🔎

Full-Text Search

Built-in tsvector / tsquery with GIN indexes. Good enough for most apps — often eliminates the need for a separate search engine at smaller scale.

🧩

Extensibility

Custom data types, operators, index types (via GiST), and languages (PL/pgSQL, PL/Python, PL/v8). PostGIS adds full GIS capabilities without leaving SQL.

🪟

Window Functions

First-class SQL window functions: RANK, LAG, LEAD, FIRST_VALUE, NTILE, ROWS BETWEEN. Essential for analytics without a data warehouse.

📋

CTEs & Recursion

Recursive CTEs enable graph traversal (org charts, bill-of-materials, social graphs) in pure SQL. No application-level graph iteration needed.

🌀

Materialized Views

Persist expensive query results on disk. Refresh manually or on a schedule. Dramatically accelerates dashboards and reporting without denormalizing the base tables.

PostgreSQL — JSONB, Full-Text Search, Recursive CTE 
-- JSONB: store and query structured config per tenant
CREATE TABLE tenants (
  id      BIGINT PRIMARY KEY,
  config  JSONB
);
CREATE INDEX idx_tenants_config ON tenants USING GIN (config);

-- Query a nested JSONB field with index support
SELECT id FROM tenants
WHERE config @> '{"plan": "enterprise", "features": ["sso"]}';

-- Full-text search with ranking
SELECT id, title,
  ts_rank(search_vector, query) AS rank
FROM articles,
  to_tsquery('english', 'database & performance') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 20;

-- Recursive CTE: traverse org hierarchy
WITH RECURSIVE org_tree AS (
  -- Base: top-level employees
  SELECT id, name, manager_id, 0 AS depth
  FROM employees WHERE manager_id IS NULL

  UNION ALL

  -- Recursive: direct reports of already-found employees
  SELECT e.id, e.name, e.manager_id, ot.depth + 1
  FROM employees e
  JOIN org_tree ot ON e.manager_id = ot.id
)
SELECT REPEAT('  ', depth) || name AS hierarchy
FROM org_tree ORDER BY depth, name;

Key PostgreSQL Configuration

postgresql.conf — Production Tuning 
# Memory
shared_buffers            = 4GB    # ~25% of RAM
work_mem                  = 64MB   # per sort/hash per query — multiply by connections!
maintenance_work_mem      = 512MB  # for VACUUM, CREATE INDEX
effective_cache_size      = 12GB   # tell planner how much OS cache exists

# WAL / Checkpoints
wal_buffers               = 64MB
checkpoint_completion_target = 0.9
max_wal_size              = 4GB

# Planner cost model — tune if using SSDs
random_page_cost          = 1.1    # default 4.0 is for spinning disks
effective_io_concurrency  = 200   # SSD: 200; spinning: 2

# Autovacuum — don't disable this
autovacuum                = on
autovacuum_vacuum_cost_delay = 2ms # reduce IO throttling on SSDs
MySQL vs PostgreSQL

Choosing Between MySQL and PostgreSQL

Both are excellent, production-hardened, and capable of handling the vast majority of application workloads. The right choice depends on what your application actually does — not which one is "better."

🐬 MySQL Wins When…

  • Your workload is primarily simple CRUD and single-table reads
  • You are running a WordPress, Drupal, or LAMP-stack application
  • Your team has deep MySQL operational expertise (backups, replication setup)
  • You need a battle-tested, low-maintenance primary database for a SaaS with straightforward data models
  • You are migrating from a legacy system already using MySQL

🐘 PostgreSQL Wins When…

  • Your application requires complex queries: window functions, CTEs, lateral joins
  • You need JSONB for flexible or schema-light data alongside relational data
  • You are building ERP, FinTech, healthcare, or any domain requiring strong data integrity
  • You want full-text search, geospatial data (PostGIS), or custom data types
  • You are building an analytics-heavy product and want to avoid a warehouse initially
Dimension MySQL 8+ PostgreSQL 16+
Clustered Primary Key Yes — InnoDB data is in the PK B-Tree No — heap files; PK is a separate index
JSONB / Document JSON type (text-stored), limited indexing JSONB (binary), GIN indexing, JSONPath
Full-Text Search FULLTEXT index (basic) tsvector + GIN (much more flexible)
Window Functions Available since MySQL 8.0 Available since PostgreSQL 8.4; more functions
Isolation Default Repeatable Read Read Committed
Replication Async & semi-sync; Group Replication Streaming + Logical replication
Extensibility Plugins (storage engines) Extensions, custom types, PostGIS
Ecosystem Huge; PlanetScale, Vitess, AWS Aurora Supabase, Neon, Timescale, Citus
"If you are starting a new project in 2026, PostgreSQL is the safer default. MySQL remains the right call for workloads where its operational model and ecosystem are already known quantities."
08 — Beyond Relational

NoSQL & Why It Emerged

NoSQL did not emerge because relational databases were bad. It emerged because Google, Amazon, and Facebook faced problems that no existing database could solve at their scale: billions of rows, global distribution, and schema flexibility across thousands of services.

The term "NoSQL" is a category, not a technology. Each subcategory solves a different problem — and choosing the wrong one is as costly as choosing an RDBMS for a use case it cannot handle.

📄 Document
MongoDB CouchDB Firestore

Stores JSON-like documents. Schema-flexible. Great for user profiles, CMS content, product catalogs. Weak at multi-document transactions and complex joins.

⚡ Key-Value
Redis DynamoDB Memcached

Simplest model: key maps to an opaque value. Sub-millisecond reads. Used for caching, sessions, rate limiting, real-time leaderboards.

📊 Wide-Column
Cassandra ScyllaDB HBase

Rows with variable columns grouped into column families. Designed for massive write throughput and horizontal scale. Queries must be modelled around access patterns at schema design time.

🔗 Graph
Neo4j Amazon Neptune ArangoDB

Nodes and edges as first-class citizens. Traversal queries (friends-of-friends, shortest path) that would require dozens of SQL joins become simple graph queries.

⏱️ Time-Series
InfluxDB TimescaleDB QuestDB

Optimised for time-stamped metrics, events, and IoT data. High ingest rates, automatic downsampling, retention policies built in. TimescaleDB runs on top of PostgreSQL.

🔍 Search Engine
Elasticsearch OpenSearch Typesense

Inverted indexes built for full-text search, faceting, and relevance ranking. Usually used alongside a relational DB — the RDBMS is the source of truth; the search engine powers the search UI.

The Polyglot Persistence Pattern
Modern production systems rarely use a single database. A typical architecture: PostgreSQL as the transactional source of truth, Redis for caching and sessions, Elasticsearch for search, and S3 (via Parquet/Iceberg) for analytics. The skill is knowing which store owns which data and how to keep them consistent.
09 — Global Scale

Modern Distributed SQL

Traditional relational databases scale vertically: you get a bigger server. Eventually you hit the limits of what a single machine can provide — and sharding your PostgreSQL manually is one of the most painful operations in software engineering. Distributed SQL databases solve this by providing a PostgreSQL-compatible interface over a horizontally scaled cluster.

🪳

CockroachDB

PostgreSQL-compatible wire protocol. Automatic sharding and rebalancing. Multi-region deployments with configurable data residency. Strong consistency via Raft consensus.

🐯

TiDB

MySQL-compatible. Separates storage (TiKV) from compute. Built-in HTAP — same cluster handles both OLTP and OLAP queries via TiFlash columnar replica.

🌍

YugabyteDB

PostgreSQL-compatible with DocDB storage layer. Designed for multi-cloud and multi-region deployments. YSQL layer is closer to full PostgreSQL compatibility than CockroachDB.

☁️

Aurora (AWS)

MySQL/PostgreSQL-compatible cloud database. Storage layer is decoupled from compute and replicated 6 ways across 3 AZs. Automatic failover in ~30s. Not truly distributed SQL but dramatically outscales standard RDS.

The CAP Theorem and What It Actually Means

The CAP theorem states that a distributed system can guarantee at most two of three properties simultaneously: Consistency, Availability, and Partition Tolerance. Since network partitions are inevitable in any distributed system, the practical choice is between CP (consistent but may reject requests during a partition) and AP (available but may return stale data).

Database CAP Choice Behaviour During Network Partition
CockroachDB CP Rejects writes that cannot achieve quorum — prefers consistency over availability
Cassandra AP Continues accepting reads and writes; resolves conflicts on read via last-write-wins
DynamoDB Tunable (AP default) Strong consistency mode available per-request; eventual consistency by default
PostgreSQL (single node) CA Not distributed — no partition tolerance; failover requires external tooling
Distributed SQL is not free
Cross-shard transactions are expensive. If your query touches rows on three different nodes, it must coordinate via a consensus protocol across all three — adding latency compared to a single-node RDBMS. Design schemas to minimise cross-shard operations: co-locate related data by tenant ID, user ID, or another high-cardinality partition key.
10 — Scaling Strategies

Scaling Relational Databases

The order of operations for scaling a relational database almost always looks the same: optimise queries first, add caching second, add read replicas third, partition tables fourth, and consider sharding only when the previous four steps are insufficient. Skipping steps is expensive.

Read Replicas

The first scaling lever. A primary handles all writes; one or more replicas receive WAL stream and replay it, staying within seconds of the primary. Route read-heavy workloads — reporting, dashboards, search — to replicas. This is the most impactful and operationally simple scaling approach for most systems.

Application
  │ writes           │ reads
Primary ─── WAL stream ──▶ Read Replica 1
                             └── WAL stream ──▶ Read Replica 2

Table Partitioning

Partitioning splits a large table into smaller physical sub-tables (partitions) while presenting a single logical table to queries. PostgreSQL declarative partitioning supports range, list, and hash strategies. The query planner performs partition pruning — if a WHERE clause filters on the partition key, only matching partitions are scanned.

PostgreSQL — Declarative Range Partitioning 
-- Parent table (no data stored here)
CREATE TABLE events (
  id         BIGINT,
  user_id    BIGINT,
  event_type TEXT,
  created_at TIMESTAMPTZ NOT NULL
) PARTITION BY RANGE (created_at);

-- Monthly partitions — only this month's partition is "hot"
CREATE TABLE events_2026_05 PARTITION OF events
  FOR VALUES FROM ('2026-05-01') TO ('2026-06-01');
CREATE TABLE events_2026_06 PARTITION OF events
  FOR VALUES FROM ('2026-06-01') TO ('2026-07-01');

-- Indexes are created per partition — or via the parent
CREATE INDEX ON events (user_id, created_at);

-- Query hits only the June partition (partition pruning)
SELECT * FROM events
WHERE created_at >= '2026-06-01'
  AND user_id = 12345;

Sharding

Sharding splits data horizontally across multiple independent database instances (shards). Each shard holds a subset of rows, typically by a shard key (user ID, tenant ID, geographic region). Unlike partitioning, shards are separate databases — there is no single node that knows the full dataset, which means cross-shard queries require application-level coordination or a query router.

🔑

Shard Key Selection

The most critical sharding decision. A good shard key distributes data evenly, keeps related data co-located, and avoids hot spots. Bad choices: sequential IDs (all new writes go to one shard), low-cardinality keys like status or country code.

🌡️

Hot Spot Problem

A viral user or large tenant creates a hot shard that receives disproportionate traffic. Mitigations: consistent hashing, virtual nodes, or a separate tier for large tenants.

🔗

Cross-Shard Joins

Cannot be done in the database — must be done in the application or a distributed query layer (Vitess, Citus). Design access patterns to avoid them. This is the primary cost of sharding.

🛠️

Vitess / Citus

Vitess (MySQL) and Citus (PostgreSQL) add a sharding layer above the database. They handle connection pooling, query routing, cross-shard aggregation, and online resharding without application changes.

Connection Pooling is Non-Optional at Scale
Every PostgreSQL connection is a separate OS process (~5–10MB RAM). At 500 concurrent connections you are using 2.5–5GB just for process overhead. Use PgBouncer (transaction mode for most workloads) or pgpool-II. Target fewer than 100 server-side connections with thousands of application-side connections going through the pooler.
11 — Modern Data Platforms

Modern Data Architecture

Production systems almost always have two distinct database needs: operational databases that serve live application traffic, and analytical databases that serve business intelligence, reporting, and data science. Trying to run both workloads on the same RDBMS is a common source of production incidents.

OLTP vs OLAP

⚙️ OLTP (Online Transaction Processing)

  • High volume of short, targeted queries
  • Reads and writes a handful of rows per query
  • Optimised for low-latency individual transactions
  • Row-oriented storage (fetch whole rows)
  • Examples: orders, payments, user sessions, inventory
  • Database: PostgreSQL, MySQL, Aurora

📊 OLAP (Online Analytical Processing)

  • Low volume of complex, full-scan queries
  • Aggregates millions to billions of rows
  • Optimised for throughput, not latency
  • Column-oriented storage (only scan needed columns)
  • Examples: revenue reports, cohort analysis, BI dashboards
  • Database: Snowflake, BigQuery, Redshift, ClickHouse

The Modern Data Stack

Application Database (PostgreSQL / MySQL — source of truth)
    │ CDC (Change Data Capture) via Debezium, Fivetran, or logical replication
Data Lake (S3 / GCS — raw Parquet / JSON files)
    │ dbt transforms, Apache Spark, or Flink for streaming
Data Warehouse (Snowflake / BigQuery / Redshift — modelled, queryable)
    │ BI tools: Metabase, Looker, Tableau, Superset
Business Intelligence Layer
❄️

Snowflake

Separates storage (S3) from compute (virtual warehouses). Pay per query. Near-unlimited concurrency. Industry standard for cloud data warehousing.

📊

BigQuery

Google's serverless warehouse. Columnar storage, automatic scaling, built-in ML. Pricing by bytes scanned — partition and cluster your tables to control costs.

🏔️

Apache Iceberg

Open table format that adds ACID transactions, schema evolution, and time-travel to data lakes. Decouples the table format from the query engine — query with Spark, Trino, Flink, or DuckDB.

🦆

DuckDB

In-process OLAP engine. Runs inside your Python script or application. Can query Parquet, CSV, and Iceberg files directly. Excellent for local analytics and replacing Pandas for medium-sized datasets.

12 — Real-World Patterns

Database Design for Production Systems

Database selection is domain-driven. Different application domains have different dominant constraints — and those constraints should drive both the database choice and the schema design philosophy.

🛒 E-Commerce
PostgreSQL Redis

ACID transactions for inventory and payments. Redis for cart state, session tokens, and product catalogue caching. Partition orders by month. Consider Elasticsearch for product search.

🏦 FinTech
PostgreSQL Append-only

Immutable ledger pattern — never UPDATE a transaction row, INSERT a correcting entry. Serializable isolation for critical transfers. Audit log in a separate table with trigger-based population.

📋 ERP / SaaS
PostgreSQL Multi-tenant

Row-level security per tenant. Schema-per-tenant or tenant_id column. Partitioning by tenant for large deployments. Citus for extreme multi-tenant scale.

📱 Social Network
PostgreSQL Redis Cassandra

User profiles in PostgreSQL. Activity feeds in Redis (sorted sets) or Cassandra (wide-column, time-series friendly). Shard by user ID at extreme scale.

🏥 Healthcare
PostgreSQL Audit Log

HIPAA compliance requires access logging and encryption at rest. Row-level security for provider-patient data access. Immutable audit tables. PostgreSQL + field-level encryption extensions.

📊 Analytics SaaS
PostgreSQL ClickHouse Iceberg

OLTP for account management in PostgreSQL. Event ingestion to Kafka → ClickHouse or S3/Iceberg. Materialized views for dashboard queries. Never run analytical queries on the OLTP primary.

The Immutable Ledger Pattern (FinTech Essential)

SQL — Immutable Ledger with Running Balance 
-- Never UPDATE. Only INSERT.
CREATE TABLE ledger_entries (
  id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  account_id  BIGINT        NOT NULL,
  amount      NUMERIC(18,6) NOT NULL,  -- positive = credit, negative = debit
  entry_type  TEXT          NOT NULL,  -- 'deposit', 'withdrawal', 'fee', 'reversal'
  reference   TEXT,                    -- external transaction ID
  created_at  TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
  created_by  BIGINT        NOT NULL   -- user or service that created this entry
);

-- Running balance via window function (no stored balance field)
SELECT
  id,
  entry_type,
  amount,
  SUM(amount) OVER (
    PARTITION BY account_id
    ORDER BY created_at
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS running_balance
FROM ledger_entries
WHERE account_id = 42
ORDER BY created_at;
13 — The Learning Path

Zero-to-Hero Learning Roadmap

Database mastery is not a checklist — it is a mental model that accumulates. Each layer of understanding changes how you design schemas, write queries, and architect systems. This is the path from writing SQL to designing data platforms.

Beginner
Foundation: tables, columns, data types, primary keys, foreign keys. CRUD with INSERT, SELECT, UPDATE, DELETE. Basic JOINs: INNER, LEFT, RIGHT. Filtering with WHERE, sorting with ORDER BY, grouping with GROUP BY and HAVING.
Intermediate
Query Power: subqueries, CTEs, window functions (ROW_NUMBER, RANK, LAG, LEAD). Indexes: B-Tree, composite, partial, covering. Transactions with explicit BEGIN / COMMIT / ROLLBACK. Schema design: normalization, foreign key constraints, check constraints. Reading basic EXPLAIN output.
Advanced
Performance & Internals: isolation levels and their real-world effects. MVCC and autovacuum in PostgreSQL. EXPLAIN ANALYZE with BUFFERS — reading plans, identifying bad estimates, fixing them. Index types: GIN for JSONB and full-text, partial indexes, expression indexes. Connection pooling with PgBouncer. Replication: streaming, logical, and their trade-offs. Table partitioning for large tables.
Senior
Architecture & Systems Thinking: CAP theorem and what it means for database selection. Sharding strategies, shard key selection, and cross-shard coordination. Distributed SQL trade-offs (CockroachDB, TiDB, YugabyteDB). OLTP vs OLAP separation — when to introduce a data warehouse. CDC and event streaming for data sync. Multi-tenant data isolation patterns. Disaster recovery: RPO and RTO, backup strategies, point-in-time recovery. Database cost optimisation at scale.
"A senior engineer doesn't ask, 'Which database is best?' They ask: 'What problem am I solving, and which database's trade-offs align with that problem?'"

The Mental Model That Changes Everything

The database world evolved through distinct eras, each solving a problem the previous era could not:

File Storage → no concurrency, no querying
    ↓
Relational Databases (1970s) → ACID, SQL, normalization
    ↓
MySQL / PostgreSQL (1990s–) → open-source, web-scale RDBMS
    ↓
NoSQL Movement (2000s) → flexible schema, horizontal scale
    ↓
NewSQL / Distributed SQL (2010s) → SQL + horizontal scale + ACID
    ↓
Cloud Warehouses (2010s–) → Snowflake, BigQuery, Redshift
    ↓
Lakehouses (2020s) → Iceberg, Delta Lake, open formats
    ↓
Modern Data Platforms (now) → polyglot persistence, streaming, AI

Quick Reference: Which Database for Which Problem

Database Optimised For Avoid When
MySQL Simple web CRUD, LAMP stack, SaaS with straightforward schemas Complex analytics, advanced SQL features, full-text search
PostgreSQL Complex queries, JSON storage, FinTech, ERP, anything needing strong integrity Massive write-heavy sharded workloads (use Citus or CockroachDB instead)
Redis Caching, sessions, rate limiting, pub/sub, leaderboards Primary data store for critical data without persistence configuration
MongoDB Flexible document storage, rapid iteration on schema Workloads requiring strong relational integrity or complex joins
Cassandra High write throughput, time-series, globally distributed data Ad-hoc queries, transactions, anything requiring flexible querying
CockroachDB Global ACID transactions, multi-region deployments Simple single-region apps where PostgreSQL complexity is unnecessary
Snowflake / BigQuery Analytics at scale, BI reporting, data warehouse Transactional workloads — OLAP databases are not designed for OLTP
Elasticsearch Full-text search, log analysis, faceted search UI Primary data store — always replicate from your source-of-truth RDBMS
Final Principle
Start with PostgreSQL. Add Redis when you need caching. Add a search engine when full-text requirements exceed what pg's tsvector can handle. Add a warehouse when analytics queries start affecting production performance. Add Kafka when you need reliable event streaming between services. Reach for sharding or distributed SQL only when vertical scaling and read replicas are genuinely exhausted. Complexity is a liability — add it only when the cost of not adding it is higher.
