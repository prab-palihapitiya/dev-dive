This guide is an adaptation of Robert C. Martin's heuristics and smells from Clean Code, with all code examples translated to JavaScript and TypeScript. The concepts apply universally across languages — the JS/TS context makes them immediately actionable for modern frontend and full-stack engineers.

How to Use This Guide
Use the rule codes (C1, G5, N3…) as shorthand in pull request comments to reference specific principles without lengthy explanations. A comment like "G5 — extract the shared discount logic" communicates precisely and educates at the same time.
"Clean code is not written by following a set of rules. Professionalism and craftsmanship come from values that drive disciplines."
— Robert C. Martin
C — Comments

Comments

A comment worth writing is worth writing well. But the best comment is often no comment at all — when code is expressive enough to speak for itself.

C1 Inappropriate Information

Comments should be reserved for technical notes about the code and design.

Meta-data belongs in dedicated systems, not source files. Change histories, author names, last-modified dates, ticket numbers, and SPR references are all better tracked in your version control system or issue tracker. When they appear in comments, they clutter the file with historical noise that grows stale and misleads future readers.

C2 Obsolete Comment

Obsolete comments tend to migrate away from the code they once described.

A comment that describes what the code used to do becomes a floating island of misdirection. Code evolves; comments often do not. When you find an obsolete comment, update it or delete it immediately. Do not leave it in place as a historical curiosity — it will mislead the next developer who reads it.

C3 Redundant Comment

Comments should say things that the code cannot say for itself.

A comment that restates what the code already clearly expresses adds noise without signal.

TypeScript✗ Redundant
i++; // increment

/**
 * Begins the process of selling an item.
 * @param sellRequest The request details for selling the item.
 * @returns A response object related to the sell operation.
 */
function beginSellItem(sellRequest: SellRequest): SellResponse { ... }

The JSDoc above adds nothing the function signature doesn't already communicate. Reserve documentation comments for non-obvious behaviour, edge cases, and design decisions that cannot be expressed in types or names alone.

C4 Poorly Written Comment

Choose your words carefully.

If a comment is worth writing, write it well. Use correct grammar. Be brief and precise. Don't ramble. Don't state the obvious. A poorly written comment is worse than no comment — it suggests carelessness, and carelessness in comments invites carelessness in the code itself.

C5 Commented-Out Code

It pollutes the modules that contain it and distracts the people who try to read it.

Commented-out code is an abomination. It calls functions that no longer exist, uses variable names that have changed, and follows conventions that are obsolete. Nobody deletes it because everyone assumes someone else needs it. Delete it. Your version control system remembers it.

TypeScript✗ Delete This
// const oldResult = calculateLegacyTotal(items);
// if (oldResult > 0) {
//   applyDiscount(oldResult);
// }
const result = calculateTotal(items);
E — Environment

Environment

The ease of building and testing a project is not a luxury — it is a quality signal. A project that requires multiple manual steps to build or test will be built and tested less often, which means bugs accumulate faster.

E1 Build Requires More Than One Step

Building a project should be a single trivial operation.

You should be able to check out a project and be running it with two commands. No hunting for config files, no undocumented environment variables, no manual setup steps that live only in someone's head.

Shell✓ Ideal Build Flow
git clone git@github.com:org/project.git
cd project
pnpm install
pnpm dev

If your README has more than a few steps before "you're running," that complexity belongs in scripts, Dockerfiles, or tooling — not in the human's head.

E2 Tests Require More Than One Step

You should be able to run all the unit tests with just one command.

Tests that are hard to run are tests that don't get run. Whether triggered from an IDE button or a shell command, the full test suite must be a single, obvious, fast invocation. This is so fundamental that it belongs in the project README's first paragraph.

Shell✓ One Command
pnpm test          # runs all unit tests
pnpm test:watch    # watch mode during development
pnpm test:coverage # with coverage report
F — Functions

Functions

Functions are the primary unit of organisation in code. Small, focused functions with clear signatures are easier to name, test, understand, and compose than large, multipurpose ones.

F1 Too Many Arguments

More than three is very questionable and should be avoided with prejudice.

Functions with many parameters are hard to call correctly, hard to test exhaustively, and a signal that the function may be doing too much. The solution is almost always to group related parameters into an object.

TypeScript✗ Bad
function createUser(
  firstName: string,
  lastName: string,
  email: string,
  age: number,
  isAdmin: boolean
): void { ... }

createUser("John", "Doe", "j@ex.com", 30, false);
TypeScript✓ Good
interface CreateUserParams {
  firstName: string;
  lastName: string;
  email: string;
  age: number;
  isAdmin: boolean;
}
function createUser(params: CreateUserParams): void { ... }

createUser({ firstName: "John", lastName: "Doe",
  email: "j@ex.com", age: 30, isAdmin: false });

F2 Output Arguments

If your function must change the state of something, have it change the state of the object it is called on.

Readers expect function arguments to be inputs, not outputs. A function that modifies one of its arguments surprises the caller and forces them to read the implementation to understand what happened. Return new values or use class methods that operate on this.

TypeScript✗ Output Argument
function updateEmail(
  profile: UserProfile,
  newEmail: string
): void {
  profile.email = newEmail; // mutates input
}
TypeScript✓ Returns New Value
function withUpdatedEmail(
  profile: UserProfile,
  newEmail: string
): UserProfile {
  return { ...profile, email: newEmail };
}

F3 Flag Arguments

It's recommended to split the function into multiple functions, each doing a single, well-defined task.

A boolean parameter loudly declares that the function does at least two different things depending on the flag. Split it into two clearly named functions instead.

TypeScript✗ Flag Argument
updateUserProfile(1, "new@ex.com", true);
// What does `true` mean here?
TypeScript✓ Split Functions
updateUserProfile(1, "new@ex.com");
sendVerificationEmail("new@ex.com");
// Intent is explicit at call site

F4 Dead Function

Keeping dead code around is wasteful.

Methods that are never called should be deleted. Every major IDE and editor has "Find All References" — use it. Your source control system remembers deleted code. Do not be afraid to remove functions that no longer serve a purpose. Dead code is cognitive overhead for every developer who reads the file.

G — General (G1–G12)

General Rules — Part 1

General heuristics span architecture, structure, and behaviour. Many of the most damaging code quality problems are not about syntax or style — they are about where things are placed and how much they know about each other.

G1 Multiple Languages in One Source File

The ideal is for a source file to contain one, and only one, language.

Mixing SQL strings, HTML templates, JSON blobs, and business logic in a single TypeScript file creates a file with no clear purpose and no clear owner. Extract SQL to repositories, HTML to template files, and configuration to dedicated modules. Minimise the number of languages cohabiting in any single file.

G2 Obvious Behaviour Is Unimplemented

Following the Principle of Least Surprise — any function should implement the behaviours another programmer could reasonably expect.

A function named stringToDay that accepts "Monday" should also handle "monday", "MONDAY", and "MON". When obvious behaviours are missing, developers lose trust in the codebase and must read every implementation in detail rather than relying on intuition.

TypeScript✓ Handles Expected Variants
function stringToDay(dayName: string): Day | undefined {
  switch (dayName.toUpperCase()) {   // normalise case first
    case "MONDAY": case "MON": return Day.MONDAY;
    case "TUESDAY": case "TUE": return Day.TUESDAY;
    // … etc
    default: return undefined;
  }
}

G3 Incorrect Behaviour at the Boundaries

There is no replacement for due diligence.

Developers write functions that "work" for the happy path and trust their intuition for the rest. Every boundary condition — null inputs, empty arrays, zero values, negative numbers, maximum values — represents a case that can break an algorithm that seems correct. Write tests for every boundary. Don't rely on intuition.

G4 Overridden Safeties

Turning off certain compiler warnings may help you get the build to succeed, but at the risk of endless debugging sessions.

Using // @ts-ignore, any everywhere, or disabling strict in tsconfig.json to silence type errors is the TypeScript equivalent of overriding a safety mechanism. The type system exists to catch errors before runtime. Every suppressed warning is a deferred bug.

TypeScript✗ Suppressing Safeties
// @ts-ignore  ← don't do this
const result = riskyOperation();

function processData(data: any) {  // ← `any` defeats the type system
  return data.someProp.nested;  // runtime error waiting to happen
}

G5 Duplication

Every time you see duplication in the code, it represents a missed opportunity for abstraction.

This is the DRY principle — Don't Repeat Yourself. Every duplication is a maintenance liability: when the logic changes, every copy must be found and updated. Three functions with slightly different discount rates all call for a single parameterised function.

TypeScript✗ Duplicated Logic
function calcStudentDiscount(n: number) {
  return n - n * 0.20;
}
function calcTeacherDiscount(n: number) {
  return n - n * 0.10;
}
function calcSeniorDiscount(n: number) {
  return n - n * 0.15;
}
TypeScript✓ Abstracted
const DISCOUNT_RATES = {
  student: 0.20,
  teacher: 0.10,
  senior:  0.15,
} as const;

function applyDiscount(
  amount: number,
  rate: number
): number {
  return amount * (1 - rate);
}

G6 Code at Wrong Level of Abstraction

You cannot lie or fake your way out of misplaced abstraction.

Higher-level concepts belong in base classes or interfaces; lower-level implementation details belong in derivatives. When percentFull() appears on a generic Stack interface, it forces all stack implementations — including unbounded ones — to implement a concept that doesn't apply to them. Keep abstractions clean and complete.

G7 Base Classes Depending on Their Derivatives

In general, base classes should know nothing about their derivatives.

When a base class contains if (type === "cat") or instanceof Dog checks, you've lost the benefit of inheritance. The base class is now coupled to every subclass — adding a new type requires modifying the base. Use polymorphism and factory patterns to decouple creation from usage.

TypeScript✗ Base Knows Derivatives
class Animal {
  static create(type: string) {
    if (type === "cat") return new Cat();
    if (type === "dog") return new Dog();
    throw new Error("Unknown");
  }
}
TypeScript✓ Factory Decoupled
abstract class Animal {
  abstract makeSound(): string;
}
class Cat extends Animal {
  makeSound() { return "meow"; }
}
// Factory lives separately from Animal

G8 Too Much Information

The fewer methods a class has, the better. The fewer variables a function knows about, the better.

A well-designed interface is narrow. It exposes the minimum surface area necessary to accomplish its purpose. Wide interfaces with many public methods and properties create high coupling — consumers depend on things they don't need, and any internal change risks breaking them. Hide data, hide utility functions, hide temporaries.

G9 Dead Code

When you find dead code, do the right thing. Give it a decent burial. Delete it from the system.

Dead code is code that can never execute: the body of an if on an impossible condition, a catch block for an exception that is never thrown, a utility function with zero callers. It compiles. It passes linting. And it slowly rots, becoming more misleading with every passing month. Delete it.

G10 Vertical Separation

The vertical distance between a variable/function declaration and its usage should be minimal.

Local variables should be declared just above their first use, with the smallest possible scope. Private helper functions should appear just below the public method that calls them. When definitions and usages are far apart, reading the code requires constantly scrolling back and forth — a cognitive overhead that compounds on every read.

G11 Inconsistency

If you do something a certain way, do all similar things in the same way.

If you name a variable response for an HttpResponse in one function, use response consistently across all functions that handle HttpResponse. If you name a method processVerificationRequest, name its sibling processDeletionRequest — not handleDeleteData or delete_request. Consistency is a courtesy to every future reader.

TypeScript✗ Inconsistent Naming
class RequestHandler {
  handleDataUpdate(request: any)   // "handle" + "request"
  processDeletionData(req: any)   // "process" + "req"
  verify_request(input: any)      // snake_case + "input"
  fetchRecords()                  // "fetch", no param at all
}

G12 Clutter

Keep your source files clean, well organised, and free of clutter.

Unused variables, empty constructors with no implementation, never-called private functions, meaningless comments — all of these are clutter. They add lines without adding information. Every line of code is a line a developer must read and process. Earn every line's presence in your file.

G — General (G13–G24)

General Rules — Part 2

G13 Artificial Coupling

It is a result of putting a variable, constant, or function in a temporarily convenient location. This is lazy and careless.

General-purpose utilities should not live inside specific classes. A formatDate utility in a User class artificially couples every consumer of formatDate to User. Take the time to find where things genuinely belong — put shared utilities in a utils module, domain constants in their owning module, and general helpers at the appropriate level.

TypeScript✗ Artificially Coupled
class User {
  // A date util has nothing to do with User
  static formatDate(date: Date): string {
    return `${date.getFullYear()}-...`;
  }
}
// Bad: order.ts now depends on User
User.formatDate(order.createdAt);
TypeScript✓ Correct Home
// date-utils.ts — no coupling to any domain class
export function formatDate(date: Date): string {
  return `${date.getFullYear()}-...`;
}

// order.ts imports directly from utils
import { formatDate } from "../utils/date-utils";

G14 Feature Envy

We want to eliminate Feature Envy because it exposes the internals of one class to another.

A method suffers from feature envy when it spends more time using the data of another class than its own. It "wishes" it were inside that other class. The fix is to move the method to the class it envies — unless doing so violates the Single Responsibility Principle (e.g. a report formatter that needs employee data but shouldn't know about reporting format).

G15 Selector Arguments

Selector arguments are just a lazy way to avoid splitting a large function into several smaller functions.

A boolean, enum, or integer argument used to select a code path within a function is a selector argument. They are confusing at call sites (calculateWeeklyPay(false) — what does false mean?) and signal that the function does more than one thing. Split the function instead.

TypeScript✗ Selector Argument
calculator.calculateWeeklyPay(false);
// false = no overtime? straight time? what?
TypeScript✓ Explicit Intent
calculator.straightPay();
calculator.overtimePay();
// Intent is self-documenting

G16 Obscured Intent

We want code to be as expressive as possible.

Run-on expressions, Hungarian notation, and magic numbers all obscure the author's intent. A dense calculation like return this.iThsWkd * this.iThsRte + Math.round(0.5 * this.iThsRte * Math.max(0, this.iThsWkd - 400)) is virtually impenetrable. Break it up. Name the parts. Write for the reader.

G17 Misplaced Responsibility

Code should be placed where a reader would naturally expect it to be.

The Principle of Least Surprise applies to code location. If a function is named getTotalHours, a reader expects it to be on the class that owns hours data — not on the report generator. Place logic where readers will naturally look for it, guided by the names of the functions and classes around it.

G18 Inappropriate Static

When in doubt, make the function non-static. If you really want a function to be static, make sure there is no chance you'll want it to behave polymorphically.

Math.max is a legitimate static method — it takes no instance state and has no meaningful polymorphic variant. But HourlyPayCalculator.calculatePay(employee, overtimeRate) should not be static if you might ever want OvertimePayCalculator and StraightTimePayCalculator to implement different strategies. Prefer instance methods; reserve static for true utilities.

G19 Use Explanatory Variables

One of the more powerful ways to make a program readable is to break calculations up into intermediate values with meaningful names.

Named intermediate variables cost nothing at runtime (compilers optimise them away) and buy enormous clarity for human readers. It is hard to overdo this.

TypeScript✗ Opaque
return {
  area: Math.PI * r * r,
  circ: 2 * Math.PI * r,
};
TypeScript✓ Transparent
const pi = Math.PI;
const radiusSquared = r * r;
const area = pi * radiusSquared;
const circumference = 2 * pi * r;
return { area, circumference };

G20 Function Names Should Say What They Do

If you have to look at the implementation to know what it does, work to find a better name.

date.add(5) — does this add days? Hours? Does it mutate the date or return a new one? The name is ambiguous. addDaysTo(date, 5) or daysLater(date, 5) leave no doubt. Function names are the primary documentation — make them earn their keep.

G21 Understand the Algorithm

Being unsure what your code does is just laziness.

Getting a function to pass tests by adding enough if statements and flags is not the same as understanding the algorithm. Before you consider a function done, be able to explain — without looking at it — exactly what it does and why. If you can't, refactor until the code itself makes the algorithm obvious.

G22 Make Logical Dependencies Physical

Physicalise logical dependencies to clarify module relationships.

If module A depends on a constant that conceptually belongs to module B, don't hardcode it in A and assume B will always be compatible. Ask B for the value explicitly. In the classic example, HourlyReporter should not hardcode PAGE_SIZE = 55 and assume HourlyReportFormatter will always accept 55-line pages. Instead, HourlyReportFormatter should expose a getMaxPageSize() method that HourlyReporter calls.

G23 Prefer Polymorphism to If/Else or Switch/Case

Every switch statement should be suspect.

Most switch statements exist because polymorphism wasn't used. Every time you add a new type, you must find every switch on that type and add a case. With polymorphism, you add a new subclass and everything works. The ONE SWITCH rule: there may be at most one switch for a given type selection — and that switch should create polymorphic objects that replace all other switches in the system.

G24 Follow Standard Conventions

The team should not need a document to describe these conventions because their code provides the examples.

Agree on a style guide (Airbnb, Google, or your own) and enforce it with ESLint and Prettier. The specific conventions matter less than their consistent application. A codebase where every file looks like it was written by the same person is dramatically easier to read than one where each developer has their own style.

G — General (G25–G36)

General Rules — Part 3

G25 Replace Magic Numbers with Named Constants

Having named constants for numbers is always a good idea.

A raw number in code carries no meaning. 86400 is just a number — SECONDS_PER_DAY is self-documenting. 7777 in a test assertion is a mystery — HOURLY_EMPLOYEE_ID communicates intent. The term "magic number" extends beyond integers to any token whose meaning isn't obvious: a magic string like "john.doe" is just as opaque as a magic integer.

TypeScript✗ Magic Numbers
if (sessionDuration > 1800) expireSession();
assertEqual(7777, employee.find("john.doe").id);
TypeScript✓ Named Constants
const SESSION_TIMEOUT_SECONDS = 1800;
if (sessionDuration > SESSION_TIMEOUT_SECONDS) expireSession();
assertEqual(HOURLY_EMPLOYEE_ID, employee.find(HOURLY_EMPLOYEE_NAME).id);

G26 Be Precise

Ambiguities and imprecision in code are either a result of disagreements or laziness.

Using number for currency calculations introduces floating-point rounding errors. Assuming a query returns at most one result when it could return many causes silent data loss. Not handling null when a function can return null causes runtime crashes. When you make a decision in code, understand it precisely and handle every exception.

TypeScript✗ Imprecise
// Floating point for money = rounding errors
const total = items.reduce(
  (sum, i) => sum + i.price * i.qty, 0.0
);
// Assumes at most one result
return items.find(i => i.id === id);
TypeScript✓ Precise
// Cents (integers) for money — no rounding errors
const totalCents = items.reduce(
  (sum, i) => sum + i.priceCents * i.qty, 0
);
// Returns all matches — caller decides
return items.filter(i => i.id === id);

G27 Structure over Convention

Enforce design decisions with structure over convention.

Conventions rely on discipline — people can forget or ignore them. Structure enforces compliance automatically. An abstract class with an abstract method makeSound() forces every subclass to implement it at compile time. A switch with a naming convention does not. Prefer mechanisms that make incorrect implementations impossible over those that merely make them unconventional.

G28 Encapsulate Conditionals

Extract complex conditionals into descriptive functions to clarify their intent.

Boolean logic is hard enough to reason about without having to parse it inside an if statement. Extract the condition into a named function whose name explains the intent.

TypeScript✗ Raw Conditional
if (timer.hasExpired() && !timer.isRecurrent()) {
  deleteTimer(timer);
}
TypeScript✓ Named Predicate
function shouldBeDeleted(timer: Timer): boolean {
  return timer.hasExpired() && !timer.isRecurrent();
}
if (shouldBeDeleted(timer)) deleteTimer(timer);

G29 Avoid Negative Conditionals

Prefer positive over negative conditionals for clearer logic.

Negatives require an extra mental step to parse. if (buffer.shouldCompact()) reads naturally. if (!buffer.shouldNotCompact()) requires the reader to flip a double negative. When you find yourself writing !isNot..., rename the predicate to the positive form.

G30 Functions Should Do One Thing

Decompose functions into smaller ones that each perform a single task.

A function that loops over employees, checks if it's payday, calculates pay, and delivers pay does four things. Each of those is a separate concern that can be named, tested, and understood independently. Decompose until each function's name precisely describes its complete behaviour.

TypeScript✗ Does Three Things
function pay(employees: Employee[]) {
  for (const e of employees) {
    if (e.isPayDay()) {
      const pay = e.calculatePay();
      e.deliverPay(pay);
    }
  }
}
TypeScript✓ One Thing Each
function pay(employees: Employee[]) {
  employees.forEach(payIfNecessary);
}
function payIfNecessary(e: Employee) {
  if (e.isPayDay()) calculateAndDeliverPay(e);
}
function calculateAndDeliverPay(e: Employee) {
  e.deliverPay(e.calculatePay());
}

G31 Hidden Temporal Couplings

Make temporal dependencies among function calls explicit to prevent misuse.

If function B must always be called after function A, and function C after B, that ordering constraint should be visible in the code — not hidden in a comment or tribal knowledge. Use a bucket-brigade pattern: make each function return a value that the next function requires as input, so calling them out of order is a type error, not a runtime surprise.

TypeScript✗ Hidden Order
diver.saturateGradient();  // must be first
diver.reticulateSplines(); // must be second
diver.diveForMoog(reason);  // must be third
TypeScript✓ Explicit Order
const gradient = diver.saturateGradient();
const splines = diver.reticulateSplines(gradient);
diver.diveForMoog(splines, reason);
// Out-of-order calls won't compile

G32 Don't Be Arbitrary

Have a reason for the way you structure your code, and make sure that reason is communicated by the structure.

If a class is nested inside another class, there should be a clear reason — it is a utility of that class and used nowhere else. If a helper function appears in a module, there should be a reason it belongs there. Arbitrary placement spreads confusion and forces future developers to guess at intent. Structure your code purposefully, and let the structure communicate that purpose.

G33 Encapsulate Boundary Conditions

Boundary conditions are hard to keep track of. Put the processing of them in one place.

Off-by-one errors scatter +1s and -1s throughout the code. When a boundary value like level + 1 appears in multiple places, extract it into a named variable: const nextLevel = level + 1. Now the boundary condition is defined once, used everywhere, and has a name that communicates its intent.

G34 Functions Should Descend Only One Level of Abstraction

Maintain a single level of abstraction per function to clarify intent.

Every statement in a function should be at the same level of abstraction — one level below what the function's name describes. A render() function that both constructs an HTML tag structure (high-level) and manually builds attribute strings (low-level) mixes levels. Extract the low-level detail into a helper so render() only does high-level orchestration.

G35 Keep Configurable Data at High Levels

Expose configurable data at high levels to make it easily accessible and modifiable.

Default values, configuration constants, and feature flags should be visible at the top level of the application — in a config module, environment variables, or a well-named constants file. Burying them in low-level functions forces developers to dig through multiple abstraction layers to find and change what should be a simple configuration decision.

TypeScript✓ Config at High Level
// config.ts — all configurable values in one obvious place
export const CONFIG = {
  DEFAULT_PORT: 3000,
  SESSION_TIMEOUT_MS: 30 * 60 * 1000,
  MAX_UPLOAD_SIZE_BYTES: 10 * 1024 * 1024,
  ITEMS_PER_PAGE: 25,
} as const;
// Low-level functions receive config values as params, not hardcoding them

G36 Avoid Transitive Navigation

Avoid having modules navigate through multiple layers of collaborators — the Law of Demeter.

If you find yourself writing a.getB().getC().doSomething(), your code knows too much about the internal structure of the object graph. This is "Train Wreck" code. If B's internal structure changes, every chain that traverses through B is broken. Instead, ask your immediate collaborator to do what you need: a.doSomethingWithBAndC().

TypeScript✗ Train Wreck
a.getB().getC().doSomething();
// A knows about B AND C
TypeScript✓ Law of Demeter
a.doSomethingWithBAndC();
// A only knows its immediate collaborator
JS — JavaScript / TypeScript

JavaScript & TypeScript Specifics

These rules apply the general principles to JS/TS-specific patterns. If you are using a framework (React, Vue, Angular, Svelte), always defer to the framework's established conventions for imports and module structure first.

JS1 Modularise and Structure Imports

Group exports in index files to simplify imports from a package or module.

Long lists of individual imports from deeply nested paths create tight coupling between import sites and file structure. Use index.ts barrel files to create a stable public API for each module. Internal rearrangements become transparent to consumers.

TypeScript✗ Scattered Imports
import { Button } from './components/Button';
import { Input } from './components/Input';
import { Modal } from './components/Modal';
TypeScript✓ Barrel Index
// components/index.ts
export { Button } from './Button';
export { Input } from './Input';
export { Modal } from './Modal';

// Usage — one import, stable path
import { Button, Input, Modal } from './components';

JS2 Use Exported Constants Rather Than Inheritance for Constants

This promotes clearer dependencies and easier maintenance.

Inheriting from a class to access its constants creates an artificial inheritance relationship for the wrong reason. Export constants directly from a dedicated module. Consumers import what they need without any class hierarchy entanglement.

TypeScript✓ Exported Constants
// constants.ts
export const TENTHS_PER_WEEK = 400;
export const OVERTIME_RATE   = 1.5;

// HourlyEmployee.ts — clean, explicit dependency
import { TENTHS_PER_WEEK, OVERTIME_RATE } from './constants';
class HourlyEmployee { ... }

JS3 Enums over Constants

Enums provide a clear, self-documenting way to represent a set of related values.

When a set of values are semantically related (different pay grades, order statuses, user roles), a TypeScript enum is better than a set of individual constants. Enums are self-documenting, grouping related values under a meaningful name, and they prevent mixing values from different domains.

TypeScript✗ Scattered Constants
const APPRENTICE_RATE    = 1.0;
const JOURNEYMAN_RATE    = 1.5;
const MASTER_RATE        = 2.0;
// No connection between these is enforced
TypeScript✓ Typed Enum
enum HourlyPayGrade {
  Apprentice  = 1.0,
  Journeyman  = 1.5,
  Master      = 2.0,
}
function calculatePay(grade: HourlyPayGrade) { ... }
N — Names

Names

Names are the primary communication channel in code. Every variable, function, and class name is a tiny piece of documentation. Choosing names well is one of the most impactful things a developer can do for long-term code health.

N1 Choose Descriptive Names

Names make the code readable and understandable.

A name is an opportunity to communicate. A single-letter variable or a cryptic abbreviation communicates nothing. The reader is forced to reverse-engineer meaning from context, which multiplies reading effort for every future read of that code.

TypeScript✗ Cryptic
function f() {
  let q = 0;
  // complex logic...
}
TypeScript✓ Descriptive
function calculateTotalScore() {
  let totalScore = 0;
  // clear, descriptive logic...
}

N2 Use Abstraction-Appropriate Names

Names should reflect the abstraction level of the class or function.

Names at a high abstraction level should be general; names at a low level can be specific. An interface named Connectable with a connect(locator: string) method is appropriately abstract — it accommodates database connections, socket connections, and HTTP connections without assuming any particular technology. Naming it MySQLConnector would bind a generic concept to a specific implementation.

N3 Follow Standard Nomenclature

If a pattern or standard approach exists, use it.

Naming things after the patterns and conventions they implement helps readers immediately understand the role of a piece of code. AutoLogoutDecorator signals the Decorator pattern. UserRepository signals the Repository pattern. OrderService signals a service layer. Do not invent new terminology when established terminology already communicates the concept precisely.

N4 Names Should Be Unambiguous

Choose names that clearly and unambiguously describe what a function or variable does.

Ambiguous names force readers to open the implementation to understand what something does. renamePageAndReferences() is unambiguous — you know exactly what it does without looking at a single line of its body. process() or handle() or doStuff() are not.

N5 Scope-Length Names

Short names for short scopes; longer, more descriptive names for wider scopes.

A loop index i is fine when it's visible for two lines. A module-level constant or a function parameter that will be used across a 50-line function needs a full descriptive name. The rule of thumb: name length should be proportional to the scope in which it appears.

TypeScript✓ Scope-Appropriate Names
// Short scope: i is fine — the loop is 2 lines
for (let i = 0; i < items.length; i++) { ... }

// Module-wide scope: full descriptive name
const defaultConnectionTimeoutMilliseconds = 30_000;

N6 Avoid Encodings in Names

Modern IDEs and TypeScript provide type information — don't encode it in names.

Hungarian notation (prefixing variables with type indicators like sName for string, nCount for number) was a workaround for editors that couldn't display type information. TypeScript's type system and modern editors have made it obsolete. It adds noise without adding information.

TypeScript✗ Hungarian Notation
let sName: string;
let nCount: number;
let bIsActive: boolean;
TypeScript✓ Clean Names
let name: string;
let count: number;
let isActive: boolean;

N7 Describe Side-Effects

Names should also hint at side effects to avoid surprises.

A function named getSession() suggests a pure read. A function that creates a session if one doesn't exist should be named to reveal that side effect: getOrCreateSession(). Functions that do more than their name suggests are land mines for future developers who call them expecting simple read behaviour.

TypeScript✓ Side-Effect Revealed in Name
// Name reveals: this may CREATE, not just get
function getOrCreateObjectOutputStream(): OutputStream { ... }

// Compare to the misleading alternative:
// function getOutputStream() — caller assumes no side effect
T — Tests

Tests

Tests are not optional polish added at the end — they are a fundamental quality discipline. A test suite that is slow, incomplete, or hard to run will not be used. Tests that are not used cannot provide the safety net that makes confident refactoring possible.

T1 Comprehensive Tests

Ensure your test suite covers all possible scenarios and edge cases — not just the happy path. A test suite that only covers the most common case gives a false sense of security. Every code path, every branch, every error condition deserves a test.

T2 Use Coverage Tools

Coverage tools identify paths through the code that no test has ever executed. Use them, but don't worship them — 100% line coverage with tests that make no assertions is meaningless. Coverage reveals blind spots; it does not guarantee correctness.

Shell — Vitest / Jest✓ Coverage Report
pnpm vitest run --coverage
# Generates coverage/index.html
# Look for uncovered branches, not just uncovered lines

T3 Don't Skip Simple Tests

Trivial tests for obvious behaviour have value both as documentation (they demonstrate how the code is expected to be used) and as regression protection (they catch accidental breakage of behaviour that everyone assumed "would never change"). There is no test too simple to write.

T4 Use Tests to Document Ambiguities

When the correct behaviour of a function in an edge case is unclear, write a test that asserts the behaviour you have decided on and add a comment explaining why. The test becomes living documentation of a decision that would otherwise be invisible or forgotten.

TypeScript — Vitest✓ Tests as Documentation
it("returns empty array for null input (not an error)", () => {
  // Decision: we treat null as "no items" rather than throwing,
  // because callers often chain this with optional data.
  expect(processItems(null)).toEqual([]);
});

T5 Test Boundary Conditions

Special attention should be given to testing edge cases: empty inputs, single-element inputs, maximum values, minimum values, exact boundary values, and zero. Algorithms that work correctly in the middle often fail at the edges — test the edges deliberately, not as an afterthought.

TypeScript — Boundary Tests✓ Test the Edges
describe("calculateDiscount", () => {
  it("returns 0 for zero amount", () => {
    expect(calculateDiscount(0, 0.1)).toBe(0);
  });
  it("handles 100% discount", () => {
    expect(calculateDiscount(100, 1.0)).toBe(0);
  });
  it("throws for negative rate", () => {
    expect(() => calculateDiscount(100, -0.1)).toThrow();
  });
});

T6 Test Thoroughly Around Bug Fixes

When you fix a bug, the area of code around the bug has already proven to be defect-prone. Add tests not just for the specific bug you fixed, but for related edge cases in the same function. Bugs rarely travel alone — they cluster in the areas of code that were the least well understood when they were written.

T7 & T8 Analyze Failure and Coverage Patterns

Test failures often cluster. A pattern of failures in the same area of code reveals a design or understanding problem, not just a bug. Similarly, coverage gaps tend to cluster in code paths that were not well-understood. Use these patterns as signals for where deeper design attention is needed.

T9 Maintain Fast Tests

Slow tests are tests that don't get run during development. A test suite that takes 10 minutes will only be run before commits, not continuously. A suite that takes 10 seconds will be run after every change. Keep unit tests fast by avoiding I/O, database calls, and network access — mock those boundaries and test logic in isolation.

The Test Speed Hierarchy
Unit tests should run in milliseconds. Integration tests in seconds. End-to-end tests in minutes. If your unit tests are slow, they probably aren't unit tests — they are integration tests in disguise. Find the I/O and mock it.
Conclusion

The Value System Behind the Rules

This list of heuristics could hardly be said to be complete. But completeness is not the goal — the goal is to imply a value system.

Clean code is not written by memorising a checklist. You don't become a software craftsperson by learning sixty rules. These heuristics are symptoms of values: care for the reader, respect for future maintainers (including your future self), intellectual honesty about what code actually does, and a commitment to leaving the codebase better than you found it.

"Professionalism and craftsmanship come from values that drive disciplines — not from following rules."
— Robert C. Martin, Clean Code
