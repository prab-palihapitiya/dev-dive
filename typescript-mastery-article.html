01 — Core Fundamentals

Primitive Types & Type Annotations

TypeScript is a superset of JavaScript. Every valid JS file is valid TS, but TypeScript adds a powerful static type system on top.

Type annotations tell TypeScript the "shape" of your data. The compiler checks these at build time — meaning bugs are caught before your code ever runs in a browser or Node server.

TypeScript 
// ── Primitive types ─────────────────────────────
let username: string  = "Alice"
let age:      number  = 30
let active:   boolean = true
let nothing:  null    = null
let undef:    undefined = undefined
let big:      bigint  = 9007199254740991n
let sym:      symbol  = Symbol("unique")

// ── Arrays ──────────────────────────────────────
const scores: number[]        = [90, 85, 92]
const names:  Array<string>   = ["Alice", "Bob"]

// ── Tuples ──────────────────────────────────────
const point: [number, number]          = [12.5, -8.3]
const entry: [string, number, boolean] = ["Alice", 30, true]

// ── Special types ───────────────────────────────
let anything: any     = "no type safety"  // ⚠ avoid
let flexible: unknown = fetchData()       // safe version of any
let neverVal: never            // unreachable code / exhaustive checks
function log(msg: string): void { console.log(msg) }
⚠ Warning — Avoid any
Using any disables all TypeScript checks for that variable. Prefer unknown when the type is truly unknown — it forces you to narrow before use.

Type Inference

TypeScript infers types automatically — you don't need to annotate everything explicitly. Annotations are most valuable at API boundaries: function parameters and return types.

TypeScript 
// TypeScript infers `count` as number, `greeting` as string
let count    = 42
let greeting = "Hello"

// Array of strings is inferred too
const fruits = ["apple", "banana", "cherry"] // string[]

// Return type inferred as number
function add(a: number, b: number) {
  return a + b
}

// But explicit is better at boundaries:
function getUser(id: string): User {
  // ...
}
02 — Safety First

Type Narrowing & Type Guards

Narrowing is the process of refining a broad type into a more specific one. It's one of TypeScript's most powerful features.

TypeScript 
// ── typeof narrowing ────────────────────────────
function formatValue(val: string | number): string {
  if (typeof val === "string") {
    return val.toUpperCase()  // val is string here
  }
  return val.toFixed(2)      // val is number here
}

// ── instanceof narrowing ─────────────────────────
function handleError(err: unknown): string {
  if (err instanceof Error) return err.message
  if (typeof err === "string") return err
  return "Unknown error"
}

// ── Custom type guard (predicate) ────────────────
interface Cat { meow: () => void }
interface Dog { bark: () => void }

function isCat(animal: Cat | Dog): animal is Cat {
  return (animal as Cat).meow !== undefined
}

function makeSound(animal: Cat | Dog) {
  if (isCat(animal)) animal.meow()  // Cat
  else             animal.bark()  // Dog
}

// ── Discriminated unions ─────────────────────────
type Result<T> =
  | { status: "ok";    data:  T      }
  | { status: "error"; error: string }

function handleResult<T>(r: Result<T>) {
  switch (r.status) {
    case "ok":    console.log(r.data);  break
    case "error": console.log(r.error); break
  }
}
💡 Pro Tip — Exhaustive Checks
Use never in the default branch of a switch to get a compile error if you forget to handle a new union member.
TypeScript — Exhaustive Check 
function assertNever(x: never): never {
  throw new Error(`Unhandled case: ${x}`)
}

type Shape = Circle | Square | Triangle

function area(s: Shape): number {
  switch (s.kind) {
    case "circle":   return Math.PI * s.r ** 2
    case "square":   return s.side ** 2
    case "triangle": return 0.5 * s.base * s.height
    default:        return assertNever(s) // TS error if case missed
  }
}
03 — Enums & Literals

Enums, Literal Types & Union Types

TypeScript 
// ── String Enum ──────────────────────────────────
enum Direction {
  Up    = "UP",
  Down  = "DOWN",
  Left  = "LEFT",
  Right = "RIGHT",
}
const move: Direction = Direction.Up

// ── Const Enum (no runtime overhead) ────────────
const enum Status { Active, Inactive, Pending }
const s = Status.Active // compiles to `0`, no object created

// ── Literal Types ────────────────────────────────
type Alignment = "left" | "center" | "right"
type Dice      = 1 | 2 | 3 | 4 | 5 | 6
type YesNo     = true  | false

function align(a: Alignment) { /* ... */ }
align("left")   // ✓
align("middle") // ✗ Type error

// ── Template Literal Types ───────────────────────
type EventName = `on${Capitalize<string>}`
// "onClick" | "onHover" | "onChange" ...

type CSSProperty = `margin-${"top" | "right" | "bottom" | "left"}`
// "margin-top" | "margin-right" | "margin-bottom" | "margin-left"
💡 Prefer Literal Union over Enum
String literal unions like "asc" | "desc" are often cleaner than enums — they produce no runtime code and work seamlessly with JSON APIs.
04 — Functions

Functions, Overloads & Parameter Types

TypeScript 
// ── Basic function signature ─────────────────────
function greet(name: string, greeting?: string): string {
  return `${greeting ?? "Hello"}, ${name}!`
}

// ── Default parameters ───────────────────────────
function createUser(name: string, role: string = "viewer") {
  return { name, role }
}

// ── Rest parameters ──────────────────────────────
function sum(...nums: number[]): number {
  return nums.reduce((a, b) => a + b, 0)
}

// ── Function type as type alias ──────────────────
type Transformer<T, U> = (input: T) => U
const stringify: Transformer<number, string> = (n) => n.toString()

// ── Function Overloads ───────────────────────────
function createElement(tag: "a"): HTMLAnchorElement
function createElement(tag: "canvas"): HTMLCanvasElement
function createElement(tag: string): HTMLElement {
  return document.createElement(tag)
}

// ── Async functions ──────────────────────────────
async function fetchUser(id: number): Promise<User> {
  const res  = await fetch(`/api/users/${id}`)
  return res.json()
}
05 — OOP

Classes, Access Modifiers & OOP

TypeScript 
class BankAccount {
  // Access modifiers: public, private, protected, readonly
  readonly id: string
  public   owner: string
  private  balance: number
  protected interestRate: number

  // Shorthand constructor — declare + assign in one go
  constructor(
    public readonly  accountId:  string,
    private          _balance:   number = 0
  ) {
    this.id      = accountId
    this.balance = _balance
  }

  // Getter / Setter
  get currentBalance(): number { return this.balance }

  set currentBalance(val: number) {
    if (val < 0) throw new Error("Negative balance")
    this.balance = val
  }

  deposit(amount: number): this {  // return `this` for chaining
    this.balance += amount
    return this
  }

  static create(id: string): BankAccount {
    return new BankAccount(id)
  }
}

// Inheritance
class SavingsAccount extends BankAccount {
  constructor(id: string, private goal: number) {
    super(id)
  }
  get progress() { return this.currentBalance / this.goal }
}

// Abstract class
abstract class Animal {
  abstract speak(): string
  describe() { return `I say: ${this.speak()}` }
}
"TypeScript's class system lets you apply the principle of least privilege — expose only what callers need to know."
06 — Contracts

Interfaces & Type Aliases

Both interface and type can describe object shapes. Know when to use which.

Feature Interface Type Alias
Object shapes ✅ Primary use ✅ Supported
Declaration merging ✅ Yes ❌ No
Union types ❌ No ✅ Yes
Mapped types ❌ No ✅ Yes
Extends / implements ✅ Yes Limited via &
Error messages Often cleaner Can be verbose
TypeScript 
// ── Interface ────────────────────────────────────
interface User {
  readonly id:  number
  name:         string
  email?:       string        // optional
  role:         Role
  address:      Address       // nested type
}

// ── Declaration merging (unique to interface) ────
interface Window { analytics: Analytics }  // extends global Window

// ── Type aliases for complex types ───────────────
type ID        = string | number
type Callback  = (err: Error | null, data?: unknown) => void
type NonEmpty  = [string, ...string[]]   // non-empty array

// ── Intersection types ───────────────────────────
type AdminUser = User & { adminLevel: number }

// ── Index signatures ─────────────────────────────
interface Dict<V> {
  [key: string]: V
}
const scores: Dict<number> = { alice: 95, bob: 87 }

// ── Callable interface ───────────────────────────
interface Formatter {
  (value: string): string
  locale: string
}
07 — Generics

Generics: Write Once, Type Safely

Generics let you write reusable, type-safe code that works across many types — without losing type information.

TypeScript 
// ── Basic generic function ───────────────────────
function identity<T>(arg: T): T { return arg }

const num = identity(42)     // T inferred as number
const str = identity("hi")   // T inferred as string

// ── Constrained generics ─────────────────────────
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key]
}
const user  = { name: "Alice", age: 30 }
const uName = getProperty(user, "name") // string

// ── Generic interface & class ────────────────────
interface Repository<T> {
  findById(id: string):   Promise<T | null>
  save(entity: T):         Promise<T>
  delete(id: string):      Promise<void>
}

class Stack<T> {
  private items: T[] = []
  push(item: T)  { this.items.push(item) }
  pop(): T | undefined { return this.items.pop() }
  peek(): T | undefined { return this.items.at(-1) }
  get size() { return this.items.length }
}

// ── Generic with default ─────────────────────────
interface ApiResponse<T = unknown> {
  data:    T
  status:  number
  message: string
}
08 — Utility Types

Built-in Utility Types

TypeScript ships with powerful generic utilities. Learn these and you'll write far less boilerplate.

TypeScript — Object Utilities 
interface User {
  id:    number
  name:  string
  email: string
  age:   number
}

Partial<User>             // all props optional
Required<User>            // all props required
Readonly<User>            // all props readonly
Pick<User, "id" | "name">  // { id, name } subset
Omit<User, "age">          // everything except age
Record<string, User>       // { [key: string]: User }

// ── Real usage ───────────────────────────────────
type CreateUserInput = Omit<User, "id">
type UpdateUserInput = Partial<Omit<User, "id">>
type UserProfile    = Pick<User, "name" | "email">
TypeScript — Function & Conditional Utilities 
// ── Function utilities ───────────────────────────
function fetchUser(id: number): Promise<User> { /* ... */ }

type FetchParams   = Parameters<typeof fetchUser>    // [number]
type FetchReturn   = ReturnType<typeof fetchUser>    // Promise<User>
type Awaited       = Awaited<FetchReturn>           // User
type CtorInstance  = InstanceType<typeof BankAccount> // BankAccount

// ── String utilities ─────────────────────────────
type U = Uppercase<"hello">    // "HELLO"
type L = Lowercase<"HELLO">    // "hello"
type C = Capitalize<"hello">   // "Hello"
type UC= Uncapitalize<"Hello"> // "hello"

// ── Conditional utility ──────────────────────────
type NonNullable<T> = T extends null | undefined ? never : T
type Clean = NonNullable<string | null | undefined> // string
09 — Advanced

Advanced Types & Type-Level Programming

This is where TypeScript gets truly expressive — conditional types, mapped types, infer, and template literals give you a complete type-level programming language.

Mapped Types

TypeScript 
// ── Mapped types ─────────────────────────────────
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K]
}
// Getters<{name: string; age: number}>
// → { getName: () => string; getAge: () => number }

// Remove readonly modifier with -readonly
type Mutable<T> = { -readonly [K in keyof T]: T[K] }

// Remove optional with -?
type Concrete<T> = { [K in keyof T]-?: T[K] }

Conditional Types & infer

TypeScript 
// ── Conditional types ────────────────────────────
type IsArray<T> = T extends any[] ? true : false

type A = IsArray<string[]>  // true
type B = IsArray<string>    // false

// ── infer — extract types ────────────────────────
// Get element type from an array
type ElementOf<T> =
  T extends (infer Item)[] ? Item : T

type Num = ElementOf<number[]>   // number
type Str = ElementOf<string[]>   // string

// Get the resolved type of a Promise
type UnPromise<T> =
  T extends Promise<infer R> ? R : T

type UserFromPromise = UnPromise<Promise<User>>  // User

// Extract first arg type
type FirstArg<T extends (...args: any) => any> =
  T extends (first: infer F, ..._: any) => any ? F : never

type FA = FirstArg<(a: number, b: string) => void>  // number

Decorators (TypeScript 5.x)

TypeScript — Decorators 
// ── Method decorator ─────────────────────────────
function log(target: any, key: string, desc: PropertyDescriptor) {
  const orig = desc.value
  desc.value = function(...args: any[]) {
    console.log(`Calling ${key} with`, args)
    return orig.apply(this, args)
  }
}

function readonly(target: any, key: string, desc: PropertyDescriptor) {
  desc.writable = false
}

class Service {
  @log
  @readonly
  greet(name: string) {
    return `Hello, ${name}`
  }
}
10 — Best Practices

Pro Tips, Patterns & Common Pitfalls

These are the things experienced TypeScript developers know that beginners don't.

1. Use as const for literal inference

TypeScript 
// Without as const — type is string[]
const colors = ["red", "green", "blue"]

// With as const — type is readonly ["red","green","blue"]
const COLORS = ["red", "green", "blue"] as const
type Color = (typeof COLORS)[number] // "red" | "green" | "blue"

// Works great for config objects too
const CONFIG = {
  endpoint: "https://api.example.com",
  timeout:  5000,
} as const
type Endpoint = typeof CONFIG.endpoint  // literal string type

2. Branded / Nominal Types

TypeScript 
// Problem: these are both `number` — easy to confuse
function transfer(from: number, to: number, amount: number) {}

// Solution: brand types
type UserId = number & { readonly _brand: "UserId"  }
type USD    = number & { readonly _brand: "USD"     }

function asUserId(n: number): UserId { return n as UserId }
function asUSD(n:    number): USD    { return n as USD    }

function transfer(from: UserId, to: UserId, amount: USD) {}

const alice = asUserId(1)
const bob   = asUserId(2)
const amt   = asUSD(100)

transfer(alice, bob, amt)    // ✓ correct order
transfer(alice, amt, bob)    // ✗ Type error! amt ≠ UserId

3. Satisfies Operator (TypeScript 4.9+)

TypeScript 
type Palette = Record<string, string | [number, number, number]>

// satisfies validates the type but PRESERVES literal inference
const palette = {
  red:   [255, 0, 0],
  green: "#00ff00",
} satisfies Palette

// `palette.red` is still `[255,0,0]` not `string | [n,n,n]`
const r = palette.red[0]  // number ✓ (not an error!)

4. Template Literal Types for Typed APIs

TypeScript 
// Build typed event systems
type EntityEvent<T extends string> =
  `${T}:created` | `${T}:updated` | `${T}:deleted`

type UserEvent = EntityEvent<"user">
// "user:created" | "user:updated" | "user:deleted"

function on(event: UserEvent, handler: () => void) {}

on("user:created", () => {})  // ✓
on("user:login",   () => {})  // ✗ Type error

5. Avoid Type Assertions — Validate Instead

❌ Unsafe assertion
const data =
  await res.json()

// Pretending it's User
const user = data as User
// Crashes at runtime
console.log(user.name.toUpperCase())
✅ Runtime validation
import { z } from "zod"
const UserSchema = z.object({
  name: z.string(),
  age:  z.number(),
})
type User = z.infer<typeof UserSchema>
// Safe parse at runtime ✓
const user = UserSchema.parse(data)
💡 Use Zod, Valibot, or io-ts
These libraries let you define a schema once and get both the runtime validator AND the TypeScript type for free. This is the gold standard for handling external data (API responses, env vars, form inputs).

8. Variance & Covariance

TypeScript 5.x — Variance Annotations 
// in = contravariant (consumer), out = covariant (producer)
interface Animal   { name: string }
interface Dog extends Animal { breed: string }

// Covariant: produces T — safe to assign Dog[] to Animal[]
interface Producer<out T> { produce(): T }

// Contravariant: consumes T — a Dog handler can be used where Animal is expected
interface Consumer<in T>  { consume(val: T): void }
