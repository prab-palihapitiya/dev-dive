01 — The Landscape

The NoSQL Landscape

NoSQL is not one thing — it is a family of database paradigms that trade relational consistency guarantees for flexibility, horizontal scale, or specialised performance. Choosing the right one requires understanding the trade-offs of each.

📄

Document (MongoDB)

Stores JSON-like documents. Rich query language, flexible schema, naturally maps to JavaScript objects. Best for product catalogues, user profiles, CMS.

⚡

Key-Value (Redis)

Ultra-fast in-memory store with optional persistence. O(1) reads and writes. Best for caching, sessions, rate limiting, pub/sub, leaderboards.

🔍

Search (Elasticsearch)

Inverted-index engine built on Lucene. Full-text search, faceting, geospatial queries. Best for log analytics, search boxes, observability.

📊

Wide-Column (Cassandra)

Distributed, linear write scale. Query by partition key. Best for time-series, IoT data, activity feeds at massive scale.

🕸️

Graph (Neo4j)

Native graph storage and traversal. Efficient for highly-connected data. Best for social networks, fraud detection, recommendation engines.

🕐

Time-Series (InfluxDB)

Optimised for timestamped data ingestion and range queries. Best for metrics, monitoring, financial tick data, sensor streams.

The CAP Theorem in Practice

Every distributed system must choose two of three guarantees when a network partition occurs. MongoDB prioritises CP (consistency + partition tolerance) with replica sets. Redis Cluster prioritises AP (availability + partition tolerance) by default.

Database CAP Consistency Model Best Use Case
MongoDB CP Tunable (eventual → strong) General purpose, documents
Redis AP Eventual (Cluster) / Strong (single) Cache, sessions, pub/sub
Elasticsearch AP Near-real-time (~1s) Search, analytics, logs
Cassandra AP Tunable (quorum) High-write time-series
PostgreSQL CP ACID / Serializable Financial, relational data
ℹ️ SQL vs NoSQL — Not Either/Or
Modern production stacks frequently use both. PostgreSQL for financial transactions, MongoDB for product data, Redis for caching, and Elasticsearch for search is a completely normal architecture. Choose based on access patterns, not hype.
02 — Schema Design

Data Modelling & Schema Design

MongoDB is schema-flexible, not schema-free. Designing documents well — knowing when to embed vs reference, how to optimise for your read patterns, and how to handle relationships — is the skill that separates good from great MongoDB engineers.

Embed vs Reference

✅ Embed — One-to-few, read together 
// Always read order with its line items
// Items won't grow unboundedly
// No cross-document transactions needed
{
  _id:        ObjectId("..."),
  userId:     ObjectId("..."),
  status:     "paid",
  items: [            // embedded array
    { sku: "T-001", qty: 2, price: 29.99 },
    { sku: "H-042", qty: 1, price: 49.99 }
  ],
  total:      109.97
}
🔗 Reference — One-to-many, update independently 
// Post has potentially thousands of comments
// Comments can be queried independently
// Avoid 16MB document limit
{
  _id:      ObjectId("..."),
  title:    "My Post",
  authorId: ObjectId("..."), // ref to users
  tagIds:   [ObjectId("...")]   // ref to tags
  // comments live in comments collection
  // comments.postId = this._id
}
JavaScript — Schema Design Decision Tree 
// Embed when ALL of these are true:
//  ✓ Data is always accessed together with the parent
//  ✓ Array won't grow unboundedly (< a few hundred items)
//  ✓ Document won't exceed 16MB
//  ✓ Data is not needed independently
//
// Reference when ANY of these are true:
//  ✗ The sub-document has its own lifecycle (CRUD independently)
//  ✗ The array can grow to thousands of items
//  ✗ Multiple parent documents reference the same child
//  ✗ You need to query the sub-documents without the parent

// ── Hybrid: partial embed for hot fields ──────────
// Users collection — embed the 5 most recent orders
// Full order lives in orders collection
{
  _id:   ObjectId("..."),
  name:  "Alice",
  email: "alice@example.com",
  recentOrders: [   // denormalised snapshot — updated on order write
    { orderId: ObjectId("..."), total: 109.97, date: "2026-06-01" }
  ]
}

// ── Always design schema for your query patterns ──
// Ask: "What are the top 5 queries this collection serves?"
// Then design the document structure to make those queries
// require zero joins (lookups) and use indexes well.
03 — Query Language

CRUD & Query Operators

MongoDB's query language is expressive and composable. Mastery means knowing all the comparison, logical, element, array, and update operators — and which to reach for when.

JavaScript — Comprehensive Query Operators 
import { MongoClient, ObjectId } from 'mongodb'

const client = await MongoClient.connect(process.env.MONGO_URI)
const db = client.db('shop')
const products = db.collection('products')

// ── Comparison operators ──────────────────────────
await products.find({
  price:  { $gte: 10, $lte: 100 },   // range
  status: { $in: ['active', 'featured'] },
  sku:    { $nin: ['DEPRECATED'] },
  stock:  { $ne: 0 },
}).toArray()

// ── Logical operators ─────────────────────────────
await products.find({
  $or: [
    { category: 'electronics' },
    { tags: 'featured' },          // matches if 'featured' in tags array
  ],
  $and: [
    { price: { $gt: 0 } },
    { stock: { $gt: 0 } },
  ]
})

// ── Element operators ─────────────────────────────
await products.find({
  discount: { $exists: true },    // field must exist
  weight:   { $type: 'number' }, // type guard
})

// ── Array operators ───────────────────────────────
await products.find({
  tags:        { $all:      ['waterproof', 'outdoor'] }, // both must exist
  images:      { $size:     3 },                          // exactly 3 images
  'specs.dims':{ $elemMatch: { val: { $gt: 10 } } },     // at least one element matches
})

// ── Projection — retrieve only needed fields ──────
await products.find({}, {
  projection: { name: 1, price: 1, _id: 0 }
})

// ── Update operators — atomic mutations ──────────
await products.updateOne(
  { _id: new ObjectId(id) },
  {
    $set:   { status: 'archived', 'meta.updatedAt': new Date() },
    $unset: { discount: '' },            // remove field
    $inc:   { viewCount: 1 },            // atomic increment
    $push:  { tags: 'sale' },           // append to array
    $addToSet: { categories: 'clearance' }, // push if not exists
    $pull:  { tags: 'new-arrival' },   // remove from array
  }
)

// ── findOneAndUpdate — atomic read-modify ─────────
const updated = await products.findOneAndUpdate(
  { sku: 'T-001', stock: { $gte: 1 } },    // filter + condition
  { $inc: { stock: -1 } },                 // decrement atomically
  { returnDocument: 'after' }            // return updated doc
)
04 — Data Processing

The Aggregation Pipeline

The aggregation pipeline is MongoDB's most powerful feature — a multi-stage data processing framework that can transform, group, join, reshape, and analyse data with sub-millisecond latency on properly indexed collections.

JavaScript — Aggregation Pipeline Mastery 
// ── Stage reference ───────────────────────────────
// $match     — filter documents (use EARLY, hits indexes)
// $project   — reshape / add / remove fields
// $group     — aggregate + accumulate ($sum, $avg, $push)
// $sort      — sort the stream
// $limit     — take N documents
// $skip      — skip N documents (use with $sort for pagination)
// $lookup    — left outer join to another collection
// $unwind    — flatten an array into separate documents
// $addFields — add computed fields without removing others
// $replaceRoot— replace root with a sub-document
// $facet     — multiple aggregations in parallel
// $bucket    — range-based grouping

// ── Real-world example: sales dashboard ──────────
const salesReport = await db.collection('orders').aggregate([
  // Stage 1: filter — hits index on createdAt + status
  { $match: {
    createdAt: { $gte: new Date('2026-01-01') },
    status:    { $in: ['paid', 'shipped'] }
  }},

  // Stage 2: add computed fields
  { $addFields: {
    month: { $dateToString: { format: '%Y-%m', date: '$createdAt' } },
    margin: { $subtract: ['$total', '$costPrice'] }
  }},

  // Stage 3: group by month + category
  { $group: {
    _id:         { month: '$month', category: '$category' },
    revenue:     { $sum:   '$total'   },
    totalMargin: { $sum:   '$margin'  },
    orderCount:  { $sum:   1          },
    avgOrderVal: { $avg:   '$total'   },
    topSkus:     { $addToSet: '$sku'   }
  }},

  // Stage 4: reshape output
  { $project: {
    month:      '$_id.month',
    category:   '$_id.category',
    revenue:    { $round: ['$revenue', 2] },
    marginPct:  { $multiply: [{ $divide: ['$totalMargin', '$revenue'] }, 100] },
    orderCount: 1, avgOrderVal: 1, _id: 0
  }},

  { $sort: { month: -1, revenue: -1 } }
]).toArray()

// ── $lookup — left join to another collection ─────
const ordersWithUsers = await db.collection('orders').aggregate([
  { $match: { status: 'paid' } },
  { $lookup: {
    from:         'users',
    localField:   'userId',
    foreignField: '_id',
    as:           'user',
    pipeline: [          // pipeline lookup — project only needed fields
      { $project: { name: 1, email: 1, tier: 1 } }
    ]
  }},
  { $unwind: { path: '$user', preserveNullAndEmptyArrays: true } }
]).toArray()
05 — Query Optimisation

Indexes & Query Optimisation

Indexes are the single most impactful performance lever in MongoDB. A missing index on a 10M-document collection turns a 1ms query into a 10-second collection scan. Know every index type and when to apply it.

JavaScript — Index Types & Creation 
// ── Single field ──────────────────────────────────
await products.createIndex({ price: 1 })           // 1=asc, -1=desc

// ── Compound — field order matters! ──────────────
// Rule: Equality → Sort → Range (ESR rule)
await products.createIndex({
  category: 1,     // Equality  — always first
  createdAt: -1,   // Sort      — second
  price:    1      // Range     — last
})

// ── Unique constraint ─────────────────────────────
await db.collection('users').createIndex(
  { email: 1 },
  { unique: true, name: 'idx_users_email_unique' }
)

// ── Text index — full-text search ────────────────
await products.createIndex(
  { name: 'text', description: 'text' },
  { weights: { name: 10, description: 5 }, name: 'idx_text_search' }
)
// Query: { $text: { $search: "wireless headphones" } }

// ── Sparse — only index docs where field exists ──
await products.createIndex(
  { discount: 1 },
  { sparse: true }  // skip docs without 'discount' field
)

// ── TTL — auto-expire documents ───────────────────
await db.collection('sessions').createIndex(
  { expiresAt: 1 },
  { expireAfterSeconds: 0 }  // expire at the expiresAt date value
)

// ── 2dsphere — geospatial ─────────────────────────
await db.collection('stores').createIndex({ location: '2dsphere' })
// Query: { location: { $near: { $geometry: { type:'Point', coordinates:[lng,lat] }, $maxDistance: 5000 } } }

// ── EXPLAIN — read the query plan ─────────────────
const plan = await products
  .find({ category: 'electronics', price: { $lt: 500 } })
  .explain('executionStats')

// Look for:
//   winningPlan.stage === 'IXSCAN'   → good (index used)
//   winningPlan.stage === 'COLLSCAN' → bad (full scan)
//   executionStats.totalDocsExamined ≈ nReturned → good (index is selective)
⚠️ The ESR Rule
For compound indexes, always order fields as: Equality fields first, Sort fields second, Range fields last. A query filtering on category (equality), sorting by createdAt, and ranging on price needs the index in exactly that order to avoid an in-memory sort.
06 — ACID in MongoDB

Transactions & ACID Guarantees

MongoDB has supported multi-document ACID transactions since version 4.0. They work across collections, databases (within a replica set), and sharded clusters — but carry overhead that makes careful design the better first choice.

JavaScript — Multi-Document Transactions 
import { MongoClient } from 'mongodb'

async function transferFunds(fromId, toId, amount) {
  const session = client.startSession()

  try {
    await session.withTransaction(async () => {
      const accounts = db.collection('accounts')

      // Debit sender — fails if insufficient funds
      const from = await accounts.findOneAndUpdate(
        { _id: fromId, balance: { $gte: amount } },
        { $inc: { balance: -amount } },
        { session, returnDocument: 'after' }
      )
      if (!from) throw new Error('Insufficient funds')

      // Credit receiver
      await accounts.updateOne(
        { _id: toId },
        { $inc: { balance: amount } },
        { session }
      )

      // Log the transfer
      await db.collection('transfers').insertOne(
        { fromId, toId, amount, at: new Date() },
        { session }
      )
    }, {
      readConcern:  { level: 'snapshot' },  // MVCC snapshot isolation
      writeConcern: { w: 'majority' },      // durable on majority of nodes
    })
  } finally {
    await session.endSession()
  }
}
💡 Design away transactions first
Transactions in MongoDB carry 2–3× the latency of a regular write because of the distributed coordination. Before reaching for a transaction, ask: can I redesign the schema to make this a single atomic document update? The $set + $inc + $push operators on a single document are always atomic without transactions.
07 — ODM Layer

Mongoose & ODM Patterns

Mongoose adds schema validation, middleware hooks, virtuals, and a rich model API on top of the native driver. Used correctly it makes codebases dramatically more maintainable. Used naïvely it hides performance problems.

JavaScript — Production Mongoose Schema 
import mongoose from 'mongoose'
const { Schema, model } = mongoose

const productSchema = new Schema({
  name:     { type: String,  required: true, trim: true, maxlength: 200 },
  slug:     { type: String,  unique: true, lowercase: true },
  price:    { type: Number,  required: true, min: 0 },
  status:   { type: String,  enum: ['draft','active','archived'], default: 'draft' },
  tags:     [String],
  authorId: { type: Schema.Types.ObjectId, ref: 'User', required: true },
  meta:     {
    views:    { type: Number, default: 0 },
    featured: { type: Boolean, default: false },
  }
}, {
  timestamps: true,        // auto createdAt + updatedAt
  toJSON: {
    virtuals: true,
    transform: (_, ret) => { delete ret.__v; return ret }
  }
})

// ── Virtual — computed field, not stored ──────────
productSchema.virtual('url').get(function() {
  return `/products/${this.slug}`
})

// ── Pre-save hook — auto-generate slug ───────────
productSchema.pre('save', function(next) {
  if (this.isModified('name'))
    this.slug = this.name.toLowerCase().replace(/[^a-z0-9]+/g, '-')
  next()
})

// ── Post-delete hook — cascade clean-up ──────────
productSchema.post('findOneAndDelete', async function(doc) {
  if (doc) await Review.deleteMany({ productId: doc._id })
})

// ── Static method — reusable query builder ────────
productSchema.statics.findActive = function(filter = {}) {
  return this.find({ ...filter, status: 'active' })
}

// ── Avoid N+1: lean() + populate only what you need
const products = await Product
  .find({ status: 'active' })
  .select('name price slug authorId')     // projection
  .populate('authorId', 'name avatar')   // only 2 fields from user
  .lean()                                  // plain JS objects, 2-3× faster
  .limit(20)

export const Product = model('Product', productSchema)
08 — In-Memory Power

Redis — Caching & Data Structures

Redis is not just a cache — it is a multi-model data store with strings, hashes, lists, sorted sets, streams, pub/sub, and Lua scripting. Its O(1) complexity on most operations makes it the first line of defence against slow DB queries.

JavaScript — Redis Data Structures in Practice 
import { Redis } from 'ioredis'

const redis = new Redis(process.env.REDIS_URL)

// ── Caching pattern with stale-while-revalidate ──
async function cachedQuery(key, ttl, fetchFn) {
  const cached = await redis.get(key)
  if (cached) {
    const ttlLeft = await redis.ttl(key)
    // Revalidate in background when TTL < 20% remaining
    if (ttlLeft < ttl * 0.2)
      fetchFn().then(v => redis.set(key, JSON.stringify(v), 'EX', ttl)).catch(noop)
    return JSON.parse(cached)
  }
  const value = await fetchFn()
  await redis.set(key, JSON.stringify(value), 'EX', ttl)
  return value
}

// ── Hash — session store ──────────────────────────
await redis.hset(`session:${sessionId}`, {
  userId:   userId,
  role:     'admin',
  ip:       req.ip,
  loginAt:  Date.now(),
})
await redis.expire(`session:${sessionId}`, 86400)
const session = await redis.hgetall(`session:${sessionId}`)

// ── Sorted set — leaderboard ──────────────────────
await redis.zadd('leaderboard:weekly', score, userId)
const top10 = await redis.zrevrange('leaderboard:weekly', 0, 9, 'WITHSCORES')
const rank   = await redis.zrevrank('leaderboard:weekly', userId)

// ── Atomic rate limiter with Lua ──────────────────
const rateLimitScript = `
local key   = KEYS[1]
local limit = tonumber(ARGV[1])
local ttl   = tonumber(ARGV[2])
local count = redis.call('INCR', key)
if count == 1 then redis.call('EXPIRE', key, ttl) end
return count
`
const count = await redis.eval(rateLimitScript, 1, `rl:${ip}`, 100, 60)
if (count > 100) throw Object.assign(new Error('Rate limited'), { status: 429 })

// ── Pub/Sub — real-time notifications ────────────
const publisher  = new Redis(process.env.REDIS_URL)
const subscriber = new Redis(process.env.REDIS_URL)

await subscriber.subscribe('order:created')
subscriber.on('message', (channel, message) => {
  const order = JSON.parse(message)
  sendNotification(order.userId, 'Order confirmed!')
})
await publisher.publish('order:created', JSON.stringify(order))
09 — Full-Text Search

Elasticsearch & Full-Text Search

Elasticsearch delivers the search experience MongoDB's text indexes can't — relevance scoring, autocomplete, fuzzy matching, faceted navigation, and sub-second full-text search across billions of documents.

JavaScript — Elasticsearch Search Patterns 
import { Client } from '@elastic/elasticsearch'

const es = new Client({ node: process.env.ELASTICSEARCH_URL })

// ── Index mapping — define field types upfront ───
await es.indices.create({
  index: 'products',
  mappings: {
    properties: {
      name:        { type: 'text',    analyzer: 'english'      },
      name_raw:    { type: 'keyword'                         }, // for sorting/exact
      description: { type: 'text',    analyzer: 'english'      },
      price:       { type: 'float'                           },
      category:    { type: 'keyword'                         }, // facets
      tags:        { type: 'keyword'                         },
      location:    { type: 'geo_point'                       },
      suggest:     { type: 'completion'                      }, // autocomplete
    }
  }
})

// ── Full-text search with facets ──────────────────
const { hits, aggregations } = await es.search({
  index: 'products',
  query: {
    bool: {
      must: [{
        multi_match: {
          query:  userQuery,
          fields: ['name^3', 'description'],   // boost name 3×
          fuzziness: 'AUTO',                     // handle typos
        }
      }],
      filter: [
        { term:  { category: selectedCategory } },
        { range: { price: { gte: minPrice, lte: maxPrice } } }
      ]
    }
  },
  aggs: {
    categories: { terms:     { field: 'category', size: 20      } },
    priceRange:  { histogram: { field: 'price',    interval: 50 } },
  },
  highlight: { fields: { name: {}, description: {} } },
  from: page * pageSize, size: pageSize,
})

// ── Autocomplete with completion suggester ────────
const suggest = await es.search({
  index: 'products',
  suggest: {
    nameSuggest: {
      prefix:     partialQuery,
      completion: { field: 'suggest', size: 5, fuzzy: { fuzziness: 1 } }
    }
  }
})
10 — Advanced Patterns

Advanced Schema Patterns

The MongoDB documentation identifies 12 common schema design patterns. These five are the ones every production MongoDB engineer needs to know deeply.

JavaScript — The Bucket, Outlier & Computed Patterns 
// ── 1. BUCKET PATTERN — time-series data ─────────
// Instead of one document per measurement (millions of docs),
// bucket by time interval (thousands of docs)
{
  sensor_id:  "sensor-042",
  hour:       new Date("2026-06-07T14:00:00Z"),
  count:      60,            // readings in this bucket
  min:        18.2,
  max:        23.7,
  sum:        1247.4,        // pre-aggregated for fast avg
  readings:  [{ t: 0, v: 21.2 }, { t: 60, v: 21.5 }]  // offset seconds
}

// ── 2. OUTLIER PATTERN — unbounded arrays ────────
// Most posts have < 1000 likes. A few viral posts have millions.
// Don't let outliers destroy your schema.
{
  _id:        ObjectId("..."),
  content:   "...",
  likes:     [ObjectId("...") /* ... up to 1000 */],
  likeCount: 2_400_000,      // always accurate counter
  hasExtras: true             // flag: more likes in overflow collection
}
// Overflow docs in 'post_likes' collection: { postId, likes: [...] }

// ── 3. COMPUTED PATTERN — pre-aggregate hot reads ─
// Recalculate on write; serve pre-computed value on read
async function addReview(productId, rating) {
  await db.collection('reviews').insertOne({ productId, rating })
  // Update pre-computed avg atomically
  await db.collection('products').updateOne(
    { _id: productId },
    {
      $inc: { 'rating.count': 1, 'rating.sum': rating },
      $set: { 'rating.avg':
        { $round: [{ $divide:
          [{ $add: ['$$CURRENT.rating.sum', rating] },
           { $add: ['$$CURRENT.rating.count', 1] }
          ]
        }, 1]
      } }
    }
  )
}

// ── 4. TREE PATTERN — hierarchical data ──────────
// Materialized paths: fast subtree queries
[
  { _id: "root",   path: ",",                  name: "Electronics"      },
  { _id: "phones", path: ",root,",            name: "Phones"           },
  { _id: "apple",  path: ",root,phones,",     name: "Apple"            },
]
// Get all descendants: { path: /,root,/ } → regex match
await db.collection('categories').find({
  path: { $regex: `,root,` }
}).toArray()
11 — Tuning

Performance Tuning

MongoDB performance problems almost always fall into four categories: missing indexes, overfetching (retrieving more data than needed), write contention on hot documents, and unbounded collection scans in aggregations.

JavaScript — Performance Diagnostics & Fixes 
// ── Find slow queries in the profiler ────────────
await db.setProfilingLevel(1, { slowms: 100 })  // log queries > 100ms
const slowQueries = await db.collection('system.profile').find({
  millis:     { $gt: 100 },
  'planSummary': { $regex: 'COLLSCAN' },   // full scans
}).sort({ millis: -1 }).limit(10).toArray()

// ── Check index usage ─────────────────────────────
const stats = await db.collection('products').aggregate([
  { $indexStats: {} }
]).toArray()
// Indexes with accesses.ops === 0 are unused — drop them!

// ── Bulk writes — batch inserts/updates ──────────
const bulk = products.initializeUnorderedBulkOp()
for (const p of productUpdates) {
  bulk.find({ sku: p.sku }).updateOne({
    $set: { price: p.price, stock: p.stock }
  })
}
await bulk.execute()  // single round-trip for all updates

// ── Connection pooling ────────────────────────────
const client = new MongoClient(uri, {
  maxPoolSize:      50,      // default: 100
  minPoolSize:      5,       // keep alive connections
  maxIdleTimeMS:    30_000,
  serverSelectionTimeoutMS: 3_000,
  socketTimeoutMS:  30_000,
})

// ── Projection — ALWAYS specify in production ─────
// Fetching 50 fields when you need 5 wastes bandwidth + memory
await products.find({}, {
  projection: { name: 1, price: 1, slug: 1, _id: 1 }
}).toArray()
Symptom Root Cause Fix
Query >100ms on small collection COLLSCAN — missing index Add compound index matching ESR rule
Aggregation slow on large collection $match not first, or after $project Always $match early to hit indexes
High write latency on one document Hot document contention Bucket pattern or atomic increment only
Memory error in aggregation Exceeds 100MB allowance Add { allowDiskUse: true } + check stages
Slow populate() in Mongoose N+1 round-trips Use $lookup aggregation or batch refs
12 — Operations

Production Operations

Running MongoDB in production means replica sets for availability, read preferences for scale, change streams for real-time events, and a disciplined backup and monitoring strategy.

JavaScript — Change Streams & Read Preferences 
// ── Change streams — real-time CDC ────────────────
// Listen to any insert/update/delete on a collection
const changeStream = db.collection('orders').watch([
  { $match: { 'operationType': { $in: ['insert', 'update'] } } },
  { $match: { 'fullDocument.status': 'paid' } }
], { fullDocument: 'updateLookup' })

for await (const change of changeStream) {
  const order = change.fullDocument
  await fulfillOrder(order)
  // Store resume token — restart from here if process crashes
  await saveResumeToken(change._id)
}

// Resume after crash — no missed events
const resumeToken = await loadResumeToken()
const stream = collection.watch([], { resumeAfter: resumeToken })

// ── Read preferences — scale reads across replica ─
// 'primary'          — always latest data (default)
// 'primaryPreferred' — primary if available, else secondary
// 'secondary'        — offload reads to secondaries
// 'nearest'          — lowest latency node

const analyticsDb = client.db('shop', {
  readPreference: 'secondary'  // heavy aggregations — don't hit primary
})

// ── Write concern — durability tuning ────────────
await db.collection('payments').insertOne(doc, {
  writeConcern: {
    w:      'majority',  // durable on majority of replica nodes
    j:      true,         // wait for journal flush
    wtimeout: 5000        // fail if majority not acknowledged in 5s
  }
})
🚨 Never run a standalone MongoDB in production
A single-node MongoDB has no replication, no automatic failover, and no oplog. A replica set with at least three nodes (primary + 2 secondaries, or primary + secondary + arbiter) is the minimum for any production workload. Use MongoDB Atlas or a managed service to skip the ops burden.

Production Readiness Checklist

  • ✅
    Replica set deployed — minimum 3 nodes. Write concern majority on critical writes. Automatic failover tested.
  • ✅
    All queries use indexes — EXPLAIN run on every query in hot paths. No COLLSCAN in production profiles. Unused indexes identified and dropped.
  • ✅
    Connection pooling configured — maxPoolSize set to floor(DB_max_connections / app_instances). Idle timeout and server selection timeout set.
  • ✅
    Schema validation enabled — $jsonSchema validator on all collections. Prevents garbage data at the DB layer, not just application layer.
  • ✅
    Backups automated — daily mongodump or continuous Atlas backups. Point-in-time restore tested. Backup encryption enabled.
  • ✅
    Slow query profiler active — slowms: 100 in production. Alerts on new COLLSCAN patterns. Weekly index review scheduled.
  • ✅
    Authentication & TLS enforced — SCRAM-SHA-256 auth. TLS on all connections. IP allowlist on Atlas. Principle of least privilege per application user.
  • ✅
    TTL indexes on ephemeral data — sessions, tokens, rate-limit keys, soft-delete records all expire automatically.
