01 — The Big Picture

The Three-Layer Architecture

A design system is not a component library. It is a product that ships products. It has three distinct layers, and confusing them is the primary reason design systems stall, fork, or get abandoned.

Most teams start by building a component library and calling it a design system. Components are the most visible layer, so the mistake is understandable — but components without a token foundation are impossible to theme, and components without documented usage patterns are impossible to adopt consistently. The three layers must be built in order: tokens first, then components, then patterns.

LAYER 3 — PATTERNS & DOCUMENTATION ↓ usage guidelines · page templates · composition rules · content models LAYER 2 — COMPONENTS ↓ atoms · molecules · organisms · hooks · utilities LAYER 1 — DESIGN TOKENS primitives · semantic tokens · component tokens · mode tokens
🪙

Design Tokens

Named, purposeful decisions: colors, spacing, radii, shadows, transitions. Platform-agnostic JSON that compiles to CSS variables, iOS Swift, Android XML.

🧱

Components

Reusable UI units built entirely from tokens. Never hard-coded values inside components — every visual decision traces back to a token.

📐

Patterns

Opinionated recipes: how to compose components into common flows — forms, navigation, data tables, onboarding. Answers the "when" and "why", not just "what".

📖

Documentation

The system's source of truth. Living Storybook stories, token visualizers, do/don't comparisons, accessibility notes, and API references.

"If a designer can change a decision in Figma without needing an engineer to ship it, you have a design system. If they can't, you have a component library."

The technical implication of this distinction is that tokens must exist as a source of truth that both the design tool (Figma variables) and the codebase (CSS custom properties, JS constants) consume — compiled from the same JSON, not maintained in parallel.

02 — The Foundation

Design Tokens in Depth

Tokens are named design decisions. They are the atoms of your system — not "blue" or "16px" but what that value means. The W3C Design Token Community Group format defines three tiers: primitive, semantic, and component-specific.

The Three-Tier Token Hierarchy

TierExample TokenExample Value
Primitive
Raw scale values color.cyan.600 #0891b2
Semantic
Intent-named aliases color.action.default {color.cyan.600}
Component
Scoped overrides button.primary.bg {color.action.default}

Primitives are never used directly in components. Components consume semantic tokens. Semantic tokens reference primitives. This indirection is what makes theming possible: you can swap an entire brand by re-mapping semantic tokens to different primitives — without touching a single component.

W3C Token Format & Style Dictionary

JSON — tokens/primitives/color.json
{
  "color": {
    "cyan": {
      "50":  { "$value": "#ecfeff", "$type": "color" },
      "100": { "$value": "#cffafe", "$type": "color" },
      "600": { "$value": "#0891b2", "$type": "color" },
      "700": { "$value": "#0e7490", "$type": "color" },
      "900": { "$value": "#164e63", "$type": "color" }
    },
    "neutral": {
      "0":   { "$value": "#ffffff", "$type": "color" },
      "900": { "$value": "#111827", "$type": "color" }
    }
  }
}
JSON — tokens/semantic/color.json
{
  "color": {
    "action": {
      "default": { "$value": "{color.cyan.600}" },
      "hover":   { "$value": "{color.cyan.700}" },
      "subtle":  { "$value": "{color.cyan.50}"  }
    },
    "surface": {
      "default": { "$value": "{color.neutral.0}"   },
      "raised":  { "$value": "{color.neutral.0}"   },
      "sunken":  { "$value": "{color.neutral.50}"  }
    },
    "text": {
      "primary":  { "$value": "{color.neutral.900}" },
      "secondary":{ "$value": "{color.neutral.600}" },
      "inverse":  { "$value": "{color.neutral.0}"   }
    }
  }
}
JavaScript — style-dictionary.config.mjs

💡 One token source, every platform
Style Dictionary compiles the same JSON into CSS custom properties, JavaScript ES6 exports, Swift structs, and Android resources in a single build step. The design tool (Figma) reads from the same JSON via the Tokens Studio plugin. Every platform stays in sync automatically.
03 — Text & Readability

Typography System

A typography system is not a list of font sizes. It is a set of named text roles — each with a fixed combination of size, line height, weight, and letter spacing — that map to the semantic meaning of the content, not to its visual outcome.

Modular Type Scale

A modular scale generates harmonious size steps from a base size and a ratio. The most common ratios are 1.25 (Major Third), 1.333 (Perfect Fourth), and 1.5 (Perfect Fifth). Each step multiplies the previous by the ratio, producing a coherent visual rhythm across headings and body text.

JSON — tokens/typography/scale.json
{
  "font": {
    "family": {
      "display": { "$value": "'Lora', Georgia, serif"    },
      "body":    { "$value": "'Outfit', system-ui, sans-serif" },
      "mono":    { "$value": "'JetBrains Mono', monospace"   }
    },
    "size": {
      // Base: 16px, Ratio: 1.25 (Major Third)
      "xs":   { "$value": "0.64rem"  },  // 10.24px
      "sm":   { "$value": "0.8rem"   },  // 12.8px
      "base": { "$value": "1rem"     },  // 16px
      "lg":   { "$value": "1.25rem"  },  // 20px
      "xl":   { "$value": "1.563rem" },  // 25px
      "2xl":  { "$value": "1.953rem" },  // 31.25px
      "3xl":  { "$value": "2.441rem" },  // 39px
      "4xl":  { "$value": "3.052rem" }   // 48.8px
    },
    "lineHeight": {
      "tight":  { "$value": "1.2" },
      "snug":   { "$value": "1.375" },
      "normal": { "$value": "1.5" },
      "relaxed":{ "$value": "1.75" }
    }
  }
}

Fluid Typography with CSS clamp()

Fluid typography removes the need for breakpoint-specific font size overrides. The clamp() function linearly interpolates between a minimum and maximum size across a defined viewport range, producing smooth scaling at every screen width.

CSS — Fluid type scale via clamp()
/*
  Formula: clamp(min, preferred, max)
  preferred = minSize + (maxSize - minSize) * ((100vw - minVW) / (maxVW - minVW))
  Simplified to a vw-based calc for CSS native use:
*/
:root {
  /* Display heading: 32px → 50px, 375px → 1280px viewport */
  --ds-text-display-size: clamp(2rem, 1.27rem + 3.09vw, 3.125rem);

  /* H1: 28px → 40px */
  --ds-text-h1-size: clamp(1.75rem, 1.31rem + 1.87vw, 2.5rem);

  /* Body: always 16px — never fluid */
  --ds-text-body-size: 1rem;
}

/* ── Semantic text roles (what components actually consume) ── */
.ds-text-display {
  font-family: var(--ds-font-family-display);
  font-size: var(--ds-text-display-size);
  font-weight: 600;
  line-height: var(--ds-font-lineHeight-tight);
  letter-spacing: -0.025em;
}
📚 Variable Fonts
If your brand allows it, a single variable font file replaces an entire weight family. A variable font with wght axis lets you use any weight from 100–900 in a single 80KB file instead of loading six separate 40KB files. Google Fonts serves variable variants for Outfit, Inter, and most modern typefaces.
04 — Visual Identity

Color System & Theming

A color system has two jobs: encode brand identity and communicate meaning. These are in constant tension. Solve both by separating the palette (raw values) from the semantic layer (what each color means in context) and the mode layer (how semantics shift between light and dark).

Palette → Semantic → Mode

CSS — Full color token cascade with dark mode
/* ── Layer 0: Raw palette (never use directly in components) ── */
:root {
  --palette-cyan-50:  #ecfeff;
  --palette-cyan-600: #0891b2;
  --palette-cyan-700: #0e7490;
  --palette-red-600:  #dc2626;
  --palette-green-600:#16a34a;
  --palette-neutral-0:  #ffffff;
  --palette-neutral-50: #f9fafb;
  --palette-neutral-900:#111827;
}

/* ── Layer 1: Semantic tokens (light mode defaults) ── */
:root {
  --ds-color-action-default:  var(--palette-cyan-600);
  --ds-color-action-hover:    var(--palette-cyan-700);
  --ds-color-surface-default: var(--palette-neutral-0);
  --ds-color-surface-sunken:  var(--palette-neutral-50);
  --ds-color-text-primary:    var(--palette-neutral-900);
  --ds-color-feedback-error:  var(--palette-red-600);
  --ds-color-feedback-success: var(--palette-green-600);
}

/* ── Layer 2: Mode overrides (dark theme) ── */
[data-theme="dark"] {
  --ds-color-action-default:  #22d3ee;  /* cyan-400 — passes 4.5:1 on dark bg */
  --ds-color-surface-default: #111827;
  --ds-color-surface-sunken:  #0f172a;
  --ds-color-text-primary:    #f9fafb;
}

/* ── Honour OS preference when no explicit toggle ── */
@media (prefers-color-scheme: dark) {
  :root:not([data-theme="light"]) {
    --ds-color-surface-default: #111827;
    --ds-color-text-primary:    #f9fafb;
    /* ...remaining overrides */
  }
}
⚠ Never create a dark mode by inverting
Inverting light colors for dark mode is a predictable failure. A cyan that reads at 4.6:1 on white may drop to 2.1:1 on a dark surface. Dark mode requires independently verified token values — not programmatic transformations. Always re-check contrast ratios for every semantic color in each mode.

Multi-brand Theming

When a single codebase must serve multiple brands (a SaaS product with white-labeling, or a company with sub-brands), CSS custom property cascading makes brand switching zero-JavaScript. The component code never changes; only the token values do.

CSS — Brand-specific token overrides via data-brand
/* Brand A (default — loaded in :root above) */

/* Brand B — loaded on the root element at runtime */
[data-brand="nova"] {
  --ds-color-action-default: #7c3aed;
  --ds-color-action-hover:   #6d28d9;
  --ds-font-family-display:  "Playfair Display", Georgia, serif;
  --ds-radius-base:          2px;   /* brand nova is angular */
}

/* Switching brands in React — no component code touched */
JSX — ThemeProvider with brand switching
import { createContext, useContext, useEffect } from 'react'

const ThemeCtx = createContext({})

export function ThemeProvider({ brand = 'default', mode = 'light', children }) {
  useEffect(() => {
    const root = document.documentElement
    root.setAttribute('data-brand', brand)
    root.setAttribute('data-theme', mode)
  }, [brand, mode])

  return (
    <ThemeCtx.Provider value={{ brand, mode }}>
      {children}
    </ThemeCtx.Provider>
  )
}

export const useTheme = () => useContext(ThemeCtx)
05 — Structure & Rhythm

Spacing, Grid & Layout

Spacing is the most underestimated token category. Inconsistent whitespace is the single most common reason a UI looks "off" without anyone being able to name why. An 8pt grid system eliminates the problem at its source.

The 8pt Grid System

Every spacing value in the system is a multiple of 8. The base unit of 8px was chosen because it divides evenly into the most common screen resolutions, aligns with the standard line-height rhythms for 16px body text (16 = 8×2), and produces a consistent vertical rhythm when applied uniformly. 4px is permitted as a half-unit for tight internal padding only.

JSON — tokens/spacing.json
{
  "space": {
    "0":   { "$value": "0"       },
    "1":   { "$value": "0.25rem" },  // 4px  — tight internal padding
    "2":   { "$value": "0.5rem"  },  // 8px  — base unit
    "3":   { "$value": "0.75rem" },  // 12px
    "4":   { "$value": "1rem"    },  // 16px
    "6":   { "$value": "1.5rem"  },  // 24px
    "8":   { "$value": "2rem"    },  // 32px
    "12":  { "$value": "3rem"    },  // 48px
    "16":  { "$value": "4rem"    },  // 64px
    "24":  { "$value": "6rem"    },  // 96px
    "32":  { "$value": "8rem"    }   // 128px — section separators
  },
  "radius": {
    "none": { "$value": "0"      },
    "sm":   { "$value": "4px"    },
    "base": { "$value": "8px"    },
    "lg":   { "$value": "12px"   },
    "full": { "$value": "9999px" }
  }
}

Container Queries for Component-level Responsiveness

CSS container queries are now widely supported and fundamentally change how responsive components work inside a design system. Instead of responding to the viewport (which depends on where a component happens to be placed), a component responds to the size of its immediate container. This makes components genuinely portable.

CSS — Container query responsive card
/* Mark the parent as a containment context */
.ds-card-wrapper {
  container-type: inline-size;
  container-name: card;
}

/* Card adapts to its container, not the viewport */
.ds-card {
  display: flex;
  flex-direction: column;
  gap: var(--ds-space-3);
}

@container card (min-width: 400px) {
  .ds-card {
    flex-direction: row;
    align-items: center;
  }
  .ds-card__image {
    width: 160px;
    flex-shrink: 0;
  }
}

@container card (min-width: 600px) {
  .ds-card__image { width: 240px; }
}
06 — Building Blocks

Atomic Component Architecture

Atomic design, introduced by Brad Frost, provides the right vocabulary for component hierarchy: atoms are the smallest possible units, molecules are composed of atoms, and organisms are composed of molecules. In a React design system, this maps directly to implementation decisions about scope, API surface, and import paths.

LevelExamplesRule
Atom Button, Input, Label, Badge, Icon No component dependencies. Uses only tokens and HTML.
Molecule FormField, SearchBar, Card, Tooltip Composes 2–5 atoms. Has a single, clear job.
Organism Navbar, DataTable, CommentThread Composed of molecules. May connect to context or store.
Template PageLayout, DashboardShell Wires organisms into a page structure. No business logic.

Compound Components Pattern

For complex stateful widgets like Tabs, Accordions, Dropdowns, or Selects, the compound component pattern gives consumers a declarative, composable API while keeping internal state encapsulated. It is the pattern used by Radix UI, HeadlessUI, and React Aria.

JSX — Compound Select component with Context
import { createContext, useContext, useState, useId } from 'react'

const SelectCtx = createContext(null)

// ── Root: owns all state ──────────────────────────────────
function Select({ value, onChange, children }) {
  const [open, setOpen] = useState(false)
  const id = useId()

  return (
    <SelectCtx.Provider value={{ value, onChange, open, setOpen, id }}>
      <div className="ds-select" role="combobox"
        aria-haspopup="listbox" aria-expanded={open}
        aria-owns={`${id}-listbox`}
      >
        {children}
      </div>
    </SelectCtx.Provider>
  )
}

// ── Trigger: renders the visible control ──────────────────
function SelectTrigger({ placeholder = 'Select…', children }) {
  const { value, open, setOpen } = useContext(SelectCtx)
  return (
    <button type="button" className="ds-select__trigger"
      onClick={() => setOpen(o => !o)}
    >
      {value ?? placeholder}
      <span aria-hidden="true">{open ? '▲' : '▼'}</span>
    </button>
  )
}

// ── Content: the dropdown panel ───────────────────────────
function SelectContent({ children }) {
  const { open, id } = useContext(SelectCtx)
  if (!open) return null
  return (
    <ul id={`${id}-listbox`} role="listbox" className="ds-select__list">
      {children}
    </ul>
  )
}

// ── Item: each selectable option ──────────────────────────
function SelectItem({ value: itemValue, children }) {
  const { value, onChange, setOpen } = useContext(SelectCtx)
  const isSelected = value === itemValue
  return (
    <li role="option" aria-selected={isSelected}
      className={`ds-select__item ${isSelected ? 'is-selected' : ''}`}
      onClick={() => { onChange(itemValue); setOpen(false) }}
    >
      {children}
    </li>
  )
}

// ── Export as namespace ───────────────────────────────────
Select.Trigger  = SelectTrigger
Select.Content = SelectContent
Select.Item    = SelectItem

// ── Consumer usage ────────────────────────────────────────
function Example() {
  const [lang, setLang] = useState(null)
  return (
    <Select value={lang} onChange={setLang}>
      <Select.Trigger placeholder="Choose language" />
      <Select.Content>
        <Select.Item value="ts">TypeScript</Select.Item>
        <Select.Item value="js">JavaScript</Select.Item>
        <Select.Item value="py">Python</Select.Item>
      </Select.Content>
    </Select>
  )
}
07 — Developer Experience

Component API Design

The component API is the interface between your design system and every product team that consumes it. A bad API generates workarounds that spread through codebases and become impossible to deprecate. A good API is predictable, composable, and constrains choices toward the right answer.

Variant-first Props with cva()

The class-variance-authority library provides a typed, composable way to define visual variants. It produces a single function that maps props to class strings, keeping component code clean and providing TypeScript autocomplete for every variant slot.

TypeScript — Button with cva variants
import { cva, VariantProps } from 'class-variance-authority'
import { forwardRef, ButtonHTMLAttributes } from 'react'

const button = cva(
  // Base classes applied to every variant
  'inline-flex items-center justify-center font-medium rounded-base'
  + ' transition-colors focus-visible:outline-none focus-visible:ring-2'
  + ' disabled:pointer-events-none disabled:opacity-50',
  {
    variants: {
      variant: {
        primary:   'bg-action text-inverse hover:bg-action-hover',
        secondary: 'bg-surface border border-border hover:bg-sunken',
        ghost:     'hover:bg-sunken hover:text-primary',
        danger:    'bg-error text-inverse hover:bg-error-hover',
      },
      size: {
        sm:   'h-8  px-3 text-sm gap-1.5',
        base: 'h-10 px-4 text-sm gap-2',
        lg:   'h-12 px-6 text-base gap-2.5',
      },
      loading: {
        true: 'cursor-wait',
      }
    },
    defaultVariants: {
      variant: 'primary',
      size:    'base',
    }
  }
)

type ButtonProps =
  ButtonHTMLAttributes<HTMLButtonElement>
  & VariantProps<typeof button>
  & { loading?: boolean }

export const Button = forwardRef<HTMLButtonElement, ButtonProps>(
  ({ variant, size, loading, disabled, className, children, ...props }, ref) => {
    return (
      <button
        ref={ref}
        className={button({ variant, size, loading, className })}
        disabled={disabled || loading}
        aria-busy={loading}
        {...props}
      >
        {loading && <Spinner aria-hidden="true" />}
        {children}
      </button>
    )
  }
)
Button.displayName = 'Button'

Polymorphic Components with the as Prop

A polymorphic component renders as a different HTML element or custom component based on the as prop. This is the correct way to build a Text or Box primitive that can render as a <p>, <span>, or <h1> without duplicating prop types or losing TypeScript inference.

TypeScript — Polymorphic Text primitive
import { ComponentPropsWithRef, ElementType } from 'react'

// ── Generic helper type ───────────────────────────────────
type PolymorphicProps<
  E extends ElementType,
  P = {}
> = ComponentPropsWithRef<E> & {
  as?: E
} & P

// ── Text-specific variant props ───────────────────────────
type TextVariant = 'display' | 'h1' | 'h2' | 'body' | 'caption'
type TextOwnProps = { variant?: TextVariant; muted?: boolean }

const variantMap: Record<TextVariant, ElementType> = {
  display: 'h1', h1: 'h1', h2: 'h2',
  body: 'p', caption: 'span'
}

export function Text<E extends ElementType = 'p'>({
  as,
  variant = 'body',
  muted,
  className,
  ...props
}: PolymorphicProps<E, TextOwnProps>) {
  // 'as' wins; then infer from variant; fallback to 'p'
  const Component = as ?? variantMap[variant] ?? 'p'

  return (
    <Component
      className={[`ds-text-${variant}`, muted && 'ds-text--muted', className]
        .filter(Boolean).join(' ')}
      {...props}
    />
  )
}

// Usage — TypeScript knows which HTML attrs are valid on each element:
// <Text variant="h1">Title</Text>                   → renders <h1>
// <Text as="label" htmlFor="id">Label</Text>        → renders <label>
// <Text as="time" dateTime="2026-06">June</Text>    → renders <time>
❌ Prop surface too wide
// Escape hatch = API contract broken
<Button
  primaryColor="#ff0000"
  textSize="20px"
  marginTop="12px"
  borderWidth="2px"
/>
✓ Constrained variant API
// Consumers pick from known variants
<Button
  variant="danger"
  size="sm"
>
  Delete
</Button>
⚠ Avoid style props on design system components
Accepting arbitrary style, color, or dimension props on design system atoms defeats the purpose of the system. If a consumer needs something the current variants don't cover, that is a signal to add a variant — not to expose an escape hatch. Escape hatches accumulate and eventually replace the system.
08 — Long-term Health

Governance & Documentation

The technical layer of a design system accounts for perhaps 30% of its success. The remaining 70% is governance: who decides what goes in, how changes are communicated, and how the system stays relevant as products evolve. Systems that skip this rot within two years.

Semantic Versioning for Design Systems

Design system versions have a stronger social contract than typical library semver because consuming teams commit design and engineering resources to upgrades. A clear versioning policy reduces upgrade anxiety and prevents teams from pinning to old versions indefinitely.

VersionWhen to bumpExamples
MAJOR Breaking API change, removed prop, token rename Renaming variant="primary" → "brand"; removing size="xs"
MINOR New component, new variant, new token Adding <Stepper>; adding size="2xl"; adding semantic color token
PATCH Bug fix, visual tweak that doesn't change API Fixing focus ring offset; correcting a contrast ratio; fixing a broken SSR prop

Contribution Checklist

  • ✅Token-first: Every new visual decision is a token before it is a component prop. No hard-coded values anywhere in the system.
  • ✅Accessibility by default: Every interactive component has correct ARIA roles, keyboard navigation, and passes axe. No component merges without a passing jest-axe test.
  • ✅API review: New props are reviewed by at least one consuming team before merge. Prop names follow the system's conventions (variant, size, as) not the implementor's preference.
  • ✅Stories before code: Write all Storybook stories first. If you can't write a story for a state, you don't yet understand what you're building.
  • ✅Visual regression tests: Chromatic (or equivalent) runs on every PR and blocks merge on unreviewed visual changes to existing components.
  • ✅Changelog discipline: Every PR includes a CHANGELOG.md entry categorised as Added, Changed, Deprecated, Removed, Fixed, or Security.
  • ✅Deprecation period: Removed APIs get a deprecation notice in a minor version, with a codemod or migration guide, before removal in the next major.
  • ✅Office hours: Design system team holds weekly open office hours. Consuming teams should never resort to forking or monkey-patching because they couldn't get a question answered.
"A design system is never finished. It reflects the living design language of your organisation. When it stops evolving, it starts dying."
