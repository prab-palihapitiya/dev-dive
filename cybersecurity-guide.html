01 — Identity

Authentication & Authorization

AuthN proves who you are. AuthZ decides what you can do. Confusing the two — or implementing either badly — is the root cause of most breach headlines.

JWT Architecture Done Right

JSON Web Tokens are stateless and portable — but misuse is rampant. The three cardinal rules: always verify the signature, never trust the alg header blindly, and keep access tokens short-lived.

JavaScript — Secure JWT Implementation
import { SignJWT, jwtVerify, importJWK } from 'jose'  // prefer jose over jsonwebtoken
import crypto from 'node:crypto'

// ── Key generation — RS256 asymmetric (preferred for APIs) ──
// Private key signs tokens; public key verifies them.
// You can publish the public key as JWKS; services verify without your secret.
const { privateKey, publicKey } = await crypto.subtle.generateKey(
  { name: 'RSASSA-PKCS1-v1_5', modulusLength: 2048, hash: 'SHA-256',
    publicExponent: new Uint8Array([1, 0, 1]) },
  true, ['sign', 'verify']
)

// ── Issue access + refresh token pair ────────────
async function issueTokens(userId, roles) {
  const now = Math.floor(Date.now() / 1000)

  const access = await new SignJWT({ sub: userId, roles, type: 'access' })
    .setProtectedHeader({ alg: 'RS256' })   // never 'none', never trust client alg
    .setIssuedAt(now)
    .setExpirationTime('15m')              // short-lived — 15 minutes max
    .setJti(crypto.randomUUID())           // unique ID for revocation
    .sign(privateKey)

  const refresh = await new SignJWT({ sub: userId, type: 'refresh' })
    .setProtectedHeader({ alg: 'RS256' })
    .setExpirationTime('7d')
    .setJti(crypto.randomUUID())
    .sign(privateKey)

  return { access, refresh }
}

// ── Verification with audience + issuer checks ────
async function verifyAccessToken(token) {
  const { payload } = await jwtVerify(token, publicKey, {
    algorithms: ['RS256'],     // whitelist — reject 'none' or 'HS256'
    issuer:     'https://auth.myapp.com',
    audience:   'https://api.myapp.com',
  })
  if (payload.type !== 'access') throw new Error('Wrong token type')
  return payload
}
🚨 Critical Vulnerability — alg:none Attack
Some JWT libraries honour {"alg":"none"} in the header, accepting unsigned tokens as valid. Always specify an algorithm allowlist server-side. Never trust the algorithm from the incoming token header.

Role-Based & Attribute-Based Access Control

JavaScript — RBAC & ABAC Middleware
// ── RBAC: simple role checks ──────────────────────
const requireRole = (...roles) => (req, res, next) => {
  const userRoles = req.user?.roles ?? []
  if (!roles.some(r => userRoles.includes(r)))
    return res.status(403).json({ error: 'Forbidden' })
  next()
}
router.delete('/posts/:id', authenticate, requireRole('admin', 'moderator'), deletePost)

// ── ABAC: fine-grained attribute checks ───────────
class PolicyEngine {
  #policies = []

  addPolicy(resource, action, condition) {
    this.#policies.push({ resource, action, condition })
    return this
  }

  can(user, action, resource) {
    return this.#policies.some(p =>
      p.resource === resource.type &&
      p.action   === action &&
      p.condition(user, resource)
    )
  }
}

const policy = new PolicyEngine()
  .addPolicy('post', 'delete', (user, post) =>
    user.id === post.authorId || user.roles.includes('admin')
  )
  .addPolicy('billing', 'view', (user, _) =>
    user.subscription === 'pro' && user.emailVerified
  )

// Usage: fail loudly if unauthorised
if (!policy.can(req.user, 'delete', post))
  return res.status(403).json({ error: 'Forbidden' })

Multi-Factor Authentication (MFA)

JavaScript — TOTP MFA (RFC 6238)
import { TOTP } from 'otpauth'
import QRCode  from 'qrcode'

// ── Enrol user — generate secret ─────────────────
async function enrolMFA(userId, email) {
  const totp = new TOTP({
    issuer:    'MyApp',
    label:     email,
    algorithm: 'SHA1',
    digits:    6,
    period:    30,
  })

  const secret = totp.secret.base32   // store encrypted in DB
  const qrUrl  = await QRCode.toDataURL(totp.toString())

  await db.users.update({
    where: { id: userId },
    data:  { mfaSecret: await encryptSecret(secret), mfaEnabled: false }
  })
  return { qrUrl, backupCodes: await generateBackupCodes(userId) }
}

// ── Verify TOTP token ─────────────────────────────
function verifyMFA(secret, token) {
  const totp = new TOTP({ secret })
  // window:1 allows ±30s clock drift
  const delta = totp.validate({ token, window: 1 })
  return delta !== null
}
02 — Attack Surface

OWASP Top 10 (2025) for JavaScript Engineers

The OWASP Top 10 is not a checklist to tick once — it is a map of where attackers look first. Know each category deeply enough to find it in a code review.

A01 — CRITICAL
Broken Access Control

Insecure direct object references, missing AuthZ checks, CORS misconfiguration. Most common critical finding in pen tests.

A02 — CRITICAL
Cryptographic Failures

Weak algorithms, hardcoded keys, plaintext secrets in logs, HTTP for sensitive data, predictable IVs.

A03 — CRITICAL
Injection

SQL, NoSQL, OS command, LDAP, and template injection. Caused by untrusted data reaching an interpreter.

A04 — HIGH
Insecure Design

Missing rate limits, no abuse cases in design, business logic flaws. Threat modeling addresses this category.

A05 — HIGH
Security Misconfiguration

Default credentials, unnecessary features enabled, verbose error messages, missing security headers.

A06 — HIGH
Vulnerable Components

Outdated npm packages with known CVEs. The Log4Shell of the JavaScript world waiting to happen.

A07 — HIGH
Broken Authentication

Weak passwords, no MFA, credential stuffing, session fixation, insecure password reset flows.

A08 — MEDIUM
Software & Data Integrity Failures

Unsigned packages, insecure CI/CD pipelines, deserializing untrusted data (prototype pollution).

A09 — MEDIUM
Security Logging Failures

Missing audit logs, not alerting on suspicious events, logging sensitive data (tokens, passwords).

A10 — MEDIUM
Server-Side Request Forgery (SSRF)

App fetches user-supplied URLs, reaching internal services (metadata APIs, databases, admin panels).

Injection — The Most Critical Category in Practice

❌ SQL Injection
// NEVER do this
const q = `SELECT * FROM users
  WHERE email = '${req.body.email}'`

// Attack input:
// ' OR '1'='1'; DROP TABLE users; --
✅ Parameterised Query
// Always use parameterised queries

  'SELECT * FROM users WHERE email = $1',
  [req.body.email]  // driver escapes it
)
JavaScript — NoSQL Injection (MongoDB)
// ❌ Vulnerable — operator injection
// req.body = { "$gt": "" } — matches all documents!

// ✅ Sanitise — ensure string type before query
import mongoSanitize from 'express-mongo-sanitize'
app.use(mongoSanitize())  // strips keys starting with '$' or containing '.'

// Or enforce types explicitly with Zod before any DB call
const LoginSchema = z.object({
  email:    z.string().email(),
  password: z.string().min(1).max(128),
})

// ── SSRF prevention ───────────────────────────────
import { Resolver } from 'node:dns/promises'

async function safeRequest(urlStr) {
  const url = new URL(urlStr)
  if (!['https:'].includes(url.protocol))
    throw new Error('Only HTTPS allowed')

  // Block private IP ranges (RFC 1918, loopback, link-local)
  const addrs = await new Resolver().resolve4(url.hostname)
  for (const ip of addrs) {
    if (/^(10\.|172\.(1[6-9]|2\d|3[01])\.|192\.168\.|127\.|169\.254\.)/.test(ip))
      throw new Error(`Blocked private IP: ${ip}`)
  }
  return fetch(url.toString())
}
03 — Credential Security

Password Security

Password security is a solved problem — if you use the right tools. The mistakes are always the same: MD5/SHA hashing, no salt, too-fast algorithms, or leaking hashes in logs.

🔴

Never Use

MD5, SHA-1, SHA-256 for passwords. These are fast — attackers can try billions/second. A 6-char password cracked in seconds.

✅

Use Instead

Argon2id (winner of Password Hashing Competition), bcrypt (time-tested), or scrypt. All are slow by design and resist GPU attacks.

⚙️

Tuning

Calibrate so hashing takes 200–500ms on your server. Too fast = weak. Too slow = DoS vector. Re-calibrate yearly.

🔑

Pepper

Server-side secret appended before hashing. Stored in env vars, not DB. If DB is compromised, attacker still can't crack without the pepper.

JavaScript — Argon2id Password Hashing
import argon2 from 'argon2'

// ── Argon2id — recommended algorithm in 2025 ─────
const ARGON2_CONFIG = {
  type:        argon2.argon2id,
  memoryCost:  65536,   // 64 MB RAM per hash — GPU-hostile
  timeCost:    3,       // iterations
  parallelism: 4,       // threads
  hashLength:  32,
}

const PEPPER = process.env.PASSWORD_PEPPER  // secret, not in DB

async function hashPassword(plaintext) {
  return argon2.hash(plaintext + PEPPER, ARGON2_CONFIG)
}

async function verifyPassword(hash, plaintext) {
  return argon2.verify(hash, plaintext + PEPPER)
}

// ── Rehash on login if config upgraded ───────────
async function login(email, password) {
  const user = await db.users.findByEmail(email)
  if (!user || !await verifyPassword(user.hash, password))
    throw new Error('Invalid credentials')  // same message — no enumeration

  // Migrate to stronger hash on successful login
  if (argon2.needsRehash(user.hash, ARGON2_CONFIG)) {
    const newHash = await hashPassword(password)
    await db.users.update({ where: { id: user.id }, data: { hash: newHash } })
  }

  return issueTokens(user.id, user.roles)
}

// ── Secure password reset flow ────────────────────
async function initiateReset(email) {
  const user = await db.users.findByEmail(email)
  // Always respond with same message — prevents user enumeration
  if (!user) return

  const token   = crypto.randomBytes(32).toString('base64url')
  const tokenHash = crypto.createHash('sha256').update(token).digest('hex')
  await db.resetTokens.create({
    data: { userId: user.id, tokenHash, expiresAt: new Date(Date.now() + 15*60*1000) }
  })
  // Only store the hash in DB; send the raw token in email
  await mail.send({ to: email, template: 'password-reset', data: { token } })
}
04 — API Hardening

Secure API Design

Every API endpoint is an attack surface. Defence-in-depth means multiple independent controls — no single layer should be the only thing between an attacker and your data.

JavaScript — Security Middleware Stack
import express      from 'express'
import helmet       from 'helmet'
import rateLimit    from 'express-rate-limit'
import slowDown     from 'express-slow-down'
import hpp          from 'hpp'
import { v4: uuidv4 } from 'uuid'

const app = express()

// ── 1. Security headers (helmet covers 14 headers) ─
app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc:     ["'self'"],
      scriptSrc:      ["'self'"],       // no unsafe-inline
      objectSrc:      ["'none'"],
      upgradeInsecureRequests: [],
    }
  },
  hsts: { maxAge: 31536000, includeSubDomains: true, preload: true }
}))

// ── 2. Request ID for tracing ──────────────────────
app.use((req, res, next) => {
  req.id = req.headers['x-request-id'] || uuidv4()
  res.setHeader('X-Request-ID', req.id)
  next()
})

// ── 3. Rate limiting with progressive slowdown ────
app.use('/api/auth', rateLimit({
  windowMs:    15 * 60 * 1000,  // 15 min window
  max:         5,                // 5 attempts per window
  skipSuccessfulRequests: true,
  handler:     (req, res) => res.status(429).json({
    error: 'Too many attempts. Try again in 15 minutes.'
  })
}))

// ── 4. HTTP Parameter Pollution protection ─────────
app.use(hpp())  // ?sort=asc&sort=desc → keeps last only

// ── 5. Body size limits ────────────────────────────
app.use(express.json({    limit: '50kb'  }))
app.use(express.urlencoded({ limit: '50kb', extended: false }))

// ── 6. Never expose stack traces in production ─────
app.use((err, req, res, _next) => {
  logger.error({ err, requestId: req.id })
  res.status(err.status || 500).json({
    error: process.env.NODE_ENV === 'production'
      ? 'Internal server error'   // never leak stack traces
      : err.message,
    requestId: req.id               // helps correlate support tickets
  })
})

API Key Rotation & Scoped Tokens

JavaScript — API Key Management
// ── Issue scoped API keys ──────────────────────────
async function createApiKey(userId, { name, scopes, expiresIn }) {
  const rawKey  = `sk_live_${crypto.randomBytes(32).toString('base64url')}`
  const keyHash  = crypto.createHash('sha256').update(rawKey).digest('hex')
  const keyPrefix = rawKey.slice(0, 12)  // show in UI for identification

  await db.apiKeys.create({
    data: { userId, name, keyHash, keyPrefix,
            scopes,  // e.g. ['read:users', 'write:posts']
            expiresAt: new Date(Date.now() + expiresIn) }
  })
  return rawKey  // shown once — never stored plaintext
}

// ── Authenticate API key ───────────────────────────
async function authenticateApiKey(req, res, next) {
  const key = req.headers['x-api-key']
  if (!key) return res.status(401).json({ error: 'API key required' })

  const hash   = crypto.createHash('sha256').update(key).digest('hex')
  const apiKey = await db.apiKeys.findUnique({ where: { keyHash: hash } })

  if (!apiKey || apiKey.revokedAt || apiKey.expiresAt < new Date())
    return res.status(401).json({ error: 'Invalid or expired API key' })

  req.apiKey = apiKey
  // Update last used (async — don't await, don't block request)
  db.apiKeys.update({ where: { id: apiKey.id }, data: { lastUsedAt: new Date() } }).catch(noop)
  next()
}
05 — Client-Side Attacks

Browser Security: XSS, CSRF & CORS

Three of the most exploited web vulnerabilities live in the browser layer. Understanding the attack mechanics is the only way to truly defend against them.

Cross-Site Scripting (XSS)

XSS lets attackers inject and execute JavaScript in a victim's browser — stealing cookies, tokens, form data, or performing actions as the user. There are three types: Stored (in DB), Reflected (in URL), and DOM-based (client-side manipulation).

❌ Vulnerable
// React — dangerouslySetInnerHTML
<div dangerouslySetInnerHTML={{
  __html: userInput  // executes scripts!
}} />

// Express template — unescaped
res.send(`
${req.query.name}
`)
✅ Safe
// React escapes by default
<div>{userInput}
// If HTML needed: DOMPurify import DOMPurify from 'isomorphic-dompurify' <div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(html) }} />
JavaScript — XSS Prevention & CSP
// ── Server-side output encoding ───────────────────
function escapeHtml(str) {
  return String(str)
    .replace(/&/g,  '&')
    .replace(/,  '<')
    .replace(/>/g,  '>')
    .replace(/"/g,  '"')
    .replace(/'/g,  ''')
}

// ── Content Security Policy (most powerful XSS defence)
// With nonces — allows specific inline scripts, blocks all others
app.use((req, res, next) => {
  res.locals.nonce = crypto.randomBytes(16).toString('base64')
  res.setHeader('Content-Security-Policy',
    `default-src 'self'; ` +
    `script-src 'self' 'nonce-${res.locals.nonce}'; ` +
    `style-src 'self' 'nonce-${res.locals.nonce}'; ` +
    `img-src 'self' data: https:; ` +
    `object-src 'none'; ` +
    `base-uri 'self'; ` +
    `frame-ancestors 'none'`  // clickjacking protection too
  )
  next()
})

Cross-Site Request Forgery (CSRF)

JavaScript — Double Submit CSRF Pattern
import csrf from 'csrf'
const tokens = new csrf()

// ── Issue CSRF token ──────────────────────────────
app.get('/csrf-token', (req, res) => {
  const secret = req.session.csrfSecret ??= tokens.secretSync()
  const token  = tokens.create(secret)
  res.json({ csrfToken: token })
})

// ── Verify on state-mutating requests ─────────────
function verifyCsrf(req, res, next) {
  const token = req.headers['x-csrf-token'] || req.body._csrf
  if (!tokens.verify(req.session.csrfSecret, token))
    return res.status(403).json({ error: 'CSRF token mismatch' })
  next()
}

// ── Cookie settings that prevent CSRF ─────────────
res.cookie('sessionId', token, {
  httpOnly:  true,      // JS can't read it (blocks XSS theft)
  secure:    true,      // HTTPS only
  sameSite:  'strict', // blocks cross-site submission (replaces CSRF tokens for simple cases)
  maxAge:    15 * 60 * 1000
})

CORS — Configured Correctly

JavaScript — Strict CORS Policy
import cors from 'cors'

const ALLOWED_ORIGINS = new Set([
  'https://app.mysite.com',
  'https://www.mysite.com',
  ... (process.env.NODE_ENV !== 'production'
    ? ['http://localhost:3000']
    : []
  )
])

app.use(cors({
  origin: (origin, cb) => {
    // null origin = same-origin or non-browser request — allow explicitly
    if (!origin || ALLOWED_ORIGINS.has(origin)) cb(null, true)
    else cb(new Error(`Origin ${origin} not allowed by CORS`))
  },
  credentials: true,      // allow cookies in cross-origin requests
  methods:     ['GET', 'POST', 'PUT', 'PATCH', 'DELETE', 'OPTIONS'],
  allowedHeaders: ['Content-Type', 'Authorization', 'X-CSRF-Token'],
  maxAge:      86400       // cache preflight for 1 day
}))

// ❌ Never do this — opens to any attacker origin:
// Access-Control-Allow-Origin: *  +  credentials: true
// This is spec-prohibited but misconfigured proxies can expose it
06 — Cloud Security

Cloud IAM & Network Security

Cloud environments give you enormous control — and enormous rope to hang yourself with. The principle of least privilege is the cornerstone of every secure cloud deployment.

AWS IAM — Least Privilege in Practice

JSON — IAM Policy: Minimal S3 Access
// ❌ Wildcard policies — never in production
{
  "Effect":   "Allow",
  "Action":   "*",
  "Resource": "*"
}

// ✅ Scoped policy — API server can only read its specific prefix
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect":    "Allow",
    "Action":    ["s3:GetObject", "s3:PutObject"],
    "Resource":  "arn:aws:s3:::my-uploads-bucket/user-uploads/*",
    "Condition": {
      "StringEquals": {
        "aws:RequestedRegion": "eu-west-1"
      }
    }
  }]
}
JavaScript — AWS SDK with Role Assumption
import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3'
import { STSClient, AssumeRoleCommand } from '@aws-sdk/client-sts'
import { getSignedUrl }               from '@aws-sdk/s3-request-presigner'

// ── Presigned URLs — client uploads directly to S3 (no proxy) ─
async function generateUploadUrl(userId, filename) {
  const s3  = new S3Client({ region: 'eu-west-1' })
  const key = `user-uploads/${userId}/${crypto.randomUUID()}-${sanitize(filename)}`

  const cmd = new PutObjectCommand({
    Bucket:      process.env.S3_BUCKET,
    Key:         key,
    ContentType: 'application/octet-stream',
    // Force content-type — prevents polyglot file attacks
    ServerSideEncryption: 'AES256',
  })

  const url = await getSignedUrl(s3, cmd, { expiresIn: 300 })  // 5 minutes
  return { url, key }
}

// ── IMDSv2 — fetch instance metadata securely ─────
// IMDSv1 is vulnerable to SSRF; IMDSv2 requires a PUT first
async function getInstanceMetadata(path) {
  const token = await fetch(
    'http://169.254.169.254/latest/api/token',
    { method: 'PUT', headers: { 'X-aws-ec2-metadata-token-ttl-seconds': '21600' } }
  ).then(r => r.text())
  return fetch(`http://169.254.169.254/latest/meta-data/${path}`,
    { headers: { 'X-aws-ec2-metadata-token': token } }
  ).then(r => r.text())
}
07 — Secrets Management

Secrets Management

A hardcoded API key in a public GitHub repo is breached within minutes. Secrets management is not optional — it is the difference between a secure system and a liability.

🚨 Never Do These
Never commit secrets to git (even private repos), never log secrets, never put secrets in environment variables on shared CI runners without masking, never store secrets in Docker image layers.
JavaScript — AWS Secrets Manager Integration
import { SecretsManagerClient, GetSecretValueCommand }
  from '@aws-sdk/client-secrets-manager'

const client = new SecretsManagerClient({ region: 'eu-west-1' })

// ── Cache secrets to avoid per-request API calls ─
const secretCache = new Map()

async function getSecret(secretName) {
  const cached = secretCache.get(secretName)
  if (cached && cached.expiresAt > Date.now()) return cached.value

  const res = await client.send(
    new GetSecretValueCommand({ SecretId: secretName })
  )
  const value = JSON.parse(res.SecretString)
  secretCache.set(secretName, { value, expiresAt: Date.now() + 5*60*1000 })
  return value
}

// ── Startup secret loading ────────────────────────
async function loadConfig() {
  const dbSecret = await getSecret('prod/myapp/database')
  const appSecret = await getSecret('prod/myapp/app-keys')

  return {
    databaseUrl:  dbSecret.connectionString,
    jwtSecret:    appSecret.jwtSecret,
    stripeKey:    appSecret.stripeSecretKey,
  }
}

// ── .env audit with dotenv-safe ───────────────────
// .env.example lists all required vars (committed to git)
// .env fills the values (gitignored)
// dotenv-safe throws at startup if any required var is missing
import 'dotenv-safe/config'  // fails fast if .env is incomplete

08 — Proactive Security

Threat Modeling

Threat modeling is structured thinking about what can go wrong before it does. Done in design reviews, it finds architectural flaws that code review and pen testing cannot.

The most widely used framework is STRIDE, developed at Microsoft. Each letter identifies a threat category. Apply it to every trust boundary in your system.

LetterThreatJS/Node ExampleMitigation
SSpoofingForged JWT, cookie theftSignature verification, httpOnly cookies
TTamperingModified request body, IDORInput validation, integrity checks
RRepudiationNo audit log of admin actionsStructured audit logging with actor ID
IInfo DisclosureStack traces in 500s, verbose errorsGeneric errors in prod, log internally
DDenial of ServiceNo rate limits, ReDoS, large payloadsRate limiting, body size caps, safe regex
EElevation of PrivilegeIDOR, missing AuthZ check, JWT role tamperingABAC policies, signed payloads

Practical Threat Modeling in 4 Steps

  • 1
    Diagram the system. Draw a Data Flow Diagram (DFD) with every component, data store, external entity, and trust boundary. Use OWASP Threat Dragon or draw.io. Every line crossing a trust boundary is a potential attack surface.
  • 2
    Apply STRIDE to each boundary. For each arrow crossing a trust boundary: can an attacker Spoof the sender? Tamper with the data? Not be identified (Repudiation)? Read it (Info Disclosure)? DoS it? Escalate privileges through it?
  • 3
    Rate each threat by DREAD or CVSS. Damage, Reproducibility, Exploitability, Affected users, Discoverability. Prioritise the high scorers for immediate mitigation before writing code.
  • 4
    Define mitigations and test cases. Each threat becomes a security requirement and a test case. Put them in the ticket backlog. Verify them in code review and pen test.
JavaScript — Security Audit Log Pattern
// ── Structured audit log — addresses Repudiation (R in STRIDE)
async function auditLog(event) {
  await db.auditLogs.create({
    data: {
      actorId:   event.userId,
      actorIp:   event.ip,
      action:    event.action,         // 'user.delete', 'api_key.create'
      resource:  event.resource,
      resourceId:event.resourceId,
      metadata:  event.metadata,      // before/after for mutation events
      result:    event.result,         // 'success' | 'failure'
      timestamp: new Date(),
      requestId: event.requestId,
    }
  })
}

// Wrap sensitive operations
async function deleteUser(req, res) {
  await db.users.delete({ where: { id: req.params.id } })
  await auditLog({

    resourceId: req.params.id,
    result:     'success',
    requestId:  req.id,
    ip:         req.ip,
  })
  res.status(204).send()
}
09 — Modern Architecture

Zero-Trust Architecture

Zero-trust is a security model based on one axiom: never trust, always verify. There is no "inside the network" — every request, from any source, is treated as potentially hostile until proven otherwise.

"Assume breach. Verify explicitly. Use least privilege access." — Microsoft Zero Trust Principles
🔒

Verify Explicitly

Authenticate and authorise every request based on all available data: identity, location, device health, service, workload.

📏

Least Privilege

JIT/JEA access, risk-based adaptive policies, data protection. No standing admin permissions. Scope every token.

💥

Assume Breach

Segment networks. Encrypt all traffic end-to-end. Use analytics for visibility. Minimise blast radius.

🔍

Continuous Validation

Re-verify trust on every request. Short token TTLs. Step-up auth for sensitive operations. Real-time anomaly detection.

JavaScript — mTLS Service-to-Service Auth
import https  from 'node:https'
import fs     from 'node:fs'

// ── Zero-trust internal service: require client cert ─
const server = https.createServer({
  cert:              fs.readFileSync('/etc/certs/server.crt'),
  key:               fs.readFileSync('/etc/certs/server.key'),
  ca:                fs.readFileSync('/etc/certs/internal-ca.crt'),
  requestCert:       true,   // demand client cert
  rejectUnauthorized: true,   // reject if not signed by our CA
}, app)

// ── Extract caller identity from cert ────────────
app.use((req, _res, next) => {
  const cert = req.socket.getPeerCertificate()
  if (!cert?.subject?.CN)
    throw Object.assign(new Error('mTLS required'), { status: 401 })

  req.callerService = cert.subject.CN  // e.g. 'payments-service'
  next()
})

// ── Service allow-list ────────────────────────────
const allowedServices = (...services) => (req, res, next) => {
  if (!services.includes(req.callerService))
    return res.status(403).json({ error: 'Service not authorised' })
  next()
}

router.post('/internal/charge',
  allowedServices('order-service', 'subscription-service'),
  processCharge
)
10 — Offensive Knowledge

Penetration Testing Basics

Understanding how attackers work makes you a better defender. These techniques are for testing systems you own or have explicit permission to test.

⚖️ Legal Reminder
Only test systems you own or have explicit written permission to test. Unauthorised pen testing is illegal in most jurisdictions under the Computer Fraud and Abuse Act (USA), Computer Misuse Act (UK), and equivalents globally.

The Pen Test Methodology

  • 1
    Reconnaissance — Passive: OSINT (Shodan, WHOIS, LinkedIn, GitHub dorking). Active: port scanning (nmap), service fingerprinting, subdomain enumeration (subfinder, amass).
  • 2
    Enumeration — Directory brute-force (gobuster, ffuf), API endpoint discovery, technology fingerprinting (Wappalyzer), SSL/TLS analysis (testssl.sh).
  • 3
    Vulnerability Scanning — Automated: OWASP ZAP, Nuclei, Burp Suite Pro scanner. Manual review of authentication, session management, and input handling.
  • 4
    Exploitation — Confirm vulnerabilities are real, not false positives. Document exact steps. Capture minimal evidence. Stop at first confirmed access — no lateral movement without scope approval.
  • 5
    Reporting — Risk-rated findings (CVSS), reproduction steps, business impact, specific remediation advice. Executive summary + technical details. Fix verification after remediation.

JavaScript — AES-256-GCM Encryption
import crypto from 'node:crypto'

const ALGORITHM = 'aes-256-gcm'
const KEY_LEN   = 32  // 256 bits
const IV_LEN    = 12  // 96 bits — GCM standard

function encrypt(plaintext, keyHex) {
  const key    = Buffer.from(keyHex, 'hex')
  const iv     = crypto.randomBytes(IV_LEN)  // fresh IV per message
  const cipher = crypto.createCipheriv(ALGORITHM, key, iv)

  const encrypted = Buffer.concat([
    cipher.update(plaintext, 'utf8'),
    cipher.final(),
  ])
  const authTag = cipher.getAuthTag()  // GCM authentication tag

  // Pack: iv(12) | authTag(16) | ciphertext — all needed for decryption
  return Buffer.concat([iv, authTag, encrypted]).toString('base64url')
}

function decrypt(ciphertextB64, keyHex) {
  const buf     = Buffer.from(ciphertextB64, 'base64url')
  const iv      = buf.subarray(0, IV_LEN)
  const authTag = buf.subarray(IV_LEN, IV_LEN + 16)
  const data    = buf.subarray(IV_LEN + 16)
  const key     = Buffer.from(keyHex, 'hex')

  const decipher = crypto.createDecipheriv(ALGORITHM, key, iv)
  decipher.setAuthTag(authTag)  // verifies integrity — throws if tampered

  return Buffer.concat([decipher.update(data), decipher.final()]).toString('utf8')
}

12 — Ecosystem Security

Supply Chain & Dependency Security

The npm ecosystem is a massive attack surface. A single compromised dependency can affect thousands of downstream projects. The event-stream and ua-parser-js incidents proved this is not theoretical.

⚔️ Known Attacks
  • Typosquatting (lodsh instead of lodash)
  • Dependency confusion (private package name hijacking)
  • Maintainer account takeover
  • Malicious postinstall scripts
  • Protestware (deliberate sabotage by maintainer)
🛡️ Defences
  • Audit with npm audit and Snyk in CI
  • Lock dependencies with package-lock.json
  • Review new deps: stars, last published, maintainers
  • Use --ignore-scripts for untrusted packages
  • Scope private packages with a registry prefix

  • ✅
    Dependencies audited — npm audit passes with no high/critical CVEs. Outdated packages reviewed.
  • ✅
    Secrets scanned — Gitleaks pre-commit hook passes. No hardcoded keys, tokens, or passwords in any file.
  • ✅
    Security headers verified — Helmet configured. CSP, HSTS, X-Frame-Options, X-Content-Type-Options all present.
  • ✅
    Input validation on all endpoints — Zod schemas, parameterised DB queries, file upload type restrictions.
  • ✅

  • ✅
    Rate limits active — Auth endpoints, API endpoints, and file upload endpoints all rate-limited.
  • ✅
    Error messages sanitised — Production errors return generic messages. Stack traces logged internally only.
  • ✅
    Audit logs in place — Auth events, admin actions, data mutations all captured with actor, action, timestamp, IP.
  • ✅
    TLS enforced everywhere — HTTPS only. HSTS preload submitted. Internal service traffic uses mTLS.
  • ✅
    Docker image hardened — Non-root user, read-only filesystem, no unnecessary capabilities, distroless or Alpine base.
