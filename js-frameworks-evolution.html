⚫ 2005–2009 · The Dark Ages
01 — Before Frameworks

The Dark Ages — Vanilla JavaScript & jQuery

Before frameworks, the web was a collection of documents. JavaScript was a scripting language for adding interactivity to static HTML pages — not a platform for building applications. The gap between what developers needed and what the browser provided was enormous.

The Web Before Frameworks

In the early 2000s, web development meant PHP or ASP generating HTML on the server, with JavaScript used sparingly for form validation and image rollovers. The browser was a document viewer with scriptability bolted on. Cross-browser compatibility was a nightmare — Internet Explorer 6 had 95% market share and its own completely different DOM API from every other browser.

Developers wrote functions like getElement() that tried document.getElementById, then document.all, then gave up. Every project duplicated the same 200 lines of compatibility shims. AJAX (popularised by Google Maps in 2005) changed expectations dramatically — suddenly pages could update without a full reload — but writing it cross-browser required understanding four different XMLHttpRequest implementations.

JavaScript — DOM-Centric Programming, 2005 Style 
// The reality of cross-browser JavaScript in 2005
function getElement(id) {
  if (document.getElementById)    return document.getElementById(id)
  if (document.all)               return document.all[id]
  if (document.layers)            return document.layers[id]
}

function addEvent(el, type, fn) {
  if (el.addEventListener)        el.addEventListener(type, fn, false)  // W3C
  else if (el.attachEvent)         el.attachEvent('on' + type, fn)     // IE
  else                             el['on' + type] = fn                // legacy
}

// AJAX: the XMLHttpRequest saga
function createXHR() {
  if (window.XMLHttpRequest)       return new XMLHttpRequest()
  try { return new ActiveXObject('Msxml2.XMLHTTP') } catch (e) {}
  try { return new ActiveXObject('Microsoft.XMLHTTP') } catch (e) {}
}
// This was every project. Every. Single. One.

Why jQuery Became Dominant

John Resig released jQuery 1.0 in August 2006 with a radical proposition: write once, work everywhere. Its CSS selector API mapped directly onto how designers thought about pages. Its chaining syntax was readable. It normalised events, animations, and AJAX across every browser. Within two years it was on the majority of websites on the internet.

JavaScript — jQuery's Radical Simplicity (2006) 
// Before jQuery: 30 lines to do this cross-browser
// After jQuery: one line
$('.submit-btn').on('click', function() {
  $('#loading').show()
  $.ajax({
    url: '/api/submit',
    method: 'POST',
    data: $('form').serialize(),
    success: function(response) {
      $('#result').html(response).fadeIn()
      $('#loading').hide()
    }
  })
})

// jQuery's genius: the DOM *is* the application state.
// Find elements. Modify them. jQuery was the perfect library
// for pages that needed scripting.
// It was the wrong tool for applications that needed architecture.
💡 jQuery's Legacy
jQuery's influence is understated. Its CSS selector syntax became the browser's native querySelector. Its AJAX API influenced the Fetch API. Its plugin architecture was the first widely-used example of open-source frontend extensibility. jQuery didn't fail — it succeeded so completely that the browser absorbed its ideas.
🌅 2010–2012 · The Rise of MVC
02 — Server Patterns Meet the Browser

The Rise of MVC Frameworks

As web applications grew more complex — Gmail, Google Maps, early Twitter — the jQuery model of finding-and-mutating DOM nodes broke down. Developers who understood Rails and Django tried to bring those server-side patterns into the browser.

Backbone.js (2010) was the first widely-adopted answer. Jeremy Ashkenas gave developers Models (data with events), Collections (arrays of models), and Views (DOM management tied to models). It imposed minimal structure and made few assumptions — which was both its strength and its weakness.

Knockout.js (2010) introduced observable-based data binding to the browser. Decorate your HTML with data-bind attributes; when the underlying JavaScript observable changed, the DOM updated automatically. It was a revelation for data-heavy UIs like dashboards.

Ember.js (2011) went further — it was an opinionated, convention-over-configuration framework with routing, a data layer (Ember Data), and strong opinions about application structure. It bet heavily on the MVC pattern and URLs as the source of truth.

JavaScript — Backbone.js: The First Attempt at Frontend Architecture 
// Backbone: Models with events + Views that re-render on change
const Todo = Backbone.Model.extend({
  defaults: { title: '', done: false }
})

const TodoView = Backbone.View.extend({
  tagName: 'li',
  events: { 'click .toggle': 'toggleDone' },
  initialize() {
    this.listenTo(this.model, 'change', this.render)  // model → view sync
  },
  render() {
    this.$el.html(this.template(this.model.toJSON()))
    return this
  },
  toggleDone() {
    this.model.save({ done: !this.model.get('done') })
  }
})
// The problem: as apps grew, views multiplied.
// Who owned which part of the DOM? Who could modify what?
// Nothing prevented two views from independently updating the same element.

These frameworks exposed a fundamental tension: the server-side MVC pattern assumes a request-response cycle where the server renders a complete, consistent view. The browser has no such clear lifecycle. State is continuous. Multiple events fire simultaneously. The same piece of data drives multiple UI elements. The patterns didn't map cleanly.

🌸 2012–2016 · The Two-Way Binding Era
03 — Automatic Synchronisation

AngularJS & the Two-Way Binding Era

AngularJS (2010, Google) was the first framework that felt like it was built for building applications rather than augmenting documents. Its HTML-first approach and automatic two-way binding felt like magic — until you understood what it was actually doing.

The Digest Cycle

AngularJS's core innovation was dirty-checking. The framework maintained a list of watchers — bindings between scope properties and DOM nodes. On any significant event, it ran the digest cycle: iterating through all watchers, comparing current values to previous values, and updating the DOM where they differed. It repeated this until no more changes were detected (stability).

JavaScript — AngularJS: Two-Way Binding & the Digest Cycle 
// AngularJS: HTML becomes declarative data binding
// <input ng-model="user.name">
// <p>Hello, {{user.name}}!</p>
// When the input changes, $scope.user.name updates.
// When $scope.user.name changes in code, the input updates.
// Automatic. Bi-directional. No event handlers written by hand.

angular.module('app').controller('UserCtrl', function($scope, $http) {
  $scope.users = []

  $scope.loadUsers = function() {
    $http.get('/api/users').then(function(res) {
      $scope.users = res.data       // assign → digest detects → DOM updates
    })
  }

  // $watch: explicit observation of scope properties
  $scope.$watch('searchTerm', function(newVal, oldVal) {
    if (newVal !== oldVal) $scope.search(newVal)
  })
})

// The scalability problem:
// Each ng-repeat item + each binding adds a watcher.
// 2,000 watchers → digest cycle takes 50ms → UI feels sluggish.
// "AngularJS best practice": keep watchers below 2,000.
// Real applications have 10,000+.

The Scalability Challenges of Two-Way Binding

AngularJS was transformative for medium-complexity applications. For large, data-dense UIs — trading dashboards, analytics platforms, real-time feeds — the digest cycle became a performance ceiling. The framework could not know what changed, only that something might have, so it checked everything. The solution (one-time bindings, $scope.$apply discipline, manual watch removal) required deep framework knowledge and undermined the "it just works" promise.

⚛️ 2013–2016 · The React Revolution
04 — Unidirectional Flow

React & the Virtual DOM Revolution

React's 2013 release was controversial. Mixing HTML into JavaScript? Throwing away the DOM on every render? It contradicted every established frontend best practice. It was also fundamentally correct about what was wrong with every approach before it.

"React is the V in MVC." — The most underselling product description in software history.

Declarative UI — Describe What, Not How

The critical insight in React was shifting from imperative DOM manipulation (find this element, change this attribute, add this class) to declarative UI description: given this data, this is what the UI should look like. The framework's job was to figure out the minimal DOM changes required to make that description real.

JavaScript — The Paradigm Shift: Imperative vs Declarative 
// ── Imperative (jQuery/Backbone): describe the transitions ────
function setUserLoading(loading) {
  if (loading) {
    $('#spinner').show()
    $('#user-list').hide()
    $('#error').hide()
  } else {
    $('#spinner').hide()
    $('#user-list').show()
  }
  // Bug-prone: what if #error was visible and loading goes true then false?
  // You must track every possible state transition manually.
}

// ── Declarative (React): describe the outcome ─────────────────
function UserSection({ loading, users, error }) {
  if (loading) return <Spinner />
  if (error)   return <ErrorMessage error={error} />
  return <UserList users={users} />
}
// Given ANY combination of loading/error/users, the output is always correct.
// No transition management. No state mutation bugs.
// This is why React fundamentally changed frontend engineering.

// ── Unidirectional data flow ───────────────────────────────────
// Data flows DOWN (parent → child via props)
// Events flow UP (child → parent via callback props)
// NEVER sideways. This makes dataflow debuggable.
function Parent() {
  const [count, setCount] = useState(0)
  return <Child count={count} onIncrement={() => setCount(c => c + 1)} />
}

The Virtual DOM was React's implementation detail, not its idea. A lightweight JavaScript representation of the DOM tree allowed React to compute diffs cheaply in JavaScript memory rather than expensively in real DOM operations. But the deeper innovation was making rendering a pure function of state — and that insight outlasted the virtual DOM implementation itself.

🧩 2014–2017 · Component Architecture
05 — The New Unit of Abstraction

Component Architecture Changed Everything

React's component model — small, composable, self-contained UI units — turned out to be the most durable idea in the history of frontend development. It changed how teams organised code, how design systems were built, and how we think about UI abstraction.

Composition Over Inheritance

Object-oriented UI inheritance (class hierarchies, widget toolkits) had failed on the web repeatedly. A ButtonBase → Button → IconButton → LoadingIconButton hierarchy becomes unmaintainable as requirements diverge. Component composition — wrapping one component in another, passing components as props — solved the same problems with less coupling.

JavaScript — Composition Patterns That Became Industry Standard 
// ── Compound components — related components sharing state ─────
function Tabs({ children }) {
  const [active, setActive] = useState(0)
  return (
    <TabsContext.Provider value={{ active, setActive }}>
      {children}
    </TabsContext.Provider>
  )
}
// Usage is readable, flexible, no prop drilling:
<Tabs>
  <Tabs.List>
    <Tabs.Tab>Overview</Tabs.Tab>
    <Tabs.Tab>Settings</Tabs.Tab>
  </Tabs.List>
  <Tabs.Panel>Overview content</Tabs.Panel>
  <Tabs.Panel>Settings content</Tabs.Panel>
</Tabs>

// ── Render props — sharing stateful logic ──────────────────────
function MouseTracker({ render }) {
  const [pos, setPos] = useState({ x: 0, y: 0 })
  return <div onMouseMove={e => setPos({ x: e.clientX, y: e.clientY })}>{render(pos)}</div>
}
// Hooks (2019) replaced render props with a cleaner API,
// but the underlying idea of composable stateful logic survived.

// ── Hooks: the final form of component logic sharing ──────────
function useMouse() {
  const [pos, setPos] = useState({ x: 0, y: 0 })
  useEffect(() => {
    const handler = (e) => setPos({ x: e.clientX, y: e.clientY })
    window.addEventListener('mousemove', handler)
    return () => window.removeEventListener('mousemove', handler)
  }, [])
  return pos
}

Component-driven development also changed tooling. Storybook (2016) let teams build and test components in isolation. Design systems — a single source of truth for UI components used across products — became the standard deliverable of frontend platform teams at every major technology company.

💚 2014–Present · The Middle Ground
06 — Progressive Adoption

Vue's Middle Ground

Evan You released Vue.js in 2014 after working on AngularJS at Google. His goal was to extract the parts he liked about Angular and build something lighter. Vue became the framework that proved you didn't have to choose between simplicity and power.

Vue's progressive adoption model was its defining characteristic. You could drop a <script> tag into an existing page and start using Vue for just that widget. Or you could build an entire SPA with Vue CLI, Vue Router, and Vuex. The same framework scaled from a jQuery replacement to a full application platform.

JavaScript — Vue 3: Composition API & the Proxy Reactivity System 
// Vue 3 Composition API — close to React Hooks but with automatic tracking
import { ref, computed, watch, onMounted } from 'vue'

export default {
  setup() {
    const count = ref(0)                          // reactive primitive
    const double = computed(() => count.value * 2)  // auto-tracked derived state

    // Vue 3 uses ES6 Proxy — no explicit dependency declaration needed.
    // The computed fn runs; Vue intercepts every .value access;
    // it knows exactly which refs the computed depends on.

    watch(count, (newVal, oldVal) => {
      console.log(`Count changed: ${oldVal} → ${newVal}`)
    })

    return { count, double }
  }
}

// <script setup> — even more concise (Vue 3.2+)
// <script setup lang="ts">
const count = ref(0)
const double = computed(() => count.value * 2)
// Automatically exported to template. No return statement. No boilerplate.

Vue's single-file components (.vue files with <template>, <script>, <style>) kept related code co-located without the JSX controversy. For many developers coming from traditional web development, it was the most approachable path to modern component architecture.

🔷 2016–Present · Enterprise Evolution
07 — TypeScript-First Architecture

Angular's Enterprise Evolution

Angular 2 (2016) was not an upgrade of AngularJS — it was a complete rewrite. It bet entirely on TypeScript when TypeScript was still controversial, introduced a DI system more sophisticated than most backend frameworks, and set a high floor for application structure that made it particularly suited to large enterprise teams.

Angular's opinionation is its defining feature. There is an Angular Way to do routing, forms, HTTP, state management, testing, internationalization, and server-side rendering. A developer joining any Angular project will find familiar patterns. This predictability is enormously valuable at enterprise scale where team turnover and codebase longevity are primary concerns.

TypeScript — Angular: DI, Services & Standalone Components 
// Angular's DI is hierarchical — injectors form a tree matching the component tree.
// Services can be scoped to the root, a module, or a component.
@Injectable({ providedIn: 'root' })  // singleton across the app
export class UserService {
  constructor(private http: HttpClient) {}

  getUsers(): Observable<User[]> {
    return this.http.get<User[]>('/api/users').pipe(
      retry(3),
      catchError(this.handleError)
    )
  }
}

// Standalone components (Angular 14+) — removed NgModule boilerplate
@Component({
  selector:    'app-user-list',
  standalone:  true,
  imports:     [CommonModule, UserCardComponent],
  template: `
    <app-user-card
      *ngFor="let user of users$ | async"
      [user]="user"
    />
  `
})
export class UserListComponent {
  users$ = inject(UserService).getUsers()  // Angular 14+ inject() function
}

Angular's adoption of Signals (Angular 16–17) marked its most significant reactivity evolution — moving from RxJS-everywhere to a fine-grained reactive primitive that brings it architecturally closer to Vue's reactivity model and SolidJS, while retaining backwards compatibility with its RxJS-based ecosystem.

⚔️ 2015–Present · The State Management Wars
08 — Centralised State Architecture

The State Management Wars

React's unidirectional data flow solved local component state elegantly. It did not solve shared state — data needed by multiple disconnected components. The ecosystem's answer was a decade of competing solutions, each embodying a different philosophy about where data should live and how it should change.

JavaScript — State Management Evolution: Redux → Zustand 
// ── REDUX (2015): explicit, predictable, verbose ───────────────
// Inspired by Elm's architecture. Single source of truth.
// Pure functions (reducers). Time-travel debugging.
// Beautiful in theory. Exhausting in practice for simple cases.

// Action type constants (often in a separate file)
const INCREMENT = 'counter/increment'

// Reducer — pure function: (state, action) → state
function counterReducer(state = { value: 0 }, action) {
  switch (action.type) {
    case INCREMENT: return { value: state.value + 1 }
    default:       return state
  }
}
// Also: action creators, selectors, connect(), mapStateToProps...
// 5 files for a counter. Developers fled.

// ── REDUX TOOLKIT (2019): Redux without the ceremony ──────────
const counterSlice = createSlice({
  name: 'counter',
  initialState: { value: 0 },
  reducers: {
    increment: state => { state.value += 1 }  // Immer under the hood
  }
})

// ── ZUSTAND (2019): minimal global state ──────────────────────
const useStore = create((set) => ({
  count:     0,
  increment: () => set(state => ({ count: state.count + 1 })),
  fetchUsers: async () => {
    const users = await api.getUsers()
    set({ users })
  }
}))

// ── TAN STACK QUERY / SWR (2019–2020): server state is different ─
// The revelation: most "state management" problems are really
// data-fetching problems. Caching, refetching, sync — not Redux concerns.
const { data, isLoading, error } = useQuery({
  queryKey:  ['users'],
  queryFn:   api.getUsers,
  staleTime: 5 * 60 * 1000,
})

The most important lesson from the state wars: server state and client state are fundamentally different problems. Redux tried to manage both with the same abstractions. TanStack Query's separation of concerns — server state handled by query/mutation hooks, UI state handled by local state or lightweight stores like Zustand — became the dominant modern pattern.

🌐 2016–Present · The SSR Renaissance
09 — Back to the Server

The SSR Renaissance

Client-side rendering promised rich applications with native-app-like experiences. By 2016, the bills were coming due: blank screens on slow networks, catastrophic SEO, and Time-to-Interactive measured in seconds. The industry began rediscovering what PHP developers had always known: servers are good at rendering HTML.

🔍

The SEO Problem

Search crawlers in 2013–2016 executed JavaScript inconsistently. A React SPA served a blank <div id="root"></div>. Google eventually improved JS rendering, but the uncertainty was costly for content sites.

📱

The Performance Problem

3G mobile users waited 8–12 seconds for a React SPA to paint. First Contentful Paint required: download HTML → download JS → parse JS → execute React → fetch data → render. Every step was a round-trip.

🔋

The Battery Problem

Client-side rendering pushed all computation to the device. Low-end Android phones spending 4 seconds parsing and executing a 500KB JS bundle to show a product list that a server could have sent in 50ms of HTML.

💧

Hydration Cost

SSR solved the first paint problem but introduced hydration: re-attaching React to server-rendered HTML required executing all the same component code client-side. Sending the same data twice, doing the same work twice.

Next.js (2016) became the definitive answer by making SSR the default rather than the exception. Its model — React components that could run on the server, with getServerSideProps and getStaticProps for data fetching — made SSR accessible without the complexity of setting up a Node.js rendering server from scratch.

🔥 2016–Present · The Compiler Revolution
10 — Eliminating the Framework Runtime

Svelte's Compiler Revolution

Rich Harris released Svelte in 2016 with an idea that was either obvious or radical depending on your perspective: frameworks are compilers, not runtime libraries. Move all the framework work to build time. Ship no virtual DOM. Ship no reactivity runtime. Ship surgically precise, plain JavaScript that directly updates the DOM.

⚛️ React: Runtime Reactivity
// React ships ~45KB (gzipped) of runtime.
// Every component call is a function call.
// Reconciler diffs virtual DOM on every render.
// Works beautifully. Has overhead you always pay.
function Counter() {
  const [n, setN] = useState(0)
  return (
    <button onClick={() => setN(n + 1)}>
      Count: {n}
    </button>
  )
}
🔥 Svelte: Compile-Time Reactivity 
<!-- Svelte source — looks like HTML -->
<script>
  let n = 0
</script>
<button on:click={() => n++}>
  Count: {n}
</button>

// Svelte compiles this to:
// document.createTextNode(n)
// button.addEventListener('click', () => {
//   n++
//   set_data(t, n) // direct DOM update
// })
// Zero virtual DOM. Zero runtime reactivity.
// The component IS the output.

Svelte 5 (2024) introduced Runes — a new reactivity primitive that addressed Svelte 4's ambiguity around reactivity in shared modules (not just components). The $state, $derived, and $effect runes are explicit signal-like primitives, aligning Svelte with the signals model adopted by Solid, Vue 3, and Angular.

⚛️ 2022–Present · Server Components
11 — Rethinking Rendering

React Server Components

React Server Components (RSC), stabilised in React 19 and shipped in Next.js App Router, represent the most fundamental rethinking of React's architecture since Hooks. They challenge the assumption that React is a client-side library that optionally renders on the server.

JavaScript — RSC: Server vs Client Components 
// ── Server Component (default in Next.js App Router) ──────────
// Runs ONLY on the server. Never sent to the client.
// Can directly access databases, file systems, secrets.
// Zero JavaScript bundle impact.
async function ProductPage({ id }) {
  // Direct DB access — no API route needed
  const product = await db.products.findById(id)
  const reviews  = await db.reviews.findByProduct(id)

  return (
    <article>
      <h1>{product.name}</h1>
      {/* Server component — no client JS */}
      <ReviewList reviews={reviews} />
      {/* Client component — has interactivity */}
      <AddToCart productId={product.id} />
    </article>
  )
}

// ── Client Component — explicit opt-in ────────────────────────
'use client'  // directive: this runs in the browser

function AddToCart({ productId }) {
  const [added, setAdded] = useState(false)
  return (
    <button onClick={async () => {
      await addToCart(productId)
      setAdded(true)
    }}>
      {added ? 'Added!' : 'Add to Cart'}
    </button>
  )
}

// The mental model shift:
// Before RSC: React is client-side. SSR is an add-on.
// After RSC:  Components can be server-only, client-only, or both.
//             Server is the default. Client is opt-in.
//             Islands of interactivity in a sea of static HTML.

RSC's most consequential innovation is eliminating the API layer for data fetching within the same application. A server component can read directly from the database — no REST endpoint, no serialisation, no client-side fetch. The network boundary is no longer "client requests server"; it is "where in your component tree does server-side code end and client-side code begin."

🌈 2020–Present · The Rendering Spectrum
12 — Modes of Rendering

The Modern Rendering Spectrum

Modern frameworks support a spectrum of rendering strategies within the same application. Understanding each mode — its performance characteristics, caching model, and appropriate use cases — is a core senior frontend skill.

Mode Where When Trade-offs Best for
CSR Browser On user interaction Slow TTFB, fast subsequent nav Dashboards, SaaS apps
SSR Server Per request Fast FCP, server load Personalised pages
SSG Build time At deploy Fastest TTFB, stale data risk Marketing, blogs, docs
ISR Build + CDN Background revalidation Near-static speed + fresh data Product pages, news
Streaming Server Progressive chunks Fast TTFB, complex error handling Data-heavy pages
Partial Hydration Both Per island Minimal JS shipped Content + interactivity mix
JavaScript — Next.js: Multiple Rendering Modes in One App 
// ── SSG: built at deploy time, served from CDN ────────────────
export async function generateStaticParams() {
  const posts = await db.posts.findAll()
  return posts.map(p => ({ slug: p.slug }))
}
async function BlogPost({ params }) {
  const post = await db.posts.findBySlug(params.slug)
  return <article>{post.content}</article>
}

// ── ISR: revalidate in background every N seconds ─────────────
async function ProductPage({ params }) {
  const product = await fetch(`/api/products/${params.id}`, {
    next: { revalidate: 60 }   // ISR: cache 60s, then regenerate
  }).then(r => r.json())
  return <ProductDetail product={product} />
}

// ── Streaming: Suspense boundaries for progressive loading ─────
async function DashboardPage() {
  return (
    <div>
      <h1>Dashboard</h1>           {/* renders immediately */}
      <Suspense fallback={<Skeleton />}>
        <RecentOrders />          {/* streams when ready */}
      </Suspense>
      <Suspense fallback={<Skeleton />}>
        <Analytics />             {/* streams independently */}
      </Suspense>
    </div>
  )
}
🏗️ 2016–Present · The Meta-Framework Era
13 — Full-Stack Frontend

The Meta-Framework Era

A meta-framework takes a UI library and adds everything needed for production: routing, data fetching, rendering strategies, build tooling, deployment targets. They are the reason "what framework should I use?" has largely been replaced by "what meta-framework should I use?"

Meta-Framework Built on Rendering model Distinctive feature
Next.js React CSR, SSR, SSG, ISR, Streaming, RSC App Router + RSC; largest ecosystem
Nuxt 3 Vue 3 CSR, SSR, SSG, hybrid Auto-imports; Nitro server engine
SvelteKit Svelte 5 CSR, SSR, SSG, streaming Progressive enhancement; file-based routing
Remix / React Router 7 React SSR-first; CSR opt-in Web standards (forms, fetch); nested routing
Astro Any (islands) SSG-first; optional hydration Zero JS by default; framework-agnostic islands
TanStack Start React SSR, SSG, streaming Type-safe routing; TanStack ecosystem
💡 Astro's Islands Architecture
Astro's core insight is that most content sites are 90% static HTML that doesn't need JavaScript at all. Its islands architecture ships zero JS by default and only hydrates the specific interactive components ("islands") that need it. A blog post with a comment form ships HTML + CSS + a small island for the comment form. No hydration cost for the article text.

15 — Choosing Wisely

Framework Selection Strategies

Framework selection is not a technical problem — it is an engineering management problem. The best framework is the one your team will be most productive with, can hire for, and can maintain over the product's lifetime.

🚀

Startups

Next.js + React. Largest hiring pool. Excellent for SEO and performance. RSC reduces backend API burden. Vercel deployment removes ops overhead. Most third-party integrations target React first.

🏢

Enterprise Systems

Angular. Strong TypeScript-first culture. Enforced architecture conventions reduce team variance. Dependency injection makes large codebases testable. Built-in i18n, forms, and accessibility tooling.

🔧

Internal Tools

React + Vite or Vue + Vite. No SEO requirements. CSR is fine. Focus on DX and iteration speed. Avoid heavyweight meta-frameworks — plain Vite is sufficient for dashboards and admin panels.

📰

Content-Heavy Sites

Astro. Maximum performance through minimal JS. Islands for interactive elements. Bring your favourite UI framework for those islands. Content collections for structured content management.

⚡

Real-Time Applications

SvelteKit or React. Svelte's fine-grained reactivity handles frequent updates efficiently. React with TanStack Query handles optimistic updates and cache invalidation well. Avoid Angular for latency-sensitive UIs.

🌍

International / High Scale

Next.js with Edge Runtime. Deploy to Vercel Edge or Cloudflare Workers. Render at the edge closest to each user. Combine with CDN caching for sub-100ms global TTFB on static and personalised pages.

🔭 2025–Beyond · The Horizon
16 — What Comes Next

The Future of Frontend Architecture

Several converging trends are reshaping frontend architecture simultaneously. The patterns that will dominate in 2028 are already visible in early adopter projects today.

⚛️

Server Components Everywhere

RSC will spread beyond React. The server-first component model — with explicit client islands — is architecturally superior for most applications. Expect Vue, Svelte, and others to adopt analogous patterns.

🌊

Edge Rendering

Application logic moving to the edge (Cloudflare Workers, Vercel Edge, Fastly). Personalised server-rendered HTML from a location 20ms from every user on earth. Eliminates the SSR latency tradeoff entirely.

🏝️

Islands Architecture Maturation

Astro proved the model. Expect fine-grained hydration control — hydrate on viewport, on interaction, on idle — to become standard across frameworks, not a special Astro feature.

🤖

AI-Assisted UI Development

LLMs are already generating production-quality component code. The highest-leverage use: generating tests, migrating between framework versions, translating design tokens to component implementations, and writing accessibility attributes.

⚙️

Compiler-Driven Frameworks

Svelte, Solid, Qwik, and Angular Signals represent a convergence toward fine-grained, compiler-optimised reactivity. The virtual DOM's era is ending — surgical DOM updates driven by explicit signals is the performance model of the next decade.

📦

Module Federation 2.0

Micro-frontend architecture with runtime module federation — loading separately deployed UI modules at runtime — is maturing. Rspack and Module Federation 2.0 make the pattern production-viable for large organisations with multiple teams.

17 — Pattern Recognition

Lessons from Twenty Years of Frontend Evolution

Every framework cycle teaches the same lessons. The engineers who recognise them make better architectural decisions, waste less time on hype cycles, and build systems that outlast the frameworks they're built on.

Patterns That Survived

✓
SINCE 2006
Declarative UI description
jQuery was imperative. Everything that succeeded it has been declarative. React made it mainstream, but the principle predates it. Describing what the UI should be rather than how to get there is structurally sound.
✓
SINCE 2014
Component-based composition
Web Components, React, Vue, Angular, Svelte — every modern approach uses components. The unit of UI is a composable, self-contained module. This pattern will outlast every current framework.
✓
SINCE 2013
Unidirectional data flow
Two-way binding (AngularJS, Knockout) lost. Unidirectional flow (React, Elm, Redux) won. Predictable data flow makes large applications debuggable. This is now the universal default.
✓
SINCE 2016
File-system routing
Next.js introduced file-system routing. Every subsequent meta-framework adopted it. The convention is now so universal that frameworks adding manual router configuration are fighting convention.

Patterns That Failed

✗
FAILED: 2010–2014
Unbounded two-way binding
AngularJS and Knockout proved that automatic two-way data binding doesn't scale. The simplicity gain at small scale is outweighed by the debugging and performance cost at large scale. React's unidirectional model won decisively.
✗
FAILED: 2015–2019
Redux for all state
Centralising server state, UI state, and form state in a single Redux store created ceremony without benefit. The revelation: server state and client state are different problems. TanStack Query, SWR, and Zustand each solved a specific slice better than Redux solved everything.
✗
FAILED: 2014–2020
Full client-side rendering for everything
The SPA model — render everything in JavaScript in the browser — failed content sites, SEO-dependent products, and low-powered devices. SSR, SSG, and streaming represent the correction. Client-side rendering is still appropriate for authenticated applications; it is not appropriate as the default.

How to Evaluate Future Frameworks

Every new framework will be presented as solving all the problems of every existing framework. Apply this filter before committing:

  • 1
    What problem does it solve, and how painful is that problem today? Svelte's compiler model solved bundle size and runtime overhead — genuinely painful at scale. Not every framework solves a real problem.
  • 2
    What does it trade away to solve that problem? Every optimisation has a cost. RSC gives you zero-cost server data access in exchange for a more complex mental model. Svelte gives you minimal runtime in exchange for a custom compile step.
  • 3
    Who is building production systems with it? GitHub stars and Hacker News enthusiasm are not production validation. Find companies with similar scale and domain using it in production and study their experience.
  • 4
    What does the migration path look like? You will eventually need to migrate off any framework. Frameworks with stable, widely-implemented primitives (components, modules, fetch) are easier to migrate from. Frameworks with deep proprietary abstractions lock you in.
  • 5
    Does it make your team faster or slower today? A framework that will pay off in three years but costs you two sprints of productivity now may not be the right choice for your current stage. Developer experience compounds.
