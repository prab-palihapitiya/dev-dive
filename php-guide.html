01 — The Runtime

Zend Engine Architecture

The Zend Engine is the core virtual machine powering PHP. Understanding it demystifies PHP's execution model — from source text to opcodes to output — and explains why certain patterns are fast, others slow, and why the language behaves the way it does.

📝

Lexer (Tokeniser)

Scans raw PHP source into a flat stream of tokens — keywords, variables, operators, strings. The first stage of compilation. Handled by ext/tokenizer.

🌳

Parser (AST Builder)

Consumes the token stream and produces an Abstract Syntax Tree. Since PHP 7, the AST is an intermediate representation — the parser no longer generates opcodes directly.

⚙️

Compiler

Walks the AST and emits Zend opcodes — a compact bytecode format. This bytecode is what OPcache stores, skipping re-compilation on subsequent requests.

🚀

Zend VM

Executes the opcode stream. Register-based, with a sophisticated dispatch mechanism. PHP 8 added JIT compilation here — hot code paths compile to native machine code.

🔧

Extensions

PDO, cURL, Redis, OPcache — all implemented as C extensions hooked into the engine. They expose functions and classes to userland while operating at native speed.

🧹

Garbage Collector

PHP uses reference counting with a cycle collector for circular references. Every zval (Zend value container) carries a refcount. When it hits zero, memory is freed immediately.

PHP 8 JIT — What It Actually Does

The Just-In-Time compiler in PHP 8 compiles hot opcode sequences directly to native CPU instructions, bypassing the VM interpreter. However, JIT in PHP 8.x primarily benefits CPU-bound workloads — mathematical computations, image processing, data transformations. Typical I/O-bound web applications see little improvement because the bottleneck is database and network, not CPU instruction dispatch.

💡 zval Internals
Every PHP value is a zval — an 16-byte struct containing a type tag and a union of the actual value. In PHP 7+ they are passed by value on the C stack where possible, which is why PHP 7 was ~2× faster than PHP 5 even without JIT: it eliminated heap allocation for most scalar operations.
02 — Execution Flow

Request Lifecycle

Every PHP request follows a deterministic path from TCP connection to response. Understanding this pipeline is essential for debugging, profiling, and knowing which extensions run at which phase.

  • 1
    SAPI Bootstrap (Server API) — PHP boots via its SAPI layer: php-fpm for web, cli for scripts, embed for libraries. FPM accepts a FastCGI connection from Nginx and passes environment variables and stdin.
  • 2
    Engine Initialisation (MINIT) — Module Initialisation. Runs once when PHP-FPM starts, not per request. All extensions register their functions, classes, and INI directives. OPcache warms its shared memory here.
  • 3
    Request Initialisation (RINIT) — Per-request setup. Superglobals ($_GET, $_POST, $_SERVER, $_COOKIE) are populated. Extension-specific request state is initialised (e.g. error handlers reset).
  • 4
    Script Execution — The autoloader fires, routing resolves, your application code runs. The Zend VM executes opcodes. This is the only phase you control as an application developer.
  • 5
    Request Shutdown (RSHUTDOWN) — Registered shutdown functions run, output buffers flush, destructors fire. Database connections close (unless persistent). Per-request memory is freed by the GC.
  • 6
    Module Shutdown (MSHUTDOWN) — Runs once when FPM stops. Extensions clean up global state. This is why PHP-FPM is extremely fast to recycle workers — the expensive initialisation happened at step 2.
ℹ️ The Shared-Nothing Model
Traditional PHP (FPM) uses a shared-nothing architecture — every request starts with a clean memory space. There is no global state leaking between requests. This makes PHP apps extraordinarily easy to reason about and scale horizontally. It is also why Octane's long-running process model requires careful attention to static state and singleton lifetimes.
PHP — Request Lifecycle Hooks 
// ── Register shutdown function — runs at RSHUTDOWN ───────────
register_shutdown_function(function() {
    // Runs even if the script exits via die() or a fatal error
    $error = error_get_last();
    if ($error && $error['type'] === E_ERROR) {
        Logger::critical('Fatal error', $error);
    }
});

// ── Measuring phases with Xdebug / tideways ──────────────────
// $_SERVER keys populated at RINIT:
$requestStart  = $_SERVER['REQUEST_TIME_FLOAT'];  // microtime of RINIT
$bootstrapTime = microtime(true) - $requestStart; // time to first line of app

// ── Autoloader registration (Composer) ───────────────────────
// Composer's generated autoloader is the first thing every app includes.
// It lazy-loads classes on first use via spl_autoload_register().
require_once __DIR__ . '/vendor/autoload.php';

// With OPcache preloading, the autoloader + all core classes are
// already compiled in shared memory — require_once hits the cache.
03 — Process Management

PHP-FPM

PHP-FPM (FastCGI Process Manager) is the de-facto standard SAPI for production PHP. It manages a pool of worker processes, handles graceful restarts, exposes a status endpoint, and implements backpressure via configurable accept queues.

Process Pool Modes

📌

static

Fixed number of workers (pm.max_children). Predictable memory usage. Best for high-traffic homogeneous workloads where you know the steady-state load.

🔄

dynamic

Starts with pm.start_servers, scales between pm.min_spare_servers and pm.max_children. Default mode. Good for variable traffic patterns.

💤

ondemand

Spawns workers only when requests arrive, kills idle ones. Zero idle memory overhead. Suitable for low-traffic pools or staging environments.

INI — php-fpm.conf: Production Pool Configuration 
[www]
user  = www-data
group = www-data

; FastCGI socket — Unix socket is faster than TCP for same-host Nginx
listen = /run/php/php8.3-fpm.sock
listen.owner = www-data
listen.group = www-data
listen.mode  = 0660

; Process management — static for consistent high-load production
pm = static
pm.max_children = 50          ; Formula: (RAM - OS overhead) / per-worker memory
                              ; e.g. (4096MB - 512MB) / 70MB ≈ 50 workers

; Slow log — catch long-running requests
slowlog = /var/log/php-fpm-slow.log
request_slowlog_timeout = 5s   ; log PHP stack trace if request > 5s

; Prevent memory leaks accumulating over time
pm.max_requests = 500          ; recycle worker after N requests

; Status and ping endpoints (enable in monitoring)
pm.status_path = /fpm-status
ping.path       = /fpm-ping

; Environment — pass through necessary vars from Nginx fastcgi_params
env[APP_ENV]  = production
env[LOG_LEVEL]= error

; Per-request limits override php.ini for this pool
php_admin_value[memory_limit]       = 256M
php_admin_value[max_execution_time] = 30

Sizing Your Worker Pool

The fundamental formula is: workers = (available RAM) ÷ (peak memory per worker). Measure actual peak memory with memory_get_peak_usage(true) in production, add 20% headroom, then divide. Over-provisioning workers doesn't add capacity — it causes thrashing when workers compete for RAM and the OS starts swapping.

Bash — Diagnose FPM Worker Memory & Pool Health 
# Average memory per FPM worker (RSS in KB)
ps -ylC php-fpm8.3 | awk '{sum+=$8; n++} END {print "Avg:", sum/n/1024, "MB"}'

# FPM status page (requires pm.status_path configured)
curl -s http://127.0.0.1/fpm-status | grep -E "active|idle|queue"

# Output format:
# pool:               www
# active processes:   12
# idle processes:     38
# listen queue:       0       ← if > 0, you need more workers
# max active:         18
# slow requests:      3       ← check slowlog

# Graceful reload (zero-downtime config reload)

04 — Compilation Cache

OPcache

OPcache stores compiled PHP opcodes in shared memory, eliminating the parse → compile cycle on every request. On a mature Laravel or Symfony application, enabling OPcache typically reduces response time by 3–7× and CPU usage proportionally. It is non-optional in production.

💾

Shared Memory Segment

A single shared memory region, accessible by all FPM workers simultaneously with no copying. Set memory_consumption large enough that your entire codebase fits — check with opcache_get_status().

🔑

Interned Strings Buffer

Deduplicates string literals across all cached files. A class name like "App\Models\User" is stored once and shared. Set interned_strings_buffer to at least 16MB for large codebases.

🏎️

Preloading (PHP 7.4+)

Load and compile specified files into shared memory at FPM startup. Framework core classes (Laravel's Container, Illuminate support) are available to all workers instantly, no autoloading at all.

🔁

File Change Detection

In development, set validate_timestamps=1 with revalidate_freq=0. In production, disable with validate_timestamps=0 — PHP never hits the filesystem to check for changes.

PHP — OPcache Preload Script 
// preload.php — referenced by opcache.preload in php.ini
// Runs once at FPM startup as the opcache.preload_user
// Preloaded classes are available WITHOUT autoloading in any worker
<?php
require_once __DIR__ . '/vendor/autoload.php';

$directory = new RecursiveDirectoryIterator(__DIR__ . '/vendor/laravel/framework/src');
$iterator  = new RecursiveIteratorIterator($directory);

foreach ($iterator as $file) {
    if ($file->getExtension() !== 'php') continue;
    try {
        opcache_compile_file($file->getRealPath());
    } catch (Throwable $e) { continue; }
}

// Laravel ships its own preload generator:
// php artisan optimize  → generates bootstrap/cache/packages.php
// php artisan config:cache, route:cache, view:cache
// These write serialised PHP arrays — no parse/compile on boot

05 — Memory Model

Memory Management

PHP's memory model is built on reference counting. Understanding it prevents memory leaks, explains copy-on-write behaviour, and makes large-data processing efficient.

PHP — Reference Counting, COW & Circular References 
// ── Copy-on-Write (COW) ───────────────────────────────────────
// Arrays are NOT copied until one is modified.
// Both $a and $b point to the same zval — refcount=2
$a = [1, 2, 3];
$b = $a;      // no copy yet — COW
$b[] = 4;     // NOW $b is copied (separated) — $a unchanged

// ── Explicit references — bypass COW ─────────────────────────
$c = &$a;     // $c is an alias — both point to the same memory
$c[] = 99;   // modifies $a too — they share the zval

// ── Circular reference — GC cycle collector handles this ─────
class Node {
    public ?Node $next = null;
}
$node       = new Node();
$node->next = $node;  // circular — refcount never hits zero!
unset($node);          // refcount = 1 (self-reference) — not freed yet
gc_collect_cycles();  // PHP's cycle collector frees it

// ── Processing large datasets without memory explosion ────────
// ❌ Loads entire file into memory at once — will exhaust memory_limit
$data = file_get_contents('huge.csv');
$rows = explode("\n", $data);

// ✅ Generator — lazy, one line at a time, constant memory
function readCsvLines(string $path): Generator
{
    $handle = fopen($path, 'r');
    try {
        while (($line = fgets($handle)) !== false) {
            yield str_getcsv(rtrim($line));
        }
    } finally {
        fclose($handle);
    }
}

foreach (readCsvLines('10GB.csv') as $row) {
    processRow($row);    // ~4KB peak memory regardless of file size
}

07 — Inversion of Control

Dependency Injection Containers

A DI container is an object graph factory. It reads constructor type-hints via reflection, resolves the full dependency tree automatically, and manages object lifetimes (singleton, transient, scoped). Understanding what the container does — and what it costs — makes you a more effective Laravel and Symfony developer.

PHP — Manual DI vs Container Autowiring 
// ── Without container — manual wiring ────────────────────────
$logger      = new PinoLogger(LogLevel::INFO);
$db          = new DatabaseConnection(getenv('DATABASE_URL'), $logger);
$userRepo    = new UserRepository($db);
$mailer      = new SmtpMailer(getenv('SMTP_DSN'));
$userService = new UserService($userRepo, $mailer, $logger);

// ── With Laravel's container — autowired via reflection ───────
class UserService
{
    public function __construct(
        private readonly UserRepository $users,
        private readonly MailerInterface $mailer,
        private readonly LoggerInterface  $logger,
    ) {}
}
// Container reads type-hints, resolves the full tree, injects automatically
$service = app(UserService::class);

// ── Binding interfaces to implementations ─────────────────────
// In a ServiceProvider:
public function register(): void
{
    // Singleton — one instance per container lifetime
    $this->app->singleton(MailerInterface::class, SmtpMailer::class);

    // Transient — new instance every time
    $this->app->bind(ReportGeneratorInterface::class, PdfReportGenerator::class);

    // Contextual binding — same interface, different impl per consumer
    $this->app->when(PhotoController::class)
              ->needs(StorageInterface::class)
              ->give(S3Storage::class);
}
⚠️ Container Performance
Reflection-based autowiring has overhead. Laravel mitigates this with a compiled binding cache. In Octane (long-running process), the container itself is reused across requests — avoid registering closures that capture request-specific state as singletons. Use $this->app->scoped() for per-request singletons in Octane.
08 — Framework Architecture

Laravel Architecture

Laravel is not just a collection of helpers — it is a cohesive application framework built around a service container, service providers, a request-response pipeline, and a set of first-party packages that cover every common concern in web application development.

The Boot Sequence

  • 1
    public/index.php — Entry point. Loads Composer autoloader, creates the Application instance, and resolves the HTTP kernel from the container.
  • 2
    Application Bootstrap — The Application class (extends the IoC Container) binds the core framework abstractions: ExceptionHandler, Kernel, ConsoleKernel.
  • 3
    HTTP Kernel Bootstrappers — Runs in order: LoadEnvironmentVariables, LoadConfiguration, HandleExceptions, RegisterFacades, RegisterProviders, BootProviders.
  • 4
    Service Provider Registration & Boot — Every ServiceProvider's register() runs first (all of them), then every boot() runs. This two-phase ensures all bindings exist before any boot logic tries to resolve them.
  • 5
    Middleware Pipeline — The request passes through the global middleware stack, then route middleware groups, then route-specific middleware. Implemented as a recursive closure chain.
  • 6
    Route Dispatch → Controller → Response — The router matches the request, the controller action is resolved from the container (injecting its dependencies), the response object is built and passed back through the middleware pipeline.
PHP — Laravel: Eloquent, Events & Jobs 
// ── Eloquent: relationships & scopes ─────────────────────────
class Order extends Model
{
    protected $casts = [
        'metadata'   => 'array',
        'placed_at'  => 'immutable_datetime',
        'status'     => OrderStatus::class,  // backed enum cast
    ];

    public function items(): HasMany
    {
        return $this->hasMany(OrderItem::class);
    }

    public function scopePaid(Builder $query): Builder
    {
        return $query->where('status', OrderStatus::Paid);
    }

    protected static function booted(): void
    {
        static::created(fn(Order $order) =>
            OrderPlaced::dispatch($order)  // fire domain event
        );
    }
}

// ── Job: async queue worker ───────────────────────────────────
class SendOrderConfirmation implements ShouldQueue
{
    use Dispatchable, InteractsWithQueue, Queueable;

    public int    $tries   = 3;
    public int    $timeout = 30;
    public string $queue   = 'notifications';

    public function __construct(public readonly Order $order) {}

    public function handle(MailerInterface $mailer): void
    {
        $mailer->to($this->order->user)
               ->send(new OrderConfirmationMail($this->order));
    }

    public function failed(Throwable $e): void
    {
        Log::error('Confirmation email failed', [
            'order_id'  => $this->order->id,
            'exception' => $e->getMessage(),
        ]);
    }
}

PHP — Symfony: Autowired Controller + Messenger 
use Symfony\Component\HttpFoundation\JsonResponse;
use Symfony\Component\Messenger\MessageBusInterface;
use Symfony\Component\Routing\Attribute\Route;
use Symfony\Component\Validator\Validator\ValidatorInterface;

// Symfony compiles the DI container — no reflection at runtime
class OrderController
{
    public function __construct(
        private readonly MessageBusInterface  $bus,
        private readonly ValidatorInterface   $validator,
    ) {}

    #[Route('/orders', methods: ['POST'])]
    public function create(PlaceOrderRequest $dto): JsonResponse
    {
        $violations = $this->validator->validate($dto);
        if (count($violations) > 0) {
            return new JsonResponse(['errors' => $violations], 422);
        }
        // Dispatch async — handled by a worker process via Redis/SQS
        $this->bus->dispatch(new PlaceOrderCommand($dto));
        return new JsonResponse(['status' => 'accepted'], 202);
    }
}
💡 Symfony's Compiled Container
Symfony's DI container generates PHP code at build time — var/cache/prod/App_KernelProdContainer.php. This compiled file has zero reflection at runtime. The container is a flat PHP class with hardcoded new calls. This is why Symfony boot time is extremely predictable regardless of service count.
10 — Structural Thinking

MVC & Clean Architecture in PHP

MVC is a UI pattern, not an application architecture. In complex PHP applications, controllers should be thin orchestrators — delegating to a domain layer that has no knowledge of HTTP, Eloquent, or any framework.

❌ Fat Controller (MVC anti-pattern) 
class OrderController extends Controller
{
    public function store(Request $request)
    {
        // ❌ Validation in controller
        $request->validate(['items' => 'required|array']);
        // ❌ Business logic in controller
        $discount = auth()->user()->isPremium() ? 0.1 : 0;
        // ❌ Database access in controller
        $order = Order::create([...]);
        // ❌ Side effects in controller
        Mail::to($order->email)->send(new Confirm());
        return response()->json($order);
    }
}
✅ Thin Controller → Use Case 
class OrderController extends Controller
{
    public function __construct(
        private PlaceOrderUseCase $useCase
    ) {}

    public function store(PlaceOrderRequest $request)
    {
        // ✅ Controller: validate + delegate + respond
        $order = $this->useCase->execute(
            $request->toDto()
        );
        return OrderResource::make($order)
            ->response()->setStatusCode(201);
    }
}

// PlaceOrderUseCase knows nothing about HTTP,
// can be tested without the framework
PHP — Domain Entity With Value Objects 
// Value object — immutable, self-validating
final readonly class Money
{
    public function __construct(
        public int    $amount,      // in cents — never use floats for money
        public string $currency,
    ) {
        if ($amount < 0) throw new \DomainException('Amount cannot be negative');
    }

    public function add(Money $other): self
    {
        if ($this->currency !== $other->currency)
            throw new \DomainException('Currency mismatch');
        return new self($this->amount + $other->amount, $this->currency);
    }

    public function format(): string
    {
        return number_format($this->amount / 100, 2) . ' ' . $this->currency;
    }
}

12 — High-Performance PHP

Octane, Swoole & Modern High-Performance PHP

Laravel Octane fundamentally changes PHP's execution model by keeping the application booted in memory across requests. Combined with Swoole or RoadRunner, it delivers Node.js-level concurrency with PHP's developer experience — easily handling tens of thousands of requests per second.

"Octane doesn't make PHP faster by magic. It makes PHP faster by eliminating the work PHP was doing on every single request: booting the framework."
🚀

Swoole

C extension that replaces PHP-FPM with an event-loop server. Coroutines, async I/O, connection pooling — all within a single long-running PHP process. Truly async PHP.

🛤️

RoadRunner

Go-based application server. PHP workers communicate via gRPC. No C extensions needed — runs anywhere. More portable than Swoole, slightly less feature-rich.

🔄

FrankenPHP

Embed PHP into a Go web server (Cadence-based). Worker mode for persistent processes. Built-in TLS, HTTP/3, and file watching. The newest and fastest-growing option.

⚡

Connection Pooling

Octane + Swoole keeps DB connections open across requests. Massive win — no TCP handshake per request. Requires careful management of connection state.

PHP — Octane: State Management & Concurrency 
// ── The critical difference: request lifecycle in Octane ──────
//
// PHP-FPM: new process (or worker reset) for EVERY request
//          Global state, singletons, static vars all reset automatically
//
// Octane: application stays booted, ONE process handles MANY requests
//         You MUST reset request-specific state between requests

// ── Register listeners in AppServiceProvider ─────────────────
use Laravel\Octane\Facades\Octane;
use Laravel\Octane\Events\RequestReceived;
use Laravel\Octane\Events\RequestTerminated;

Octane::listen(RequestReceived::class, function (RequestReceived $event) {
    // Reset per-request state at the start of each request
    RequestContext::clear();
    DB::reconnect();   // reconnect if connection was dropped
});

Octane::listen(RequestTerminated::class, function () {
    DB::purge();      // release connections back to pool
});

// ── Swoole coroutines — true concurrency within one worker ────
// Octane::concurrently() runs tasks in parallel coroutines
use Laravel\Octane\Facades\Octane;

[$users, $products, $stats] = Octane::concurrently([
    fn() => User::active()->count(),
    fn() => Product::featured()->get(),
    fn() => OrderStats::today(),
]);
// All three DB queries execute simultaneously — total time = slowest query
// vs sequential: sum of all query times

// ── Common Octane pitfalls ────────────────────────────────────
// ❌ Static properties accumulate across requests
class RequestLogger {
    private static array $log = [];    // grows forever in Octane!
    public static function add(string $msg): void {
        self::$log[] = $msg;            // ← MEMORY LEAK
    }
}
// ✅ Use $app->scoped() for per-request singletons in Octane
// ✅ Register RequestReceived listener to reset static state
// ✅ Run `php artisan octane:check` to scan for memory leak patterns

Performance Benchmarks: Traditional PHP vs Octane

Scenario PHP-FPM Octane + Swoole Improvement
Simple JSON response ~2,000 req/s ~30,000 req/s 15×
Laravel with 1 DB query ~800 req/s ~8,000 req/s 10×
Laravel hello world ~3,000 req/s ~40,000 req/s 13×
Memory per worker ~70MB (reset each req) ~100MB (persistent) ~1.4× more RAM
🚨 Octane is Not PHP-FPM
Octane eliminates the shared-nothing model. Memory leaks, static state pollution, and non-thread-safe code that worked fine in FPM (because the process recycled) will silently corrupt responses or grow memory without bound. Always test your application with php artisan octane:check and load-test with realistic traffic before deploying.
