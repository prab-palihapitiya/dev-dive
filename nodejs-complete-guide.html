01 — The Foundation

Architecture & the Event Loop

Node.js is a single-threaded, non-blocking runtime built on V8 and libuv. Understanding its event loop is non-negotiable for writing performant server-side code.

Unlike threaded servers (Apache, Java), Node.js uses a single thread with an asynchronous I/O model. When I/O happens (file reads, network calls, DB queries), Node delegates to libuv's thread pool and moves on — the callback fires when the operation completes. This is the secret to handling tens of thousands of concurrent connections on commodity hardware.

🔄

Event Loop

Single thread that continuously checks queues of callbacks to execute. Six distinct phases, each with its own FIFO queue.

⚙️

libuv Thread Pool

4 threads by default (configurable with UV_THREADPOOL_SIZE) for CPU-bound I/O: file system, DNS, crypto.

🔢

V8 Engine

Google's JavaScript engine. Compiles JS to machine code via JIT. Manages the heap, garbage collector, and JS execution.

📡

Non-blocking I/O

Network and I/O operations are handled by the OS kernel directly, signaling Node via epoll/kqueue when data is ready.

The Six Event Loop Phases

JavaScript — Event Loop Phase Order
// Event loop phases (simplified):
//  1. timers        — setTimeout, setInterval callbacks
//  2. pending I/O   — deferred I/O callbacks from previous iteration
//  3. idle/prepare  — internal use only
//  4. poll          — retrieve new I/O events; execute I/O callbacks
//  5. check         — setImmediate callbacks
//  6. close         — close event callbacks (socket.on('close'))
//
// Between each phase: process.nextTick + Promise microtasks drain first

setTimeout(() => console.log('setTimeout'),   0)
setImmediate(() => console.log('setImmediate'))
Promise.resolve().then(() => console.log('Promise microtask'))
process.nextTick(() => console.log('nextTick'))
console.log('synchronous')

// Output order:
//   synchronous         ← executes immediately
//   nextTick            ← drains before any I/O phase
//   Promise microtask   ← microtask queue (after nextTick queue)
//   setTimeout          ← timers phase (next loop iteration)
//   setImmediate        ← check phase (same or next iteration)

// ── Blocking the event loop — NEVER DO THIS ──────
// Bad: synchronous heavy computation blocks all requests
app.get('/compute', (req, res) => {
  let sum = 0
  for (let i = 0; i < 1e9; i++) sum += i  // blocks for ~1s!
  res.json({ sum })
})

// Good: offload to worker thread (see section 11)
app.get('/compute', async (req, res) => {
  const sum = await runInWorker(heavyComputation)
  res.json({ sum })
})
⚠️ Event Loop Blocking
Any synchronous operation over ~1–2ms will delay all other requests. JSON parsing of large payloads, complex regex on untrusted input, and synchronous crypto are the most common offenders. Use --prof and clinic.js to identify blockage.
02 — Module System

Module System: CommonJS vs ESM

Node.js supports two module systems. CommonJS is the legacy default; ECMAScript Modules (ESM) is the modern standard. Understanding both — and how to interop — is essential.

CommonJS (CJS)
// Synchronous, dynamic requires ok
const path = require('path')
const { readFile } = require('fs')

// Conditional import at runtime
if (isDev) require('./debug')

module.exports = { myFn }
// package.json: "type": "commonjs"
// or .cjs extension
ESM (Modern)
// Static, hoisted, tree-shakeable
import path from 'node:path'
import { readFile } from 'node:fs/promises'

// Dynamic import still possible
const mod = await import(path)

export { myFn }
export default myFn
// package.json: "type": "module"
// or .mjs extension
JavaScript — Module Best Practices
// ── Always use 'node:' prefix for built-ins (Node 14.18+) ──
import fs   from 'node:fs/promises'   // not 'fs'
import path from 'node:path'
import http from 'node:http'
import { createHash } from 'node:crypto'

// ── __dirname / __filename equivalents in ESM ──────────────
import { fileURLToPath } from 'node:url'
import { dirname }      from 'node:path'

const __filename = fileURLToPath(import.meta.url)
const __dirname  = dirname(__filename)

// ── Barrel exports pattern ──────────────────────────────────
// services/index.js — centralise public API
export { UserService }  from './user.service.js'
export { AuthService }  from './auth.service.js'
export { MailService }  from './mail.service.js'

// ── Interop: require a CJS package in ESM ─────────────────

const require = createRequire(import.meta.url)
const legacyLib = require('legacy-cjs-package')
03 — Asynchronous Node.js

Async Patterns & Robust Error Handling

Node's single-threaded model means unhandled rejections and synchronous throws can crash your entire process. Structured async patterns are not optional — they are the job.

Async/Await with Proper Error Handling

JavaScript — Production Async Patterns
// ── Result-type pattern — no more try/catch soup ──
async function safe(promise) {
  try {
    return [null, await promise]
  } catch (err) {
    return [err, null]
  }
}

const [err, user] = await safe(db.findUser(id))
if (err) return res.status(500).json({ error: err.message })

// ── Retry with exponential backoff ───────────────
async function withRetry(fn, { retries = 3, base = 200 } = {}) {
  for (let attempt = 0; attempt <= retries; attempt++) {
    try {
      return await fn()
    } catch (err) {
      if (attempt === retries) throw err
      const delay = base * 2 ** attempt + Math.random() * 100
      await new Promise(r => setTimeout(r, delay))
    }
  }
}

// ── Timeout wrapper ───────────────────────────────
function withTimeout(promise, ms) {
  const timeout = new Promise((_, reject) =>
    setTimeout(() => reject(new Error(`Timed out after ${ms}ms`)), ms)
  )
  return Promise.race([promise, timeout])
}

// ── Global safety net — always add these ─────────
process.on('unhandledRejection', (reason) => {
  logger.error({ reason }, 'Unhandled promise rejection')
  process.exit(1)
})
process.on('uncaughtException', (err) => {
  logger.fatal({ err }, 'Uncaught exception — shutting down')
  process.exit(1)
})

Async Iteration & Concurrency Control

JavaScript — Concurrency Patterns
// ── Limit concurrency — avoid overwhelming DB/API ─
async function pLimit(tasks, concurrency) {
  const results = []
  const executing = new Set()

  for (const task of tasks) {
    const p = task().then(res => {
      results.push(res)
      executing.delete(p)
    })
    executing.add(p)
    if (executing.size >= concurrency)
      await Promise.race(executing)
  }
  await Promise.allSettled(executing)
  return results
}

// Process 1000 URLs, 10 at a time
const tasks = urls.map(url => () => fetch(url))
const results = await pLimit(tasks, 10)

// ── Async generators for lazy streaming ──────────
async function* paginate(url) {
  let page = 1
  while (true) {
    const data = await fetch(`${url}?page=${page}`).then(r => r.json())
    if (!data.items.length) break
    yield* data.items
    page++
  }
}

for await (const item of paginate('/api/products')) {
  await processItem(item)  // memory-efficient, lazy
}
04 — Data in Motion

Streams & Buffers

Streams are Node's most powerful and underused primitive. They let you process gigabytes of data with constant memory usage by operating on chunks rather than loading everything at once.

There are four stream types: Readable (source), Writable (destination), Duplex (both), and Transform (duplex that modifies data). Every stream is an EventEmitter.

JavaScript — Stream Fundamentals
import { createReadStream, createWriteStream } from 'node:fs'
import { createGzip }             from 'node:zlib'
import { pipeline }               from 'node:stream/promises'
import { Transform }              from 'node:stream'

// ── ALWAYS use pipeline — it handles backpressure + cleanup ─
await pipeline(
  createReadStream('huge-file.csv'),
  createGzip(),
  createWriteStream('huge-file.csv.gz')
)
// Compresses a 2GB file using ~64KB of memory — try that with readFileSync!

// ── Custom Transform stream ───────────────────────
class CSVToJSON extends Transform {
  #headers = null
  #remainder = ''

  _transform(chunk, encoding, callback) {
    const lines = (this.#remainder + chunk.toString()).split('\n')
    this.#remainder = lines.pop()  // save incomplete last line

    for (const line of lines) {
      if (!this.#headers) {
        this.#headers = line.split(',')
        continue
      }
      const values = line.split(',')
      const obj = Object.fromEntries(this.#headers.map((h, i) => [h, values[i]]))
      this.push(JSON.stringify(obj) + '\n')
    }
    callback()
  }
}

// Stream a 500MB CSV to JSON without loading it in memory
await pipeline(
  createReadStream('big.csv'),
  new CSVToJSON(),
  createWriteStream('output.jsonl')
)

Buffers — Understanding Binary Data

JavaScript — Buffer Operations
// ── Buffer creation ───────────────────────────────
const buf1 = Buffer.from('hello world', 'utf8')
const buf2 = Buffer.alloc(16)          // zero-filled, safe
const buf3 = Buffer.allocUnsafe(16)    // faster, not zeroed

// ── Encoding / decoding ───────────────────────────
buf1.toString('hex')     // '68656c6c6f...'
buf1.toString('base64')  // 'aGVsbG8gd29ybGQ='

// ── Constant-time comparison (timing-safe!) ───────
import { timingSafeEqual } from 'node:crypto'
const a = Buffer.from(tokenFromUser)
const b = Buffer.from(storedToken)
const valid = a.length === b.length && timingSafeEqual(a, b)
💡 Backpressure
When a writable stream's internal buffer fills up, write() returns false. You must pause the readable until the writable emits 'drain'. pipeline() handles this automatically — it's the main reason to always prefer it over manual piping.
05 — Building Servers

HTTP Servers & Express

Node's built-in http module is lean and powerful. Express adds routing and middleware. Understanding both makes you a better API developer at either layer.

Bare HTTP Server — The Foundation

JavaScript — Native HTTP Server
import http from 'node:http'

const server = http.createServer((req, res) => {
  // Read body manually (req is a Readable stream)
  const chunks = []
  req.on('data', chunk => chunks.push(chunk))
  req.on('end', () => {
    const body = Buffer.concat(chunks).toString()
    res.writeHead(200, { 'Content-Type': 'application/json' })
    res.end(JSON.stringify({ received: body.length }))
  })
})

server.listen(3000, () => console.log('Server on :3000'))

Express — Production API Structure

JavaScript — Express Production Setup
import express     from 'express'
import helmet      from 'helmet'
import rateLimit   from 'express-rate-limit'
import compression from 'compression'

const app = express()

// ── Security middleware ───────────────────────────
app.use(helmet())               // Sets 14 security headers
app.use(compression())          // gzip responses
app.use(express.json({ limit: '100kb' }))  // limit body size

app.use(rateLimit({
  windowMs: 15 * 60 * 1000,   // 15 minutes
  max:       100,              // 100 requests per window
  standardHeaders: true,
}))

// ── Router separation ─────────────────────────────
const userRouter = express.Router()

userRouter.get('/:id',         asyncHandler(getUser))
userRouter.post('/',           asyncHandler(createUser))
userRouter.patch('/:id',       asyncHandler(updateUser))
userRouter.delete('/:id',      asyncHandler(deleteUser))

app.use('/api/users', authenticate, userRouter)

// ── asyncHandler — no more try/catch in controllers
function asyncHandler(fn) {
  return (req, res, next) => Promise.resolve(fn(req, res, next)).catch(next)
}

// ── Centralized error handler (last middleware) ───
app.use((err, req, res, _next) => {
  const status = err.status || 500
  logger.error({ err, req: req.path })
  res.status(status).json({
    error: status < 500 ? err.message : 'Internal server error'
  })
})

07 — Architecture

Design Patterns in Node.js

Node's module system, EventEmitter, and async-first design lend themselves to patterns that feel different from classical OOP. These are the ones that matter most at scale.

Dependency Injection Container

JavaScript — DI & Service Layer
// ── Simple DI container ───────────────────────────
class Container {
  #registry = new Map()

  register(name, factory) {
    this.#registry.set(name, { factory, instance: null })
    return this
  }
  resolve(name) {
    const entry = this.#registry.get(name)
    if (!entry) throw new Error(`Unknown service: ${name}`)
    entry.instance ??= entry.factory(this)  // singleton
    return entry.instance
  }
}

const container = new Container()
  .register('db',   ()    => new Database(process.env.DATABASE_URL))
  .register('mail', ()    => new MailService(process.env.SMTP_URL))
  .register('users', (c)  => new UserService(c.resolve('db'), c.resolve('mail')))

// ── EventEmitter for decoupled events ────────────
import { EventEmitter } from 'node:events'

class OrderService extends EventEmitter {
  async create(data) {
    const order = await db.orders.create(data)
    this.emit('order.created', order)  // decouple side effects
    return order
  }
}

const orders = new OrderService()
orders.on('order.created', sendConfirmationEmail)
orders.on('order.created', updateInventory)
orders.on('order.created', notifyWarehouse)
08 — Data Layer

Databases & ORMs

Node.js works well with every major database. Connection pooling and query efficiency are where most performance gains — and losses — live.

Prisma — Type-Safe ORM

JavaScript — Prisma ORM Patterns
import { PrismaClient } from '@prisma/client'

// ── Single shared instance — critical for pooling ─
const prisma = globalThis.prisma ??= new PrismaClient({
  log: ['query', 'error']
})

// ── Eager loading prevents N+1 queries ───────────
const users = await prisma.user.findMany({
  include: { posts: { include: { tags: true } } },
  where:   { active: true },
  orderBy: { createdAt: 'desc' },
  skip:    (page - 1) * pageSize,
  take:    pageSize
})

// ── Transactions ──────────────────────────────────
const [transfer, _] = await prisma.$transaction([
  prisma.account.update({
    where: { id: fromId },
    data:  { balance: { decrement: amount } }
  }),
  prisma.account.update({
    where: { id: toId },
    data:  { balance: { increment: amount } }
  })
])

// ── Raw queries when ORM isn't enough ────────────
const stats = await prisma.$queryRaw`
  SELECT DATE_TRUNC('month', created_at) AS month,
         COUNT(*)::int                   AS count
  FROM orders
  GROUP BY 1 ORDER BY 1 DESC LIMIT 12
`
💡 Connection Pool Sizing
For PostgreSQL with Prisma, the default pool is min: 2, max: 10. In a clustered app with 4 workers, you'll have up to 40 connections. Most Postgres servers cap at 100. Set connection_limit to Math.floor(100 / numWorkers) and use PgBouncer in production.
09 — Hardening

Security Hardening

Node.js apps face the same OWASP Top 10 threats as any web app — plus a few Node-specific pitfalls. Security is not a middleware you add at the end; it is woven through every decision.

JWT Authentication Done Right

JavaScript — JWT & Auth Patterns
import jwt    from 'jsonwebtoken'
import bcrypt from 'bcrypt'

// ── Password hashing ──────────────────────────────
const SALT_ROUNDS = 12  // tune so hash takes ~300ms
const hash     = await bcrypt.hash(password, SALT_ROUNDS)
const isValid  = await bcrypt.compare(password, hash)

// ── Token pair: access (short) + refresh (long) ──
function issueTokens(userId) {
  const access = jwt.sign(
    { sub: userId, type: 'access' },
    process.env.JWT_SECRET,
    { expiresIn: '15m', algorithm: 'HS256' }
  )
  const refresh = jwt.sign(
    { sub: userId, type: 'refresh', jti: crypto.randomUUID() },
    process.env.JWT_REFRESH_SECRET,
    { expiresIn: '7d' }
  )
  return { access, refresh }
}

// ── Middleware: verify & attach user ─────────────
async function authenticate(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1]
  if (!token) return res.status(401).json({ error: 'Unauthorised' })
  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET)
    next()
  } catch {
    res.status(401).json({ error: 'Invalid or expired token' })
  }
}

⚠️ Prototype Pollution
Merging user-controlled objects (like req.body) using naive spread or deep-merge utilities can pollute Object.prototype, affecting all objects globally. Always use Object.create(null) as merge targets, or libraries patched against this attack (e.g. lodash 4.17.21+).

11 — Scaling

Clustering & Worker Threads

A single Node.js process uses one CPU core. To use all cores: cluster spawns multiple processes (each with its own event loop), while worker threads share memory in the same process for CPU-intensive work.

JavaScript — Cluster Module
import cluster from 'node:cluster'
import os      from 'node:os'
import { createServer } from 'node:http'

if (cluster.isPrimary) {
  const numCPUs = os.cpus().length
  console.log(`Primary ${process.pid} spawning ${numCPUs} workers`)

  for (let i = 0; i < numCPUs; i++) cluster.fork()

  cluster.on('exit', (worker, code) => {
    console.error(`Worker ${worker.process.pid} died (${code}) — restarting`)
    cluster.fork()  // auto-restart crashed workers
  })
} else {
  // Worker: each runs the full Express/http app
  const { app } = await import('./app.js')
  app.listen(3000)
}

Worker Threads for CPU-Bound Tasks

JavaScript — Worker Thread Pool
import { Worker, isMainThread, parentPort, workerData } from 'node:worker_threads'

// ── worker.js — the CPU-bound work ────────────────
if (!isMainThread) {
  const { data } = workerData
  const result = heavyProcessing(data)  // runs on separate thread

// ── main thread — dispatch to worker ─────────────
function runInWorker(data) {
  return new Promise((resolve, reject) => {
    const worker = new Worker(new URL('./worker.js', import.meta.url), {
      workerData: { data }
    })
    worker.on('message', resolve)
    worker.on('error', reject)
  })
}

12 — Going Fast

Performance & Profiling

Profile before optimising. The most common Node.js bottlenecks are I/O queue starvation, GC pressure from too many allocations, and JSON parsing of large payloads.

JavaScript — Performance Techniques
// ── Object pooling — reduce GC pressure ──────────
class ObjectPool {
  #pool = []
  #create

  constructor(create) { this.#create = create }

  acquire()     { return this.#pool.pop() ?? this.#create() }
  release(obj)  { this.#pool.push(obj) }
}

const bufPool = new ObjectPool(() => Buffer.allocUnsafe(64 * 1024))

// ── Caching with LRU ──────────────────────────────
import { LRUCache } from 'lru-cache'

const cache = new LRUCache({
  max:     500,            // max 500 items
  ttl:     60_000,         // 60 second TTL
  maxSize: 5_000_000,      // 5MB max total
  sizeCalculation: v => JSON.stringify(v).length
})

async function getCachedUser(id) {
  const cached = cache.get(id)
  if (cached) return cached

  const user = await db.users.findUnique({ where: { id } })
  cache.set(id, user)
  return user
}

// ── Fast JSON with simdjson / orjson alternatives ─
// For large payloads, consider fast-json-stringify:
import fastJson from 'fast-json-stringify'
const stringify = fastJson({
  type: 'object',
  properties: {
    id:    { type: 'string' },
    name:  { type: 'string' },
    score: { type: 'number' }
  }
})  // 2-3× faster than JSON.stringify for known schemas
13 — Shipping to Production

Docker & Deployment

A well-crafted Dockerfile is the difference between a 1.2GB image and a 120MB one. Multi-stage builds and non-root users are non-negotiable for production.

Dockerfile — Multi-Stage Production Build
# ── Stage 1: install dependencies ────────────────
FROM node:22-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

# ── Stage 2: final image ──────────────────────────
FROM node:22-alpine AS runner
WORKDIR /app

# Run as non-root
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

COPY --from=deps /app/node_modules ./node_modules
COPY --chown=appuser:appgroup . .

# Tell Node.js it's in production
ENV NODE_ENV=production
ENV PORT=3000
EXPOSE 3000

JavaScript — Graceful Shutdown Handler
// ── Essential for Kubernetes rolling deploys ──────
async function gracefulShutdown(signal) {
  logger.info(`Received ${signal} — starting graceful shutdown`)

  // 1. Stop accepting new connections
  await new Promise((resolve, reject) =>
    server.close(err => err ? reject(err) : resolve())
  )

  // 2. Finish in-flight requests (wait up to 10s)
  await withTimeout(waitForActiveRequests(), 10_000)

  // 3. Close DB connections
  await prisma.$disconnect()

  logger.info('Shutdown complete')
  process.exit(0)
}

process.on('SIGTERM', gracefulShutdown)
process.on('SIGINT',  gracefulShutdown)

You can't improve what you can't measure. The three pillars of observability — logs, metrics, traces — each answer a different question about your running system.

JavaScript — Structured Logging with Pino
import pino from 'pino'

// ── Structured JSON logging (fastest Node.js logger)
const logger = pino({
  level:    process.env.LOG_LEVEL || 'info',
  redact:  ['req.headers.authorization', '*.password'],  // auto-redact secrets
  base:    { service: 'api', version: process.env.npm_package_version }
})

// ── Request logging middleware ────────────────────
function requestLogger(req, res, next) {
  const start = Date.now()
  res.on('finish', () => {
    logger.info({
      method:      req.method,
      url:         req.url,
      status:      res.statusCode,
      durationMs:  Date.now() - start,
      ip:          req.ip,
    }, 'request')
  })
  next()
}
