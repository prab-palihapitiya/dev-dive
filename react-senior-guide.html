01 — Origins

Why React Exists

React was not created to be a component library. It was created to solve a specific, painful problem that all large web applications hit: as UI state grows, keeping the DOM in sync with application state becomes exponentially more complex and unpredictable.

The Frontend Before React

Before React, the dominant model was imperative DOM manipulation. jQuery made it easy; it did not make it correct. When your application had dozens of UI elements that all needed to reflect the same piece of state, you wrote code like:

JavaScript — jQuery Era
// Every time user data changes, manually update every element
function onUserLoaded(user) {
  $('#header-name').text(user.name);
  $('#sidebar-avatar').attr('src', user.avatar);
  $('#profile-email').text(user.email);
  $('#dashboard-greeting').text('Hello, ' + user.name);
  // ... and 20 more elements scattered across the page
  // Miss one? Silent bug. State is out of sync.
}

The core problem: state lived in the DOM itself. The DOM was both the data store and the view. As applications grew, keeping these in sync required increasingly complex event chains, and bugs became almost impossible to trace.

React's Core Insight

React's founding idea, articulated in the 2013 JSConf presentation, was deceptively simple:

"UI is a function of state. Given the same state, the UI should always look the same. Re-render everything on every state change, and let the framework figure out the minimal DOM update."

This shifted frontend from imperative (tell the DOM what to change) to declarative (describe what the UI should look like for any given state, and React handles the rest).

Imperative (jQuery): Find element → mutate it → hope everything is consistent
Declarative (React): UI = f(state) → React diffs → minimal DOM update

State changes? Re-run f(state). React calculates the diff. Done.
No manual DOM tracking. No synchronization bugs. Predictable.
02 — Core Concepts

React Fundamentals

Components, JSX, props, and state are React's four foundational primitives. Understanding what each one actually is — not just how to use it — is the difference between using React and understanding React.

Components

A React component is a pure function that maps props to a React element tree. The function takes an input object (props) and returns a description of what to render. React calls this function, compares the result to the previous result, and updates only what changed.

TypeScript — Component Anatomy
import { useState } from 'react'

// Props contract — typed explicitly
interface UserCardProps {
  name: string
  role: string
  avatarUrl: string
  onSelect?: (name: string) => void
}

// Component = pure function: props → React element tree
export function UserCard({ name, role, avatarUrl, onSelect }: UserCardProps) {
  const [expanded, setExpanded] = useState(false)

  return (
    <div className="card" onClick={() => onSelect?.(name)}>
      <img src={avatarUrl} alt={name} />
      <h3>{name}</h3>
      {expanded && <p>{role}</p>}
      <button onClick={(e) => { e.stopPropagation(); setExpanded(e => !e) }}>
        {expanded ? 'Less' : 'More'}
      </button>
    </div>
  )
}

JSX — Not HTML

JSX is syntactic sugar that compiles to React.createElement() calls. Understanding this is important because it explains why JSX has certain constraints (single root element, className not class, camelCase attributes) and why React can render to non-DOM targets like React Native, terminals, and PDFs.

JSX → JavaScript
// What you write
const el = <Button variant="primary" onClick={handleClick}>Save</Button>

// What the compiler emits (React 17+ automatic JSX transform)
import { jsx } from 'react/jsx-runtime'
const el = jsx(Button, { variant: 'primary', onClick: handleClick, children: 'Save' })

// This returns a plain JS object (a React element):
// { type: Button, props: { variant: 'primary', onClick: f, children: 'Save' } }
// It is NOT a DOM element. React creates the DOM element later.

03 — Internals

Rendering, Reconciliation & Fiber

React's rendering model is its most important and least understood subsystem. Every performance problem, every unexpected re-render, every subtle state bug traces back to how React decides what to render, when to render it, and how to reconcile the result with the previous tree.

The Render Pipeline

Trigger — setState(), new props, context change, forceUpdate()
    ↓
Render Phase (pure, interruptible in Concurrent Mode)
    React calls component functions → builds new virtual DOM tree
    Diffs new tree against previous tree (reconciliation)
    Produces a list of "effects" (DOM mutations needed)
    ↓
Commit Phase (synchronous, non-interruptible)
    Applies DOM mutations (beforeMutation → mutation → layout)
    Runs useLayoutEffect synchronously
    Schedules useEffect for async execution
    ↓
Passive Effects — useEffect runs after paint
Key insight
The render phase is pure and side-effect free. This is why React can safely interrupt, restart, and throw away in-progress renders in Concurrent Mode. The commit phase is where side effects happen — and it is always synchronous and complete.

React Fiber Architecture

The old React reconciler (pre-v16) used a recursive, synchronous call stack. A large re-render would block the main thread for its full duration — causing dropped frames and janky UIs. Fiber replaced it with a linked-list work unit model that can be interrupted.

🧵

Fiber Nodes

Each component instance has a corresponding Fiber node — a plain JS object that tracks component type, props, state, effects, and pointers to parent/child/sibling fibers.

⏸️

Interruptible Work

React processes fibers one at a time. Between each unit of work, it checks if higher-priority work has arrived. If so, it pauses and handles the priority work first.

🔀

Priority Lanes

React 18 introduced Lanes — a bitmask-based priority system. User interactions get high-priority lanes; background work gets low-priority lanes. This is what powers useTransition.

🌲

Double Buffering

React maintains two trees: the "current" tree (what's on screen) and the "work-in-progress" tree (what's being computed). Completed work-in-progress becomes the new current atomically at commit.

Reconciliation — The Diffing Algorithm

React's reconciler needs to diff two trees efficiently. A naive tree diff is O(n³). React achieves O(n) by applying two assumptions: elements of different types produce different trees, and key props identify which list elements are stable across renders.

TypeScript — Keys and Reconciliation
// ❌ Without keys: React can't match items across renders
// On re-order, React unmounts/remounts everything → loses input state
{users.map(u => <UserRow user={u} />)}

// ❌ Index as key: still breaks on re-order (index changes)
{users.map((u, i) => <UserRow key={i} user={u} />)}

// ✅ Stable, unique key: React tracks identity across renders
{users.map(u => <UserRow key={u.id} user={u} />)}

// Keys also force remount — useful to reset component state:
<ProfileForm key={selectedUserId} userId={selectedUserId} />
// Changing selectedUserId unmounts old form (clearing state) and mounts fresh one

Concurrent Rendering

React 18's Concurrent Mode enables React to prepare multiple UI versions simultaneously and interrupt low-priority renders when high-priority work arrives. This is what makes useTransition and useDeferredValue possible — they tell React which updates can be deferred without blocking user interaction.

TypeScript — useTransition for Non-Blocking Updates
import { useState, useTransition } from 'react'

function SearchPage() {
  const [query, setQuery] = useState('')
  const [results, setResults] = useState([])
  const [isPending, startTransition] = useTransition()

  function handleSearch(value: string) {
    setQuery(value)              // Urgent: update input immediately
    startTransition(() => {   // Non-urgent: filter 10k items
      setResults(filterItems(value))
    })
  }

  return (
    <>
      <input value={query} onChange={e => handleSearch(e.target.value)} />
      {isPending ? <Spinner /> : <ResultList items={results} />}
    </>
  )
}
04 — Hooks

Hooks Deep Dive

Hooks are not just a syntactic convenience over class components. They are a fundamentally better model for composing stateful logic — one that makes it possible to extract, test, and reuse behaviour that was previously trapped inside component lifecycle methods.

useState — Snapshot Semantics

The most important mental model for useState: state is a snapshot at render time. Calling the setter does not mutate the current value — it schedules a re-render with the new value. This is why closures over state can go stale in async code.

TypeScript — useState Snapshot Trap & Functional Updates
// ❌ Stale closure: count is captured at render time (0)
function BuggyCounter() {
  const [count, setCount] = useState(0)
  const handleClick = () => {
    setTimeout(() => setCount(count + 1), 1000) // always sets to 1
  }
}

// ✅ Functional update: receives the LATEST state, not the closure
function CorrectCounter() {
  const [count, setCount] = useState(0)
  const handleClick = () => {
    setTimeout(() => setCount(prev => prev + 1), 1000) // correct
  }
}

useEffect — Synchronization, Not Lifecycle

The mental model that makes useEffect click: it is not "run code after mount" — it is "synchronize this side effect with these values." The cleanup function un-synchronizes when the values change or the component unmounts. Thinking of it as synchronization eliminates most dependency array bugs.

TypeScript — useEffect Synchronization Mental Model
// Think: "Synchronize a subscription to roomId"
useEffect(() => {
  const socket = connectToRoom(roomId)           // synchronize
  socket.on('message', handleMessage)

  return () => {
    socket.off('message', handleMessage)
    socket.disconnect()                          // un-synchronize
  }
}, [roomId])
// When roomId changes: cleanup old room → connect to new room
// Not "run on mount" — "keep in sync with roomId"
useEffect is rarely the right tool for data fetching
Effects run after paint, create waterfalls in nested components, and require manual cleanup of race conditions. Use TanStack Query, SWR, or React Router loaders for data fetching. Reserve useEffect for browser APIs, subscriptions, and third-party library synchronization.

useMemo & useCallback — When They Actually Help

Both hooks skip re-computation between renders when dependencies haven't changed. The cost: every render checks the dependency array and stores the previous result. They only provide a net benefit when the saved work exceeds the memoization overhead.

TypeScript — Memoization Decision Guide
// ❌ useMemo for cheap operations — overhead exceeds benefit
const fullName = useMemo(() => `${first} ${last}`, [first, last])

// ✅ useMemo for genuinely expensive computation
const sortedFilteredItems = useMemo(() =>
  items
    .filter(i => i.category === activeCategory)
    .sort((a, b) => a.name.localeCompare(b.name)),
  [items, activeCategory]
)  // items has 5,000 entries — sorting is expensive

// ✅ useCallback to stabilize a ref passed to a memoized child
const handleDelete = useCallback((id: string) => {
  deleteItem(id)
}, [])  // stable reference → React.memo child won't re-render

Custom Hooks — The Real Power

Custom hooks are the primary mechanism for code reuse in React. Any stateful logic — data fetching, form handling, WebSocket connections, media queries, authentication — can be extracted into a custom hook that is testable in isolation and composable across components.

TypeScript — Production Custom Hooks
// useLocalStorage — persists state to localStorage with SSR safety
function useLocalStorage<T>(key: string, initialValue: T) {
  const [value, setValue] = useState<T>(() => {
    if (typeof window === 'undefined') return initialValue
    try {
      const item = window.localStorage.getItem(key)
      return item ? JSON.parse(item) as T : initialValue
    } catch { return initialValue }
  })

  const setStoredValue = useCallback((newValue: T | ((prev: T) => T)) => {
    const resolved = newValue instanceof Function
      ? newValue(value) : newValue
    setValue(resolved)
    window.localStorage.setItem(key, JSON.stringify(resolved))
  }, [key, value])

  return [value, setStoredValue] as const
}

// useDebounce — debounces a value (for search inputs, etc.)
function useDebounce<T>(value: T, delayMs: number): T {
  const [debounced, setDebounced] = useState(value)
  useEffect(() => {
    const timer = setTimeout(() => setDebounced(value), delayMs)
    return () => clearTimeout(timer)
  }, [value, delayMs])
  return debounced
}
05 — State Architecture

State Architecture at Scale

State management is the hardest design problem in React applications. Not because the tools are complex, but because developers consistently misclassify state — putting server data in Redux, URL state in useState, and local UI state in a global store. The solution is a clear taxonomy.

The State Classification Framework

📍

Local State

Owned by one component. Examples: modal open/closed, input value, accordion expanded. Tool: useState, useReducer.

🌐

Shared UI State

Needed by multiple components in the same subtree. Examples: selected item, active tab. Tool: lift state up, or useContext.

🗂️

Global UI State

App-wide UI decisions: theme, language, sidebar collapsed. Tool: Context API, Zustand. Rarely needs Redux.

🌍

Server State

Data that lives on the server: users, orders, products. The client has a cached copy. Tool: TanStack Query, SWR — not Redux.

🔗

URL State

State that should survive page refresh and be shareable: filters, pagination, selected IDs. Tool: query params via useSearchParams.

💾

Session State

Auth tokens, user preferences, feature flags. Survives navigation but not new sessions. Tool: localStorage hook, session storage, or auth context.

The most common state architecture mistake
Putting server data in Redux. Redux is for client-side state that your application owns. Users, orders, and products live on the server — you have a temporary cache. TanStack Query manages cache lifetime, background refetching, loading states, and error handling. Redux cannot do this elegantly.

Zustand — Minimal Global State

TypeScript — Zustand Store Pattern
import { create } from 'zustand'
import { immer } from 'zustand/middleware/immer'

interface CartStore {
  items: CartItem[]
  addItem(item: CartItem): void
  removeItem(id: string): void
  total: number
}

export const useCartStore = create<CartStore>()(
  immer((set, get) => ({
    items: [],
    addItem(item) {
      set(state => {
        const existing = state.items.find(i => i.id === item.id)
        if (existing) existing.quantity++
        else state.items.push({ ...item, quantity: 1 })
      })
    },
    removeItem(id) {
      set(state => {
        state.items = state.items.filter(i => i.id !== id)
      })
    },
    get total() {
      return get().items.reduce((sum, i) => sum + i.price * i.quantity, 0)
    }
  }))
)
06 — Data Layer

Data Fetching & Server State

Data fetching is not a React problem — it is an async state management problem. React itself provides no opinions. The ecosystem has converged on TanStack Query as the standard answer, and for good reason: it solves loading states, caching, background refetching, deduplication, and race conditions out of the box.

TanStack Query — The Server State Standard

TypeScript — TanStack Query Patterns
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query'

// Read: auto-cached, background-refetched, deduped across components
function useUser(userId: string) {
  return useQuery({
    queryKey: ['user', userId],
    queryFn: () => fetchUser(userId),
    staleTime: 5 * 60 * 1000,   // consider fresh for 5 mins
    gcTime: 10 * 60 * 1000,    // keep in cache for 10 mins
    retry: 2,
  })
}

// Write: mutation with optimistic update + rollback
function useUpdateUser() {
  const queryClient = useQueryClient()
  return useMutation({
    mutationFn: (data: UpdateUserPayload) => updateUser(data),
    async onMutate(data) {
      // Cancel in-flight refetches to avoid overwriting optimistic update
      await queryClient.cancelQueries({ queryKey: ['user', data.id] })
      const snapshot = queryClient.getQueryData(['user', data.id])
      queryClient.setQueryData(['user', data.id], prev => ({ ...prev, ...data }))
      return { snapshot }            // return snapshot for rollback
    },
    onError(_err, data, ctx) {
      queryClient.setQueryData(['user', data.id], ctx?.snapshot) // rollback
    },
    onSettled(_data, _err, data) {
      queryClient.invalidateQueries({ queryKey: ['user', data.id] }) // sync
    },
  })
}

07 — Performance Engineering

React Performance Engineering

React performance is not about useMemo everywhere. It is about understanding why renders happen, measuring their actual cost, and applying targeted optimizations. Premature optimization is the enemy of readable code; unmeasured optimization is the enemy of correct code.

Diagnosing Re-renders

Before optimizing, measure. The React DevTools Profiler shows exactly which components rendered, why they rendered, and how long each render took. The "why did this render?" feature is your first diagnostic tool — not memoization.

📊

React DevTools Profiler

Record a session, identify components with high "self time." The flamegraph shows the render cascade; "why did this render?" shows prop/state/context changes.

🔬

React Scan

Open-source tool (2024) that visually highlights components as they re-render in real time. Far faster than the DevTools profiler for identifying hot spots.

📉

Performance Budgets

Define explicit limits for JS bundle size, LCP, INP, and CLS. Enforce them in CI. Without a budget, performance degrades silently one small change at a time.

⚡

Chrome Performance

For commit-phase analysis: Long Tasks, Layout, Paint. React Profiler shows React work; Chrome Performance shows the full browser picture including CSS and layout.

React.memo — Bail Out of Renders

TypeScript — React.memo Correctly Applied
import { memo, useCallback } from 'react'

// ✅ Memoize when: expensive render AND parent re-renders frequently
const DataGrid = memo(function DataGrid({ rows, onRowClick }: DataGridProps) {
  // Renders 500 rows — expensive. memo skips if rows & onRowClick are stable.
  return ...
})

// Parent must stabilize the callback too, or memo is useless
function Dashboard() {
  const [filter, setFilter] = useState('')

  // Without useCallback: new function reference every render → memo is bypassed
  const handleRowClick = useCallback((id: string) => {
    navigateTo(`/items/${id}`)
  }, [])  // stable — no dependencies

  return <DataGrid rows={rows} onRowClick={handleRowClick} />
}

Virtualization for Large Lists

TypeScript — TanStack Virtual
import { useVirtualizer } from '@tanstack/react-virtual'

function VirtualList({ items }: { items: Item[] }) {
  const parentRef = useRef<HTMLDivElement>(null)

  const virtualizer = useVirtualizer({
    count: items.length,
    getScrollElement:() => parentRef.current,
    estimateSize: () => 48,    // estimated row height in px
    overscan: 5,               // render 5 extra rows above/below viewport
  })

  return (
    <div ref={parentRef} style={{ height: '600px', overflow: 'auto' }}>
      <div style={{ height: virtualizer.getTotalSize() }}>
        {virtualizer.getVirtualItems().map(vItem => (
          <div
            key={vItem.key}
            style={{ transform: `translateY(${vItem.start}px)`, position: 'absolute' }}
          >
            <ItemRow item={items[vItem.index]} />
          </div>
        ))}
      </div>
    </div>
  )
}

Memory Leaks — The Silent Killer
The most common memory leaks in React: event listeners added in useEffect without cleanup, WebSocket subscriptions that outlive the component, intervals not cleared on unmount, and closures retaining large arrays in callbacks. Always return a cleanup function from effects that create subscriptions or timers.
08 — Modern Architecture

Modern React Architecture

The most impactful architectural decision in a React codebase is not which state library to use — it is how you organize code. Feature-based architecture aligns the codebase with the business domain, which makes it scale with team growth rather than against it.

Feature-Based Architecture

Traditional layer-based organization (components/, hooks/, utils/) breaks down when a codebase grows beyond one team. A change to the "orders" feature touches files scattered across every directory. Feature-based organization co-locates everything a feature needs.

src/
  features/
    users/
      components/ <-- UserCard, UserTable, UserForm
      hooks/ <-- useUsers, useUser, useCreateUser
      api/ <-- usersApi.ts (TanStack Query functions)
      store/ <-- usersSlice.ts or usersStore.ts
      types/ <-- User.ts, UserRole.ts
      index.ts <-- public API barrel (export only what others need)
    orders/
    products/
  shared/ <-- Button, Input, Modal, useDebounce
  pages/ <-- route-level components (thin wrappers)
  app/ <-- router, providers, global error boundary

Clean Architecture in React

For enterprise applications, a four-layer model provides clear separation between business logic, application orchestration, and infrastructure concerns — making the application testable and framework-independent at the domain level.

🖼️

UI Layer

React components. Pure rendering — no business logic, no direct API calls. Receives data as props or from hooks. Testable in isolation with Storybook.

⚙️

Application Layer

Custom hooks that orchestrate use cases. Calls domain services, manages loading/error state, handles navigation side effects. The "controller" of the frontend.

🏛️

Domain Layer

Pure TypeScript — no React. Business entities, validation rules, calculation logic. Fully unit-testable with zero React dependencies.

🔌

Infrastructure Layer

API clients, localStorage adapters, WebSocket connections. Implements interfaces defined by the domain layer. Swappable without touching business logic.

Design Systems Engineering

A design system is not a component library — it is a shared language between design and engineering. The architecture flows from abstract tokens to concrete components to application usage.

Design Tokens — color-primary-500, spacing-4, font-size-lg (CSS variables)
    ↓ consumed by
Primitive Components — Button, Input, Modal, Badge (no business logic)
    ↓ composed into
Pattern Components — DataTable, FormSection, PageLayout
    ↓ assembled into
Feature Components — UserTable, OrderForm, ProductGrid
    ↓ composed on
Pages — UsersPage, OrdersPage (route-level)
Accessibility is architecture, not compliance
Accessibility built into the design system level propagates throughout the entire application automatically. A Button component that handles keyboard interaction, focus management, and ARIA roles correctly means every feature that uses Button gets this for free. Retrofitting accessibility feature-by-feature costs 10x more and results in inconsistency.

Micro-Frontends — When and Why

✅ Use Micro-Frontends When

  • Multiple teams need truly independent release cycles
  • Different parts use genuinely different tech stacks
  • Organizational boundaries map to product boundaries
  • Team autonomy is worth the integration overhead

⚠️ Use Modular Monolith Instead When

  • One team owns the whole frontend
  • Shared components/state are core to the UX
  • You want a single deployment unit
  • The overhead of Module Federation isn't justified
09 — Next.js & RSC

Next.js & React Server Components

React alone is a UI library — it solves rendering, not routing, SSR, or deployment. Next.js has become React's de facto production framework. React Server Components, introduced in Next.js 13 and stabilised in Next.js 14–15, are the most significant architectural shift in React since hooks.

Rendering Strategies

SSR (Server-Side)On the server per requestDynamic content with SEO requirementsSlower TTFB than static; server load

React Server Components — The Paradigm Shift

Server Components run only on the server. They can directly query databases, access the filesystem, and call internal APIs — all without sending any JavaScript to the client. They render to a serialized payload (the "Flight" format) that the client uses to hydrate the tree. The result: dramatically less client-side JavaScript for data-heavy UIs.

Server Component Tree (no useState, no event handlers, direct DB access)
    ↓ renders to
RSC Payload (Flight format — JSON-like serialized tree)
    ↓ streamed to browser
"use client" boundaries (interactive islands hydrated with JS)
    ↓
Final UI (server-rendered shell + interactive client components)

Streaming SSR

Next.js App Router streams HTML to the browser as each part of the page becomes ready — rather than waiting for the entire page to render before sending anything. This dramatically improves Time-to-First-Byte (TTFB) and Largest Contentful Paint (LCP) for pages with slow data fetches. Suspense boundaries mark where streaming chunks begin.

10 — Frontend System Design

Frontend System Design

System design is where senior engineers distinguish themselves. Designing a real-time dashboard, a multi-tenant SaaS, or a collaborative editor requires thinking about data flow, rendering architecture, and state synchronization holistically — not component by component.

Real-Time Dashboards

Real-time dashboards combine high-frequency data streams with expensive rendering. The architecture must separate concerns clearly: the data layer fetches and caches, the state layer manages subscriptions, and the UI layer renders only what changed.

WebSocket / SSE connection
    │ raw event stream
Event normalizer — maps raw events to domain types
    │
Zustand store — holds current snapshot; sliced selectors prevent over-rendering
    │
React.memo + useCallback — memoized chart/metric components
    │
Virtualized list — only visible rows rendered; stable keys preserve scroll
⏱️

Throttle Updates

A WebSocket emitting 100 events/second will cause 100 renders/second. Throttle or batch state updates — update the store at most every 100ms using a flush interval.

🔪

Slice Selectors

If the whole store re-renders every chart on every tick, performance collapses. Use Zustand's selector pattern: useStore(s => s.metricA) so each component only re-renders when its specific slice changes.

🔌

Reconnection Logic

WebSockets disconnect. Your hook must detect disconnection, implement exponential backoff, and re-request missed events on reconnect. Never expose raw WebSocket connections to components.

💾

Buffer Overflow

A time-series store keeping all data in memory will grow without bound. Implement a sliding window — keep only the last N data points per metric, evicting old entries on each update.

Multi-Tenant SaaS Frontend

Multi-tenant applications must isolate tenant state, configuration, and branding without creating a separate codebase per tenant. The architecture centres on a tenant context that drives feature flags, theming, and permission checks throughout the tree.

TypeScript — Tenant Context Pattern
interface TenantConfig {
  id: string
  name: string
  theme: { primaryColor: string; logoUrl: string }
  features: Record<string, boolean>
  permissions: Permission[]
}

const TenantContext = createContext<TenantConfig | null>(null)

export function useTenant() {
  const ctx = useContext(TenantContext)
  if (!ctx) throw new Error('useTenant must be used inside TenantProvider')
  return ctx
}

// Feature flag gate — renders children only if feature is enabled for tenant
export function Feature({ name, children, fallback }: {
  name: string, children: ReactNode, fallback?: ReactNode
}) {
  const { features } = useTenant()
  return <>{features[name] ? children : (fallback ?? null)}</>
}

// Usage: invisible to tenants without the 'advanced-reporting' feature
<Feature name="advanced-reporting">
  <ReportBuilder />
</Feature>

Collaborative Applications (CRDTs)

Collaborative editors (think Google Docs, Figma, Notion) require multiple users to edit the same document simultaneously and converge on the same state without conflicts. Conflict-free Replicated Data Types (CRDTs) provide a mathematical guarantee that any two clients that have seen the same set of operations will have the same state — regardless of order.

Local Edit
    │ apply immediately to local state (optimistic)
CRDT Operation (Yjs, Automerge) — encodes the edit as a commutative op
    │ broadcast to all peers via WebSocket
Remote Operations arrive — CRDT merges regardless of order
    │
All clients converge to identical state — no server arbitration needed

ERP & Complex Form Frontends

ERP frontends are characterised by massive forms, complex conditional logic, deep permission models, and data grids with thousands of rows. The key architectural decisions are schema-driven form generation, row-level permission evaluation, and server-side pagination with client-side sorting.

URL State is underused
Filter values, pagination state, sort keys, and selected IDs should almost always live in the URL. Users can share a link and land on the same view. Browser back/forward work. Page refresh doesn't reset the UI. Use useSearchParams from React Router or nuqs for type-safe URL state management.
11 — Security & Testing

Security & Testing

Security and testing are not separate disciplines — they are part of the same quality system. A test suite that never tests authorization logic, XSS vectors, or error boundaries gives false confidence. Senior engineers design both systems together.

XSS Prevention

React escapes all values rendered in JSX by default — {userContent} is always safe. The only XSS risk in React comes from explicitly bypassing this protection with dangerouslySetInnerHTML, which should be treated as a last resort and always combined with a sanitizer.

TypeScript — Safe HTML Rendering
import DOMPurify from 'dompurify'

// ❌ Never: direct unsanitized HTML from user/server
<div dangerouslySetInnerHTML={{ __html: userContent }} />

// ✅ If HTML rendering is genuinely needed: sanitize first
const clean = DOMPurify.sanitize(userContent, {
  ALLOWED_TAGS: ['b', 'i', 'em', 'strong', 'a', 'p'],
  ALLOWED_ATTR: ['href', 'target'],
})
<div dangerouslySetInnerHTML={{ __html: clean }} />

// ✅ CSP header (set server-side) as defence-in-depth
// Content-Security-Policy: default-src 'self'; script-src 'self'

Authentication & Authorization Patterns

🔑

Auth with OIDC

Use a proven provider (Auth0, Clerk, Supabase Auth). Never implement your own OAuth flow. Store tokens in httpOnly cookies — never localStorage — to prevent XSS token theft.

🔒

RBAC on the Frontend

Role-based access control on the client is for UX only — it hides UI elements. It is never a security boundary. The server must enforce every permission on every API call.

🔄

Token Rotation

Short-lived access tokens (15 min) refreshed silently via a refresh token flow. Intercept 401 responses, refresh the token, and replay the failed request — transparent to the user.

🛡️

Route Guards

Protect routes at the router level — not by checking auth state inside every page component. A single auth guard in the router tree is the single source of truth for route access.

Testing Strategy

The testing trophy (not pyramid) is the model that works for React: the majority of coverage should come from integration tests that render real component trees with mocked API boundaries — not unit tests of isolated functions or full E2E tests.

TypeScript — React Testing Library Best Practices
import { render, screen, userEvent } from '@testing-library/react'
import { http, HttpResponse } from 'msw'
import { server } from '../mocks/server'

// Test behaviour, not implementation
describe('UserProfile', () => {
  it('shows updated name after edit and save', async () => {
    // Mock the API response for this specific test
    server.use(
      http.patch('/api/users/:id', () =>
        HttpResponse.json({ id: '1', name: 'Jane Doe' })
      )
    )

    render(<UserProfile userId="1" />)

    // Query by accessible role — not by class name or test ID
    const nameInput = screen.getByRole('textbox', { name: /name/i })
    await userEvent.clear(nameInput)
    await userEvent.type(nameInput, 'Jane Doe')
    await userEvent.click(screen.getByRole('button', { name: /save/i }))

    // Assert on visible outcome — not on state or function calls
    await screen.findByText('Jane Doe')
    expect(screen.queryByRole('alert')).not.toBeInTheDocument()
  })
})

12 — The Learning Path

Senior → Staff Engineer Roadmap

Most React engineers plateau at the senior level because they keep deepening implementation knowledge rather than broadening architectural thinking. The transition from Senior to Staff is a transition from building React applications to designing frontend platforms and systems.

Beginner
Foundation: JSX, components, props, state, event handling. Build real UIs with useState. Understand why React re-renders. Learn conditional rendering, list rendering with keys, and controlled inputs. Read the official React docs from start to finish — they are excellent.
Intermediate
Hooks & data: All core hooks with deep understanding — not just syntax. React Router, data fetching with TanStack Query, Context API. Build a multi-page app with authentication. Write component tests with RTL. Understand the render cycle and what triggers re-renders.
Advanced
Performance & architecture: React Profiler, memoization strategies, virtualization. Feature-based architecture. Global state design with Zustand or Redux Toolkit. Next.js App Router — SSR, SSG, ISR. TypeScript deeply. Design systems. E2E testing with Playwright. Custom hooks that abstract complex behaviour.
Senior
Systems thinking: React Fiber and Concurrent Mode internals. React Server Components and the Flight protocol. Frontend system design: design a real-time dashboard, a multi-tenant SaaS, a collaborative editor. Performance budgets and INP optimization. Authentication architecture. Error boundaries and observability. Leading architecture decisions with written ADRs.
Staff+
Platform engineering: Micro-frontend strategy and Module Federation. Design system governance — versioning, deprecation, accessibility SLAs. Frontend observability: RUM, error budgets, performance SLOs. Scaling frontend teams on a single codebase (20+ engineers). Technology selection and migration planning. Mentoring senior engineers. Writing technical strategy documents that influence product direction.

The Three Fundamentals — Unchanged Since 2013

Every innovation in React's history — hooks, context, concurrent mode, server components — is an attempt to better solve the same three problems:

📦

State

Where does state live? Who owns it? How does it flow? Local, shared, global, server, URL — classifying state correctly is the most important architectural skill in React.

🔄

Rendering

What triggers a render? What is the cost? Can it be deferred? Understanding the Fiber scheduler, memoization, and Concurrent Mode separates engineers who guess from those who know.

🌊

Data Flow

How does data get from the server to the UI? Props down, events up, context across, TanStack Query for server state, Zustand for global UI state. Each mechanism has the right use case.
